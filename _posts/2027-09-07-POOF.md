## Overview

**POOF** is an Easy HackTheBox **Forensics** challenge built around a PyPI supply-chain ransomware incident. A developer installed a "cutting-edge" game library suggested on a private Discord; it turned out to be ransomware that AES-encrypted their files and appended a `.boo` extension. We're handed a Linux memory dump, a Volatility 2 profile, a packet capture, and one encrypted file — `candy_dungeon.pdf.boo`. A docker Q&A service asks six questions; answering all six reveals the flag.

The key realisation: **the memory dump is a red herring.** The pcap delivered the malware binary, and the binary hardcodes its own AES key, so the encrypted file is fully recoverable offline — no Volatility needed.

## The technique

Symmetric, self-contained ransomware carries its own key so it can run offline with no C2 round-trip. That makes it its own kill switch: recover the binary, recover the key, decrypt. The chain is **carve the binary from network traffic → unpack the PyInstaller ELF → decompile → read the hardcoded AES key/IV → decrypt**. This is a [hardcoded cryptographic key](https://cwe.mitre.org/data/definitions/321.html) weakness ([CWE-321](https://cwe.mitre.org/data/definitions/321.html) / [CWE-798](https://cwe.mitre.org/data/definitions/798.html)) — real ransomware uses *asymmetric* crypto where the attacker keeps the private key, so this challenge is deliberately the wrong way to do it.

## Solution

### 1. Find the download URL (pcap)

```bash
tshark -r poof_capture.pcap -Y "http.request" -T fields -e http.request.full_uri
# http://files.pypi-install.com/packages/a5/61/.../pygaming-dev-13.37.tar.gz
```

`files.pypi-install.com` typosquats the real `files.pythonhosted.org` — the supply-chain vector.

### 2. Carve the payload (pcap)

```bash
tshark -r poof_capture.pcap --export-objects http,carved
tar xf carved/pygaming-dev-13.37.tar.gz
file pygaming-dev-13.37/configure
# ELF 64-bit ... stripped ; strings shows libpython3.6m -> PyInstaller
md5sum pygaming-dev-13.37/configure
```

The `md5sum` of `configure` answers the "md5sum of the ransomware file" question.

### 3. Unpack + decompile (PyInstaller → pyc → source)

```bash
python3 pyinstxtractor.py configure
uncompyle6 configure_extracted/configure.pyc
```

The decompiled source hands you everything — the encryption function name, the language, and the hardcoded key/IV:

```python
def mv18jiVh6TJI9lzY(filename):
    data = open(filename, "rb").read()
    key = "vN0nb7ZshjAWiCzv"
    iv  = b'ffTC776Wt59Qawe1'
    cipher = AES.new(key.encode("utf-8"), AES.MODE_CFB, iv)
    ct = cipher.encrypt(data)
    # renames file -> filename.boo
```

### 4. Decrypt (the durable artifact)

AES-CFB is a stream mode: the keystream depends only on key+IV, so calling `.decrypt()` with the same key/IV byte-for-byte undoes `.encrypt()`. Use the *identical* pycryptodome call (default `segment_size=8`) so the segmentation matches.

Create `solve.py`:

```python
#!/usr/bin/env python3
from Crypto.Cipher import AES
import sys, hashlib

key = "vN0nb7ZshjAWiCzv".encode("utf-8")
iv  = b'ffTC776Wt59Qawe1'

boo = sys.argv[1] if len(sys.argv) > 1 else "candy_dungeon.pdf.boo"
ct = open(boo, "rb").read()
pt = AES.new(key, AES.MODE_CFB, iv).decrypt(ct)   # identical call to the encryptor

open("candy_dungeon.pdf", "wb").write(pt)
print("magic:", pt[:8])                            # b'%PDF-1.4'
print("md5  :", hashlib.md5(pt).hexdigest())
```

```bash
python3 solve.py candy_dungeon.pdf.boo
# magic: b'%PDF-1.4'   -> valid PDF, decryption confirmed
```

The md5 of the recovered PDF answers the final question.

### 5. Answer the Q&A service → flag

| Question | Source |
|---|---|
| malicious URL | `http.request.full_uri` from the pcap |
| malicious process | `configure` (the binary name) |
| md5 of ransomware | `md5sum configure` |
| language | `python` (PyInstaller / libpython3.6m) |
| encryption function | `mv18jiVh6TJI9lzY` (decompiled) |
| decrypted file md5 | md5 of the recovered PDF |

All six correct prints the flag: `HTB{...}` *(redacted)*.

## Why it worked

The malware is symmetric and self-contained — it ships the key so it can encrypt offline. Once the binary is recovered from the network capture and decompiled, the key falls out of the source and decryption is a one-liner. There is no secret the attacker kept off the victim's machine, so recovery is total.

## Fix / defense

- Never `pip install` from untrusted indexes or links shared on Discord; pin to PyPI with hashes (`pip install --require-hashes`) so a typosquatted package can't substitute.
- Real ransomware uses **asymmetric** crypto (an attacker-held public key encrypts a per-victim symmetric key), making recovery from the binary alone impossible — never rely on a symmetric, embedded key for confidentiality ([CWE-321](https://cwe.mitre.org/data/definitions/321.html)).
- Monitor for outbound fetches to look-alike domains (`pypi-install.com` vs `pythonhosted.org`) — typosquats are a recurring supply-chain tell.
