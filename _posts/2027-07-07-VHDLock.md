## Overview

VHDLock is an HTB Hardware challenge (Easy) built around VHDL — the hardware description language used in FPGAs and ASICs. The circuit takes pairs of 4-bit inputs, encodes each with a one-hot decoder, then XOR-masks them with a cycling 4-byte key. The output pairs are given in `out.txt`. The key is unknown but brute-force recoverable in milliseconds since each of the 4 key positions can be attacked independently.

---

## The Technique

### Understanding the VHDL circuit

`lock.vhd` defines two components chained together:

**`decoder_4x16`**: a standard 4-to-16 one-hot decoder. Input is a 4-bit nibble (0–15); output is a 16-bit value with exactly bit `input` set:

```
input 0 → 0000000000000001
input 1 → 0000000000000010
...
input 8 → 0000000100000000
```

**`xor_get`**: plain 16-bit XOR.

**`main`**: takes two 4-bit inputs and one 16-bit `xorKey`:
```
decoder1 = decode(input_1)
decoder2 = decode(input_2)
output1  = decoder1 XOR xorKey
output2  = decoder2 XOR xorKey
```

`out.txt` contains 44 `(output1, output2)` pairs. The description states the key is a 4-byte sequence — i.e., it cycles with period 4, one byte (zero-extended to 16 bits) per pair position.

### Recovering the key by brute force

For a given key byte `k` and pair `(o1, o2)`:
- `o1 XOR k` must be a power of 2 (valid one-hot output)
- `o2 XOR k` must be a power of 2

There are only 65 536 possible 16-bit values. Testing each against all same-position pairs eliminates all wrong candidates instantly. The 4 key positions are independent, so each is solved in isolation:

```
key[0] = k where all pairs[0,4,8,...]: is_pow2(o1^k) and is_pow2(o2^k)
key[1] = k where all pairs[1,5,9,...]: ...
key[2] = k where all pairs[2,6,10,...]: ...
key[3] = k where all pairs[3,7,11,...]: ...
```

Result: `key = [0x33, 0x31, 0x31, 0x37]` = ASCII `"3117"`.

### Decoding the flag

Once the key is known, recover each 4-bit input pair and reconstruct characters:
```
nibble1 = log2(o1 XOR k)   # which bit is set → input_1
nibble2 = log2(o2 XOR k)   # which bit is set → input_2
char    = chr((nibble1 << 4) | nibble2)
```

## Solution

`solve.py`:

```python
#!/usr/bin/env python3
import sys

def is_pow2(n):
    return n > 0 and (n & (n-1)) == 0

pairs = []
with open(sys.argv[1] if len(sys.argv) > 1 else "out.txt") as f:
    for line in f:
        a, b = map(int, line.strip().split())
        pairs.append((a, b))

# Key cycles with period 4; brute-force each position (0-65535)
key = []
for pos in range(4):
    for k in range(65536):
        if all(is_pow2(o1^k) and is_pow2(o2^k) for o1,o2 in pairs[pos::4]):
            key.append(k)
            break

flag = ""
for i, (o1, o2) in enumerate(pairs):
    k = key[i % 4]
    n1 = (o1^k).bit_length()-1
    n2 = (o2^k).bit_length()-1
    flag += chr((n1<<4)|n2)

print(f"[+] Key bytes: {[hex(k) for k in key]}")
print(f"[+] FLAG: {flag}")
```

```bash
python3 solve.py out.txt
```

## Why it worked

The one-hot encoding creates a strong constraint: a valid decrypted value must have exactly one bit set. The 4-byte key cycling means each key position affects only a quarter of the pairs, so the 4 positions are independent. A simple brute-force through all 65 536 possible 16-bit values, validated against the one-hot constraint, finds each key byte in O(65536) iterations.

---

## Takeaway

One-hot encoding is not a cipher — it is a bijection with a detectable structural property (power-of-2 values). Any XOR mask applied to one-hot outputs is immediately recoverable using the "exactly one bit set" oracle.
