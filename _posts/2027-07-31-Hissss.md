## Overview

Hissss is an Easy HTB Reversing challenge consisting of a single PyInstaller-packed Python 3.8 ELF binary named `auth`. The binary prompts for a 12-character password and validates it against a series of arithmetic, XOR, and character-literal constraints embedded directly in the bytecode. Extracting the archive with `pyinstxtractor` and disassembling the resulting `.pyc` with `pydisasm` exposes all constraints in plain bytecode, making algebraic recovery trivial — no brute force or dynamic analysis needed.

---

## The Technique

[CWE-656](https://cwe.mitre.org/data/definitions/656.html) — Reliance on Security Through Obscurity. PyInstaller bundles a Python interpreter, dependencies, and a `.pyc` file into a single ELF. The packing is transparent: `pyinstxtractor` extracts the CArchive in seconds, and `pydisasm` (from `xdis`) produces a readable instruction listing that includes every `LOAD_CONST` value — the same constants that define the authentication constraints.

The `auth.pyc` bytecode runs a sequence of 12 independent checks against the input string. Because all comparison targets are `LOAD_CONST` literals and all operations are standard Python opcodes (`BINARY_XOR`, `BINARY_MULTIPLY`, `BINARY_SUBTRACT`, `BINARY_ADD`), reading the disassembly is equivalent to reading the source.

## Solution

### Step 1 — Identify the packer

```bash
strings auth | grep -i meipass
```

The presence of `_MEIPASS2` and `_MEIPASS` strings confirms PyInstaller.

### Step 2 — Extract the bundle

```bash
python3 pyinstxtractor.py auth
```

This creates `auth_extracted/` containing `auth.pyc` (the main script) alongside bundled `.so` files for Python 3.8.

> **Note:** `uncompyle6` fails with `Parse error at or near LOAD_CONST at offset 0` on this bytecode. Use `pydisasm` (from `xdis`) instead:

```bash
pip3 install xdis --break-system-packages
pydisasm auth_extracted/auth.pyc 2>/dev/null | grep -E 'LOAD_CONST|COMPARE_OP|BINARY_(XOR|MULTIPLY|ADD|SUBTRACT)'
```

### Step 3 — Read the constants header

`pydisasm` prints all constants before the instruction listing:

```
# Constants:
#    0: 0          #    5: 48         #    7: '!'
#    6: 11         #   16: 5610       #   17: 'p'
#   20: -46        #   21: 64         #   23: 166
#   24: 'n'        #   27: 'Well Done! HTB{'    #   28: '}'
```

The string `'Well Done! HTB{'` reveals the flag format: `HTB{<12-char-password>}`.

### Step 4 — Extract constraints from the disassembly

The 12 constraints, in the order they appear in the bytecode:

| Check | Constraint |
|---|---|
| Length | `len(password) == 12` |
| `[0]` | `ord(password[0]) == 48` |
| `[11]` | `password[11] == '!'` |
| `[7],[5]` | `ord(password[7]) == ord(password[5])` |
| `[4]` | `143 - ord(password[0]) == ord(password[4])` |
| `[1],[3]` | `ord(password[1]) ^ ord(password[3]) == 30` |
| `[2],[3]` | `ord(password[2]) * ord(password[3]) == 5610` |
| `[1]` | `password[1] == 'p'` |
| `[6],[8]` | `ord(password[6]) - ord(password[8]) == -46` |
| `[6],[7]` | `ord(password[6]) ^ ord(password[7]) == 64` |
| `[10],[5]` | `ord(password[10]) + ord(password[5]) == 166` |
| `[9]` | `ord('n') - ord(password[9]) == 1` |
| `[10]` | `password[10] == str(3)` |

### Step 5 — Solve algebraically

Assign known values first, then derive dependents:

```python
password[0]  = chr(48)                        # '0'  (ord == 48)
password[1]  = 'p'                            # direct literal
password[11] = '!'                            # direct literal
password[3]  = chr(ord('p') ^ 30)            # 112 ^ 30 = 110 = 'n'
password[2]  = chr(5610 // ord(password[3])) # 5610 / 110 = 51 = '3'
password[4]  = chr(143 - ord(password[0]))   # 143 - 48 = 95 = '_'
password[10] = str(3)                         # '3'
password[9]  = chr(ord('n') - 1)             # 110 - 1 = 109 = 'm'
password[5]  = chr(166 - ord(password[10]))  # 166 - 51 = 115 = 's'
password[7]  = password[5]                   # 's'
password[6]  = chr(64 ^ ord(password[7]))    # 64 ^ 115 = 51 = '3'
password[8]  = chr(ord(password[6]) + 46)    # 51 + 46 = 97 = 'a'
```

Full `solve.py`:

```python
#!/usr/bin/env python3
password = [''] * 12
password[0] = chr(48)
password[1] = 'p'
password[3] = chr(ord('p') ^ 30)
password[2] = chr(5610 // ord(password[3]))
password[4] = chr(143 - ord(password[0]))
password[10] = str(3)
password[9] = chr(ord('n') - 1)
password[5] = chr(166 - ord(password[10]))
password[7] = password[5]
password[6] = chr(64 ^ ord(password[7]))
password[8] = chr(ord(password[6]) + 46)
password[11] = '!'
pwd = ''.join(password)
print(f"HTB{{{pwd}}}")
```

```bash
python3 solve.py
# Password: 0p3n_s3sam3!
```

Confirm against the binary:

```bash
echo "0p3n_s3sam3!" | ./auth
# Enter password> Well Done! HTB{...}
```

Flag: `HTB{...}` *(redacted)*

## Why It Worked

PyInstaller is a deployment tool, not a protection mechanism. Its CArchive format is fully documented, and `pyinstxtractor` has been public for years. Once extracted, Python `.pyc` files are structured bytecode: every literal in the source becomes a `LOAD_CONST` opcode, and every comparison becomes a `COMPARE_OP`. There is no encryption, obfuscation, or runtime secret — the constraints are as readable as the original Python source code, and solvable with basic algebra in under a minute.

---

## Fix / Defense

Client-side authentication is inherently bypassable. The correct architecture is to validate credentials server-side — the client sends a guess, the server replies accepted or rejected, and no checking logic (or its constants) ever ships to the user:

```python
import requests
pwd = input("Enter password> ")
resp = requests.post("https://auth.example.com/verify", json={"password": pwd})
print("Access granted" if resp.json().get("ok") else "Denied")
```

If offline validation is a hard requirement, at minimum use a one-way hash with a server-stored salt — embedding reversible arithmetic constraints in shipped bytecode is equivalent to shipping the plaintext password.
