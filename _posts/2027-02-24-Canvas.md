## Overview

**Canvas** is an Easy HackTheBox **Misc** challenge. You download a tiny static site — `index.html`, a stylesheet, a decoy `dashboard.html`, and a heavily obfuscated `js/login.js`. The script was run through [javascript-obfuscator](https://obfuscator.io/) (the obfuscator.io tool), which makes it *look* impenetrable. It isn't: obfuscation is not encryption, and the flag is a leftover developer artifact encoded as a `String.fromCharCode(...)` array on the very last line. This is a textbook case of [putting secrets in client-side code](https://cwe.mitre.org/data/definitions/1331.html).

## The technique

`js/login.js` carries three layers of noise, none of which protect anything:

- **Rotated string-array** — `var _0x4e0b=[...]` holds every string (hex-escaped), and an IIFE rotates it with `arr.push(arr.shift())`. A decoder `_0x20fe(i)` returns `arr[i]`, so calls like `_0x4d1a17('0x12')` are just array lookups that resolve to plain strings (e.g. `'admin'`).
- **Self-defending stubs** — a `Function('return (function(){}…))()` console/debugger hijack plus a regex-`toString` integrity check. Pure anti-tamper theatre; it has zero effect on *static* recovery.
- **A decoy login** — `validate()` compares the username and password to `admin`/`admin` entirely client-side and redirects to `dashboard.html`, which literally contains `HTB{🤦‍♂️}`. A troll, not the flag.

The real flag never goes through the obfuscator's machinery — it sits in the open:

```js
var res = String.fromCharCode(0x48, 0x54, 0x42, 0x7b, /* … */ 0x7d);
```

## Solution

Don't unwind any of the obfuscation. Grep the source for the trailing `String.fromCharCode` integer list and decode it. The argument list begins with `0x48` (`'H'` of `HTB{`), which makes it easy to anchor a regex on.

Create `solve.py`:

```python
#!/usr/bin/env python3
import re
s = open("files/js/login.js").read()
m = re.search(r"\((0x48(?:,0x[0-9a-fA-F]+)*)\)", s)   # arg list starts with 'H' = 0x48
codes = [int(x, 16) for x in m.group(1).split(",")]
print("".join(chr(c) for c in codes).strip())
```

Run it:

```bash
python3 solve.py
# HTB{...}
```

Equivalent one-liners: evaluate the `String.fromCharCode(...)` call directly in a browser/node console, or run the whole file through [`webcrack`](https://github.com/j4k0xb/webcrack) / `js-beautify` first to collapse the string-array layer.

```bash
node -e 'const s=require("fs").readFileSync("files/js/login.js","utf8");const m=s.match(/\((0x[0-9a-fA-F]+(?:,0x[0-9a-fA-F]+)*)\)/g).pop();console.log(String.fromCharCode(...m.slice(1,-1).split(",").map(x=>parseInt(x,16))))'
```

The script prints the flag — value redacted here as `HTB{...}`.

## Why it worked

Anything shipped to the browser must be decoded and run by the browser, so the attacker can decode it too. javascript-obfuscator raises the effort to read the code but provides no confidentiality — the string array, the `fromCharCode` literals, and the redirect target are all recoverable from the served source. The challenge leans on a [false sense of security from obscurity](https://cwe.mitre.org/data/definitions/656.html): a leftover secret was left in the bundle behind a layer of obfuscation and a decoy UI flow.

## Fix / defense

- Never embed secrets, flags, or authorization decisions in client-side code ([CWE-1331](https://cwe.mitre.org/data/definitions/1331.html), [CWE-540](https://cwe.mitre.org/data/definitions/540.html) information exposure through source code).
- Authenticate and authorize **server-side** on every privileged route; the client cannot be trusted to gate access.
- Treat obfuscation (minify/rot13/base64/string-array) as a readability speed bump, not as [encryption or access control](https://cwe.mitre.org/data/definitions/656.html).
