An Easy Secure Coding challenge where you patch an [Insecure Direct Object Reference (IDOR)](https://cwe.mitre.org/data/definitions/639.html) vulnerability in a Node.js Express message API, then submit the fix through a Monaco code editor to receive the flag.

## Overview

**Commnet** is an Easy HackTheBox Secure Coding challenge. The scenario is a post-apocalyptic survivor communication network: a Node.js/Express backend with a SQLite database of users and private messages, served alongside an in-browser Monaco editor.

The vulnerability is a classic [IDOR](https://cwe.mitre.org/data/definitions/639.html) ([CWE-639](https://cwe.mitre.org/data/definitions/639.html)): the `GET /messages/:id` endpoint authenticates the caller but never checks whether they are the message's sender or recipient. Any logged-in user can read any message by guessing its numeric ID.

The Secure Coding category flips the usual challenge model — instead of exploiting the vulnerability, you must **patch** it. The bundled `exploit/solver.py` confirms the bug; `GET /api/verify` runs the exploit against your patched code and returns the flag only once it fails.

## The Technique

The vulnerable route in `routes/messages.js` queries by ID alone:

```javascript
router.get('/:id', requireAuth, (req, res) => {
  const messageId = req.params.id;

  req.db.get(`
    SELECT m.*, sender.username as sender_username, ...
    FROM messages m
    LEFT JOIN users sender ON m.sender_id = sender.id
    LEFT JOIN users recipient ON m.recipient_id = recipient.id
    WHERE m.id = ?
  `, [messageId], (err, message) => {
    if (err || !message) return res.status(404).json({ ... });
    res.json({ success: true, message: message });
  });
});
```

`requireAuth` confirms a valid session exists — but does nothing to verify the session user owns the message. The fix is a single SQL clause binding the query to the authenticated principal:

```javascript
router.get('/:id', requireAuth, (req, res) => {
  const messageId = req.params.id;
  const userId = req.session.userId;

  req.db.get(`
    SELECT m.*, sender.username as sender_username, ...
    FROM messages m
    LEFT JOIN users sender ON m.sender_id = sender.id
    LEFT JOIN users recipient ON m.recipient_id = recipient.id
    WHERE m.id = ? AND (m.sender_id = ? OR m.recipient_id = ? OR m.recipient_id IS NULL)
  `, [messageId, userId, userId], (err, message) => {
    if (err || !message) return res.status(404).json({ ... });
    res.json({ success: true, message: message });
  });
});
```

`m.recipient_id IS NULL` preserves broadcast messages that every user should be able to read.

## Solution

The editor communicates with a backend API. The workflow is:

1. Read the current file to obtain its MD5 (required for the optimistic-concurrency save check).
2. Submit the patched content via a socket.io `save` event.
3. Restart the Node.js server so the patched route is loaded.
4. Call `GET /api/verify` — it runs the bundled exploit; a successful patch causes the exploit to fail and the endpoint returns the flag.

`solve.py` (the durable artifact):

```python
#!/usr/bin/env python3
import requests, json, hashlib, re, time

TARGET = "http://<rhost>:<port>"
FILE   = "routes/messages.js"

FIXED = """import express from 'express';
import { requireAuth } from '../utils/middleware.js';

const router = express.Router();

router.get('/', requireAuth, (req, res) => {
  const userId = req.session.userId;
  req.db.all(`SELECT m.*, sender.username as sender_username,
    sender.enclave as sender_enclave,
    recipient.username as recipient_username,
    recipient.enclave as recipient_enclave
    FROM messages m
    LEFT JOIN users sender ON m.sender_id = sender.id
    LEFT JOIN users recipient ON m.recipient_id = recipient.id
    WHERE m.sender_id = ? OR m.recipient_id = ? OR m.recipient_id IS NULL
    ORDER BY m.created_at DESC`, [userId, userId], (err, messages) => {
    if (err) return res.status(500).json({ success: false, error: 'Failed to fetch messages' });
    res.json({ success: true, messages: messages });
  });
});

router.get('/:id', requireAuth, (req, res) => {
  const messageId = req.params.id;
  const userId = req.session.userId;
  req.db.get(`SELECT m.*, sender.username as sender_username,
    sender.enclave as sender_enclave,
    recipient.username as recipient_username,
    recipient.enclave as recipient_enclave
    FROM messages m
    LEFT JOIN users sender ON m.sender_id = sender.id
    LEFT JOIN users recipient ON m.recipient_id = recipient.id
    WHERE m.id = ? AND (m.sender_id = ? OR m.recipient_id = ? OR m.recipient_id IS NULL)`,
    [messageId, userId, userId], (err, message) => {
    if (err || !message) return res.status(404).json({ success: false, error: 'Message not found' });
    res.json({ success: true, message: message });
  });
});

router.post('/', requireAuth, (req, res) => {
  const { recipient_id, subject, content, message_type, priority } = req.body;
  const sender_id = req.session.userId;
  if (!subject || !content) return res.status(400).json({ success: false, error: 'Subject and content required' });
  req.db.run(`INSERT INTO messages (sender_id, recipient_id, subject, content, message_type, priority)
    VALUES (?, ?, ?, ?, ?, ?)`,
    [sender_id, recipient_id, subject, content, message_type || 'personal', priority || 'normal'],
    function(err) {
      if (err) return res.status(500).json({ success: false, error: 'Failed to send message' });
      res.json({ success: true, message: 'Message sent successfully', messageId: this.lastID });
    });
});

export { router as messageRoutes };"""

# 1. Read current file → MD5 for optimistic-lock check
cur = requests.get(f"{TARGET}/api/file?path={FILE}").json()["content"]
cur_md5 = hashlib.md5(cur.encode()).hexdigest()

# 2. Connect socket.io (EIO4 polling — websocket upgrade is firewalled on challenge instances)
s   = requests.Session()
sid = re.search(r'"sid":"([^"]+)"',
      s.get(f"{TARGET}/socket.io/?EIO=4&transport=polling").text).group(1)
s.post(f"{TARGET}/socket.io/?EIO=4&transport=polling&sid={sid}", data="40")
s.get(f"{TARGET}/socket.io/?EIO=4&transport=polling&sid={sid}")

# 3. Save patch — md5 = current file hash (not the new content's hash)
s.post(f"{TARGET}/socket.io/?EIO=4&transport=polling&sid={sid}",
       data="42" + json.dumps(["message", {"type": "save",
           "data": {"fileName": FILE, "content": FIXED, "md5": cur_md5}}]))
time.sleep(0.5)
s.get(f"{TARGET}/socket.io/?EIO=4&transport=polling&sid={sid}")  # drain save_success

# 4. Restart so Express reloads the patched routes
requests.post(f"{TARGET}/api/restart"); time.sleep(3)

# 5. Verify
r = requests.get(f"{TARGET}/api/verify").json()
print(r)
flag = re.search(r'HTB\{[^}]+\}', str(r))
if flag: print("FLAG:", flag.group())
```

Running it against the spawned instance returns `HTB{...}`.

## Why it worked

The `requireAuth` middleware only answered the question "is the caller logged in?" — never "does the caller own this message?". Without an ownership predicate in the SQL `WHERE` clause, the database happily returned any row matching the supplied ID.

This is the textbook definition of [CWE-639](https://cwe.mitre.org/data/definitions/639.html) (Authorization Bypass Through User-Controlled Key): the resource identifier (message ID) is caller-supplied, and the authorization decision is based solely on the ID rather than on a binding between the ID and the authenticated principal.

The `md5` field in the socket.io save payload is a concurrency guard — the server rejects a save if the hash doesn't match what it currently has on disk (preventing one editor session from overwriting another's concurrent edit). Sending the MD5 of the *new* content rather than the *current* content triggers a `save_conflict` error.

## Fix / defense

- **Bind every object query to the session principal.** Never fetch a resource by bare ID; always include `owner_id = session.userId` (or equivalent) in the WHERE clause.
- **Deny by default at the data layer.** A 404 (rather than 403) on a message the caller doesn't own avoids confirming the ID exists.
- **Automated ownership tests in CI.** Register two users; assert that user B's session cannot retrieve user A's private resources — any 200 response for a cross-user fetch is a test failure.
