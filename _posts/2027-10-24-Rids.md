A factory RFID lock has a flash memory chip soldered inside it, marked **W25Q128**. The challenge hands you a network bridge that behaves like an FT2232H USB-to-SPI adapter wired to that chip, and asks you to "uncover the secret encryption keys stored within." There is no decryption to do — the keys are sitting in cleartext at the very start of the flash. This is [CWE-312: cleartext storage of sensitive information](https://cwe.mitre.org/data/definitions/312.html), and reading it out is just a matter of speaking SPI to the chip.

## Overview

**Rids** is an Easy Hardware challenge. The only file provided is a `client.py` that shows how to talk to a live `pyftdi` SPI bridge over a JSON protocol. By sending the standard SPI flash command set — JEDEC ID to fingerprint the chip, then Read Data to dump it — the flag falls out of the first 64 bytes of flash, no logic-analyzer capture or decryption required.

## The technique

A SPI NOR flash speaks a tiny command language. You clock out a command byte (plus a 24-bit address for read commands), then clock bytes back in on the MISO line. The two commands needed here:

| Command | Bytes out | Bytes back | Purpose |
|---|---|---|---|
| `0x9F` JEDEC ID | `9F` | 3 | Identify manufacturer + capacity |
| `0x03` Read Data | `03 AA AA AA` (24-bit addr) | N | Read N bytes from the address |

The provided `client.py` wraps this in JSON: `data_out` is the list of bytes clocked out on MOSI, `readlen` is how many bytes to read back on MISO, and the response is a JSON array of those bytes. So the whole challenge reduces to issuing the right SPI commands over a socket.

## Solution

The durable artifact is `solve.py`. It fingerprints the chip, then reads the first 64 bytes of flash and strips the trailing erased bytes (`0xFF`):

```python
import socket, json, sys

HOST = sys.argv[1]
PORT = int(sys.argv[2])

def exchange(hex_list, readlen=0):
    cmd = {
        "tool": "pyftdi",
        "cs_pin": 0,
        "url": "ftdi://ftdi:2232h/1",
        "data_out": [hex(x) for x in hex_list],
        "readlen": readlen,
    }
    with socket.socket() as s:
        s.connect((HOST, PORT))
        s.sendall(json.dumps(cmd).encode())
        data = b''
        while not data.endswith(b']'):       # response is a JSON array
            data += s.recv(4096)
    return json.loads(data.decode())

def read_data(addr, n):
    a = [(addr >> 16) & 0xff, (addr >> 8) & 0xff, addr & 0xff]
    return bytes(exchange([0x03] + a, n))

mfg, mtype, cap = exchange([0x9F], 3)         # JEDEC ID
print(f"JEDEC: {mfg:02x} {mtype:02x} {cap:02x}")   # ef 40 18 = Winbond W25Q128

blob = read_data(0x000000, 64)                # Read Data from address 0
print("FLAG:", blob.split(b'\xff')[0].decode())
```

Running it against the live instance:

```bash
python3 solve.py <target-ip> <target-port>
```

Step 1 returns the JEDEC ID `EF 40 18` — `0xEF` is Winbond, `0x40` is the memory type, `0x18` is the capacity code for a 16&nbsp;MB (128&nbsp;Mbit) part, confirming the **W25Q128** from the challenge prompt. Step 2 reads 64 bytes from address `0x000000`; the flash holds the flag as plain ASCII followed by `0xFF` erased padding, so splitting on the first `0xFF` byte yields:

```
HTB{...}
```

## Why it worked

The "encryption keys" were never encrypted — they were written to the main memory array at address `0x000000` in cleartext. External SPI flash is completely unauthenticated: any standard read command returns whatever is stored, and anyone with physical access (and a ~$5 FT2232H, Bus Pirate, or `flashrom` rig) can pull the entire chip in seconds. Storing a secret there in the clear gives it zero confidentiality.

If the main array had been blank, the next places to check would be the chip's **Read Security Registers** (`0x48` + 3-byte address + 1 dummy byte — the W25Q128 has three at `0x001000`, `0x002000`, `0x003000`) and the **Read Unique ID** (`0x4B`) region.

## Fix / defense

- Never store secrets in cleartext on external flash. Encrypt them at rest with a key the flash never sees.
- Use the flash's security registers together with the status-register lock bits (SRP/SRL) and read protection so the sensitive region cannot be read back.
- Better still, keep keys in the MCU's one-time-programmable (OTP) fuses or a dedicated secure element, so the secret never travels across the SPI bus at all.
