## Overview

`Anti Flag` is an Easy HackTheBox **Reversing** challenge: a single 64-bit PIE, stripped ELF and no live target. Run it any way you like and it just says `No flag for you :(`. The catch is in the name — it punishes you for *tracing* it, and the genuine flag routine ships in the binary but is never reached at runtime. The fix is one byte.

## The technique

Two traps are stacked on top of each other:

1. **Inverted ptrace anti-debug.** `ptrace(PTRACE_TRACEME, 0, 1, 0)` returns `-1` when the process is *already* being traced. Here the author routes that `== -1` case to a **decoy** success string (`Well done!!`). So if you attach gdb/strace expecting the "good" branch, you get a fake win and never see the flag. The instruction-level tell is `cmp rax, -1 ; jne <real>` immediately after `call ptrace` — meaning the path you want is the **untraced** one. Run it; don't attach.
2. **Dead-code flag decoder.** The real decode-and-print is guarded by a constant compare on a variable that is initialised to `0` and never written again:

   ```asm
   cmp dword [var], 0x539     ; 0x539 = 1337
   je   0x1525                ; the decode+puts path — never taken (0 != 1337)
   ```

   So the genuine flag path is present in `.text` but unreachable. That decode routine is textbook **RC4**: a 256-iteration key-schedule over the `.rodata` key `2asdf-012=14`, then a PRGA that XORs a 25-byte `.rodata` blob into an output buffer and `puts` it.

## Solution

The clean move is to make the dead branch live and let the binary run its *own* RC4, rather than reimplementing the cipher (no risk of a port bug, and the flag is derived live). At file offset `0x1510` the gate's `je` (opcode `0x74`) becomes a `jmp` (`0xeb`) — an "always-take" flip, the cousin of NOPing a `jne`. Then run normally: no debugger means `ptrace != -1`, so control falls through to the now-unconditional decode.

Because this is a PIE ELF of this size, the load base is `0`, so the runtime virtual address `0x1510` equals the file offset (radare2 confirms `paddr == vaddr` on the `.rodata` strings). On a binary where they differ, resolve the offset with `objdump`/`rabin2` first.

Create `solve.py`:

```python
import shutil, subprocess, os

src = "files/anti_flag"
patched = src + "_patched"
shutil.copy(src, patched)

with open(patched, "r+b") as f:
    f.seek(0x1510)
    assert f.read(1) == b"\x74", "gate byte not 0x74 (wrong build?)"
    f.seek(0x1510)
    f.write(b"\xeb")              # je -> jmp: the dead decode branch always runs now
os.chmod(patched, 0o755)

out = subprocess.run([f"./{patched}"], capture_output=True, text=True, timeout=10).stdout.strip()
print("[+] FLAG:", out)
```

Run it:

```bash
python3 solve.py
# [+] FLAG: HTB{...}
```

The binary decrypts its own flag with its own RC4 routine — `HTB{...}`.

## Why it worked

Neither protection is cryptographically enforced. The anti-debug is a runtime check you simply don't trigger, and the success gate is a constant comparison you can rewrite. The RC4 key and ciphertext both live in `.rodata`, and the decoder lives in `.text` — so flipping a single conditional jump and running the binary untraced beats both traps at once and yields the true flag.

## Fix / defense

- Don't use anti-debug as the gate to decryption when the key and ciphertext ship in the binary — an analyst just patches the branch and runs your decryptor. Derive the decryption key from something a patch or debugger *perturbs* (a checksum of `.text`, timing, or the `ptrace` result itself) so a tampered binary decrypts to garbage.
- Avoid leaving the success path behind a constant-guarded branch (`cmp .., 0x539`); constants are trivially flipped. Tie real control flow to the verified secret rather than a magic number.
- Stripping and string obfuscation are not protection here: the key `2asdf-012=14` and the encoded blob were recoverable straight from `.rodata`.
