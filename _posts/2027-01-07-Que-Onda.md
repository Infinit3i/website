## Overview

`Que Onda` is a Very Easy HackTheBox **Pwn** challenge. It is the gentle on-ramp that comes before the real memory-corruption work: there is no bug to exploit. The binary is fully hardened — Full RELRO, a stack canary, NX, and PIE — yet none of those protections are ever exercised, because the program never overflows anything. The "exploit" is to connect to the remote instance and send it the four-byte string `flag`.

## The technique

There is no vulnerability class here. The point of the challenge is operational: spawn the Docker instance, `nc` to the host/port, read the prompt, and send the exact trigger word the program is waiting for. The skill being taught is *interacting with a remote pwn service* — the same first step every harder Pwn challenge starts from.

Reversing `que_onda` shows the entire logic lives in one function. `main()` only calls `banner()`:

```
main()  ->  banner()
```

`banner()` (from `objdump -d -M intel que_onda`):

1. `srand(time(0)); rand() % 6` picks one of six greeting strings, `puts()` it, then clears the screen.
2. Prints the panel: *"Send me the string \"flag\" and I will give you uno grande prize!!"*.
3. `read(0, buf /* rbp-0x20 */, 6)` — reads up to 6 bytes into a stack buffer.
4. `strncmp(buf, "flag", 4)` — compares the first four bytes against `"flag"`.
   - equal  -> `read_flag()` opens the flag file and prints it.
   - not    -> `error()`.

The 6-byte read into a buffer that sits well clear of the saved return address and the canary means there is nothing to smash. The gate is a single `strncmp`.

## Solution

Spawn the instance, then run the solver against its host and port. The script drains the banner (including the ANSI screen-clear codes), sends the trigger, and prints whatever comes back.

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys
from pwn import remote
host, port = sys.argv[1], int(sys.argv[2])
io = remote(host, port)
io.recv(timeout=5)          # drain banner + ANSI screen-clear codes
io.send(b'flag')            # 4-byte trigger; strncmp checks first 4 of the 6 read
print(io.recvall(timeout=8).decode(errors='replace'))
```

Run it:

```bash
python3 solve.py <ip> <port>
```

The flag prints after the `$` prompt:

```
$ HTB{...}
```

## Why it worked

The program's only decision is `strncmp(input, "flag", 4)`. Satisfy that comparison and the intended `read_flag()` path runs and prints the flag. No leak, no ROP, no canary defeat — the binary's protections are decorative for this particular flow because the only `read()` is length-bounded to a size far smaller than the distance to anything sensitive on the stack.

## Fix / defense

There is nothing to "fix" — this is a teaching stub, not a vulnerable service. The lesson for the next tier of Pwn challenges is the contrast: keep that same `read()` length-bounded (`read(0, buf, 6)` into a buffer with room to spare) and the stack stays safe. Make it unbounded instead — `read(0, buf, 0x100)` into a `0x20`-byte buffer — and you get a classic [stack buffer overflow](https://cwe.mitre.org/data/definitions/121.html) ([CWE-121](https://cwe.mitre.org/data/definitions/121.html)), the exact situation where the canary, NX, and PIE on display here finally start to matter.
