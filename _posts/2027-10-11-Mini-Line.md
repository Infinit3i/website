## Overview

**Mini Line** is an Easy Hardware challenge built from a single download — `firmware.hex` — and no server at all. The brief says a firmware *flashing tool* is backdoored and exfiltrates data over the SPI bus of an **LPC2148** (ARM7TDMI) microcontroller, and the leaked data is the flag. The whole solve is static: convert the Intel HEX to a flat image, find the obfuscated constants, and invert the trivially-reversible per-byte transforms that "protect" them.

## The technique

The flag is never stored in cleartext. It lives in **three constant buffers**, each scrambled with a different cheap per-byte operation that the firmware reverses at runtime before transmitting. Because the de-obfuscation routine ships inside the binary, you simply read the operation and invert it offline — a classic case of [reversible client-side obfuscation as the only barrier](https://cwe.mitre.org/data/definitions/656.html) ([CWE-656](https://cwe.mitre.org/data/definitions/656.html)) and [inadequate encryption strength](https://cwe.mitre.org/data/definitions/326.html) ([CWE-326](https://cwe.mitre.org/data/definitions/326.html)). Obfuscation is not encryption; there is no key you don't already have.

## Solution

### 1. Intel HEX → flat binary

Each `:` record is `:LLAAAATT<payload><cksum>`; the type byte `TT` drives the parse — `00` data, `02`/`04` extended segment/linear address, `01` EOF. Flash address ends up equal to file offset.

### 2. Find the encoded data

`strings -t x` over the image surfaces the SPI status text (`SPI Initialization`, `Transmitting data to slave`, `Data transfer is completed`) and one suspicious printable run at offset `0x8ac`: `RNXax.h)Ew)n.viE`.

### 3. Recover the XOR key from the `HTB{` crib

The flag begins `HTB{`, so XOR the first encoded bytes against the known prefix:

```text
'R' ^ 'H' = 0x1a   'N' ^ 'T' = 0x1a   'X' ^ 'B' = 0x1a   'a' ^ '{' = 0x1a
```

Key `0x1a`. Decoding the 16-byte buffer gives `HTB{b4r3_m3t4ls_` — but no closing brace, so the flag is split.

### 4. Peel the second layer

The remaining two buffers decode with `(byte >> 1) ^ 0x39`: a single int at `0x738` → `4r3_`, and four ints at `0x8bc` → `fun_...!!!}`. Concatenating the three buffers yields the full flag.

The full, runnable solver:

```python
#!/usr/bin/env python3
def parse_ihex(path):
    data, ext = {}, 0
    for line in open(path):
        line = line.strip()
        if not line.startswith(':'):
            continue
        b = bytes.fromhex(line[1:])
        ln, addr, rt, payload = b[0], (b[1] << 8) | b[2], b[3], b[4:4 + b[0]]
        if rt == 0x00:
            for i, by in enumerate(payload):
                data[ext + addr + i] = by
        elif rt == 0x02:
            ext = ((payload[0] << 8) | payload[1]) << 4
        elif rt == 0x04:
            ext = ((payload[0] << 8) | payload[1]) << 16
        elif rt == 0x01:
            break
    lo, hi = min(data), max(data)
    out = bytearray(hi - lo + 1)
    for a, v in data.items():
        out[a - lo] = v
    return bytes(out)

img = parse_ihex('firmware.hex')

buf1 = bytes(c ^ 0x1a for c in img[0x8ac:0x8ac + 16])   # key from the HTB{ crib
dec = bytes(((c >> 1) ^ 0x39) & 0xff for c in img)
buf2 = dec[dec.find(b'4r3_'):][:4]                      # 1 int @ 0x738
raw3 = dec[0x8bc:0x8bc + 16]                            # 4 ints @ 0x8bc
buf3 = raw3[:raw3.find(b'}') + 1]

print((buf1 + buf2 + buf3).decode())                    # HTB{...}
```

Running it prints the flag: `HTB{...}` (redacted).

## Why it worked

The author tried to defeat a naïve single-key XOR pass by encoding pieces of the flag with *different* transforms. Two habits beat it: **decode the whole region around a `strings` hit** — `strings` only shows the bytes that stayed ASCII after encoding, so the rest of the buffer is invisible to it — and **scan under several cheap transforms** (`b ^ k`, `(b + k)`, `(b >> 1) ^ k`) when one key only recovers part of the flag.

## Fix / defense

Never store a sensitive value in a shipped binary — it is always recoverable. If a value must live on-device, derive it from a hardware secret element or fused key, and never put it on a bus protected only by a per-byte transform. XOR/add/shift "obfuscation" adds no security.
