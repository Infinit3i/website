## Overview

`Recovery` is an easy HackTheBox **Blockchain** challenge — but there is no smart contract
and no cryptography to break. You are given SSH credentials to the hacker's box, which holds
the **seed phrase** of an [Electrum](https://electrum.org) Bitcoin wallet (on a private
**regtest** chain) that still contains the stolen coins. A seed phrase *is* the master private
key, so the whole challenge is: read the seed, restore the wallet, and **sweep the funds back**
to the address a handler service asks for. The handler watches the hacker's balance and prints
the flag the instant it reaches zero.

## The technique

A wallet **seed/mnemonic stored in cleartext on a reachable host** is total fund compromise
([CWE-522: Insufficiently Protected Credentials](https://cwe.mitre.org/data/definitions/522.html)).
Possession of the phrase is unconditional spend authority over every address the wallet
derives — no password, no signing ceremony, no second factor. "Recovering" the coins is not an
attack on Bitcoin's cryptography at all; you simply restore the wallet from the stolen seed and
move the money.

A spawned instance exposes three TCP services (roles vary per spawn — fingerprint them):

| Service | Identify by | Use |
|---|---|---|
| **SSH** | `SSH-2.0-OpenSSH…` banner | log in, read the seed file |
| **electrs** | replies JSON-RPC 2.0 (`electrs/0.10.1`) to `server.version` | wallet sync + tx broadcast |
| **handler** | prints `Hello fella, help us recover our bitcoins…` | gives the return address + the flag |

## Solution

First, pull the seed off the box and ask the handler where to send the coins:

```bash
ssh satoshi@<ip> -p <ssh-port>     # password: L4mb0Pr0j3ct
cat ~/wallet/electrum-wallet-seed.txt
# e.g.  neutral burst become happy snack jump force movie cactus rate mask damp

nc <ip> <handler-port>
# Return our Bitcoins to the following address: bcrt1qds7y63eydauekhvu2grg2clzjml480epjurdsf
#   - Electrum server to connect to blockchain: 0.0.0.0:50002:t   (t = plain TCP, not SSL)
# Hacker wallet must have 0 balance to earn your flag.
```

An Electrum seed is **not** a BIP-39 mnemonic — it uses Electrum's own wordlist and derivation,
so it must be loaded with the real Electrum client. Restore the wallet, point Electrum at the
challenge's electrs server, and build a max-value (`!`) sweep to the target address. `payto`
returns the raw **signed** transaction hex:

```bash
electrum --regtest -D edata --offline restore "<seed phrase>" -w edata/wallet
electrum --regtest -D edata daemon -d
electrum --regtest -D edata load_wallet -w edata/wallet
electrum --regtest -D edata getbalance -w edata/wallet      # {"confirmed":"1"}
RAW=$(electrum --regtest -D edata payto <return-addr> '!' -w edata/wallet)
```

The Electrum daemon's own `broadcast` is unreliable here, so push the signed transaction
**straight to electrs** over its line-delimited JSON-RPC protocol — mempool acceptance is enough
for the balance-watching handler:

```bash
printf '{"id":1,"method":"blockchain.transaction.broadcast","params":["%s"]}\n' "$RAW" \
  | nc <ip> <electrs-port>
# {"id":1,"jsonrpc":"2.0","result":"38118dcf…8ce5fd4a"}   (the txid)
```

With the hacker's wallet drained to zero, ask the handler for the flag:

```bash
printf '1\n' | nc <ip> <handler-port>
# Enter your choice: HTB{...}
```

Flag value redacted.

### Gotchas worth knowing

A handful of Electrum quirks eat most of the solve time:

- **`setconfig server host:port:t` silently drops the `:t` protocol.** Electrum then assumes
  SSL, fails to handshake against the TCP-only electrs, and falls back to its default regtest
  server `127.0.0.1:51001`. Write `edata/regtest/config` JSON directly instead, keeping
  `"server":"host:port:t"`, `"oneserver":true`, `"auto_connect":false`. (In **zsh**, a literal
  `:t` at the end of a parameter expansion is the *tail* history-modifier and gets eaten too.)
- **`getinfo` reports `connected:false` / `server:localhost` even when the wallet has synced.**
  Don't trust that field — check `getbalance` / `listunspent`.
- Electrum 4.x hard-requires `libsecp256k1`; with no root, `pip install coincurve` and symlink
  its bundled `_libsecp256k1*.so` to `…/electrum/libsecp256k1.so.0`.

## Why it worked

The hacker stored a **plaintext wallet seed on a reachable host**. A seed phrase is the private
keys — anyone who can read `electrum-wallet-seed.txt` simply *is* the wallet and can move every
coin. The challenge frames this as "recovery," but it is exactly the
[insufficiently-protected-credential](https://cwe.mitre.org/data/definitions/522.html) class
that drains real wallets when a key file leaks.

## Fix / defense

- **Never store a seed/mnemonic in plaintext on a networked machine.** Keep it offline on
  paper/metal or in a hardware wallet.
- Encrypt the wallet at rest (Electrum supports a wallet password) and never co-locate the
  decryption secret with the file.
- Segregate signing from internet-facing infrastructure: an online watch-only wallet builds an
  unsigned PSBT, an air-gapped signer holds the keys.
- Treat any host that has touched a seed as compromised — rotate to a fresh wallet and move
  funds immediately, which is precisely what this challenge has you do.
