## Overview

**Anti Flag** is an Easy HackTheBox **Reversing** challenge — a 64-bit, stripped, PIE
ELF that takes no input and, however you run it, prints `No flag for you :(`. The flag
is genuinely inside the binary as an RC4 ciphertext; it's just hidden behind two static
tricks designed to punish the reflex of "throw it in a debugger." Recognise the traps,
flip a single byte, and the program's own decoder hands you the flag.

## The technique

Two traps are stacked so the binary *always* fails at runtime while the real decoder
ships intact in `.text`:

1. **Inverted ptrace anti-debug.** `ptrace(PTRACE_TRACEME, 0, 1, 0)` returns `-1` when
   the process is *already* being traced. The author routes that `== -1` case to a
   **decoy** success message (`Well done!!`) and sends the normal, untraced run toward
   the flag logic. So attaching gdb/ltrace expecting the "good" path hands you a fake
   win and you never see the flag — the whole joke is in the flag text itself.
2. **Dead-code flag decoder.** The genuine decode + `puts` only runs when a counter
   equals `0x539` (1337), but that counter is initialised to `0` and never written
   again. `0 != 1337` always, so the decode branch is unreachable at runtime even
   though it's fully present in the binary.

The decoder is textbook **RC4**: one helper is the 256-iteration key schedule (KSA) over
the `.rodata` key `2asdf-012=14`, the other is the PRGA that XORs a 25-byte `.rodata`
ciphertext blob into an output buffer.

`main` in pseudocode:

```c
int counter = 0;                                 // set once, never touched again
char *key = "2asdf-012=14";                      // 0x2004
char *enc = (char *)0x2011;                       // 25-byte ciphertext blob
char *out = malloc(strlen(enc) * 4);

if (ptrace(PTRACE_TRACEME, 0, 1, 0) == -1)        // -1 => a debugger is ALREADY attached
    puts("Well done!!");                          //   <-- DECOY success path
else {                                             // normal run, no debugger
    if (counter == 0x539)                         // 1337 — but counter is always 0
        rc4_decode(key, enc, out), puts(out);     //   <-- REAL flag path = DEAD CODE
    else
        puts("No flag for you :(");
}
```

## Solution

The decoder itself is correct — only the *gate* in front of it is rigged. Rather than
reimplement RC4 and risk a transcription bug, flip the dead conditional jump so the
decode branch is always taken, then run the binary **normally** (no debugger, so
`ptrace` returns `0` and control falls through to the gate). At `0x1510` the gate is:

```
0x1509  cmp dword [counter], 0x539
0x1510  je   0x1525            ; 74 13  -> taken only if counter == 1337
```

Flip `je` (`0x74`) to `jmp` (`0xeb`) so it always jumps into the decoder. For a PIE this
size the load base is 0, so the runtime vaddr `0x1510` equals the file offset.

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys, shutil, subprocess, os

src = sys.argv[1] if len(sys.argv) > 1 else "anti_flag"
patched = src + "_patched"
shutil.copy(src, patched)

with open(patched, "r+b") as f:
    f.seek(0x1510); assert f.read(1) == b"\x74", "gate byte not 0x74 (wrong build?)"
    f.seek(0x1510); f.write(b"\xeb")           # je -> jmp : make the dead branch live
os.chmod(patched, 0o755)

out = subprocess.run([os.path.abspath(patched)], capture_output=True, text=True,
                     timeout=10).stdout.strip()
print("[+] FLAG:", out)
```

```bash
python3 solve.py anti_flag
# [+] FLAG: HTB{...}
```

Because the *real* RC4 routine inside the binary runs, the output is the true flag —
nothing is copied from a writeup. As a cross-check, the same key + ciphertext decrypted
by a standalone RC4 implementation produces the identical string:

```python
KEY = b"2asdf-012=14"
CT  = bytes.fromhex("d0c0354f0cd9232a484b090b543cf8c0e5dfd7790f3ddb35c4")

def rc4(key, data):
    S = list(range(256)); j = 0
    for i in range(256):                          # KSA
        j = (j + S[i] + key[i % len(key)]) & 0xff
        S[i], S[j] = S[j], S[i]
    out = bytearray(); i = j = 0
    for c in data:                                # PRGA
        i = (i + 1) & 0xff
        j = (j + S[i]) & 0xff
        S[i], S[j] = S[j], S[i]
        out.append(c ^ S[(S[i] + S[j]) & 0xff])
    return bytes(out)

print(rc4(KEY, CT).decode())                      # HTB{...}
```

## Why it worked

The protection is entirely *static*: an anti-debug check you simply don't trigger (run
it, don't trace it) and a constant comparison you can rewrite with one byte. None of it
is cryptographically enforced — the RC4 key and ciphertext live in `.rodata` and the
decoder lives in `.text`, fully recoverable. The right mindset is to treat
`ptrace(PTRACE_TRACEME)` as *data to read*, not a wall to defeat: if you never run the
binary under a debugger, there is nothing to bypass.

## Fix / defense

- Anti-debugging is obfuscation, not security — a static analyst recovers embedded
  secrets regardless. Don't ship a key or secret inside a client binary
  ([use of hard-coded credentials](https://cwe.mitre.org/data/definitions/798.html)),
  and don't rely on `ptrace` self-tracing as protection
  ([leftover debug code](https://cwe.mitre.org/data/definitions/489.html)).
- A success path guarded by a constant comparison (`cmp ..,0x539`) is trivially flipped.
  Real anti-tamper derives the decryption key from something a patch or debugger
  perturbs — a checksum of `.text`, timing, or the `ptrace` result itself — so a flipped
  branch decrypts to garbage instead of the flag.
- If a value must stay secret, it cannot live on the client. Decrypt server-side.
