## Overview

**Pinned** is an Easy HackTheBox **Mobile** challenge. You get a single Android
`.apk` and no server. The app is full of TLS **certificate-pinning** code, which
is exactly the misdirection the name invites — pinning protects the network
channel, not a secret already baked into the app. The flag never touches the
network: it is AES-decrypted on-device, with the key, ciphertext, and even the
algorithm string split across obfuscated helper methods. Reassemble them
statically and the [hard-coded cryptographic key](https://cwe.mitre.org/data/definitions/321.html)
([CWE-321](https://cwe.mitre.org/data/definitions/321.html)) decrypts the flag.

## The technique

Decompiling `classes.dex` shows the real package is `com.example.pinned`. The
pinning logic in `MainActivity$b` throws *"Connection is not honoured as the
bundled certificate does not match..."* on a cert mismatch — a decoy. The actual
prize is in `MainActivity.x()`, gated behind hard-coded credentials:

```java
if (uname.equals("bnavarro") && pass.equals("1234567890987654")) {
    // reassemble AES key + base64 ciphertext, decrypt, show flag
}
```

On that branch the app rebuilds three values by indexing tiny obfuscated helper
classes (`c.b.a.a` … `c.b.a.i`, `b.q.h`). Each helper is just a decoy list with a
single real return:

```java
public static String a() {
    ArrayList v = new ArrayList();
    v.add("fQn/gdiEvh"); v.add("dEd0Xv"); /* ...10 entries... */
    return (String) v.get(1);   // only ONE index is ever returned
}
```

Three things get assembled from those constants:

1. **Algorithm** — `c.b.a.g.a()` = `f[5].upper + a[7] + i[5].upper` = `"AES"`.
2. **Ciphertext (base64)** — ten helper strings concatenated.
3. **Key (16 bytes)** — sixteen `charAt(n)` picks across the helpers, some wrapped
   in `.toUpperCase()`.

Java's `Cipher.getInstance("AES")` with no mode/padding defaults to
**`AES/ECB/PKCS5Padding`**, so once you have the 16-byte key and the base64
ciphertext, decryption is a one-liner.

## Solution

No mobile RE tooling needed beyond a Python venv — `androguard` decompiles the dex
headlessly when jadx/apktool are unavailable:

```bash
python3 -m venv venv && ./venv/bin/pip install androguard pycryptodome
./venv/bin/python -c "from androguard.misc import AnalyzeAPK; a,d,dx=AnalyzeAPK('pinned.apk'); [print(c.get_source()) for x in d for c in x.get_classes() if 'example' in c.get_name() or 'c/b/a' in c.get_name()]"
```

Resolve every `get(idx)` and `charAt(n)` statically and AES-decrypt:

Create `solve.py`:

```python
import base64
from Crypto.Cipher import AES
from Crypto.Util.Padding import unpad

# Each obfuscated helper returns one element of its list:
a="fQn/gdiEvh"; b="dEd0Xv"; c="98Cc1Z"; d="zlg4rj"; e="lw+sVs"
f="k3ED9a"; h="wel80q"; i="1TPpCs"; q="2jOu89"

algo = f[5].upper()+a[7]+i[5].upper()           # -> "AES"
ct_b64 = d+b+h+c+i+"LPGt63"+e+f+"yRCYDm"+a      # concat order from x()
key = (q[3]+q[0]+c[0]+a[8].upper()+h[1]+q[0]+i[5].upper()+a[7]
       +b[4]+e[4]+e[4]+i[5].upper()+d[3]+d[5]+h[1]+h[1])  # -> u29Ve2SEXVVS4jee

ct = base64.b64decode(ct_b64)
pt = unpad(AES.new(key.encode(), AES.MODE_ECB).decrypt(ct), 16)
print(pt.decode())   # HTB{...}
```

Run it:

```bash
./venv/bin/python solve.py
# HTB{...}
```

## Why it worked

The "encryption" is **obfuscation, not protection**. Splitting a key and
ciphertext across array-list lookups and `charAt` picks raises the reading effort
by a few minutes — every constant is sitting in the dex in plaintext. The
certificate pinning is a thematic red herring placed to lure you toward a
network/MITM path that has nothing to do with the flag.

## Fix / defense

- **Never embed secrets or cryptographic keys in client bytecode.** Anything
  shipped in an APK is recoverable; obfuscation buys minutes, not security.
- Derive keys from a user secret via a KDF (PBKDF2/scrypt/Argon2), or bind them to
  the Android Keystore / hardware-backed StrongBox so a static analyst can never
  read the raw bytes.
- Keep real secrets server-side and return data only after a genuine auth check.
- Certificate pinning is worthwhile for channel integrity, but it is
  **orthogonal** to protecting client-side secrets — don't conflate the two.
