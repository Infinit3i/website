## Overview

Ransom is an Easy [Reversing](https://cwe.mitre.org/data/definitions/327.html) challenge. A system administrator ran a fake "critical Windows update" (`windows_update.exe`) and it encrypted the company spreadsheet to `login.xlsx.enc`. Our job is to recover the original `login.xlsx` and read the flag from a cell. The binary uses its own [repeating-key stream cipher](https://cwe.mitre.org/data/definitions/327.html) with a hard-coded key — which means it is fully reversible offline, no malware execution required.

## The technique

Two things make a hand-rolled "ransomware" cipher recoverable:

1. **A constant repeating key provides no confidentiality.** The encryption is `cipher[i] = (plain[i] + KEY[i % len(KEY)]) & 0xff` — a per-byte *addition* of a fixed keystream. Addition is invertible: decryption is just *subtraction*. (Had it used XOR, the inverse would be XOR again.)
2. **The key is recoverable even when `strings` hides it.** The key never appears as a `.rdata` string — it is assembled on the stack at runtime from consecutive `mov` immediates, so `strings` leaks only the fragment `SUPERSEC`. Reading those immediates in order reconstructs the whole key.

Disassembling the encryption routine shows the key construction and the loop:

```asm
movabs rax, "SUPERSEC"        ; 8 bytes -> stack
mov    word [rbp-0x07], "UR"  ; +2      => KEY = "SUPERSECURE" (11 bytes)
mov    byte [rbp-0x05], 'E'   ; +1
...
movzx  r8d, byte [buf + i]    ; plaintext[i]
movzx  edx, byte [key + i%11] ; keystream byte  (i%11 via the 0x2e8ba2e8ba2e8ba3 magic divide)
add    edx, r8d               ; cipher = plain + key  (mod 256)
mov    byte [buf + i], dl
```

The `0x2e8ba2e8ba2e8ba3` constant in the index math is a compiler-emitted divide-by-11 — i.e. the key index is simply `i % 11`.

You don't even need to trust the disassembly: an XLSX file is a ZIP, so the plaintext must start with the magic bytes `PK\x03\x04`. Subtracting those from the first ciphertext bytes recovers the start of the key (`KEY[i] = (cipher[i] - magic[i]) & 0xff`), confirming `SUPERSECURE`.

## Solution

Create `solve.py`:

```python
#!/usr/bin/env python3
import zipfile, io, re

KEY = b"SUPERSECURE"
ct = open("login.xlsx.enc", "rb").read()
pt = bytes((c - KEY[i % len(KEY)]) & 0xff for i, c in enumerate(ct))   # subtract = decrypt

open("login.xlsx", "wb").write(pt)
assert pt[:4] == b"PK\x03\x04"                                          # valid ZIP/XLSX

z = zipfile.ZipFile(io.BytesIO(pt))                                    # flag is a spreadsheet cell
blob = b"".join(z.read(n) for n in z.namelist())                       # -> xl/sharedStrings.xml
print(re.search(rb"HTB\{[^}]+\}", blob).group().decode())
```

Run it against the provided ciphertext:

```bash
python3 solve.py
# [+] decrypted -> login.xlsx (valid XLSX/ZIP)
# HTB{...}
```

The decrypted `login.xlsx` opens as a normal spreadsheet; the flag lives in `xl/sharedStrings.xml`.

## Why it worked

The cipher is encoding, not encryption. A byte-wise `cipher = f(plaintext, constant_key)` with a fixed repeating key is invertible by construction — knowing the key (or just the plaintext's file magic) is enough to undo it. Building the key on the stack defeats a naive `strings` grep but is plainly visible in the disassembly, and using `add` instead of a real cipher means the inverse is a one-line subtraction.

## Fix / defense

Never roll your own cipher. A constant 11-byte additive keystream is checksum-grade obfuscation, not confidentiality ([CWE-327](https://cwe.mitre.org/data/definitions/327.html) broken algorithm, [CWE-326](https://cwe.mitre.org/data/definitions/326.html) inadequate strength, [CWE-798](https://cwe.mitre.org/data/definitions/798.html) hard-coded key). Use an authenticated AEAD such as AES-GCM or libsodium's `crypto_secretbox` with a random per-file key and nonce derived from a real KDF — never a secret shipped inside the binary.
