## Overview

`WIDE` (the "Widely Inflated Dimension Editor") is a Very Easy HackTheBox **Reversing**
challenge. You get an ELF64 binary, `wide`, and a data file, `db.ex`. The program lists
seven "dimensions"; the encrypted one, *Flaggle Alpha*, only reveals its contents when you
supply the correct decryption key. The whole challenge hangs on a single idea: the key is
stored in a text encoding that the usual first-pass tool, `strings`, silently skips.

## The technique

Running the binary lists the dimensions and marks Flaggle Alpha (entry 6) as encrypted:

```
[X] Flaggle Alpha    | admin secret power hidden        |       *        [*]
Which dimension would you like to examine? 6
[X] That entry is encrypted - please enter your WIDE decryption key:
```

The instinct is `strings wide | grep -i key`, which finds nothing useful. That is the point.
The key is stored as a **`wchar_t` string** — on Linux that means **UTF-32LE**, where every
character is one printable byte followed by **three NUL bytes** (`'s' = 73 00 00 00`). Plain
`strings` only looks for single-byte ASCII/UTF-8 runs, so it walks straight past it.

The disassembly confirms the encoding choice: the key is compared with **`wcscmp`** (wide-char
compare), not `strcmp`.

```bash
objdump -d wide | grep -i wcs
#   call   800 <wcscmp@plt>
```

A `wcscmp`/`wcsncmp` import plus an empty ASCII `strings` result is the signature of a wide-char
secret. Two ways to pull it out:

```bash
strings -eL wide   # -eL = 32-bit little-endian (UTF-32LE); -el = 16-bit (UTF-16LE)
```

```bash
python3 -c "import re,sys;d=open(sys.argv[1],'rb').read();print('\n'.join(''.join(chr(d[i]) for i in range(m.start(),m.end(),4)) for m in re.finditer(rb'(?:[\x20-\x7e]\x00\x00\x00){5,}',d)))" wide
# sup3rs3cr3tw1d3
```

## Solution

The cleanest solve recovers the key from the binary itself (nothing hard-coded) and then
derives the flag live by driving the program — selecting dimension `6` and feeding the key.

Create `solve.py`:

```python
import re, subprocess, os

HERE = os.path.dirname(os.path.abspath(__file__))
BIN  = os.path.join(HERE, "files/rev_wide/wide")
DB   = os.path.join(HERE, "files/rev_wide/db.ex")

def recover_key(path):
    data = open(path, "rb").read()
    runs, cur = [], b""
    for i in range(0, len(data) - 3, 4):          # step 4 bytes = one UTF-32LE char
        w = data[i:i+4]
        if 0x20 <= w[0] < 0x7f and w[1:] == b"\x00\x00\x00":
            cur += bytes([w[0]])
        else:
            if len(cur) >= 8:
                runs.append(cur.decode())
            cur = b""
    return next(s.strip() for s in runs if "w1d3" in s)

key = recover_key(BIN)                            # 'sup3rs3cr3tw1d3'
print("[*] key:", key)

p = subprocess.Popen([BIN, DB], stdin=subprocess.PIPE,
                     stdout=subprocess.PIPE, stderr=subprocess.STDOUT)
p.stdin.write(f"6\n{key}\n".encode())
p.stdin.flush()
p.stdin.close()                                   # EOF -> program flushes its output
out = p.stdout.read(4096).decode(errors="replace")
p.kill()
print("[+]", re.search(r"HTB\{[^}]+\}", out).group(0))
```

Run it:

```bash
python3 solve.py
# [*] key: sup3rs3cr3tw1d3
# [+] HTB{...}
```

## Why it worked

The author relied on a tooling blind spot. `strings` is everyone's first move on a binary, and
its default encoding is single-byte, so a secret stored as a wide string is invisible to it. The
key was never encrypted — only re-encoded — which is enough to defeat a careless first pass but
trivial once you spot the `wcscmp` import and switch `strings` to a wide encoding.

## Fix / defense

- **For analysts:** when ASCII `strings` comes up empty but a comparison clearly happens, look
  for wide strings (`strings -el` / `-eL`) and `wcs*` imports. Encoding is not encryption.
- **For developers:** never embed secrets or keys as literals in a binary in *any* encoding —
  wide-char, base64, and XOR are all recoverable by static analysis. Keep secrets out of the
  artifact entirely: derive them at runtime or fetch them from a secure store.
