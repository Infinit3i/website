## Overview

CrossFit2 is an insane-difficulty OpenBSD machine. The path to user is a web chain: a WebSocket-backed chat bot is SQL-injectable, but the real foothold comes from abusing a password-reset form that trusts the `Host` header together with Cross-Site WebSocket Hijacking to steal the administrator's chat — which leaks an SSH password for `david`. This post covers recon through the user flag.

Custom web-chaining: a WebSocket blind SQLi combined with Host-header-trusting password reset and Cross-Site WebSocket Hijacking to exfiltrate the admin chat for an SSH password; bespoke multi-primitive foothold, no CVE.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22/tcp | OpenSSH | OpenBSD |
| 80/tcp | OpenBSD httpd | gym website |
| 8953/tcp | ub-dns-control | unbound remote control (TLS-gated) |

```bash
nmap -sC -sV 10.129.7.241
```

Port 80 hosts a CrossFit gym site whose **MEMBER AREA** link points to `employees.crossfit.htb`. Add the discovered vhosts to your hosts file:

```bash
echo "10.129.7.241 employees.crossfit.htb gym.crossfit.htb crossfit-club.htb" | sudo tee -a /etc/hosts
```

## Enumeration

The index page loads `js/ws.min.js`, which opens a WebSocket to `ws://gym.crossfit.htb/ws/`. Requests under `/ws*` are proxied differently (OpenBSD `relayd` with WebSocket support), and the chat exposes commands like `memberships`. The membership-availability message passes a parameter into a SQL query:

```bash
# intercept the availability request in Burp; the params value is injectable
# 6-months plan is "unavailable" -> easy boolean oracle: append `or 1=1`
```

A UNION-based blind injection over the WebSocket dumps databases. The `employees` table yields usernames, password hashes (uncrackable) and the admin email `david.palmer@crossfit.htb`.

## Foothold

The `password-reset.php` page builds its reset link from the `Host` header, and the chat WebSocket performs no origin check (Cross-Site WebSocket Hijacking). Hosting a malicious `password-reset.php` that registers a user and then drives the admin's authenticated browser, the admin's own chat — containing a freshly issued SSH password — is exfiltrated.

```bash
# 1. request a reset for david.palmer@crossfit.htb (Host header -> attacker)
# 2. CSWSH payload registers a member + reads the admin chat via the victim's session
# 3. one chat message contains david's SSH password
sshpass -p '<redacted>' ssh david@10.129.7.241
```

```
$ id

```
