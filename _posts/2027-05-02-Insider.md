## Overview

**Insider** is an Easy Forensics challenge. You're handed a raw dump of a Windows
user's Firefox profile (`Mozilla/Firefox/Profiles/2542z9mo.default-release/`) and
told a potential insider threat has been reported — find out what they accessed.
The path is pure browser forensics: triage the profile's SQLite/JSON artifacts to
see where the insider went, then recover the credential they saved for the
internal service they were poking at. Because no Firefox Primary (master) password
was ever set, that saved password decrypts to cleartext — and it *is* the flag.

## The technique

A Firefox profile directory is a self-contained forensic goldmine, all in SQLite
and JSON:

| Artifact | What it holds |
|---|---|
| `places.sqlite` | browsing history (`moz_places`), bookmarks |
| `formhistory.sqlite` | typed form / search-bar values |
| `cookies.sqlite` | session cookies |
| `logins.json` + `key4.db` | saved usernames/passwords (encrypted) |
| `sessionstore-backups/*.lz4` | open tabs / restore state (mozLz4) |

Firefox stores saved website logins in `logins.json` with the username and
password individually encrypted (3DES/AES-CBC). The key that unwraps them lives in
the NSS key database `key4.db`. The crucial detail: that wrapping key is derived
from the account's **Primary (master) password** — and when the user never sets
one (the default), the key is derived from the **empty string**. So anyone holding
the profile directory can decrypt every saved credential offline, with no password
and no cracking. This is [CWE-522](https://cwe.mitre.org/data/definitions/522.html)
(Insufficiently Protected Credentials) / effective
[cleartext storage](https://cwe.mitre.org/data/definitions/312.html).

## Solution

First, triage the history to see what the insider was after:

```bash
P="Mozilla/Firefox/Profiles/2542z9mo.default-release"
sqlite3 "$P/places.sqlite" "SELECT datetime(last_visit_date/1000000,'unixepoch'),url FROM moz_places WHERE last_visit_date IS NOT NULL ORDER BY last_visit_date DESC;"
```

The recent history points squarely at an internal target:

```
http://acc01:8080/manager/html      # Tomcat Manager
http://22.22.22.129:8080/
google: "wsl on old windows"
```

`logins.json` confirms a saved credential for exactly that host — realm
*"Tomcat Manager Application"* on `http://acc01:8080`, with `encryptedUsername`
and `encryptedPassword` blobs. To decrypt them we use
[firefox_decrypt](https://github.com/unode/firefox_decrypt), which loads `key4.db`
through `libnss3` and decrypts every entry in `logins.json`. The challenge ships
the durable artifact as `solve.py`:

```python
#!/usr/bin/env python3
import subprocess, sys, os
PROFILE = os.path.join(os.path.dirname(__file__),
    "files/Mozilla/Firefox/Profiles/2542z9mo.default-release")
out = subprocess.run([sys.executable, "firefox_decrypt.py", PROFILE],
                     input=b"", capture_output=True)
print(out.stdout.decode(errors="replace"))
```

Running it feeds the empty password (the `input=b""` here-string) to unwrap the
key — no Primary password means it just works:

```bash
python3 solve.py
```

```
Website:   http://acc01:8080
Username: 'admin'
Password: 'HTB{...}'
```

The decrypted Tomcat Manager password is the flag.

## Why it worked

A browser password store is only as strong as its master password. Firefox sets
none by default, so the key in `key4.db` is wrapped with the empty string and is
trivially recoverable from the profile directory alone. The insider saved their
privileged service credential in the browser, and the moment that profile was
imaged, every saved password — Tomcat admin included — became readable cleartext.
The "encryption" protected nothing without a master password gating the key.

## Fix / defense

- Set a Firefox **Primary (master) password** (`about:preferences#privacy`) so the
  `key4.db` key is PBE-derived from a real secret rather than the empty string.
- Don't store privileged or service credentials (Tomcat Manager, admin panels,
  VPN) in a browser password manager — use a dedicated vault with MFA.
- On managed endpoints, enforce policy `signon.rememberSignons = false` to disable
  browser password saving entirely.
- Full-disk encryption plus restricting access to profile directories limits the
  blast radius of a stolen or backed-up profile.
