## Overview

**Logger** is an Easy HackTheBox **Forensics** challenge. A workstation was
flagged as possibly infected, and the evidence is a packet capture of its USB
traffic (`keystrokes.pcapng`). The whole challenge is reconstructing what was
typed on the keyboard from the raw USB Human Interface Device (HID) reports in
the capture.

## The technique

USB keyboards report key presses to the host as small **8-byte HID reports**.
The layout is fixed and public:

```
byte[0]  byte[1]   byte[2]      byte[3..7]
modifier reserved  keycode #1   keycodes #2..6
```

- `byte[0] & 0x22` → a Shift key is held (`0x02` = Left Shift, `0x20` = Right Shift).
- `byte[2]` → the HID **usage ID** of the pressed key (`0x04`–`0x1d` = `a`–`z`,
  `0x1e`–`0x27` = `1`–`0`, `0x2c` = space, `0x28` = Enter, `0x2a` = Backspace,
  `0x39` = Caps Lock toggle).
- An all-zero report = "all keys released" — skip it.

Because the protocol is unencrypted, anyone with the capture can replay the key
sequence and recover everything typed. This is a textbook **information exposure**
([CWE-200](https://cwe.mitre.org/data/definitions/200.html)) scenario.

## Solution

Confirm the artifact, then locate the HID data. The catch: on **Linux** captures
(`usbmon`) the bytes live in `usb.capdata`, but this is a **Windows USBPcap**
capture, where they live in `usbhid.data`. Always try both — `usb.capdata`
returns nothing here:

```bash
file keystrokes.pcapng
tshark -r keystrokes.pcapng -Y "usbhid.data" -T fields -e usbhid.data > hid.txt
```

Two device formats are interleaved (6-byte reports from another device, 8-byte
keyboard reports). Filter to the 16-hex-char (8-byte) keyboard reports and decode
them, tracking Shift, Caps Lock, and Backspace.

Create `solve.py`:

```python
#!/usr/bin/env python3
KEYS = {0x04:('a','A'),0x05:('b','B'),0x06:('c','C'),0x07:('d','D'),0x08:('e','E'),
 0x09:('f','F'),0x0a:('g','G'),0x0b:('h','H'),0x0c:('i','I'),0x0d:('j','J'),
 0x0e:('k','K'),0x0f:('l','L'),0x10:('m','M'),0x11:('n','N'),0x12:('o','O'),
 0x13:('p','P'),0x14:('q','Q'),0x15:('r','R'),0x16:('s','S'),0x17:('t','T'),
 0x18:('u','U'),0x19:('v','V'),0x1a:('w','W'),0x1b:('x','X'),0x1c:('y','Y'),
 0x1d:('z','Z'),0x1e:('1','!'),0x1f:('2','@'),0x20:('3','#'),0x21:('4','$'),
 0x22:('5','%'),0x23:('6','^'),0x24:('7','&'),0x25:('8','*'),0x26:('9','('),
 0x27:('0',')'),0x28:('\n','\n'),0x2a:('[BKSP]','[BKSP]'),0x2c:(' ',' '),
 0x2d:('-','_'),0x2e:('=','+'),0x2f:('[','{'),0x30:(']','}'),0x31:('\\','|'),
 0x33:(';',':'),0x34:("'",'"'),0x35:('`','~'),0x36:(',','<'),0x37:('.','>'),
 0x38:('/','?'),0x39:('[CAPS]','[CAPS]')}

caps=False; out=[]
for line in open('hid.txt'):
    h=line.strip()
    if len(h)!=16: continue
    b=bytes.fromhex(h); mod=b[0]; key=b[2]
    if key==0: continue
    shift=bool(mod & 0x22)
    if key==0x39: caps=not caps; continue
    if key in KEYS:
        un,sh=KEYS[key]
        if un=='[BKSP]':
            if out: out.pop()
            continue
        c = sh if shift else un
        if c.isalpha() and caps:
            c = c.upper() if not shift else c.lower()
        out.append(c)
print(''.join(out))
```

```bash
python3 solve.py
# HTB{...}
```

## Why it worked

USB HID is an unencrypted, fully documented protocol, and the keycode-to-character
mapping is a fixed public table. A capture of the USB bus therefore contains a
verbatim record of every key press. The only obfuscation was the leetspeak the
flag itself was typed in.

## Fix / defense

- Treat USB-bus captures as highly sensitive — they contain everything typed,
  including passwords and other secrets.
- On high-security endpoints, disable unused USB ports and apply USB
  data-blocking / device-control policies; prefer input channels that are
  encrypted when the threat model includes a logger on the wire.
- Detection: a process or driver capturing the USB bus (a USBPcap driver install,
  or `usbmon` access) on a normal workstation is itself the indicator of
  compromise the scenario points at.
