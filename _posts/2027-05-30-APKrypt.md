## Overview

APKrypt is an Easy [Mobile](https://cwe.mitre.org/data/definitions/321.html) challenge. The app asks for a "VIP code"; enter the right one and it reveals a hidden ticket, otherwise it just says *"Wrong VIP code!"*. Decompiling the APK shows the VIP-code gate is pure theatre — the ticket is AES-encrypted with a key hard-coded right next to the ciphertext, so we never need the code, an emulator, or any instrumentation. We lift both literals and decrypt offline.

## The technique

The weakness is [use of a hard-coded cryptographic key](https://cwe.mitre.org/data/definitions/321.html) (CWE-321). A client application can never keep a secret from someone who owns the binary: shipping the AES key inside the app means every install carries everything needed to decrypt the protected data. The "VIP code" check (`md5(input) == <hash>`) only decides which message to show — it does not key or guard the cipher, so satisfying it is unnecessary.

## Solution

Decompile the APK with `jadx`:

```bash
jadx -d jadx_out APKrypt.apk
```

The only interesting class is `com/example/apkrypt/MainActivity.java`:

```java
// Button onClick:
if (MainActivity.md5(ed1.getText().toString())
        .equals("735c3628699822c4c1c09219f317a8e9")) {
    Toast.makeText(ctx,
        MainActivity.decrypt("k+RLD5J86JRYnluaZLF3Zs/yJrVdVfGo1CQy5k0+tCZDJZTozBWPn2lExQYDHH1l"),
        1).show();                       // <-- the ticket
} else {
    Toast.makeText(ctx, "Wrong VIP code!", 0).show();
}

public static String decrypt(String s) throws Exception {
    Cipher c = Cipher.getInstance("AES");          // no mode = AES/ECB/PKCS5Padding
    c.init(Cipher.DECRYPT_MODE, generateKey());
    return new String(c.doFinal(Base64.decode(s, 0)), "utf-8");
}

private static Key generateKey() throws Exception {
    return new SecretKeySpec("Dgu8Trf6Ge4Ki9Lb".getBytes(), "AES");  // hard-coded key
}
```

Three things give the game away: the key is a 16-byte string literal (`Dgu8Trf6Ge4Ki9Lb`, AES-128), the ciphertext is a base64 literal, and `Cipher.getInstance("AES")` with no mode defaults to **AES/ECB/PKCS5Padding** (no IV needed). That's everything required to decrypt offline.

Create `solve.py`:

```python
#!/usr/bin/env python3
import base64
from Crypto.Cipher import AES
from Crypto.Util.Padding import unpad

KEY    = b"Dgu8Trf6Ge4Ki9Lb"   # SecretKeySpec literal from generateKey()
CT_B64 = "k+RLD5J86JRYnluaZLF3Zs/yJrVdVfGo1CQy5k0+tCZDJZTozBWPn2lExQYDHH1l"

ct = base64.b64decode(CT_B64)
pt = unpad(AES.new(KEY, AES.MODE_ECB).decrypt(ct), AES.block_size)
print(pt.decode())
```

```bash
$ python3 solve.py

```

## Why it worked

The whole confidentiality story rests on a key the attacker already holds. Because the key lives in the dex, "protecting" the ticket with AES adds no security — recovering the key is just reading a string. The MD5 "VIP code" gate guards a UI branch, not the secret, and Java's bare `Cipher.getInstance("AES")` silently selecting ECB makes the decrypt a one-liner with no IV to worry about.

## Fix / defense

- **Never embed cryptographic keys** in an app. Derive them from a user secret with a KDF (PBKDF2/scrypt/Argon2), or store them in the **Android Keystore / hardware StrongBox**, where the bytes are non-extractable.
- Don't treat a client-side check (an MD5 of an input code) as a security boundary — anything in the dex is attacker-visible.
- Avoid **ECB**; use an authenticated mode such as AES-GCM with a fresh random IV per message.
