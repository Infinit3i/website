## Overview

Hourcle is a Crypto (Easy) challenge. The service "seals" your name by running
`AES.new(KEY, MODE_CBC, iv=IV).decrypt(pad(user + password))` — it calls the **decryption**
routine to encrypt. Because a 20-character secret `password` is appended to our input, and CBC
decryption exposes the block cipher's inverse permutation directly, this collapses to the same
[byte-at-a-time recovery](https://cwe.mitre.org/data/definitions/327.html) you would run against an
ECB oracle. We recover the password one byte at a time and use it to open the Forbidden Sanctum.

## The technique

The vulnerable function:

```python
def encrypt_creds(user):
    padded = pad((user + password).encode(), 16)   # password = 20-char server secret
    IV = os.urandom(16)
    cipher = AES.new(KEY, AES.MODE_CBC, iv=IV)
    ciphertext = cipher.decrypt(padded)            # decrypt(), used as "encrypt"
    return ciphertext
```

CBC **decryption** produces each output block as:

```
O[k] = D_K(B[k]) XOR B[k-1]          (B[-1] = IV)
```

where `B` are the input blocks — here `pad(user + password)`, which we mostly control. Rearranged,
for any block index `k ≥ 1` (so the previous block `B[k-1]` sits inside our prefix and is known):

```
D_K(B[k]) = O[k] XOR B[k-1]
```

That is a **free oracle for the fixed permutation `D_K`** on any 16-byte block we choose. Submit
`user = AAAA…(16) || X`: block 1 is `X`, block 0 is `'A'*16`, so `dk(X) = O[1] XOR ('A'*16)`. The
random per-message `IV` only affects `O[0]`, which we never use — so re-randomising it each call
changes nothing.

Since `password` is appended right after `user`, we align it block by block: pad so the next unknown
secret byte is the **last** byte of a block, read that block's true value `O[t] XOR B[t-1]`, then
brute the final byte by querying `dk([15 known] || guess)` over the `[A-Za-z0-9]` charset. The
username filter `^\w{16,}$` allows only word characters — our `A` filler and the password charset
all qualify. The menu loops, so every query runs over one persistent connection.

## Solution

`solve.py`:

```python
import sys, string, re
from pwn import remote, context
context.log_level = 'error'
HOST, PORT = sys.argv[1], int(sys.argv[2])
CHARSET = (string.ascii_letters + string.digits).encode()
BS = 16
io = remote(HOST, PORT)

def query(user):                       # menu opt 1 -> list of 16-byte output blocks
    io.recvuntil(b'traveler :: '); io.sendline(b'1')
    io.recvuntil(b'archives :: ');  io.sendline(user)
    line = io.recvuntil(b'\n')
    ct = bytes.fromhex(re.search(rb'scrolls: ([0-9a-f]+)', line).group(1).decode())
    return [ct[i:i+BS] for i in range(0, len(ct), BS)]

def x(a, b): return bytes(p ^ q for p, q in zip(a, b))

A = b'A' * BS
def dk(block):                          # oracle for D_K on any chosen block
    return x(query(A + block)[1], A)    # O[1] = D_K(block) ^ A

known = b''
while len(known) < 20:
    pad_len = (15 - len(known)) % BS
    user = b'A' * BS + b'A' * pad_len           # >=16, all \w; secret byte is last in its block
    O = query(user)
    p = len(user) + len(known)                  # plaintext index of next secret byte
    t = p // BS                                  # its block index (0-based)
    pt = user + known
    tg = x(O[t], pt[(t-1)*BS:t*BS])              # D_K(block_t) = O[t] ^ B[t-1]
    k15 = pt[t*BS:t*BS+15]
    known += bytes([next(c for c in CHARSET if dk(k15 + bytes([c])) == tg)])
    print(f"[+] {len(known):2}/20 {known.decode()}")

pw = known.decode()
io.recvuntil(b'traveler :: '); io.sendline(b'2')
io.recvuntil(b'Sanctum :: ');  io.sendline(pw.encode())
print(io.recvuntil(b'}').decode())               # the flag
```

Run it against a fresh instance:

```bash
python3 solve.py <host> <port>
```

It recovers all 20 password bytes (~20 seconds) and then submits the password to menu option 2,
which prints the flag:

```
[+] 20/20 8PsQh96b9eHHTXn2FZtK
HTB{...}
```

## Why it worked

Calling a block cipher's `decrypt()` to "encrypt" hides nothing. The inverse permutation `D_K` is
just as deterministic as `E_K`, and CBC's feedback equation exposes it directly the moment one
neighbouring block is attacker-known. Concatenating attacker input with a secret into a single
re-queryable cipher input then turns the problem into the classic ECB
[`prefix || secret`](https://cwe.mitre.org/data/definitions/327.html) byte-at-a-time recovery —
the flag text even spells it out: encrypting with CBC decryption is as insecure as ECB.

## Fix / defense

- **Never call a cipher's `decrypt()` routine to encrypt data.** Use authenticated encryption
  (AES-GCM / libsodium `secretbox`) — the output is integrity-protected and is not a malleable,
  queryable block oracle.
- **Don't put a secret in the same cipher input as attacker-controlled data** that the attacker can
  re-encrypt at will.
- **Compare credentials with a constant-time MAC** (`hmac.compare_digest`) over a fixed message,
  not by encrypting `user + password`.

```python
c = AES.new(KEY, AES.MODE_GCM)
ct, tag = c.encrypt_and_digest(pad(user.encode(), 16))   # never (user + secret)
```
