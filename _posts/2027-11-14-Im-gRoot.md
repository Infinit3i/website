## Overview

`I'm gRoot` is an easy HackTheBox **Crypto** challenge. A "blockchain" service mines a block of
eight transactions, builds a Merkle tree over their signatures, and prints the root. It then offers
to hand you the flag if you can submit a **different** list of signatures that reproduces the **same
root**. The tree is built with `pymerkle.InmemoryTree(security=False)`, which disables domain
separation — and that single flag is enough to forge a [second
preimage](https://cwe.mitre.org/data/definitions/345.html) without breaking SHA-256.

## The technique

A Merkle tree hashes data in two places: each **leaf** is the hash of a data item, and each
**internal node** is the hash of its two children. To stay secure against forgery, those two hash
operations must be *domain-separated* — RFC 6962 does this by prepending `0x00` before leaf data
and `0x01` before a node pair, so a leaf hash can never be confused with a node hash.

`pymerkle`'s `security=False` mode drops those prefixes. The two operations collapse into the
**identical** function:

```
leaf hash = sha256(data)
node hash = sha256(left ‖ right)
```

So a parent node `sha256(L0 ‖ L1)` is byte-for-byte indistinguishable from a *leaf* whose data is
the 64-byte blob `L0 ‖ L1`. And the verifier rebuilds the tree from a client-supplied,
comma-separated hex list using `bytes.fromhex`, which imposes **no length limit** on each element —
so nothing stops us from handing it 64-byte "leaves".

```python
def valid(self, _signatures):
    _mt = MerkleTree(security=False)
    for _signature in _signatures:        # client controls the list AND each element's length
        _mt.append(_signature)
    return self.mt.get_state() == _mt.get_state()
```

## Solution

The original block has 8 leaves `Li = sha256(sig_i)`, giving this tree:

```
            ROOT = H(H(P0,P1), H(P2,P3))
           /                              \
      P0=H(L0,L1)  P1=H(L2,L3)      P2=H(L4,L5)  P3=H(L6,L7)
```

Instead of 8 leaves, submit **four** 64-byte "signatures", each the concatenation of two adjacent
leaf hashes:

```
S0 = L0‖L1   →  leaf = sha256(L0‖L1) = P0
S1 = L2‖L3   →  leaf = P1
S2 = L4‖L5   →  leaf = P2
S3 = L6‖L7   →  leaf = P3
```

The new 4-leaf tree's leaves are exactly the original level-1 nodes `P0..P3`, so every node above is
identical → **same root**, but the submitted list differs from the original. Second preimage, zero
hashing work.

One important gotcha: match the remote library version. The challenge image uses
`pymerkle==5.0.3`, which exposes `InmemoryTree.append` and builds the balanced tree the attack
assumes. The current `6.1.0` renamed it to `append_entry` **and** changed the tree topology, so pin
the old version locally before trusting the forgery:

```sh
pip install pymerkle==5.0.3
```

`solve.py` reads the eight signatures from menu option 1, builds the four collapsed leaves, and
submits them via menu option 2:

```python
import sys, re
from hashlib import sha256
from pwn import remote

host, port = sys.argv[1], int(sys.argv[2])
io = remote(host, port)

io.sendlineafter(b"> ", b"1")                                  # dump mined blocks
data = io.recvuntil(b"[1] Explore", drop=True).decode(errors="replace")
tx_lines = re.findall(r"Transactions: \[(.*?)\]", data)        # last block's tx sigs
sigs = [bytes.fromhex(h) for h in re.findall(r"[0-9a-f]{64}", tx_lines[-1])]

leaf   = [sha256(s).digest() for s in sigs]                    # leaf hash = sha256(sig)
forged = [leaf[i] + leaf[i + 1] for i in range(0, 8, 2)]       # 64-byte "leaves" = level-1 nodes
payload = ",".join(e.hex() for e in forged)

io.sendlineafter(b"> ", b"2")
io.sendlineafter(b"signatures: ", payload.encode())
print(io.recvall(timeout=5).decode())                          # HTB{...}
```

```sh
python3 solve.py <target-host> <target-port>
# ...
# HTB{...}
```

## Why it worked

The integrity of a Merkle commitment rests on "same root ⇒ same data". That guarantee only holds
when a node hash can never be reinterpreted as a leaf hash. Removing domain separation
(`security=False`) breaks exactly that invariant: an internal node value is a perfectly valid leaf
input, so we can splice the tree at any level and present fewer, longer "leaves" that hash up to the
same root. No collision in SHA-256 is required — the weakness is entirely in how the tree composes
the hash, not in the hash itself.

## Fix / defense

- **Domain-separate leaves and internal nodes.** Prefix leaf hashes with `0x00` and node hashes
  with `0x01` (RFC 6962 / pymerkle `security=True`) so a node hash can never be a valid leaf input.
- **Validate structure, not just the root.** Reject submissions whose leaf count or element lengths
  differ from what the protocol expects, instead of trusting an arbitrary client-supplied list.
- **Bind the arity into the commitment.** Including the leaf count in the committed root removes the
  ambiguity that lets an 8-leaf tree and a 4-leaf tree share a root.
