## Overview

FormulaX is a hard-difficulty Linux machine fronted by a Node.js **chatbot** web application. The path to user starts in the chat: messages are stored and later rendered to a support "admin" bot **as HTML**, giving a stored cross-site-scripting (XSS) primitive. That XSS runs in the bot's context — inside the box — and reaches an internal-only `dev-git-auto-update` service whose `simple-git` library is vulnerable to **CVE-2022-25912** (argument-injection RCE). That drops a shell as `www-data`. From there, the app's **MongoDB** hands over a bcrypt password hash that cracks with rockyou, and the recovered password is reused over SSH to land the user flag. This post stops at user.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 22   | OpenSSH 8.9p1 (Ubuntu) | standard, no anon access |
| 80   | nginx → Node chatbot app | login + registration, Socket.IO chat |

```bash
nmap -sC -sV 10.129.230.190
```

Two ports. The web app on 80 is the whole front door — a chat application that lets you register, log in, and message a support agent. SSH is just the eventual landing spot once credentials surface.

## Enumeration

Registering and using the chat reveals a support/admin agent that **reads** user messages. The client connects with `io('/', { withCredentials: true })` and the message content is rendered into the agent's view without sanitization — a classic **stored XSS** sink. An injected `<script>` runs in the agent's browser, which lives on the server side and can reach services not exposed to the internet.

Pointing the XSS payload at internal resources surfaces a virtual host used for automated repository updates:

```
dev-git-auto-update.chatbot.htb
```

This service drives Node's `simple-git` to clone/pull repositories — and the bundled version is vulnerable to **CVE-2022-25912**.

## Foothold

`simple-git` before 3.15.0 passes an attacker-influenced repository URL straight to the `git` binary. A URL crafted as a remote-helper / option-injection string (`ext::sh -c <cmd>`) is interpreted by git as transport directives, executing an arbitrary command on the host. Driving the internal git-update service (via the XSS-hijacked bot) with such a URL runs commands as the web user:

```bash
# repository URL submitted to the internal git-auto-update endpoint
ext::sh -c "<reverse-shell-or-command>"
```

That yields a shell as `www-data`. The application config points at a local MongoDB:

```bash
# /var/www/app/configuration/connect_db.js
# URL_DATABASE="mongodb://localhost:27017"
mongo
# > use chatbot
# > db.users.find()    # dumps bcrypt password hashes for the app users
```

One of the dumped bcrypt hashes belongs to `frank_dorky`. Cracking it offline with rockyou succeeds:

```bash
# requires: /usr/share/wordlists/rockyou.txt
john -w=/usr/share/wordlists/rockyou.txt frank_hash    # -> <redacted>
```

The recovered password is reused as `frank_dorky`'s SSH password:

```bash
sshpass -p '<redacted>' ssh -o StrictHostKeyChecking=no frank_dorky@10.129.230.190
id    # uid=1002(frank_dorky)
```
