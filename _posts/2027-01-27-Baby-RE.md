## Overview

`Baby RE` is an Easy HackTheBox **Reversing** challenge. You get a single
x86-64 ELF that asks for a key and either prints a flag or `Try again later.`.
There is no packing, no anti-debug, and no real math — the whole challenge is one
[hardcoded comparison](https://cwe.mitre.org/data/definitions/798.html): read the
key out of the disassembly, feed it back, and the program prints the flag itself.

## The technique

The binary's `main` does exactly two interesting things: it `fgets` your input
and `strcmp`s it against a fixed string baked into `.rodata`. If the comparison
succeeds, it assembles the flag from four immediate `mov` instructions and
`puts` it. Both the gate (the key) and the prize (the flag bytes) are plaintext
data inside the file — the only "protection" is misdirection.

That misdirection is worth calling out, because it is the actual lesson. The
`.rodata` section literally contains the string *"Dont run `strings` on this
challenge, that is not the way!!!!"* and seeds a fake `HTB{B4BYH...` fragment so
that a reflexive `strings baby | grep HTB` returns a wrong, truncated flag.
Reading the disassembly instead of grepping the string table is the intended
path.

## Solution

First, unpack the archive. The download is an **AES-encrypted zip** (compression
method 99), so plain `unzip` fails — use `7z`:

```bash
7z x -phackthebox files.zip
file baby   # ELF 64-bit LSB pie executable, x86-64, not stripped
```

Disassemble `main` and read the check:

```bash
objdump -d -M intel baby | sed -n '/<main>:/,/ret/p'
```

The relevant logic is:

```asm
call   fgets@plt                       ; fgets(buf, 0x14, stdin)
lea    rsi,[rip+0xebc]   # 0x2053      ; the static key
call   strcmp@plt
test   eax,eax
jne    fail                            ; mismatch -> "Try again later."
movabs rax,0x594234427b425448          ; "HTB{B4BY"
movabs rdx,0x3448545f5633525f          ; "_R3V_TH4"
mov    DWORD PTR [rbp-0x30],0x455f5354  ; "TS_E"
mov    WORD  PTR [rbp-0x2c],0x7d5a       ; "Z}"
call   puts@plt                        ; prints the assembled flag
```

The comparison target lives at `.rodata:0x2053`. Dump it directly instead of
trusting `strings`:

```bash
objdump -s -j .rodata baby
```

That reveals the key string `abcde122313` (the stored copy includes the trailing
newline, which `fgets` also keeps). Feed it back and let the binary derive the
flag live:

Create `solve.py`:

```python
#!/usr/bin/env python3
from pwn import process
p = process('./baby')
p.sendline(b'abcde122313')
print(p.recvall(timeout=3).decode(errors='replace'))
```

```bash
python3 solve.py
# Insert key:
# HTB{...}
```

Flag value redacted — it is the string the program itself `puts` on a correct
key, not anything you hand-decode from the immediates.

## Why it worked

The "secret" was never secret: the key and the flag are both static data inside
the binary, recoverable with `objdump`. This is the textbook shape of a
[hardcoded credential / hardcoded secret](https://cwe.mitre.org/data/definitions/798.html)
weakness — anything a client must validate or reveal locally can be read straight
out of the client. The decoy strings only delay someone who greps instead of
disassembling.

## Fix / defense

- Never gate access on a local, hardcoded comparison value: both the key and the
  protected data are trivially recoverable from a shipped binary.
- A secret a user must not learn cannot live client-side at all. It has to be
  derived server-side from a real authentication step, not `strcmp`'d against a
  literal compiled into the program.
- Obfuscating the string table (the "don't run `strings`" troll) is not a control
  — it changes nothing for anyone reading the disassembly.
