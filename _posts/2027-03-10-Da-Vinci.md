## Overview

**Da Vinci** is an Easy HackTheBox **Misc** (steganography) challenge. You get three
JPEGs — `monalisa.jpg`, `Plans.jpg`, and a third whose *filename* is a riddle — and the
prompt teases a secret hidden in the pictures. It's a layered chain: each image yields the
key needed to open the next, ending in a triple-base64-encoded flag. The whole thing is
scripted end-to-end in `solve.py` below.

## The technique

The core skill is recognizing **multi-stage stego**: never tunnel on one file, because each
artifact is just a key for the next stage.

1. **The passphrase is hinted in plain sight.** The file
   `Thepassword_is_the_small_name_of_the_actor_named_Hanks.jpg` literally spells it out —
   the "small name" (first name) of actor *Hanks* is **TOM**, and the word `TOM` is also
   painted on Tom Hanks' head in the image. `steghide` accepts that as the passphrase. Case
   matters: only uppercase `TOM` worked.
2. **A 32-char "encrypted" string is a hash, not a cipher.** The note that steghide drops
   says *"I used an encryption with 32 characters"* and gives `020e60c6a84db8c5d4c2d56a4e4fe082`.
   That length is the tell for **MD5** — crack it (it's `leonardo`), don't try to "decrypt" it.
3. **A JPEG can carry an appended ZIP.** `binwalk monalisa.jpg` reveals a ZIP archive glued
   on after the image data — image viewers ignore the trailing bytes. Inside is a ZipCrypto'd
   `Mona.jpg` that opens with the cracked password `leonardo`.
4. **Follow out-of-band clues.** `strings Plans.jpg` leaks a YouTube link whose video is
   titled *"Guernica 3D"*. That gives both the next steghide passphrase (`Guernica`) **and**
   the decode count — the "3D" means base64-decode the recovered blob **three** times.

## Solution

The full chain, automated. The flag is re-derived live by running the script — never copied
from a writeup:

`Create solve.py:`

```python
#!/usr/bin/env python3
import base64, hashlib, subprocess, os

D = os.path.dirname(os.path.abspath(__file__))
F = os.path.join(D, "files")

# 1) filename hint -> steghide passphrase TOM -> note with a 32-char hex "key"
hint = os.path.join(F, "Thepassword_is_the_small_name_of_the_actor_named_Hanks.jpg")
subprocess.run(["steghide","extract","-sf",hint,"-p","TOM","-xf","/tmp/hint.out","-f"], check=True)
note = open("/tmp/hint.out").read()
md5key = note.split("key:")[1].split(")")[0].strip()

# 2) "encryption with 32 characters" = MD5 -> crack to 'leonardo'
assert hashlib.md5(b"leonardo").hexdigest() == md5key
zippw = "leonardo"

# 3) monalisa.jpg has an appended ZIP (famous.zip -> Mona.jpg, ZipCrypto'd with leonardo)
subprocess.run(["unzip","-o","-P","tom",os.path.join(F,"monalisa.jpg"),"-d","/tmp/dv1"], check=False)
subprocess.run(["unzip","-o","-P",zippw,"/tmp/dv1/famous.zip","-d","/tmp/dv2"], check=True)

# 4) Plans.jpg -> YouTube "Guernica 3D": passphrase 'Guernica', "3D" = base64-decode x3
subprocess.run(["steghide","extract","-sf","/tmp/dv2/Mona.jpg","-p","Guernica","-xf","/tmp/key.out","-f"], check=True)
d = open("/tmp/key.out","rb").read().strip()
for _ in range(3):
    d = base64.b64decode(d)
print(d.decode())
```

Run it from the challenge directory (with the three JPEGs in `files/`):

```bash
python3 solve.py
# HTB{...}
```

The manual equivalent, step by step:

```bash
steghide extract -sf Thepassword_is_the_small_name_of_the_actor_named_Hanks.jpg -p TOM -xf note.txt
cat note.txt                         # reveals key:020e60c6a84db8c5d4c2d56a4e4fe082
echo -n leonardo | md5sum            # 020e60c6... -> confirms the hash is 'leonardo'
binwalk monalisa.jpg                 # shows an appended ZIP (famous.zip / Mona.jpg)
binwalk -e monalisa.jpg              # carve it out
unzip -P leonardo famous.zip         # extracts Mona.jpg
strings Plans.jpg | grep -i youtube  # leaks the "Guernica 3D" clue
steghide extract -sf Mona.jpg -p Guernica -xf key.b64
base64 -d key.b64 | base64 -d | base64 -d   # 3x decode -> flag
```

The flag prints as `HTB{...}` (redacted here).

## Why it worked

Steganography challenges hide the *next secret*, not the flag itself — and the authors leave
the keys lying around: in filenames, in text painted onto the image, in EXIF, or in linked
media. The progression here only works because each clue points to the next: the riddle
filename → an MD5 hash that masquerades as "encryption" → a ZIP smuggled past a JPEG's
end-of-image marker → a YouTube title that encodes both a passphrase and a decode count.

## Fix / defense

Stego is obscurity, not security. Don't ship secrets inside image metadata or as data
appended after a file's real EOF — `file`, `binwalk`, and `strings` surface them instantly.
Never store passwords as unsalted MD5, and don't reach for base64 ("3D") when you mean to
*encrypt* — encoding is reversible by anyone who notices it.
