## Overview

**fs0ciety** is an Easy HackTheBox **Misc** challenge — a matryoshka of encodings. You unzip an outer archive to find a *password-protected* inner ZIP, crack that password offline with `john` against `rockyou`, extract a file that claims to hold "encrypted SSH credentials", and discover the contents are just **base64 → space-separated binary → ASCII**. Peel each layer and the flag falls out. No service, no exploit — pure obfuscation.

## The technique

Two ideas carry the whole challenge:

1. **Cracking a ZipCrypto archive.** A ZIP encrypted with the legacy PKZIP stream cipher (*ZipCrypto*) has no key-derivation function and no rate limiting — each password guess is a cheap CRC/stream check. `zip2john` turns the archive's encryption header into a hash that John understands, and a `rockyou` dictionary run finishes almost instantly. (AES-256 zips are far slower — check the `Method` line before you commit to a wordlist.)
2. **"Encoding is not encryption."** The payload file is labelled as encrypted credentials, but base64-decoding it doesn't give text — it gives a string of `0`/`1` bytes separated by spaces. Each 8-bit group is one ASCII character. Recognising *which* decode applies at each layer (base64 alphabet, then a 0/1-only string) is the actual puzzle.

## Solution

```bash
# 1. Outer archive — standard HTB password
unzip -P hackthebox files.zip          # -> fsociety.zip

# 2. Inner ZIP is encrypted (ZipCrypto). Confirm the method, extract the hash.
7z l -slt fsociety.zip | grep -i Method   # Method = ZipCrypto Deflate
zip2john fsociety.zip > zip.hash

# 3. Crack with rockyou (ZipCrypto -> sub-second)
john --wordlist=/usr/share/wordlists/rockyou.txt zip.hash
john --show zip.hash                    # password recovered

# 4. Extract and inspect the payload
unzip -P <password> fsociety.zip        # -> sshcreds_datacenter.txt
cat sshcreds_datacenter.txt             # a long base64 blob

# 5. Decode the chain
python3 solve.py sshcreds_datacenter.txt
```

The decode chain is short enough to keep as a durable artifact:

Create `solve.py`:

```python
#!/usr/bin/env python3
import base64, re, sys
data   = open(sys.argv[1]).read()
b64    = re.search(r"[A-Za-z0-9+/=]{16,}", data).group(0)   # grab the blob
binstr = base64.b64decode(b64).decode()                     # "01101001 0110..."
flag   = "".join(chr(int(b, 2)) for b in binstr.split())    # binary -> chars
print("FLAG:", flag)
```

Running it prints `HTB{...}`.

## Why it worked

ZipCrypto provides essentially no protection against an offline dictionary attack: there is no slow KDF, so guessing is limited only by how fast you can stream candidates, and the password here lives in `rockyou`. Once you're inside, the "encryption" of the credentials file is just two reversible encodings stacked — base64 and binary-as-text — neither of which keeps a secret from anyone who recognises the character set.

## Fix / defense

- Don't protect secrets with ZipCrypto. Use AES-256 zips with a high-entropy passphrase, or an actual secret store. A password that appears in any public wordlist offers no real protection.
- Never treat base64/hex/binary encoding as confidentiality. Encode for transport; encrypt with a real key for secrecy.
