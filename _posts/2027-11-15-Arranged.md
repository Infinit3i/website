## Overview

`Arranged` is an easy HackTheBox **Crypto** challenge. A SageMath script performs an elliptic-curve
Diffie–Hellman exchange and seals the flag with AES-CBC under the shared secret, while trying to add
security by keeping the field prime `p` and curve coefficient `b` secret. Neither matters: the
parameters are recoverable from the public points, and the generator was chosen from a tiny subgroup,
so the [discrete-log problem](https://cwe.mitre.org/data/definitions/1240.html) that ECDH relies on
is no problem at all.

## The technique

We are given `main.sage` and `output.txt`. The curve is `y^2 = x^3 + 726*x + b (mod p)` with `p`, `b`,
`priv_a`, and `priv_b` all secret. The output hands us the public points `A = priv_a*G` and
`B = priv_b*G`, the AES IV, and the ciphertext. The AES key is
`sha256(long_to_bytes(int(C[0])))[16:32]` where `C = priv_a*B` is the shared secret.

Two independent weaknesses break it:

1. **The "secret" parameters are not secret.** Rearrange the curve equation:
   `f(x,y) = y^2 - x^3 - 726*x ≡ b (mod p)` — and `726` is public, so *every* point on the curve
   evaluates `f` to the same value `b (mod p)`. The difference of `f` over any two published points is
   therefore an exact multiple of `p`. With three points (`G`, `A`, `B`), `p` falls out of a GCD:
   `p = gcd(f(G) - f(A), f(G) - f(B))` (strip tiny cofactors), and then `b = f(G) mod p`.

2. **The generator has a tiny order.** `G` lives in a small subgroup, so the discrete log
   `A = priv_a*G` is recovered by simply walking multiples of `G` until one equals `A` — here
   `priv_a = 7`. No Pohlig–Hellman, no Sage, no index calculus.

A useful tell: the published point coordinates are *larger* than any modulus you might guess, which is
the hint that the real `p` is hidden — never assume it, always derive it from the points.

## Solution

The whole solve is pure Python with big integers. `solve.py`:

```python
from Crypto.Cipher import AES
from Crypto.Util.number import long_to_bytes
from hashlib import sha256
from math import gcd

a = 726
# Gx, Gy, Ax, Ay, Bx, By, ct, iv taken verbatim from output.txt
# ...

# 1. Recover the secret prime p and coefficient b from the public points.
f = lambda x, y: y*y - x*x*x - a*x
p = gcd(f(Gx, Gy) - f(Ax, Ay), f(Gx, Gy) - f(Bx, By))
for s in range(2, 1000):
    while p % s == 0 and p // s > 2**400:
        p //= s
b = f(Gx, Gy) % p

# 2. Pure-Python elliptic-curve point addition over GF(p).
def inv(v): return pow(v, -1, p)
def add(P, Q):
    if P is None: return Q
    if Q is None: return P
    if P[0] == Q[0] and (P[1] + Q[1]) % p == 0: return None
    m = (3*P[0]*P[0] + a) * inv(2*P[1]) % p if P == Q else (Q[1]-P[1]) * inv(Q[0]-P[0]) % p
    x = (m*m - P[0] - Q[0]) % p
    return (x, (m*(P[0]-x) - P[1]) % p)

G, A, B = (Gx, Gy), (Ax, Ay), (Bx, By)

# 3. Brute-force the discrete log: the generator's order is tiny.
P = G
for k in range(1, 1000000):
    if P == A:
        priv_a = k
        break
    P = add(P, G)

# 4. Rebuild the shared secret, derive the AES key, decrypt.
C = None
for _ in range(priv_a):
    C = add(C, B)
key = sha256(long_to_bytes(int(C[0]))).digest()[16:32]
print(AES.new(key, AES.MODE_CBC, iv).decrypt(ct))
```

Running it recovers a 512-bit prime, finds `priv_a = 7`, and decrypts the flag:

```bash
python3 solve.py   # HTB{...}
```

## Why it worked

Cryptographic security must rest on the hardness of a problem stated over **public** parameters, not on
hiding the parameters. ECC domain parameters (`p`, `a`, `b`, `G`) are meant to be public and chosen so
that the elliptic-curve discrete-log problem is hard. This challenge got both wrong: the parameters were
algebraically recoverable from any points on the curve, and the base point was drawn from a small
subgroup, collapsing the discrete log to a handful of additions. The same idea — *a secret that takes
part in published arithmetic is usually exposed by a difference or a GCD* — also recovers RSA factors
from a `c1 - c2` difference.

## Fix / defense

```python
from cryptography.hazmat.primitives.asymmetric import ec
priv = ec.generate_private_key(ec.SECP256R1())   # large prime-order generator, vetted public params
```

- Use a vetted, standard curve with a **large prime-order** generator (secp256r1, Curve25519); reject
  low-order / small-cofactor base points.
- Treat domain parameters as public and choose them for hardness — never depend on hiding `p` or `b`.
- Validate the generator's order at key generation and reject points in small subgroups.
- Don't hand-roll ECDH — use a reviewed library that enforces these checks.
