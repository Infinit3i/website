## Overview

`Olympus` is a Medium Linux machine. The interesting path is short but layered: a misconfigured DNS server allows an anonymous zone transfer that leaks a secret [port-knock](https://cwe.mitre.org/data/definitions/200.html) sequence *and* a set of SSH credentials. Knocking the three secret ports briefly opens SSH; the leaked credentials log us in; and because that user is in the `docker` group, escalating to root is a single command.

## Recon

| Port | Service |
|------|---------|
| 22   | OpenSSH (filtered — port-knock gated) |
| 53   | DNS (bind) |
| 80   | HTTP (Apache, Xdebug) |
| 2222 | OpenSSH (docker container) |

```bash
nmap -sC -sV 10.129.19.250
```

Port 22 shows as `filtered` — it only becomes reachable after a correct port-knock sequence. The standout is port 53: a DNS server we can query directly.

## Enumeration

DNS servers should never allow arbitrary clients to copy an entire zone. This one does. A zone transfer (`AXFR`) dumps every record, including a telling TXT entry.

```bash
dig axfr @10.129.19.250 ctfolympus.htb
```

The zone contains a TXT record that reads, in plain text, *"prometheus, open a temporal portal to Hades (3456 8234 62431) and St34l_th3_F1re!"* — handing us the secret knock sequence (`3456 8234 62431`) and a username/password in one shot.

![dns axfr leaks knock sequence and creds](/assets/Images/olympus-003_enum_dns-axfr-knock-creds.png)

## Foothold

SSH on port 22 is gated by `knockd`: it only opens for 10 seconds after it observes connection attempts to ports `3456`, `8234`, `62431` in that exact order.

A practical gotcha: the usual `knock`/`nmap` one-liners can fail here, because their TCP sockets to the filtered knock ports get retransmitted by the kernel, and the retransmits arrive out of order and reset `knockd`'s sequence counter. Sending exactly one clean SYN per port — a non-blocking `connect`, then an immediate `close` so the kernel never retransmits — reliably opens the port. SSH must follow immediately, inside the 10-second window.

```bash
python3 -c "import socket,time;[(s:=socket.socket(),s.setblocking(False),s.connect_ex(('10.129.19.250',p)),s.close(),time.sleep(0.12)) for p in (3456,8234,62431)]" && sshpass -p '<redacted>' ssh -o StrictHostKeyChecking=no prometheus@10.129.19.250
```

We land a shell as `prometheus`.

## Privilege Escalation

`id` shows `prometheus` belongs to the `docker` group. Docker group membership is effectively root: the daemon runs as root, and a group member can ask it to start a container with an arbitrary host bind-mount. This is a documented [improper privilege management](https://cwe.mitre.org/data/definitions/269.html) failure, not a Docker bug.

```bash
id
docker images --all
docker run -v /:/mnt --rm olympia chroot /mnt /bin/sh -c 'id; cat /root/root.txt'
```

Mounting the host's `/` into a container (whose process runs as root) and `chroot`-ing into it gives a root view of the real filesystem.
