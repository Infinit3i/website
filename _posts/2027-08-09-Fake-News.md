## Overview

**Fake News** is an Easy HackTheBox **Forensics** challenge. You're handed the captured webroot of a compromised WordPress site (a school newspaper, "The Magic Informer") and asked to work out what happened. The flag is split across **two attacker artifacts**: a rogue plugin carrying a base64'd PHP reverse shell, and an [obfuscated](https://cwe.mitre.org/data/definitions/506.html) JavaScript dropper on a phishing page that smuggles a binary into the browser.

## The technique

Two ideas carry the challenge:

1. **Find the rogue WordPress plugin.** A non-default plugin whose entire body is `eval(base64_decode("..."))` is a webshell until proven otherwise — here it's a pentestmonkey PHP reverse shell with the first flag fragment baked in.
2. **HTML smuggling** ([CWE-506](https://cwe.mitre.org/data/definitions/506.html)). The phishing page runs an [obfuscator.io](https://obfuscator.io)-style script that base64-decodes + XOR-decrypts a payload entirely **in the browser**, wraps it in a `Blob`, and force-downloads it as `official_invitation.iso`. The payload never crosses the network, so proxy/AV inspection sees nothing. The cleanest recovery is to **execute the dropper under a headless Node DOM shim** and intercept the `Blob`, rather than hand-reversing the obfuscation.

## Solution

### Part 1 — the rogue plugin

A plugin that doesn't ship with WordPress stands out: `wp-content/plugins/plugin-manager/plugin-manager.php`. Its whole body is the classic webshell tell:

```php
<?php eval(base64_decode("c2V0X3RpbWVfbGltaXQ...")); ?>
```

Find and decode it:

```bash
grep -rl 'eval(base64_decode' wp-content/plugins/
php -r 'eval("?>".base64_decode("c2V0X3RpbWVfbGltaXQ..."));'   # or python base64
```

It's a pentestmonkey reverse shell with the first fragment as a variable:

```php
$part1 = "HTB{C0m3_0n";
```

### Part 2 — execute the HTML-smuggling dropper

The phishing page `wp-blogs/2022/11/index.php` embeds an obfuscator.io script (rotated string-array decoder, `base64ToArrayBuffer()`, and an `itstime()` XOR routine) that builds a binary client-side and triggers `a.download = 'official_invitation.iso'; a.click()`.

Instead of reversing it, extract the inline `<script>` to `drop.js` and run it in Node with just enough fake DOM to let it run and to catch the `Blob`. One gotcha: the download code is assigned to `window[<prop>] = function(){...}` (a browser fires it on an event), so after `eval` you must invoke that window function yourself.

Create `run_drop.js`:

```js
const fs = require('fs');
let captured = null;
const w = {};
global.window = new Proxy(w, {
  get(t, p) {
    if (p === 'atob') return s => Buffer.from(s, 'base64').toString('binary');
    if (p === 'URL')  return { createObjectURL: () => 'blob:' };
    return t[p];
  },
  set(t, p, v) { t[p] = v; return true; }
});
global.URL = { createObjectURL: () => 'blob:' };
global.Blob = function (parts) { captured = parts && parts[0]; };
global.document = { createElement: () => ({ click(){}, setAttribute(){} }), body: { appendChild(){} } };
global.console = { log(){}, error(){}, warn(){} };

eval(fs.readFileSync('drop.js', 'utf8'));
for (const k of Object.keys(w))
  if (typeof w[k] === 'function') { try { w[k].call(global.window, {}); } catch (e) {} }

fs.writeFileSync('official_invitation.iso', Buffer.from(captured));
```

Run it, open the recovered ISO, and read the marker out of the inner executable:

```bash
node run_drop.js
7z x official_invitation.iso          # -> official_invitation.exe
strings -n4 official_invitation.exe | grep -oE 'part2:[^ }]*}' | sort | uniq -c | sort -rn | head
# => part2:_1t_w4s_t00_g00d_t0_b3_tru3}
```

The `.NET` binary repeats the marker many times — misaligned `strings` hits are noise, so take the most-frequent clean token.

### Assemble

```
part1 = HTB{C0m3_0n
part2 = _1t_w4s_t00_g00d_t0_b3_tru3}
flag  = HTB{...}        # redacted
```

## Why it worked

Obfuscation is not encryption — the browser has to decode and run the dropper, so an analyst can too. Emulating a thin slice of the DOM (`atob`, `Blob`, `URL`, `document`) lets the script execute and lets you intercept the exact bytes it builds, which is faster and less error-prone than unwinding the string-array + XOR by hand. The delivery itself — HTML smuggling — is what makes this dangerous in the wild: the malicious binary is assembled client-side and dropped via an `<a download>` click, so it never appears on the wire for network defenses to catch.

## Fix / defense

- Block ISO/IMG/VHD as email and download attachments at the gateway, and propagate Mark-of-the-Web to the contents of mounted disk images so the inner executable can't run silently.
- Alert on `blob:`/`data:` anchor downloads of executable containers in proxy logs.
- Enforce a Content-Security-Policy script allowlist so inline obfuscated dropper scripts can't execute.
- File-integrity monitoring on `wp-content/plugins/`, disable the WordPress plugin/theme file editor, and treat any plugin whose source is a single `eval(base64_decode(...))` as a backdoor.
