## Overview

RLotto is an Easy HackTheBox **Crypto** challenge — a container-based `nc` service themed as a lottery machine ("Guess the Random Lotto Numbers"). It draws five numbers and reveals them, then draws the *next* five and asks you to type them back. Both draws come from a single Python `random` (Mersenne Twister / MT19937) stream seeded with `int(time.time())`, which is a [use of a predictable seed in a PRNG](https://cwe.mitre.org/data/definitions/337.html). Because the seed is just the current clock second, you don't need any heavyweight MT state-recovery — you brute a ~20-value window, replay the stream, and read off the answer.

## The technique

The whole game lives in `server.py`, which seeds the generator once per connection:

```python
seed = int(time.time())          # attacker can guess this within a few seconds
random.seed(seed)

# first 5 draws are SHOWN to us (the "EXTRACTION"):
while len(extracted) < 5:
    r = random.randint(1, 90)
    if r not in extracted: extracted.append(r)

# next 5 draws (the SOLUTION) come from the SAME stream:
while len(next_five) < 5:
    r = random.randint(1, 90)
    if r not in next_five: next_five.append(r)
```

Two fatal choices:

1. **`random` is MT19937** — a deterministic, non-cryptographic PRNG. Same seed ⇒ identical sequence, forever.
2. **The seed is `int(time.time())`** — there are only a handful of plausible values (the current second), so the seed space is ~20, not 2³². This is a [predictable seed](https://cwe.mitre.org/data/definitions/337.html), the specific sub-case of [insufficiently random values](https://cwe.mitre.org/data/definitions/330.html).

The "next five" are not independent of the "first five" — they're simply the *continuation* of one stream. So observing any output of a time-seeded MT exposes everything that comes after it.

## Solution

Capture the five revealed numbers, brute the seed in a small window around our own clock, replay the *exact* draw logic (matching the dedup rules byte-for-byte — note the second extraction dedups only against itself, not against the first), and continue the stream to get the next five.

`solve.py`:

```python
#!/usr/bin/env python3
import socket, time, random, re, sys

HOST, PORT = sys.argv[1], int(sys.argv[2])

def draw(seed):
    rng = random.Random(); rng.seed(seed)
    extracted = []
    while len(extracted) < 5:
        r = rng.randint(1, 90)
        if r not in extracted: extracted.append(r)
    next_five = []
    while len(next_five) < 5:
        r = rng.randint(1, 90)
        if r not in next_five: next_five.append(r)   # dedup only within next_five
    return extracted, next_five

s = socket.socket(); s.connect((HOST, PORT))
t0 = int(time.time()); s.settimeout(20)

buf = b""
while b"Guess the next extraction" not in buf:
    try: chunk = s.recv(4096)
    except socket.timeout: break
    if not chunk: break
    buf += chunk
text = buf.decode("latin1")

observed = [int(x) for x in re.findall(r"EXTRACTION:\s*([\d ]+)", text)[-1].split()]

seed = next(c for c in range(t0 - 10, t0 + 10) if draw(c)[0] == observed)
answer = " ".join(str(x) for x in draw(seed)[1])
print(f"[+] seed={seed}  next 5 = {answer}")

try: s.recv(4096)
except socket.timeout: pass
s.sendall((answer + "\n").encode()); time.sleep(1)
print(s.recv(4096).decode("latin1"))
```

Run it against the spawned instance:

```bash
python3 solve.py <host> <port>
# [+] seed=1781826503  next 5 = 72 22 28 5 53
# Good Job!  HTB{...}
```

`random.Random().randint` uses the identical MT19937 + rejection-sampling algorithm as the module-level `random`, so the replay is bit-exact and the match is unambiguous.

## Why it worked

A predictable seed turns a "random" sequence into a fully recoverable one. We didn't even need the textbook MT19937 attack (untwist 624 consecutive outputs to clone the internal state) — the time-seed window is so small that a direct brute over `[now-10, now+10]`, confirmed against the five revealed numbers, pins the exact seed in microseconds. From there, every future draw is just more stream.

## Fix / defense

- Never seed a PRNG with time/PID/counter for anything an attacker shouldn't predict.
- Use a CSPRNG: `secrets.randbelow(90) + 1` or `os.urandom`, seeded from OS entropy.
- Never let a client observe outputs of the same generator/stream that produces the secret — even one observed value of a time-seeded MT leaks all of it.
