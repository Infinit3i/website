## Overview

yoU ART is an easy HackTheBox **Hardware** challenge — and the name is the whole hint: *yoU ART → UART*.
There is no firmware file and no `.sal`/`.vcd` logic-analyzer capture to download. Instead, connecting to
the challenge's TCP port streams a single long string of ASCII `0` and `1` characters: the raw, sampled
line-state of a device's serial **TX pin**. Decode that bitstream as UART and you recover the device's boot
log, which leaks the flag inside a diagnostic "file not found" message.

## The technique

UART (Universal Asynchronous Receiver/Transmitter) is the classic two-wire serial debug interface on
embedded devices. It has **no clock line** — the receiver has to stay phase-locked by itself, using a known
baud rate and a fixed frame shape. The standard frame is **8N1**:

```
 idle   start   d0 d1 d2 d3 d4 d5 d6 d7   stop   idle
  1   →   0   →  <--- 8 data bits, LSB --->   1   →  1 ...
```

- The line **idles high** (`1`). A falling edge (`1`→`0`) marks a **start bit**.
- The next **8 data bits are sent LSB-first**.
- A **stop bit** (`1`) closes the 10-bit frame, then the line returns to idle.

Here the capture is **one sample per bit** (no oversampling), so every character in the stream is exactly one
bit. The whole transmission ends in a long run of `1`s — the line going idle. That's [exposed debug/diagnostic
interface output](https://cwe.mitre.org/data/definitions/1191.html) ([CWE-1191](https://cwe.mitre.org/data/definitions/1191.html)):
a production-style device printing its boot diagnostics — including secrets — to a serial console anyone can tap.

### The one gotcha that eats an hour

The naive decoder — "find a `0`, read 10 bits, then hunt for the *next* `0` and repeat" — produces pure
garbage. Inside a data byte, a `0` *data* bit is indistinguishable from a start bit, so re-hunting a start on
every frame destroys the phase alignment.

The fix: lock onto the **first** start bit, then read frames **contiguously** — advance a full 10-bit frame
each time — and only re-hunt a start if the expected stop bit isn't `1` (a genuine desync). That single change
turns noise into a clean boot log. (Fractional-baud sweeps and edge-position fitting are red herrings here —
the bit period is just one sample.)

## Solution

Connect, drain the bitstream until the line goes idle, then UART-decode it 8N1 / LSB-first.

Create `solve.py`:

```python
import socket, sys, re

def capture(ip, port):
    s = socket.socket(); s.settimeout(6); s.connect((ip, int(port)))
    buf = b""
    try:
        while True:
            c = s.recv(65536)
            if not c:
                break
            buf += c
    except socket.timeout:
        pass
    return buf.decode().strip()

def uart_decode(bits, nbits=8, lsb_first=True):
    N = len(bits); frame = 1 + nbits + 1     # start + 8 data + stop
    i = bits.index('0')                      # FIRST falling edge = first start bit
    out = bytearray()
    while i + frame <= N:
        f = bits[i:i + frame]
        if f[0] != '0':                      # only re-hunt on real desync
            j = bits.find('0', i + 1)
            if j < 0:
                break
            i = j; continue
        d = f[1:1 + nbits]
        if lsb_first:
            d = d[::-1]                       # UART is LSB-first
        out.append(int(d, 2))
        i += frame                            # advance a WHOLE frame — never resync per byte
    return bytes(out)

bits = capture(sys.argv[1], sys.argv[2])
log  = uart_decode(bits).decode('latin1')
print(log)
m = re.search(r'HTB\{[^}]+\}', log)
if m:
    print("FLAG:", m.group())
```

Run it against the spawned instance:

```bash
python3 solve.py <target-ip> <target-port>
```

Out comes the device's boot transcript:

```
BusyBox v1.19.4 (2020-10-20 22:02:10 CST) built-in shell (ash)
...
root@ArcherC7v5:/# [LEO]uhttpd start
[error]: open file failed : /tmp/client_list.json
[error]: open file failed : /tmp/proxy_mac_list.json
[error]: open file failed : /tmp/HTB{...}

SFE STARTED---
```

The "missing files" the brief asked us to identify are the `open file failed` lines in the boot diagnostics —
and one of those filenames *is* the flag (`HTB{...}`, redacted here).

## Why it worked

The device shipped with its serial debug console live and unauthenticated, dumping the full boot log to the TX
pin in the clear. Once you tap that pin (or, in this challenge, are handed the sampled line-state), there is no
cryptography and no access control standing between you and everything the firmware prints — including a secret
that ended up in a diagnostic error string. Recovering it is a pure protocol-decode exercise: know that UART is
8N1, idle-high, LSB-first, sampled one bit per character, and read the frames in order.

## Fix / defense

Treat anything emitted on a debug UART as world-readable to anyone with physical (or, here, network) access.

- **Disable the serial console in release firmware** — drop `console=ttyS0,...` from the kernel command line /
  bootloader, or point it at `null`.
- **Gate any debug interface behind authentication** and ship it disabled by default.
- **Depopulate or remove UART/JTAG test pads** on production boards.
- **Never print secrets, keys, or sensitive paths to diagnostic output** — a "file not found" message should
  not contain a flag.
