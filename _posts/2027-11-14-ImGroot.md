## Overview

`I'm gRoot` is an easy HackTheBox **Crypto** challenge. A toy "blockchain" server mines a block of
eight transactions, builds a [Merkle tree](https://cwe.mitre.org/data/definitions/345.html) over
their signatures, prints the root hash, and then asks you to submit a **different** list of
signatures that produces the **same root**. The tree is built with `pymerkle`'s domain separation
turned off, which makes a leaf hash indistinguishable from an internal-node hash — the textbook
condition for a Merkle **second-preimage forgery**. We fold the eight leaves into four over-length
"leaves" and hand back a root-equal but byte-different list.

## The technique

A Merkle tree hashes data twice: once to turn each entry into a *leaf*, and again to combine pairs
of nodes up the tree to the *root*. Security depends on those two operations being
**distinguishable** — otherwise an internal node can be passed off as a leaf.

`pymerkle`'s `security=True` mode enforces this by prepending a **domain-separation byte**: `0x00`
before leaf data, `0x01` before a node pair (the same scheme RFC 6962 uses). The challenge disables
it:

```python
self.mt = MerkleTree(security=False)
for transaction in block.transactions():
    self.mt.append(transaction.signature())   # signature = sha256(_from + _to), 32 bytes
root_hash = self.mt.get_state()
...
def valid(self, _signatures):
    _mt = MerkleTree(security=False)
    for _signature in _signatures:             # bytes.fromhex per CSV field — NO length limit
        _mt.append(_signature)
    return self.mt.get_state() == _mt.get_state()
```

With `security=False` the prefixes vanish, so the two hash functions become **identical**:

```
leaf hash = sha256(data)
node hash = sha256(left ‖ right)
```

A parent node `sha256(L0‖L1)` is now indistinguishable from a *leaf* whose data happens to be the
64-byte blob `L0‖L1`. And because `valid()` rebuilds the tree from a comma-separated list using
`bytes.fromhex` — which imposes no length check on each element — we are free to submit 64-byte
"signatures".

## Solution

The original 8-leaf tree (with `Li = sha256(sig_i)`):

```
            ROOT = H(H(P0,P1), H(P2,P3))
           /                              \
      P0=H(L0,L1)  P1=H(L2,L3)     P2=H(L4,L5)  P3=H(L6,L7)
```

Submit **four** signatures, each the 64-byte concatenation of two adjacent leaf hashes. Each one is
hashed as a leaf to exactly the original level-1 node:

```
S0 = L0‖L1  →  sha256(L0‖L1) = P0
S1 = L2‖L3  →  P1
S2 = L4‖L5  →  P2
S3 = L6‖L7  →  P3
```

The new 4-leaf tree's leaves *are* the original level-1 nodes, so every node above them is
byte-for-byte identical — same root, but the submitted list (4×64 bytes) differs from the original
(8×32 bytes). Second preimage achieved with no hashing work at all.

One gotcha worth flagging: match the remote's `pymerkle` version. The current `6.1.0` renamed
`append` → `append_entry` **and** changed the tree topology, breaking the balanced-tree assumption.
The challenge image uses **`pymerkle==5.0.3`** — pin that locally and reproduce the published root
before trusting the forgery.

Create `solve.py`:

```python
import sys, re
from hashlib import sha256
from pwn import remote

host, port = sys.argv[1], int(sys.argv[2])
io = remote(host, port)

io.sendlineafter(b"> ", b"1")
data = io.recvuntil(b"[1] Explore", drop=True).decode(errors="replace")
tx_lines = re.findall(r"Transactions: \[(.*?)\]", data)
sigs = [bytes.fromhex(h) for h in re.findall(r"[0-9a-f]{64}", tx_lines[-1])]

leaf   = [sha256(s).digest() for s in sigs]
forged = [leaf[i] + leaf[i + 1] for i in range(0, 8, 2)]
payload = ",".join(e.hex() for e in forged)

io.sendlineafter(b"> ", b"2")
io.sendlineafter(b"signatures: ", payload.encode())
print(io.recvall(timeout=5).decode())
```

```bash
python3 solve.py <host> <port>
# ... HTB{...}
```

The flag prints straight from the server when the forged root matches.

## Why it worked

Merkle-tree integrity rests on one assumption: *the same root implies the same data*. That holds
only when an attacker cannot reinterpret an internal node as a leaf. Domain separation
(`0x00`/`0x01` prefixes) guarantees that. Disabling it makes leaf hashing and node hashing the
identical function, so a single 64-byte input collapses an entire tree level — the canonical
[second-preimage weakness](https://cwe.mitre.org/data/definitions/345.html) that also underlies the
classic Bitcoin "64-byte transaction = two 32-byte leaves" attack.

## Fix / defense

- **Use domain separation.** Prefix leaves and internal nodes with distinct bytes (`pymerkle`'s
  `security=True`, or RFC 6962's `0x00`/`0x01` tags) so a leaf hash can never be confused with a
  node hash. This kills the second-preimage attack outright.
- **Validate the expected leaf count and structure** server-side instead of trusting an arbitrary
  client-supplied list, and reject over-length leaf inputs.
- Maps to [CWE-345](https://cwe.mitre.org/data/definitions/345.html) (insufficient verification of
  data authenticity) / [CWE-328](https://cwe.mitre.org/data/definitions/328.html) (weak/ambiguous
  hash construction).
